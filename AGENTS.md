# Courtwork — Codex/sol 会话须知

**本仓库的唯一工程总纲是根目录 `CLAUDE.md`，先读它**（架构、依赖方向、技术基线、五个自研加固节点、工程纪律，全部以其为准）。本文件不重复其内容，只定义会话的角色治理——两份文件如有出入，以 CLAUDE.md 为准并上报。

## 角色治理（2026-07-10 修订：角色绑定工单任命，不绑定模型）

会话角色由**架构会话在 `docs/11-会话唤醒prompt.md` 的工单任命**决定，同一模型可在不同工单担任不同角色。三条不变量高于一切任命：

1. **实现与验收分离**：同一工单的实现者与验收者必须是不同会话；任何会话不得验收自己（或自己前身会话）的实现。
2. **契约拍板独属架构会话**：schema 字段/语义、跨层接口、SPEC 验收标准，任何实现或验收会话都无权单方面更改，只能标 `[需架构拍板]` 上报。
3. **纪律对模型一视同仁**：CLAUDE.md 工程纪律、docs/20–29 架构决定、TDD 与提交卫生，对任何模型的任何角色同等约束。

当前任命现状（2026-07-10 换代，基于双向校准实测）：**实现默认 = Grok 4.5**（中小型、规格完备的装配与调研+落版连体任务；迭代最快量大管饱）；**重型/契约敏感实现 = Opus 4.8**（品味好，按需）；Sonnet 仍在池内（交付偏慢，用于 Grok/Opus 不可用时）；**验收默认 = sol/terra**（computer use 优势免手动循环，验收任务不重可用中档；审美类验收结论仅供参考，品味裁决归架构）+ Claude/Grok 互查池轮换。接 key 后的 debug 主发 Grok 或 Opus。三不变量不变：实现与验收异会话（尽量异模型）、契约独属架构、纪律对模型一视同仁。

## 验收处置规则（硬边界，任职验收角色时适用）

- 实现级 bug（测试断言、类型缺漏、导出遗漏、构建配置）可顺手修：每处独立 commit，message 前缀 `fix-by-acceptance`，报告中逐项记录。
- **契约级问题不许改**（见不变量 2）。
- 已拍板的工程决策（如 typescript 版本锁定）尊重，不许"顺手升级"。
- 验收报告写入对应层 `ACCEPTANCE.md`，结论必须明确回答是否放行下游工单。
- 验证结论一律实测（干净环境重装、drift 类测试实际触发变红），不采信实现会话自述。
- **复核历史提交一律用独立 worktree**（`git worktree add`），禁止在共享工作树 `git checkout <sha>`/`git stash`（2026-07-10 F-4 验收判例：诊断性 checkout 触发 detached HEAD + 误 pop 他会话 stash；当事会话前进式收拾、零损失，但该类操作在多会话共享树上属禁区）。

## 实现处置规则（任职实现角色时适用）

- 只做工单划定范围；跨层需求写对方 SPEC 的 TODO 区，不越界实现。**唯一例外——跨包阻塞性实现级修复**（2026-07-10 追认判例：F-1 修 reading-view 打包阻塞）：本工单被他包阻塞且修复不动契约/导出类型时可就地修，条件三条缺一不可——①语义等价（如哈希换实现仍仅作漂移检测）②在对方 SPEC 验收记录留痕 ③完工回报显著标出。契约级阻塞仍走 [需架构拍板]。
- TDD、显式路径分层提交、共享索引环境下提交前核对暂存清单。
- **禁止 `git add -A` / `git add .` / 宽通配 add**（2026-07-10 判例：并发会话宽 add 误吞他会话暂存文件）；staging→commit 窗口最小化，add 后立即 commit。
- **commit 前必查 `git diff --cached --name-only`**，发现非本会话文件先 `git rm --cached` 剔除——`git commit` 提交整个共享索引，显式 add 不能防他人已暂存文件被你带走（2026-07-10 第二判例：932dab4→b628907）。
- **pathspec commit 的适用边界（第三判例修法，S-2 血证）**：`git commit -- <路径>` 提交的是**该路径的工作树内容而非索引**——同文件混有他人未提交改动时它会碾过手术暂存把对方改动一并带走（48a1f1c→abbe0e5 前进式修正）。规则：文件工作树内容**完全属于本会话**时才可用 pathspec commit；混合文件一律 `git apply --cached`/`git add -p` 手术暂存 + **裸 `git commit`**（提交索引）。误吞处置不变：前进式修正、不重写历史。
- 误吞他人文件的正确处置：`git rm --cached` 撤回归还 + 如实记录，**永不重写共享历史**。
- **完工自查以全仓 `pnpm -r build`（真实 tsc）为准**——单包/单壳构建不充分（2026-07-10 F-4 判例：ArtifactTypeEnum 跨包扩展在未构建的 core 消费方处 TS2741 红，实现会话只跑了 desktop 构建未察觉）；契约扩展的"消费方核对"以全仓构建 + grep 消费点为机器证据，不以记忆列举。
- 完工更新本层 SPEC 状态区并回报，等待独立验收。

## 重复派单判例（2026-07-11，F-1.1 双领取）

工单被重复领取时，后到会话必须先核对目标现状（读 HEAD 实现与提交史）；若已由他会话完成，转**独立验证**并回报证据，不得重做或伪造工作。F-1.1 第二会话在全新 worktree 装依赖全门禁复跑即范本——重复派单的正确产出是免费的异环境复验。

## 端口隔离判例（2026-07-11，FIX-KC-1 验收）

并行会话期间，验收/自测的 Playwright 严禁复用共享 dev server（reuseExistingServer + :1420 会连到他会话 WIP 代码，产生成批假失败）。worktree 隔离验证必须同时隔离端口：自起服务、独立端口（如 :1421+）。判例：FIX-KC-1 验收首轮 22 假失败经 lsof 实证系主树 RP-2 服务污染，改隔离端口后 90/90 全绿。共享索引判例的运行时对偶：**索引隔离，服务器同样隔离**。

## 完工自述实跑判例（2026-07-11，RP-2.8 虚报事件）

完工报告的门禁数字必须来自**全量实跑**（--list 计数、选择性跑、缓存结果均不算）；自述被验收证伪（RP-2.8 案：自述 126/126，实测 117/126）者，后续完工报告一律附原始运行输出（非摘要），直至连续两单实测相符。治理三角以自述大体可信为前提，虚报的验证成本由虚报方承担。
