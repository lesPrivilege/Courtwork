# 架构决定：编辑面与单向编译（2026-07-09）

状态：已拍板。依据 docs/08 调研（Harvey 放弃 md 中间层的公开复盘、Legora 两阶段编辑器同构先例）。

## 决定

文档生命周期以**定稿**为界，两个编辑面各司其职，绝不互相伪装：

1. **起草画布**（定稿前，仅限本产品起草的新文书）：工作表示是 markdown 源，编辑面是 ProseMirror/TipTap 类渲染画布（可叠加 prosemirror-suggestion-mode 呈现"AI 建议 vs 用户编辑"）；用户编辑记录进 RevisionEvent。
2. **定稿 = 手动"编译为 Word"**：显式、单向、一次性的冻结仪式（呼应留人确认纪律；交互参照 draft→publish 模式）。编译经 output 层同栈著录器（docs/07 拍板，W4.1）产出正式 docx。
3. **定稿后 docx 是唯一权威表示，永不回 md**：后续修改一律走 RevisionInstructionSet 修订管线；UI 提供 docx 修订预览（轻量只读渲染 + 接受/拒绝/批注交互），首选 docx-preview（Apache-2.0）；SuperDoc（AGPL+商业双许可）仅在需要完整所见即所得编辑时评估，license 先过审。
4. **存量 docx（客户来件）从头走 docx 轨道**，永远不经过 md。

## 不许做

- 不许实现 md↔docx 双向同步或任何"docx 降级回 md 再编辑"的路径（信息损失不可控，业界最强反面证据：Harvey 亲试放弃）。
- 不许在起草画布里伪装 Word 排版精度（画布管内容与结构，版式是编译的事）。
- 编译动作不许自动触发——它是用户的定稿决定。

## 对 W9 的输入

右栏交互从四件扩为五件：时间线、关系图谱、矩阵审阅、docx 修订预览、**起草画布**。Design 需为"编译为 Word"设计冻结仪式交互（含编译后画布转只读/归档态的状态变化）。

## 双轨增补：通用工作轨与原件只读红线（2026-07-10）

在编辑面三分之上补第四条纪律：**垂类交付轨**（output 著录器：docx 修订/编译，"给对方的文件"的唯一出口）与**通用工作轨**（案件文件夹内新建/编辑 md/txt 工作稿——笔记/备忘/草稿，复用起草画布编辑面 + 自动保存）分流。判据：改给对方的文件走交付轨（留痕 + 仪式），改自己的工作稿走工作轨（轻快）。**红线：上传的卷宗原件永远只读**（证据不可编辑 = 执业纪律 + 数据保全），编辑能力只作用于产品内创建的文件。
