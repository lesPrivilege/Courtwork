# 收束与对外材料计划（2026-07-11 立项，BUILD-1 收尾后开展）

状态：已拍板分工。本 session（架构会话）keep polish Courtwork 并派工单；career kit 由**另开 Cowork session + 独立 career-kit 目录**承接合并通览。

## 一、性质与口径

对外材料是 docs/53 披露口径纪律的正面应用：**分层深度与私密度成正比**——越公开讲结果与难题，越私密（面试对谈）才展开机制。全部材料"讲难题与表现，不讲解法"的红线不变；协议机制细节（优先级律实现、握手规格）任何层不出。

## 二、留在本仓库的两单

1. **RM-1 收束 README**（架构会话亲笔，关键文案不派）：面向"第一次打开仓库的技术读者"——base 定形版完整叙事：论纲（docs/92 一句话）→ 架构五层 → 治理三角与判例 → 门禁体系 → 验证方式（quick verify 命令）→ 路线图。替换现 README 的状态版。
2. **MIRROR-1 公开镜像方案**（工单可派）：squash-mirror 清单制——白名单列举哪些 docs 进镜像（战略/泛化观察/宣言结论层），黑名单（协议机制、评测集题面、样板案语料、工单史细节）留私仓；镜像 README 独立撰写；执行为一次性脚本 + 手册纪律（每次对外发布走清单核对）。

## 命名注记（2026-07-11）

产品名 Courtwork 沿用（用户主导前提下）：Court=专业场域、work=工作，词序即论纲——先宣告场域归属，再谈工作方式；刻意避开 AI/copilot/buddy 词系（通用陪伴姿态），与"协作者而非数字员工""垂类场景而非通用 buddy"同构。推广范式：垂类即依赖包，**场域+work 即包名**。SITE-1 与 career kit 叙事中"为什么叫 Courtwork"留两句话的位置。

## 投递策略（2026-07-11，用户定调）

**目标排序**：① 未在 coding 前沿内卷的 base model 厂（首选字节系——飞书设计语言相合、docs/48 飞书适配器一鱼三吃既有伏笔；对它们"铺飞轮"论纲是雪中送炭）；② 退而求其次：垂类服务商、二线厂商。定性："可遇而不可求"的高品味远见选择，故值得**大量积极接触**——不押单点。

**叙事锚点**：相比"AI 取代人类"，基建 + 协作者切入更接近未来——大量工作因草台班子属性难被 AI 兼容整体取代，但可以被工具渗透：客户面极广、原业务照做、**有的工具旦用难回**（粘性来自秩序而非锁定）。投递物照披露口径纪律（docs/53）：评测集 + 宣言结论层 + Demo 实物，机制不出仓。

### 命名彩蛋（2026-07-11）

Cowork + **u r t** = Courtwork——"you are there"：多出的三个字母即产品哲学——没有人被取代，你就置身 agent loop 之中（留人确认/门禁/裁量区间的 HITL 体系一字双关）。用法：SITE-1 工艺区或 About 页一句话彩蛋、面试讲稿开场备选；不作主叙事（克制），让发现它的人会心。
