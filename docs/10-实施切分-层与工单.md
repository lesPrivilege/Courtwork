# 实施切分：层与工单（2026-07-09）

实现方式：各层由独立 Claude Code 会话（Sonnet 5）认领，按下面的顺序与依赖开工。每张工单 = 一个可独立验收的交付物。UI（apps/desktop）在 core MVP（W1–W6）完成后再做 Design 与排期。

## 开工顺序与依赖

```
W1 schemas ──┬── W2 registry ──┐
             ├── W3 ingest-spike    ├── W6 core
             ├── W4 output          │
             └── W5 tools ──────────┘
W7 eval（与 W3 起并行）
W8 ingest-v1（W3 spike 结论后立项）
W9 desktop（W1–W6 验收后，先出 Design）
```

## 工单

**W1｜packages/schemas — 领域 schema 定义**（先行，其余全部依赖它）
定义五个核心 artifact 的 TypeScript 类型 + JSON Schema + 校验器：案件档案（CaseFile：卷内文书清单与分类）、事件时间线（Timeline：事件/日期/来源引用锚点）、当事人关系图谱（PartyGraph：节点=主体，边=关系，边带证据引用）、风险清单（RiskList：风险点/等级/依据/处置状态）、矩阵审阅（ReviewMatrix：行=文档，列=问题，格=答案+溯源）。另加两个基础类型：来源引用锚（SourceAnchor：文件+页+区域，一切可溯源交互的地基）、场景修正事件（RevisionEvent：schema 级修正的统一记录格式，反馈标注的载体）。验收：类型 + 校验 + 每个 schema 的合法/非法样例测试。

**W2｜packages/registry — 场景注册表**
场景定义 schema（触发条件 + 输入 schema + 工具集 + 产出 schema + UI 模板标识 + 确认节点定义）、加载与校验、场景清单查询 API。MVP 四场景（S1 卷宗阅卷 / S2 矩阵审阅 / S3 合同审查 / S4 文书起草）以声明文件形式内置。验收：注册表加载四场景并通过校验；非法场景定义被拒绝。

**W3｜services/ingest — 技术 spike（先于 v1 立项）**
用 3–5 套真实（脱敏）扫描卷宗验证：PaddleOCR-VL + PP-StructureV3 主链路 vs DeepSeek-OCR 备选，在印章遮挡、倾斜、手写批注三类样张上的量化结果；产出 spike 报告：主链路选型结论 + 手写批注的现实处理策略（识别 or 标记人工）+ "午休之内出时间线"的端到端耗时预估。**此工单结论决定 S1 滩头假设是否成立，优先级最高。**

**W4｜packages/output — docx 修订产出管线**
输入：原 docx + 修改指令集（JSON，符合 schemas）；输出：带修订痕迹（tracked changes）+ 批注（附依据文本与引用）的 .docx。基线 Python-Redlines / docx4j 二选一（spike 后定），**WPS 打开正常渲染是验收标准的一部分**（自研加固点）。验收：golden files 快照测试 + Word/WPS 双端人工核验清单。

**W5｜packages/tools — 确定性接口层**
统一的工具契约（输入/输出 schema、超时、缓存、失败降级为"标记未核验"而非模型猜测）。MVP 两个工具：主体核验（企查查/天眼查，先 mock 后接真）、引用校验（法条/判例存在性校验，先接现有库）。验收：契约测试 + mock 服务联调。

**W6｜packages/core — agent core**
基于 pi-mono 借壳：headless、协议化（会话/事件流对外暴露，UI 是纯客户端）、provider 无关封装、场景执行器（从 registry 取场景定义 → 编排工具与生成 → 产出 schemas 合规 artifact → 停在确认节点）。验收：无 UI 下用 CLI/测试脚本跑通 S3 合同审查全流程（输入合同 → 风险清单 artifact → 修订指令集 → W4 产出 docx）。

**W7｜eval — 评测框架与数据集骨架**
promptfoo/DeepEval 底座；评测集目录规范（任务/输入/专业标准答案/评分规则）；先建 S3、S4 两个场景的最小评测集（各 20 例起）；模型选型跑分脚本（甜点区重选机制的工程落点）。与 W3 并行，spike 卷宗直接进评测素材。

**W8｜services/ingest v1**
按 W3 结论实现完整管线：OCR → 版面结构化 → 文书分类 → 跨文档实体对齐（自研加固点，无现成方案）→ 输出 CaseFile/Timeline/PartyGraph JSON。HTTP API + 队列化批处理 + 进度事件（右侧 UI 的进度条数据源）。验收：spike 卷宗全量跑通，实体对齐准确率有量化基线。

**W9｜apps/desktop — UI（core MVP 后启动）**
前置：W1–W6 验收 + 专项 Design（三栏布局、场景卡片、时间线/图谱/矩阵/修订预览四个右栏交互、确认节点交互）。壳选型按 docs/05 建议 Tauri v2 优先，Design 阶段最终定。**本工单在 Design 完成前不开工。**

## 测试策略（分层）

- **W1–W8（无 UI 阶段）**：全部 headless——单元测试 + golden files + W7 评测回归。不需要 computer use。
- **W9 后（有 UI）**：回归测试用 Playwright 驱动（Tauri/Electron 均支持，可进 CI，确定性）；**computer use 类 agent（Codex / Claude computer use）定位为探索性 QA**——按场景剧本自由操作 app、发现交互断点与视觉问题，产出缺陷报告，不做回归基础。理由：computer use 跑测非确定性、慢、贵，适合"找没想到的问题"，不适合"守住已知的对"。可在 W9 排期时立一张 spike 工单验证 Codex 跑探索性 QA 的实际效果。

## 会话认领规则

一张工单 = 一个 Claude Code 会话（Sonnet 5）。开工读 CLAUDE.md + 本文件 + 本层 SPEC.md；完工更新 SPEC.md 的状态区并留验收记录。跨层需求写 TODO 进对方 SPEC，不越界实现。
