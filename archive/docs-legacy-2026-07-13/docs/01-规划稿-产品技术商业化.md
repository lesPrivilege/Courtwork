# 法律垂类 Agent 规划稿：产品·技术·商业化（内部对齐口径，2026-07-09）

本稿承接《部署意见稿-法律垂类模型与产品形态》的既定判断（壁垒在"数据 + 判断逻辑 + 接口 + 生态"，形态走"协作者"不走"数字员工"），在此基础上给出可执行的分层架构、阶段路线与商业化框架，并为长期场景预留结构位。

## 〇、北极星目标及其拆解

**目标表述：当一名法律从业人员进入某个团队时，"这个团队是否配备本产品"成为其真实的考虑因素。** 即产品从"工具"升级为"团队基础设施"——类似检索数据库之于律所、行情终端之于金融机构的地位。

这个目标可拆解为四个必须同时成立的条件，后文所有规划都向它们对齐：

1. **高频**：日常场景（审合同、整卷宗、起草文书）日均触达，而非偶发使用。
2. **沉淀**：团队的卷宗、合同、主体图谱经摄取管线结构化后沉在系统内，形成迁移即重做的切换成本。
3. **独占数据面**：类案同判库、官方接口等机构级合作，竞品与通用 agent 短期无法复制。
4. **新人零门槛**：入口是场景按钮而非提示词能力，新成员入职即可用团队已配置好的场景与数据资产。

条件 2 与 3 是"配不配备"产生实际差距的根源；条件 1 与 4 决定差距能否被日常感知。

## 一、定位与不变原则

- **场景驱动的软件，agent 藏在里面。** 目标用户不是高认知律师（他们会直接用通用 agent），而是普通法律从业者与机构团队。产品入口是场景清单，不是聊天框；采购方（律所、企业法务）买的是可演示、可培训的场景，不是"能力"。
- **不与通用 agent 比通用能力。** 基座模型选推理能力够用的"甜点区"，provider 无关。垂类任务边界被 harness 锁死（抽取、比对、按 schema 生成、有限步工具调用），中档模型即可胜任且成本可控——这是 credits 经济模型成立的前提。甜点区是移动目标，抗折旧的资产是**自有垂类 eval 集**（真实卷宗任务 + 专业标准答案），每代新模型跑一遍、按性价比换。
- **一核多壳，场景层与 agent 核心解耦。** Agent core 做成 headless 服务、讲协议；场景识别、UI、注册表由产品团队独立迭代，不动核心。
- **壁垒的工程排序**：卷宗摄取管线 ≥ 领域 schema + 确定性工具 > 反馈捕获设计 > agent 框架与 UI 壳选型。前三样自建自有，最后一样基于开源验证交互逻辑、随时可换，不重复造轮子。

## 二、架构分层与长期预留

自下而上六层。标 ★ 的是必须从第一天放在自有仓库、不长在开源 fork 里的资产。

1. **Agent core**：headless、协议化，基于开源项目（pi 一类极简框架或同等物）验证交互逻辑。Tool use 复杂度有限，loop 本身是商品。
2. **★ 卷宗/合同摄取管线**：OCR 前置、文书类型识别、跨文档实体对齐、结构化入库。管线质量决定产品上限，同时是切换成本的来源。
3. **★ 领域 schema 与确定性工具层**：当事人关系图谱、事件时间线、履约主体识别、案情经过等均为有类型定义的结构化产物；关键事实走确定性接口（企查查/天眼查、类案库、法规校验），不让模型猜。
4. **★ 场景注册表**：一个场景 = 触发条件 + 输入 schema + 工具集 + 产出 schema + UI 模板。产品团队按周上新，与核心解耦；场景识别分类器可用使用数据持续训练。
5. **壳层**：桌面 app（旗舰体验）、WPS/Word 插件（国内法律市场的真实入口）、Web/私有化部署。内部以 markdown 为通用底层格式，面向用户统一产出**带修订痕迹与批注（附法条/判例依据）的 .docx**——用户认 Office，不认 md。
6. **插件生态**：邮箱、内部工作流、案管系统对接。兼容 skill 生态，非瓶颈，按客户需求补。

**为长期场景预留的三个结构位（现在留口，以后细盘）：**

- **事件驱动/定时运行能力**：核心需支持非交互的后台任务（监控、订阅、推送），这是合同长期监控类场景的前提，交互式 loop 无法事后补。
- **数据订阅连接器框架**：为接入政企法检等官方系统的持续数据流预留统一接口位，与现有一次性查询接口区分。
- **主体图谱作为持久资产**：当事人/企业图谱从"单次案件的产物"升级为跨案件、跨合同复用的团队级图谱库——投资主体穿透、母公司诉讼风险等分析都构建在它之上。

## 三、阶段路线

- **Phase 0 — 验证**：开源底座跑通"文件夹级协作"交互；摄取管线 MVP（OCR + 实体对齐）；3–5 个滩头场景（卷宗整理、合同审查、文书起草），产出即 .docx 修订模式。目标是"用户第一眼觉得方便"，不是功能全。
- **Phase 1 — 滩头产品**：场景注册表机制上线；WPS/Word 插件入口；席位制试点客户（若干律所/法务团队）；建立驻场 + 用量遥测的场景挖掘机制——目标用户表达不出需求，只能从行为里挖。
- **Phase 2 — 复利启动**：反馈捕获结构化（律师改了哪段草稿、否决了哪个风险点、修正了图谱哪条边，都是 schema 级操作而非自由文本，直接成为标注）；eval 集成型并接入模型选型流程；第一轮精调；场景库扩张。
- **Phase 3 — 资产监控层（长期预留，场景后续细盘）**：合同长期履约监控（节点提醒、对手方工商/涉诉变动推送）；政企法检官方数据自动接入；投资主体/母公司诉讼风险、履约风险穿透分析。**这一层的战略意义在于改变价值性质：交互场景是"用了更快"，监控场景是"不用就有损失"——持续性的、机构级的损失规避，才是"团队标配"最终成立的地方。**

**口径提醒**：Phase 3 的监控是"推送事实 + 留人判断"，不是自动决策。与意见稿"协作者内核、对自主叙事保持克制"的判断一致——监控类场景恰好是既能做重、又不越过判断权边界的自主形态。

## 四、商业化

- **席位保底 + credits 计量重活。** 席位费覆盖日常高频场景（鼓励用量，用量即数据复利）；credits 只计量重计算任务：卷宗摄取、批量处理、深度检索。**原则：credits 不挡在高频交互前面**——低认知用户会因"怕花钱"不敢用，用量断则复利断。
- **监控订阅作为 Plan 附加包。** 长期监控按主体/合同数订阅计费，预算可预期，符合机构采购习惯；也可折算 credits，向小团队开放。
- **采购与交付**：决策人是律所主任/法务负责人，按场景清单演示与培训；私有化部署为可选交付形态（一核多壳架构已保证），针对数据出域敏感客户。
- **摄取即锁定**：历史卷宗批量摄取可作为签约动作（一次性 credits 包），既是收入，也是把客户资产沉进系统的时点。

## 五、"团队标配"的达成机制

个人习惯（高频场景 + 零门槛入口）→ 团队协作（共享卷宗库、团队图谱、场景配置）→ 机构知识资产（历史案卷结构化 + 反馈标注沉淀）→ 新人入职即用（继承团队场景与数据）→ 招聘市场信号（从业者主动询问团队是否配备）。每一环对应前文的具体产品机制；跳环即断链，尤其不能在个人习惯未成立时提前推机构资产叙事。

## 六、风险与对冲

- **甜点区漂移**：上游模型迭代改变性价比格局 → eval 集常态化跑分对冲，架构上 provider 无关已兜底。
- **场景挖掘失速**：目标用户表达不出需求 → 驻场 + 遥测机制是 Phase 1 的交付物而非附属品。
- **官方接口的政策依赖**：独占合作是壁垒也是单点 → 数据源多元化预留，接口层与场景层解耦以便替换。
- **过度承诺风险**：监控/自主类场景最易在宣传中越界 → 对外叙事严守"协作者 + 事实推送 + 留人判断"口径。

## 总结

北极星是"团队标配"，路径是：先用滩头场景和 Office-first 交付换来高频，用摄取管线和图谱资产换来沉淀，用独占数据面拉开代差，最后用资产监控层把价值性质从"更快"变成"不可缺"。架构上一核多壳、场景解耦、三个长期结构位现在留口；商业上席位保高频、credits 计重活、监控做订阅。基座可换、框架可换、壳可换——摄取管线、schema、场景注册表、eval 集、反馈标注这五样自有资产，换不掉。
