# 架构决定：文件操作分级（2026-07-10）

状态：已拍板。信任前提：误删一次 = 信任归零，销毁类动词永不进入能力面。

## 动词三级

- **无损级（直执）**：copy、mkdir、reveal-in-folder、open-file——加法操作，agent 可直接执行，UI 有可见反馈。
- **移形级（四段式）**：move、rename——计划 artifact（逐条可勾选的整理计划表）→ 确认门禁（分层确认，大批量强制抽看）→ 执行 + 报告 → **一键撤销**（移形操作全量记入可逆事务日志，撤销 = 逆向重放）。文件操作是场景执行器的标准形状（计划 = typed artifact、确认 = 门禁、修正 = RevisionEvent），不是新能力面。
- **销毁级（永不）**：delete、覆盖写不提供。重复文件移入"重复项"子文件夹；归档代替删除（docs/46 裁决 7 同哲学）。

## 原件红线精细化（衔接 docs/23）

内容不可变 ≠ 位置不可变：原件可被移动/重命名（整理卷宗的现实需要），但**移动前后内容哈希比对留痕**（证明证据零字节变动）+ 原始文件名永久记录于 CaseFile 条目。路径域仍限案件文件夹。

## 杀手场景：卷宗整理

乱文件（邮件附件/下载扫描件/多版本合同）拖入 → agent 出整理计划（分类归档/规范重命名/版本归组/重复项隔离）→ 确认 → 一键成卷 + 可撤销。作为场景进注册表（整理计划的 artifact 类型走 schemas 提案）。衍生：整案打包导出（无损复制类，直执级）。

## 不许做

- 任何路径不得出现 delete/覆盖语义；撤销日志本身不可删（留痕）。
- 批量操作不得跳过计划预览直接执行（哪怕单文件 move 也走轻确认）。

## 来源路由与联动预留（2026-07-10 增补）

- **sources 声明**：案件/工作区级配置声明用户授权的来源目录（下载/邮件附件目录等），注册一次免每次提醒。纪律：授权显式（选目录，永不扫全盘）、只读扫描（产出整理计划提案，动文件走四段式）、**触发方式与来源声明解耦**——MVP 仅手动触发，Stage 3 scheduled 复用同一声明（定时扫 = 换触发器不换声明）。
- **S7 候选·发票整理**（泛化首个非法律实证）：出差发票收集/重命名/打包/报销单 = 同一 FileOpsPlan 机器 + 不同场景声明；workflow 接入后"自动提交"自动到确认门禁为止（对外提交不可逆，推事实留判断）。当期仅声明预留，零新实现。
