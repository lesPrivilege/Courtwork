# Chat 目录即卷宗：容器同构论与角色扮演垂类（2026-07-15）

产品负责人论断 + 行业信号（ChatGPT/Codex 合并后 chatbot 缩为可挂起小窗）。调研原稿，不具约束力。

## 行业信号

Chatbot session 正在变为非承重结构：临时、堆积仅供检索、用户无手动管理；临时 chat 起草方案 → 有需要晋升 work project。与 Courtwork Chat/Work 双轨同构，且我们早一步把静默化做在底层（ADR-013）而非 UI 层。

## 容器同构论（ADR-013 的逻辑终点，待其自动压缩/index 实现时立 ADR 吸收）

chat 目录整体视作一个卷宗/垂类项目：内部不做 session 管理，而是**按工作语义改变每次请求的注入**——

- 连续请求：完整上下文，保证前缀缓存命中（窗口语义已落地）；
- 缓存失效后：只注入高价值/压缩的工作语义（memory 蒸馏已落地；自动压缩属 ADR-013 既列后续）；
- 原始请求数据：降级为仅供查询的参考层，做**渐进披露 index**（个人工作项目数量少，渐进披露效果最佳）——未来 CHAT 线候选工单；
- context 与续行全自动，用户零管理面。

## 角色扮演垂类（第三个可表达性案例，纸面）

一个角色 = 一个卷宗。通行 roleplay chatbot 的核心痛点正是续行与记忆管理（用户手写记忆卡、角色失忆）；本底座四层现成机制换 substrate 即可：

- 角色档案 = schema 化结构（人设/关系/世界观，verified 层，系统裁决一致性）；
- 摘要记忆 = 蒸馏缓存层（高质量语义摘要，自动写入可撤销）；
- 原始对话 = 渐进披露原文（按用户指令召回，不全量回灌）；
- pets/虚拟形象与可视化交互 = preview schema 切入（ADR-014 tab=一张 schema 表的自然消费者）。

判定预期：与算命/炒股同构，core 零改动、纯轻量包表达；「做得比通行 roleplay 更好」的差异点全部来自结构（续行质量、记忆可审计、渐进召回），不来自模型能力。

## 落点

不立即立单。容器同构论待 CHAT 自动压缩/渐进披露 index 立项时升 ADR；roleplay 为投递叙事的第三案例与未来轻量包候选。
