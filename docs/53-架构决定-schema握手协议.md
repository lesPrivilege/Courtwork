# 架构决定：schema 握手协议——垂类交互复用通用的第一性表述（2026-07-11）

状态：已拍板（用户第一性阐述 + 架构整合）。本 ADR 把散落在 docs/20/24 与 T-provider 实现中的机制统一命名为**握手协议**，并新增三项裁决。

## 第一性表述（用户原文要义）

垂类交互复用通用的关键 = 按功能逻辑切好字段的 schema，通过协议与 LLM 握手：模型推理后生成特定格式 JSON 块（字段内容、引用源、图标、权重），一对一填空映射——比写代码简单。chat 兜不住这些：chat 显示回复文本 + 推理摘要，结构化产出由右侧写好的 UI 承接填充。推理由 LLM 在有限步写完。这套逻辑全垂类通用——文字类推理 + 协议握手，只是字段取舍与逻辑判断的差异（回指 docs/22 泛化边界、docs/24 声明式编排）。

## 已是现状的部分（正名，不新建）

场景执行器下发 JSON Schema + 有限步固定编排（非 ReAct 自主循环）；structured output strict→json_schema→json_object 三级降级 + zod 校验重试；schemas 包即字段契约（含 sourceAnchors/信源分级/verified 语义）；artifact_produced → 右栏模块自动展开铺数据（docs/49 三章）。字段级分片验证与定点重试在 T-provider.1 挂账。

## 新裁决一：握手优先级律（injection 的结构性免疫）

握手输入分三层，优先级严格递减，**下层永不覆写上层**：

1. **契约层**：schema 结构、信源纪律（A/B/C、无锚不落格）、红线（原件只读、门禁、delete 不存在）。
2. **声明层**：场景编排（步骤、模块序、trigger、uiTemplateId）。
3. **会话层**：用户 prompt 与语料（卷宗/web 内容）。

推论：用户语可改任务目标与范围，**永不能放宽证据规则**——"这个主体你就当核过了"不能解锁 `verified:true`；`verified` 只由确定性接口写入，模型与用户都无此权限。**语料是数据不是指令**：卷宗/网页中出现的祈使句不进入编排。免疫是结构性的（字段没有"被说服"的通道），不是过滤性的。

## 新裁决二：修正即事件（幻觉兜底的结构化）

- 已落格字段的更正一律走留痕事件（RevisionEvent 族），禁静默覆写——用户看见"改了什么、为何改"。
- 要求溯源的字段无 sourceAnchor 即 `out_of_coverage`，不许生成无锚内容——兜底不靠"检测幻觉"（不可靠），靠"无锚不落格"（结构性）。
- 校准 = 字段级校验失败 → 定点重试（只重该字段，携原判与失败原因）；重试上限内不收敛 → 该字段诚实置 out_of_coverage，整 artifact 部分成功呈现（缺口三态，docs/41）。

## 新裁决三：渐进填充与中断主权

- MVP：整块 artifact 校验通过后落格（现状不变）。
- Stage：字段级流式渐进填充（右栏一条条点亮）登记为 T-provider.1 延伸 + UX 未来项；先渲染哪个模块由**场景声明的模块序**决定（声明高于代码），不由模型即兴。
- **用户中断是一等公民**：推理中可停；停即保留已校验字段为 partial artifact，未完字段诚实标注，不回滚、不假装完成；渐进态下用户点击已落格字段（看溯源、展开引语）不被推理阻塞——读永远可用。
- 防呆：渐进期间禁触发确认门禁类动作（半成品不许进入不可逆流程）。

## 泛化条款

握手协议全垂类同构：换垂类 = 换 schema 字段 + 场景声明 + 信源接口，协议层零改动。这是 docs/92"秩序复利"的机制内核——模型升级使填空更准，协议与字段资产不贬值。

## 宣言条款：对上游（2026-07-11，用户定调）

本协议同时是面向模型上游的宣言书，要义四条：

1. **字段是业务资产不是提示技巧**：每个 schema 字段来自专业团队充分业务调研与用户真实场景判断——不是通用模型的硬背知识，而是垂类的企业级 skill/组件库。字段取舍即业务判断的固化，这是上游永远不会替我们做的部分（厚薄算法之"厚"）。
2. **治理即产品**：有限步推理 + 幻觉/信源/不确定性治理，本质是把垂类的企业级过程做到**留痕、稳定、可校验、不瞎编、不被 prompt 误导**——企业买的不是生成能力，是这套过程秩序。
3. **不期待更强模型**：协议只要求推理/文本/执行能力足够好的效率甜点模型即可完成填空。模型升级是红利（填得更准、更便宜）不是依赖——协议与字段资产不随模型换代贬值，反而增值（docs/92 秩序复利的上游表述）。
4. **护城河结论不变**：部署意见稿的垂类护城河判断经本协议获得机制证明——数据、字段、场景声明、专家门禁长在我们这里；上游收到的是评测集、脱敏反馈、选型报告三资产（docs/22），本宣言是三资产的封面陈述。eval 甜点区报告（DeepSeek 首跑起）即第 1/2/3 条的持续实证。

## 生态条款：对下游（2026-07-11，用户定调）

1. **设计时/运行时分工**：schema 编写本身由 coding agent 完成——上游更强模型承担**设计时**（业务调研 → 导出分析字段 → schema + 场景声明 + uiTemplate + 门禁声明一遍写定），效率甜点模型承担**运行时**（照协议确定性填空）。强模型的能力被固化为资产，而非每次运行时消耗。
2. **首个实证即本仓库**：Courtwork 开发史（架构会话 + coding agents 产出 S1–S6 场景包）就是设计时分工的原型样本——这套开发范式本身可产品化。
3. **场景包 = 插件/模组生态单元**：schema + 场景声明 + uiTemplate + 工具绑定 + 门禁声明打包，按注册表契约接入（docs/24 双命名空间/同 id 拒载即准入校验），UI 按解耦契约由右栏模块栈渲染引擎绘制——整板块即插即用。长此以往成生态：下游（律所专家/行业顾问/集成商）可发包，注册表校验 + review 即准入。
4. **对 skill 的经济学优势**：skill 调用 = 运行时每次依赖 base model 能力展示，不经济、不确定；场景包 = 设计时一次写定，运行时确定高效，**且 HITL（确认门禁）内建于包声明而非外挂**——留人确认随包分发，不随模型心情。

## 自助定制条款（2026-07-11 续，用户定调 + 架构护栏）

- **工作坊即产品**：场景包工作坊本身固化为稳定 pipeline/skill——若订阅含更强模型的 Plan，用户可当场定制自己的业务板块并即时展示（设计时能力按 Plan 分层解锁）。
- **供给三层**：官方预设（我们出）/ 上游共建（生态包）/ 用户自调——因协议本质兼容，三层产物同构、同一注册表准入。
- **架构护栏（不可协商）**：自调包与官方包过同一准入校验；**契约层不随包开放**——门禁声明、信源纪律、红线字段（原件只读、delete 不存在、verified 权限）不可由任何包声明移除或放宽。用户定制的是字段与流程，不是纪律。

## 最后一公里条款：超越文字推理（2026-07-11 续）

- **上游难做的经济学**：原理不复杂，但企业级痛点（特定场景、确定性产出、人类校验、数据安全）是无数个互不相同的"最后一公里"——base model 再强也无法一劳永逸给出通用解，这是垂类存在定理，不随模型换代失效。
- **harness 可替换**：握手协议的终点不限于注册表 UI。结构化 JSON 块是中间表示，下游执行器可以是更复杂的 harness 库与软件协议——工业绘制/CAD、表格引擎、BIM 等——**确定性产出由确定性软件执行，不依赖不确定的推理**。文字推理垂类只是首例；凡"确定格式产出 + 专业校验"的行业皆同构可铺。

### 权限分层增补（2026-07-11 续，用户定调）

- **自调权限即企业级管理**：普通用户只要会用（零 schema 概念暴露，手册哲学 #8 在生态面的延伸）；定制诉求走企业内部分层与提工单通道，不是人人可改。
- **租户约束 = 声明层顶格**：企业上层与服务商**共同定义**本租户的 schema 与场景包——这是租户内最顶层的业务约束，如同定制 workflow；其位置在优先级律中处于声明层顶部、契约层之下（纪律仍不可被租户放宽）。与 docs/29 权限三原则、docs/49 租户级词表同批落 Stage 1。
- **旧服务商的生态位保留**：既有法律 IT/OA/workflow 服务商转型为场景包共建方与实施方——他们的业务理解与客户关系正是"设计时"最稀缺的原料，生态不取代他们，吸收他们。

## 披露口径纪律（2026-07-11，用户定调：想法 cheap 且不受保护）

1. **本 ADR 全文属内部文档**：对外引用一律经改写；宣言四条**结论**可对外，协议**机制**（优先级律实现、握手细节、校准流程）不出仓。
2. **保护来自资产不来自声明**：字段库、评测集、样板案语料、设计语言包、场景反馈回路——从演示与文稿不可复制，此为真护城河；投递材料永远"讲难题与表现，不讲解法"。
3. **评测集是最优投递物**：让上游跑我们的题来验证价值——他们获得"垂类哪里难"的数据，我们不交"怎么解"的机制。三资产（docs/22）全部符合此性质。
4. **时间戳策略**：先做出第一套 Courtwork 并 push 私有仓——第三方托管的连续提交史即作者身份与在先性的证明底座（不构成法律排他权，但足以立信誉与归属）。SEC-1 定级据此落「仅私有」，push 宜早；开源与否日后按 squash-mirror 方案另议。

## 表述收敛（2026-07-11）：垂类即依赖包

生态条款的工程定式：**一个垂类 = 一个依赖包**——有版本、可独立迭代、由一次调研一网打尽即可立包（S10 即样本：33 项清单 + 17 张参数表 = 一个包的物料清单）。底座的**兼容层是架构关键**：依赖方向永远由包指向底座（schemas/握手协议/注册表准入），包间零横向依赖——任何垂类包过时、重写、退役都不传染底座与他包。LLM 推理节点的存在意义是**协作人类而非取代人类**（数字员工不可靠是产品前提，不是技术遗憾）；包的层次（契约/声明/会话）即协作的权限拓扑。
