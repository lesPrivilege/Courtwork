# 61 调研 · PM 垂类场景普查与 MVP 包定义草案（2026-07-11）

> 定位：为第二垂类包（PM：产品经理为主、项目经理兼顾）立案，承接 docs/48"第二垂类选型：PM 包"拍板（接口依赖最少 / 受众即用户 / "换垂类=换声明"的最强证明载体）。本文只立案不拍板：场景全谱、容器语义、MVP 包草案、差异缺口、竞品一瞥五件套，供架构会话过目后切 PM-PKG 工单。
> 机制回指：垂类即依赖包（docs/53 表述收敛）= schema（字段契约+SourceAnchor）+ 场景声明（trigger 三维/固定编排/产出/门禁）+ 渲染词表消费（docs/57 七族 18 原语）+ 续行投影（docs/25 结构化再水化）。核心纪律全文预设不复述：模型只生成不裁决、无锚不落格、留人确认、原件只读、产出落用户文件夹。

## 〇、先立一根总纲：PM 工作的机制底色与法律同构

法律包处理的是"卷宗 → 可核查的法律判断草案"；PM 包处理的是"语料（反馈/纪要/数据）→ 可核查的产品判断草案"。两者的机制骨架完全同构：

| 机制位 | 法律包 | PM 包 |
|---|---|---|
| 原始材料（只读） | 卷宗（起诉状/证据/合同） | 用户反馈/访谈记录/会议纪要/指标导出/竞品材料 |
| 溯源锚 | 文书段落 SourceAnchor | 反馈条目/纪要段落/数据行 SourceAnchor |
| "只生成不裁决" | 裁量项永不出单值（赔偿出区间） | 优先级/结论永为提案（RICE 分是排序建议非裁决，实验结论出"数据陈述+待判读"） |
| 确定性计算 | S10 计算器族（诉讼费/利息） | 打分公式族（RICE/ICE/WSJF）——同一 CalcArtifact 机制，参数表更轻 |
| 门禁 | 风险清单确认后才出修订文书 | 归类/打分/行动项确认后才进 PRD/周报/需求池 |
| 产出 | 带修订痕迹 docx 文书 | PRD/评审意见/发布说明/周报（docx/PDF 走同一 output 管线） |

一句话：**PM 是"轻信源、重结构"的法律**——没有法条与主体核验的硬数据资产依赖，但字段裁剪、溯源纪律、门禁拓扑原样复用。这正是它作为第二垂类的证明价值：底座不动，换一套声明就是另一个行业。

## 一、场景普查：PM 日常工作全谱（16 项）

图例：**原语** = docs/57 词表原语；**门禁** = 什么动作必须留人确认；**同构** = 与法律包机制的映射。频率按"周内多次 / 周级 / 迭代级 / 版本级"标注。

### 1.1 输入整理域（PM 的"阅卷"）

**P1 用户反馈/评论/工单归类**（频率：周内多次）
- 输入 → 产出：反馈语料文件夹（App 商店评论导出、客服工单、NPS 开放题、社区帖，md/txt/csv/docx）→ 反馈洞察集（主题聚类 × 情感 × 声量 × 逐条溯源），可选反馈时间线（声量随版本变化）。
- 原语：`table`(matrix 变体：主题 × 版本/渠道交叉)、`list`（洞察条目+展开原声引语）、`chart`（主题声量条）、`timeline`（声量演变）。
- 门禁：归类结果确认后才可写入需求池 / 引用进 PRD；单条反馈的主题改判走 amend 留痕。
- 同构：≈ **S1 卷宗阅卷**（摄取 → 分类 → 时间线/结构化 artifact）。反馈原声引语 = 证据引语（mono + verifiedBg 通道），AI 归纳的主题命名 = generated 通道，一屏两通道永不混排——法律包 provenance 双通道直接复用。

**P2 用户访谈语料结构化**（频率：迭代级）
- 输入 → 产出：访谈逐字稿/录音转写（md/docx）→ 访谈纪要 artifact（痛点清单 × 原话锚点 × 用户画像标签 × 待验证假设）。
- 原语：`list`（痛点+展开原话）、`tree`（痛点 → 场景 → 原话的层级）、`table`。
- 门禁："待验证假设"字段确认后才可进需求池（假设是生成物，不是事实）。
- 同构：≈ S1 的单文书精读变体；"原话不许改写后冒充引语"= 无锚不落格在访谈域的表达。

**P3 会议纪要 → 行动项/决议抽取**（频率：周内多次）
- 输入 → 产出：会议纪要（docx/md，飞书/腾讯文档导出）→ 行动项清单（事项 × 负责人 × 截止 × 依据段落锚）+ 决议清单 + 悬而未决清单。
- 原语：`list`（三态清单）、`schedule`（截止日期板）、`gate`（逐条确认）。
- 门禁：行动项确认后才可进入 schedule/派发；**负责人与截止日是提案不是指派**（协作者非数字员工——系统永不替 PM 向他人下任务）。
- 同构：≈ S1 时间线抽取 + docs/48 时效族（截止日期 = 期限条目，临期告警走 severity 合法通道）。

**P4 竞品动态与材料归档分析**（频率：周级）
- 输入 → 产出：竞品发布说明/官网抓取/评测文章（web fetch 或用户投放文件）→ 竞品对照矩阵（竞品 × 能力维度 × 出处）+ 差异摘要。
- 原语：`table`(matrix)、`compare`（我方 vs 单个竞品字段对照）、`list`。
- 门禁：外部来源（web）字段一律标 C 级信源角标，写入对外材料前逐条确认。
- 同构：≈ **S2 矩阵审阅**（行=竞品、列=维度、格=带锚结论）；信源分级机制原样复用，只是等级语义换词（见第四节）。

### 1.2 决策支持域（PM 的"审查与计算"）

**P5 需求池管理与去重合并**（频率：周内多次）
- 输入 → 产出：散落需求（反馈洞察、老板一句话、纪要决议）→ 需求池台账（需求 × 来源锚 × 状态 × 关联反馈计数）；疑似重复项合并提案。
- 原语：`table`、`list`、`board`（状态看板，挂账原语——MVP 用 table+status 列覆盖，与法律包同一裁决 docs/57）。
- 门禁：合并/关闭需求走 confirm（全站无 delete，关闭=状态变更留痕）；**需求池是台账不是聊天记录**——续行投影的核心载体。
- 同构：≈ 法律包证据台账 + docs/26"数据维护即场景"。

**P6 需求优先级打分（RICE/ICE/WSJF/KANO）**（频率：迭代级，演示冲击力极高）
- 输入 → 产出：需求池条目 → 打分表（每需求：Reach/Impact/Confidence/Effort 预填值 × 取值依据锚 × 公式得分 × 排序提案）。
- 原语：`form`（参数预填+人核改）、`table`（得分排序表）、`chart`（价值-成本散点降维为分档条）。
- 门禁：预填参数逐项可改，排序提案确认后才写回需求池；**Confidence/Impact 是裁量项——模型预填必须给依据锚或诚实置空，得分永远标注"排序建议"**。
- 同构：≈ **S10 计算器族**逐字同构（docs/48 裁决四条全适用）：公式核心零 LLM 节点、LLM 只做填表员从语料抽参数、分项+依据+参数溯源的 CalcArtifact、裁量项永不出单值（Impact 预填出档位区间供人定）。参数表比法律轻一个量级：RICE 权重是团队自定义单表，零外部数据年更。

**P7 A/B 实验记录与结论归档**（频率：迭代级）
- 输入 → 产出：实验设计 + 指标导出（csv）→ 实验档案（假设 × 分组 × 指标对照 × 显著性计算 × 数据陈述）。
- 原语：`form`（实验参数）、`compare`（对照组 vs 实验组）、`statcard`、`chart`。
- 门禁：显著性计算走确定性脚本（零 LLM）；**"是否推全"的结论永远是待确认提案**——模型只生成"数据陈述"，推全决策是人的裁决。
- 同构：显著性检验 ≈ 计算器族；"不出胜率单值"≈ 法律包 L14 案情评估"只出焦点+依据不出胜率"的同一条纪律。

**P8 风险登记与跟踪（项目经理侧）**（频率：周级）
- 输入 → 产出：项目材料/纪要 → 风险登记册（风险 × 概率/影响档位 × 缓解措施提案 × 责任人提案 × 来源锚）。
- 原语：`list`（severity 徽章+展开依据——审阅面，语义色合法）、`table`(matrix：概率 × 影响矩阵)。
- 门禁：风险等级与缓解措施逐条确认；升级动作（上报）到门禁为止。
- 同构：≈ **S3 RiskList 直接同构**——字段几乎一一对应（riskItem/severity/basis/sourceAnchor），是"换垂类=换字段语义"的最直观展品。

**P9 干系人矩阵与沟通计划**（频率：版本级）
- 输入 → 产出：项目干系人信息 → 干系人图谱/矩阵（权力 × 利益四象限、沟通频率与口径提案）。
- 原语：`graph`（干系人关系）、`table`(matrix 四象限)。
- 门禁：对外沟通口径确认后才可用于起草邮件/汇报。
- 同构：≈ **S2 PartyGraph**（当事人图谱 → 干系人图谱，节点边 schema 同形，无需主体核验工具）。

### 1.3 产出起草域（PM 的"文书"）

**P10 PRD/需求文档起草**（频率：迭代级，PM 的第一文书）
- 输入 → 产出：需求池条目 + 反馈洞察 + 访谈锚点 → PRD 草案（背景/目标/用户故事/功能规格/埋点/非目标，每节的事实性陈述带锚）。
- 原语：`document`（起草画布，编译为 Word/PDF 冻结仪式）、`tree`（文档结构大纲）。
- 门禁：起草是提案态；编译导出是单向仪式（docs/23）；引用的反馈/数据无锚不落格。
- 同构：≈ **S4 文书起草**（起草画布 + RevisionInstructionSet 同族机制）。

**P11 PRD/需求文档评审**（频率：迭代级，与 P10 成对）
- 输入 → 产出：PRD 稿（docx/md）→ 评审意见集（缺陷项 × 维度[完整性/可测性/一致性/边界遗漏/埋点缺失] × 严重度 × 定位锚 × 修改建议）。
- 原语：`table`(matrix：章节 × 评审维度)、`list`、`revision`（接受建议后的修订痕迹预览）。
- 门禁：逐条 confirm/reject/amend，确认后才生成修订稿——**"需求评审矩阵"≈"合同审查矩阵"**，本包立包论据里最锋利的一条同构。
- 同构：≈ **S3 合同审查**全流程（审查 → 风险清单 → 门禁 → 带修订痕迹产出）。

**P12 版本发布说明/更新日志**（频率：版本级）
- 输入 → 产出：本版需求池已完成条目 + PRD 节选 → 发布说明（用户视角改写 × 条目溯源 × 分渠道版本[商店/官网/内部]）。
- 原语：`document`、`list`。
- 门禁：每条改写映射回需求条目锚；对外发布动作到门禁为止（产出落文件夹，发布是人的动作）。
- 同构：≈ S4 变体（模板化文书 + 溯源改写）。

**P13 数据周报/双周报叙事**（频率：周级，高频且苦役）
- 输入 → 产出：指标导出（csv/表格）+ 本周事项台账 → 周报页（指标卡 × 趋势图 × 事项清单 × AI 起草叙事段待确认）。
- 原语：`statcard`（头部三卡）、`chart`（趋势）、`list`（事项）、`narrative`（叙事段）、section 组合页 → 导出 docx/PDF。
- 门禁：叙事段（generated 通道）逐段确认；数字永远来自数据字段（verified 通道），**叙事不许改写数字**。
- 同构：≈ **S11 企业汇报族**原封不动（docs/48：向模板填表而比 PPT 更自由）——PM 周报就是 S11 的第一个具体租户。

**P14 Roadmap/里程碑规划**（频率：版本级）
- 输入 → 产出：需求池优先级 + 资源约束 → roadmap 提案（迭代 × 纳入需求 × 里程碑日期）。
- 原语：`timeline`（回看）、`schedule`（前瞻里程碑板）、`gantt`（挂账，多线并行需求出现再评——与法律包"多案并行审限"同一挂账理由）。
- 门禁：roadmap 是提案态；日期承诺对外同步到门禁为止。
- 同构：≈ 时效族 + timeline；"排期即承诺"的门禁语义与"期限即执业事故"同源，但可靠性等级可低一档（错排期不是事故，误期限才是——PM 包不需要时效族的高可靠多通道推送）。

**P15 上线 checklist/发布核对**（频率：版本级）
- 输入 → 产出：发布类型 → 核对单（埋点/灰度/回滚预案/客服口径/商店物料 × 是否齐备 × 证据锚）。
- 原语：`checklist`（勾选=记录核对事实，零执行语义）、`gate`（全勾完只解锁下游门禁可用态）。
- 同构：≈ **S9 招投标资质核对**逐字同构（checklist 原语的第二租户）。

**P16 迭代复盘/项目复盘**（频率：迭代级）
- 输入 → 产出：迭代台账 + 周报存档 → 复盘页（计划 vs 实际对照 × 里程碑年表 × 整改项核销单）。
- 原语：`compare`（计划 vs 实际）、`timeline`、`checklist`（整改核销）、`narrative`。
- 同构：≈ S11 复盘变体（docs/57 各原语的"汇报实例"列几乎全为本场景预写）。

### 1.4 普查小结

- 16 项中 **13 项纯"声"**（既有 schema 机制 + 新字段声明即可），3 项"声+构"（P6 打分需 CalcArtifact 的 PM 参数表、P1 需反馈洞察 schema、P13 需组合页上架）；**零"具"零"资"**（无新工具接口、无外部数据资产年更承诺）——与法律包 62 项普查中大量"资/具"形成对照，印证 docs/48 选型理由第一条。
- 渲染词表覆盖度：16 场景消费的原语全部在 18 原语词表内，无一新增诉求；挂账的 gantt/board 在 PM 域同样可先由 timeline/table 覆盖——**词表的领域无关命名纪律（docs/57）在第二垂类兑现**。

## 二、容器语义：PM 的"案件"是什么

### 2.1 容器 = 项目（产品线/大版本），阶段 = 迭代/里程碑

裁决建议：**容器对齐"一段有始有终的工作"而非"一个永续产品"**。产品本身是永续的，直接做容器会变成无边界的杂物抽屉；PM 的自然工作切片是"一个版本/一个专项/一个季度目标"，与"一个案件"同粒度（有目标、有材料、有产出、有结案态）。

- **容器 = 项目**：如"栖屋 App 3.0 版本""增长专项 Q3"。永续产品线的连续性由续行投影承担（需求池台账、复盘存档跨容器可引用——机制同法律包"系列案"，Stage 后话）。
- **阶段 = 迭代/里程碑**：法律包容器声明"阶段（立案/举证/开庭…）+三区"，PM 包声明"迭代（规划/评审/开发/验收/发布/复盘）+三区"——docs/49 第十章已明示左栏容器结构是**包的声明面**（stage-list/zone-tree/meta-row 词表组件），"招投标包可声明标段、科研包可声明文献区"，PM 包即第十章预言的第三个实例，左栏零改码。
- 容器名称词表：律所"案件"/法务"项目"已定为租户级词表（docs/49 二章 4），PM 包直接取"项目"，同一机制。

### 2.2 三区语义映射

| 三区 | 法律包 | PM 包 | 红线 |
|---|---|---|---|
| 原始材料区 | 卷宗（原件只读） | **素材区**：用户反馈导出、访谈逐字稿、会议纪要、指标 csv、竞品材料、上游 PRD/邮件 | **只读不可声明掉**（docs/49 第十章契约级红线）——反馈原声与纪要原文是溯源锚的宿主，改了原文锚就死了 |
| 工作稿 | 起草中的文书 | 起草中的 PRD/周报/评审意见、需求池台账工作态 | 同法律包：工作稿轨领域无关（docs/22 红利） |
| 产出 | 待确认 artifact → docx 文书 | 确认后的 PRD/评审报告/发布说明/周报/复盘页（docx/PDF） | 产出落用户文件夹，同 docs/46 裁决 10 |

一处需要明说的语义差：法律的"原件只读"有证据法意义，PM 的素材（如需求池）天然是活数据。处置：**需求池归工作稿/台账轨（可增改、全程留痕），进过容器素材区的"原文"（反馈导出、纪要）才受只读红线**——即只读约束的对象是"作为证据的语料"而非"作为工作对象的台账"，这与法律包"卷宗只读、证据台账可维护"的既有切分完全一致，无需新机制。

### 2.3 续行投影

重开项目容器的再水化输入 = 需求池台账 + 行动项账本 + 打分表 + PRD 版本状态 + RevisionEvent 流，transcript 为附录——docs/25"续行由工作语义保证"原样适用。PM 域反而是该机制的更强展品：PM 的周会/迭代天然是"隔几天回来续"的节奏，chatbot 竞品在此全靠翻聊天记录。

## 三、MVP 包定义草案：四场景

选型判据 = 频率 × 既有能力覆盖度 × 演示冲击力。选定：**PM-1 反馈归集（P1）、PM-2 PRD 评审（P11）、PM-3 优先级打分（P6）、PM-4 纪要行动项（P3）**。落选说明：P13 周报冲击力高但依赖 section 组合页与 narrative/statcard/chart 三原语上架，排为包内第二批（词表第一批 4 单完成后声明即上架）；P10 PRD 起草复用 S4 机制成熟但演示区分度低于评审（起草人人会 demo，带锚评审矩阵没人有）。

四场景合成一条演示动线（见 3.5 样板项目）：**倒一文件夹反馈进来 → 归出洞察 → 打分排出优先级 → 评审 PRD → 会后落行动项**——每步产出喂下一步，与临江案 S1→S4 的连贯叙事同构。

### 3.1 PM-1 反馈归集与洞察

```yaml
id: PM1
name: 反馈归集
trigger:
  fileTypes: [md, txt, csv, docx]
  userActions: [start-feedback-digest]
  classifierTags: [user-feedback, review-export, support-ticket]
inputArtifacts: [ProjectFile]           # PM 包的容器档案，≈ CaseFile 同形
toolIds: []                              # 零工具：纯文本摄取走 reading-view
outputArtifacts: [FeedbackInsightSet, FeedbackTimeline]
uiTemplateId: feedback-digest-panel      # matrix + list + chart 组合
confirmationGates:
  - artifact: FeedbackInsightSet
    label: 确认主题归类与代表引语后，方可写入需求池与后续引用
```

**FeedbackInsightSet 字段表（产出 schema 草案）**：

| 字段 | 类型 | 说明 | 纪律 |
|---|---|---|---|
| themes[] | object[] | 主题聚类 | 主题名为 generated 通道 |
| themes[].label / summary | string | 主题命名与归纳 | 生成物，可 amend |
| themes[].volume | number | 归入条数 | 确定性计数，非模型输出 |
| themes[].sentiment | enum(pos/neu/neg/mixed) | 情感分布档 | 出档位不出小数分 |
| themes[].quotes[] | object[] | 代表原声引语 | **逐字摘录 + sourceAnchor，无锚不落格** |
| themes[].channels[] / versions[] | enum[] | 渠道/版本交叉维度 | 来自语料元数据 |
| items[].themeRef / anchor | ref | 逐条反馈的归类判定 | 改判走 RevisionEvent |
| outOfCoverage[] | object[] | 无法归类条目 | 诚实置出，不硬塞 |

渲染组合：`table`(matrix：主题 × 版本) + `list`（主题卡+展开引语）+ `chart`（声量条，color.data.mono 单色阶）。

**PriorityScoreSheet 字段表（CalcArtifact 同族）**：

| 字段 | 类型 | 说明 | 纪律 |
|---|---|---|---|

### 3.5 demo 样板项目（类比临江案，虚构纪律照 docs/21）

**故事线：虚构公司"栖澜科技"的智能家居 App「栖屋」3.0 版本项目。** 唯一事实来源 project-bible.md，全部语料由其派生、日期指标交叉核对。

- **背景冲突**（≈临江案的"合同纠纷"）：2.4 版发布后商店评分从 4.6 跌到 3.9，差评激增；管理层要求 3.0 版"一个月内挽回口碑"，但需求池里躺着 30 条方向互斥的需求。
- **素材区语料**：约 60 条商店评论 + 25 条客服工单导出（csv/md）、3 份用户访谈逐字稿、4 份周会纪要、上一版 PRD（2.4，docx）、2 份竞品发布说明、周指标 csv。
- **预埋考点**（≈临江案 4 处预埋矛盾）：
  1. 分散在评论/工单/访谈三个渠道的 5 条反馈实际指向**同一根因**（设备离线推送延迟）——考归集的跨渠道聚类，演示"没有工具只有 schema 也能对齐"；
  2. 一条**高声量低价值**需求（换皮肤，30+ 条提及但 Reach 高 Impact 低）与一条**低声量高价值**需求（老人模式，3 条提及但客单与留存数据支撑）——考打分的反直觉时刻，RICE 排序与"按嗓门排序"打出的差异即演示高潮；
  3. 上一版 PRD 预埋 6 处评审缺陷（无验收标准的功能项、与访谈原话矛盾的用户假设、缺埋点、边界未定义、两节自相矛盾、引用了不存在的数据）——考评审矩阵逐处带锚定位；
  4. 4 份周会纪要中同一行动项（"补齐推送埋点"）**三次换负责人且未闭环**——考 carryOvers 接续比对；
  5. 2 条反馈为故意的不可归类噪声（乱码/无关广告）——考 out_of_coverage 诚实置出。
- **虚构纪律**：公司/产品/人名全虚构（栖澜科技、封令仪、桑之远等，沿用临江案自然人虚构命名风格）；指标数据自洽但明显演示化（DEMO 前缀口径注记）；语料一律 `source:"demo-fixture"`；数据落 `packages/demo-data` 或未来 PM 包自带 fixtures（归属存疑，见 4.3-⑤）。
- **demo 即 onboarding**（docs/21 终局增补同样适用）：PM 用户在栖屋项目里学会全部操作后归档。

## 四、差异与缺口

### 4.1 PM 包相对法律包**不需要**的

| 法律包资产 | PM 包处置 | 说明 |
|---|---|---|
| cite-check 法条/判例引用库 | 不需要 | PM 无"法条"级权威文本；引用对象是容器内语料，锚即校验 |
| party-verify 主体核验 + 工商数据 | 不需要 | 干系人图谱纯声明数据，无核验刚需 |
| OCR/ingest Python 服务 | 不需要 | 语料全为 born-digital 文本（docx/md/csv），reading-view 直接吃——选型理由第一条的兑现处 |
| 法规参数库年更承诺 | 不需要 | 打分权重表是团队自定义单表，零维护（vs 17 张法律参数表分级维护） |
| 时效族高可靠多通道推送 | 降级保留 | 截止日仍用 schedule，但无"误一次=执业事故"等级，普通提醒即可 |
| 法律文书 docx 模板（WPS 兼容性攻坚级） | 降级保留 | PRD/周报模板远宽容于法院文书格式，output 管线现成能力富余 |
| 信源分级 A/B/C | **机制保留，语义换词** | 分级机制是底座（docs/20），等级语义属包：A=容器内原文（反馈/纪要/数据导出）、B=企业内部库（Stage 后话）、C=web（竞品材料）。C 级进对外产出须逐条确认，纪律同法律包 |

### 4.2 PM 包**需要**的新东西（全部为"声明/数据"，无新机制）

1. **PM 产出 schema 族**：FeedbackInsightSet / ReviewFindingSet / PriorityScoreSheet / ActionItemSet + ProjectFile 容器档案（≈CaseFile 同形改字段）。均为字段声明，SourceAnchor/severity/门禁三态复用既有类型。
2. **打分公式表 = 计算器族既有机制**：score-calc 是 S10 CalcArtifact 机制的第二租户，脚本设计时一遍写定，公式（RICE/ICE/WSJF）公开且稳定，比法律计算器第一批 7 项更轻——建议实施时作为 CalcArtifact schema 的**同批验证用例**（一个 schema 两个垂类消费，契约质量的最好测试）。
3. **容器结构声明**："项目 + 迭代阶段 + 素材/工作稿/产出三区"一份声明，消费 docs/49 第十章预留的 stage-list/zone-tree 宿主接口。
4. **classifierTags 新词**：user-feedback/prd/meeting-minutes/backlog 等——trigger 三维的 fileTypes/classifierTags 均为自由字符串（registry SPEC 验收记录明示无枚举白名单），纯数据追加。
5. **提示词模板归包**：PM 语域的 prompt 文本随包迭代（docs/53 归包裁决"调 prompt 不动 core"），零底座改动。

### 4.3 零 core 改动上架：结论与存疑处

**结论：机制层零改动成立**——注册表 YAML strict 校验、握手协议、门禁强制非空、PreviewHost 按 uiTemplateId 挂载、output 管线、reading-view 全部现成。但有五处**依赖在途工单或需架构裁量**，如实列出：

1. **ArtifactTypeEnum 集中在 schemas 包**（registry 复用其枚举防名单漂移）——新增 PM 产出类型当前需改 `packages/schemas`。这是"垂类 schema 随包分发 vs 集中契约包"的结构张力，**正是 SCHEMA-SPEC-1 应裁决的第一案**（docs/48 已排 PM 包与 SCHEMA-SPEC-1/HARNESS-1 同期，顺序上先 SPEC 后 PM 即可消解）。
2. **渲染原语上架进度**：PM 四场景消费 list/form/table(matrix)/schedule/chart/revision——matrix/revision/timeline 既有，list/form+checklist/statcard+chart 恰在词表第一批 4 单内（docs/49 七章回册），schedule 在待上架清单。PM 包排收束后第一批，时序对得上，但需在 PM-PKG 工单里显式声明依赖。
3. **容器结构声明宿主**：docs/49 第十章"接口预留即完全兼容迭代"，当期左栏照法律声明硬装——PM 包上架前需第十章接口真正落地（左栏读包声明），此为底座侧一次性工程，属"兑现已拍板接口"而非"新机制"。
4. **`context: case | workspace | any` 字段属注册表 v2**（docs/49 落地清单）——PM 场景挂"项目"容器需要该过滤字段或等价物；v2 清单已有，需与 PM-PKG 同批排期。
5. **demo 语料归属**：临江案住 `packages/demo-data`；PM 样板项目建议同住该包（新增 `pm/` 子树）还是随 PM 包自带 fixtures，涉及"垂类即依赖包"的数据分发形态，留架构拍板。

|---|---|---|

| **Linear AI** | Triage Intelligence 自动分类/路由/优先级 issue（号称砍 70% triage 时间）、Asks 表单化收需求、agent 建 issue 占比 25% | 研发工具内嵌 agent，强在 issue 域、弱在文档与语料域 |

## 六、供架构过目的裁决清单

1. MVP 四场景选型（PM-1/2/3/4）与周报族排第二批——是否照案。
2. 容器语义"项目=容器、迭代=阶段、素材区只读对象=语料非台账"——是否照案。

5. 投递材料展品页建议：合同审查矩阵 vs PRD 评审矩阵并排 + 同一份底座代码零 diff——"换垂类=换声明"的一图证明。
