# 文档编辑面与 md↔Office 转译范式调研

- 调研日期：2026-07-09
- 调研方式：公开网络检索（官方文档/博客、产品发布页、开源仓库、第三方测评），中英文合计 24 组 WebSearch 查询 + 3 次官方博客/产品页原文抓取（Legora、Harvey、SuperDoc）
- 用途：为 Courtwork「起草画布（md）→ 手动编译为 Word（单向定稿）→ 定稿后走 docx 修订痕迹管线」的架构假设找业界证据，供 UI 层（apps/desktop）与 output 层（packages/output）设计参考
- 待验证假设：单向编译 + 阶段切换。反面假设：md↔docx 双向持续同步

---

## 0. 结论先行

**假设成立。** 证据链相当一致，且级别越高的头部产品越倾向支持这个方向：

1. 所有调研到的通用 agent 消费级产品（ChatGPT Canvas、Gemini Canvas、Notion AI、Coda AI）都是「专有/半专有编辑面 + 显式单向导出」，**没有一家做 md↔docx 的持续双向同步**。
2. Office 原生 AI（Microsoft Copilot in Word、Google Docs 的 Gemini）走的是另一条正交路径：**完全不引入中间表示**，直接在原生文档模型内操作，AI 的修改天然就是 tracked changes / suggestion。这恰好反过来印证：一旦编辑对象本身就是 docx/gdoc 原生格式，正确架构是"域内编辑 + 原生修订态"，不是"引入外部中间格式再译回"。
3. 法律垂类头部产品里，**Legora 在 2026-01 把"起草环境"和"Word 修订环境"显式拆成两个东西**（Editor 用于起草、Word Add-in 用于对已有 docx 做 redline），并且 Editor "导出到 Word" 是单向动作；**Harvey 在 2026-03 的技术博客里公开说明自己主动放弃了"以 markdown 为中间层编辑 docx"的路线**，因为 OOXML 的复杂度（run 拆分、numbering.xml 间接引用、修订元数据）撑不住通用中间表示，转而自建"docx 原生 in-memory 编辑 + 工具化 tracked changes 生成"管线。这是本次调研中最直接的第一手反例证据，来自法律 AI 头部厂商自己的工程博客。
4. 本次唯一找到的"声称做到双向同步"的案例（OpenKnowledge，用 yjs CRDT 做 markdown↔ProseMirror 双观察者同步）同步的两端是**markdown 文本与 ProseMirror 内部富文本模型**，两者结构复杂度接近（都是"轻量标记 + 线性文本"）。这不是 md↔完整 OOXML docx 的双向同步，反而通过对比划清了边界：**轻量标记语言之间的双向同步是可行的技术问题，标记语言与完整 Office 二进制/XML 格式之间的持续双向同步，在生产级产品里没有找到成功先例**。

## 1. 通用 agent / Canvas 范式盘点

| 产品 | 编辑面表示 | 导出 Office 机制 | AI 编辑与用户编辑的合并 |
|---|---|---|---|
| **ChatGPT Canvas** | 内部专有富文本 canvas 文档模型 | Canvas 右上角一键导出为 PDF / Word(.docx) / Markdown / 代码文件；web/desktop/mobile 三端一致；导出是**一次性动作**，不回写 | 同一个 canvas 内联合并，无中间格式往返 |
| **Claude / Cowork 的 md 编辑** | Markdown 源码即权威表示，渲染展示 | 生成 Word 时"先写 markdown，再转 docx"两段管线；反向则用 mammoth(docx→html)+turndown(html→md GFM) 两段管线做降级 | md 是唯一权威源，docx 只是导出格物，没有"编辑 docx 再合回 md"的产品级闭环 |
| **Gemini Canvas（Google Workspace）** | Canvas 专有文档/表格模型，自动保存(auto-save) | 需要 Workspace 许可 + Drive 数据存储连接；点击"Export to Docs"生成**独立的 Google Docs 副本**，导出后与 Canvas 不再保持同步 | Canvas 内部自行合并，导出后是快照 |
| **Notion AI** | 专有块级模型（database/callout/toggle 等） | 官方仅原生支持 PDF/Markdown/HTML 导出，**没有原生 docx 导出**；docx 需先转 md 再借第三方转换器生成，官方与第三方资料都明确指出"专有块结构不能直接映射到 Word 文档模型，简单复制粘贴会丢失表格布局、callout、toggle" | 无跨格式合并机制，纯靠人工二次调整 |
| **Coda AI** | 专有文档 + 表格块模型 | 官方只提供 PDF/CSV/文本导出；docx 需要"Export to Word Pro" / Documentero 等第三方 pack，走的是"把表格数据映射进 .docx 模板占位符"的填充模式，而非通用文档转译 | 不涉及双向合并 |

**小结**：四类通用 agent/协作产品无一例外走"专有编辑面 + 显式单向导出"，docx 从未被当作可持续同步的第二权威源。

## 2. Office 原生 AI（无转译问题的对照组）

### Microsoft Copilot in Word（2026-04 更新）
直接在 docx 域内操作，不存在任何格式转译。2026-04 更新后，"Edit with Copilot"在开启 Track Changes 时**原生产出 Word 级 tracked changes**（word-level diff，"every change reviewable, accept-or-rejectable, and audit-ready"）；同时支持 contextual comments（锚定在具体文本上的评论线程）、多步编辑时的实时进度展示。**局限**：Copilot 自己不能开关/批量接受或拒绝 tracked changes，这部分必须由用户在 Word 原生审阅 UI 里操作——即"AI 生成修订态，人工用原生审阅机制处理"，没有另建一套审阅 UI。

### Google Docs 的 Gemini（"Help me write" / Collaborate with Gemini）
同样完全在 Google Docs 原生文档模型内操作。建议以 Docs 原生的 suggestion/tracked changes 形式出现，"nothing gets applied until you approve it"；可以逐条 Accept suggestion，也可以 Accept all / Reject all；在应用前只有当前用户可见。

两家的收敛点几乎一致：**Office 原生 AI 的正确姿势是"不引入中间表示，直接在原生域内生成建议态/修订态，交给原生审阅机制处理"**。这跟 Courtwork 的"定稿后 docx 唯一权威表示、改动走修订指令管线"完全同构——只是 Courtwork 面对的是自建 agent 需要自己实现这层，而不是像 MS/Google 那样直接拥有文档引擎。

### WPS AI（国内基准）
官方主打"伴写"（续写，0.5 秒内根据标题续写内容，可加载云文档/网页作参考材料）和"智能改写"（一键润色/改写文体/调整长度），走的是**生成式覆盖或续写**，不是逐词级 track changes 的强调；对涉及数据/名言/法规的生成内容，靠"标注/引用来源"辅助校验，不是靠修订态展示。同时，第三方帖子与 Microsoft Q&A 上多次记录 WPS 修订模式与 Word OOXML 的兼容性问题（样式自动更新导致格式跳变，需要在"配置和修复工具"里手动切换兼容选项）。**这意味着国内 WPS 生态里"AI 编辑"与"tracked changes 审阅"目前是两条相对独立、尚未像 Copilot in Word 那样收敛统一的功能线**——这恰好印证 CLAUDE.md 里把"docx 的 WPS 兼容性"列为五个必须自研加固节点之一是站得住脚的，也是 Courtwork 用自建 docx 修订管线补位的机会点。

| **Legora** | **两阶段架构**：①**Editor**（2026-01 新发布，官方称"dedicated drafting environment"，非 Word 插件，"is not a standalone editor, it's part of the Legora platform"）——从 Tabular Review/研究结论"一键生成初稿"，团队协作、AI 建议全在此环境内完成，**"export to Word using your own templates and formatting"是单向动作**；②**Word Add-in**——专门用于对已导出/客户提供的既有 Word 文档做 redline、局部修改。官方博客明确划分使用场景："Use Editor when drafting and collaborating from your work done in Legora"，"Use the Word Add-in when working inside Word" | **与 Courtwork 假设几乎同构**：起草期用专属画布（Legora 是专有富文本协作环境，Courtwork 拟用 md 渲染画布）→ 单向导出到 Word → 对已有 docx 的后续修改走另一套 Word 原生工具，不回到起草画布 |
| **Harvey** | 2026-03 博客《Building an Agent for Complex Document Drafting and Editing》：第一代系统是"编排器切块 + 子 agent 一次性重写整段"，遇到 OOXML 复杂度天花板（加粗一个词就把段落拆成三个 run 元素；编号列表不与文本同存，活在独立 numbering.xml 里，经过多层间接引用才渲染出"1."或"(a)"这样的可见标记）。2025 年秋测试过用 Claude 的 docx skill（起 Python 脚本直接编辑 docx），**发现太慢、质量低，且业界最常用的 python-docx 库"已不再积极维护、OOXML 支持不全"**（本次调研核实：截至 2026-07，python-docx 在 GitHub 上仍有未解决的 issue，如不支持 .docm、无法处理 strict OOXML、.dotx 模板加载失败等）。最终自建：docx 解析为**可变的 in-memory 文档表示**，agent 用工具化接口在其上迭代编辑，用 `get_diff` 工具做自我校验，最终 diff 出结构化编辑集，**在 Word 插件里通过 OfficeJS 转成 word-level tracked changes，在 Web 应用里直接写回 docx 服务**。原文明确写道："in the web app, document editing has moved beyond markdown to a full docx editing experience"——**Harvey 是把"以 markdown 为中间态编辑再导出 docx"的路线，主动升级为"docx 原生结构直接编辑"，因为中间层在复杂法律文档（列表、样式、追踪修改）上扛不住** | **本次调研最强的第一手反例证据**：法律 AI 头部厂商公开承认并解释了"md 作为最终定稿态中间层"不可行的技术原因，与 Courtwork"定稿后 docx 唯一权威、不回 md"的假设完全一致 |
| **幂律智能 PowerOffice** | 自建网页编辑器（非 Word 插件），MeFlow 内嵌"在线 Office 编辑器"用于合同模板制作 | 印证 docs/02 已有结论：国内因 WPS 生态弱、网页协作是主流习惯，走自研编辑器路线而非依赖 Word 插件生态；Courtwork 面向中国用户理应对齐"自建网页画布"路线 |

## 4. 技术层：转译损失清单与编辑器/预览方案

### 4.1 md → docx「编译」信息损失清单（基于 pandoc 实测与 issue 追踪）
- 代码块的语言标注在往返转换中会丢失，需要 Lua filter 手动补
- H1 与 YAML frontmatter 的 title 语义会混淆，pandoc 默认把 H1 当普通标题处理，容易导致标题层级被拉平或误判
- 基于公司模板"Normal"样式派生的自定义 Word 样式，转换后只剩最小语义信息，样式命名与继承关系基本丢失
- 图片尺寸在 md→docx 转换时会被写死为固定像素尺寸（pandoc 官方 issue #976）
- 脚注、交叉引用域代码、批注(comments)、修订痕迹(`w:ins`/`w:del`)、编号列表的 `numbering.xml` 间接机制，pandoc 均不原生支持双向精确重建
- pandoc 官方文档自己的表述："round-tripping with format-specific features (Word macros, complex tables, footnotes) often loses fidelity"

### 4.2 docx → md「降级」信息损失清单
- 业界常用管线是 mammoth（docx→html）+ turndown（html→GFM markdown）两段式，Claude 生态内部即采用类似两段管线
- 只能保留基本富文本（标题/加粗/列表/表格），会丢失：tracked changes 的 `w:ins`/`w:del` 元数据、批注(comments)、页眉页脚、复杂表格合并单元格、样式命名、编号列表的间接引用、目录/交叉引用域代码、嵌入对象（OLE/图表）

### 4.3 起草画布的技术底座：ProseMirror 类富文本框架
- **prosemirror-suggestion-mode**（davefowler）与 **prosemirror-suggest-changes**（handlewithcarecollective / 现被 BlockNote 收纳为 `@blocknote/prosemirror-suggest-changes`）：两者都实现了"Google Docs 式建议模式"——用 mark 标记增删而非真正改变文档节点，可回收接受/拒绝。技术成熟度高，是"AI 建议 vs 用户编辑合并展示"的直接可复用组件。
- **OpenKnowledge**（Inkeep，2026-06 开源）：用 yjs CRDT 做"双观察者"，让 ProseMirror 富文本文档与其 markdown 表示保持连续无损同步，允许 Claude Code / Codex 直接写 md 文件、WYSIWYG 侧实时反映，声称"lossless"。**但同步的两端是 markdown 文本 ↔ ProseMirror 内部模型**，并非 md ↔ 完整 OOXML docx——两者结构复杂度接近，不构成对"md↔docx 双向同步不可行"结论的反驳，反而划清了边界。

### 4.4 docx 浏览器内渲染/编辑预览方案
| 方案 | 定位 | License | 适用场景 |
|---|---|---|---|
| **docx-preview**（VolodymyrBaydalka/docxjs） | 轻量 docx→HTML 渲染库，`renderChanges` 选项可显示修订痕迹（标注为 experimental） | Apache-2.0 | 只读预览，不适合作为可编辑修订底座 |
| **SuperDoc**（harbour-enterprises） | 定位"Modern DOCX editor and Agent SDK"，浏览器/无头 Node/AI agent workflow 中渲染、编辑、自动化 .docx；Yjs CRDT 实时协作，原生支持 comments/tracked changes/冲突自动解决 | 双许可：AGPL-3.0 开源 + 商业 Enterprise License | 目前开源生态里唯一同时覆盖"docx 编辑 + tracked changes + agent 可编程"的成熟项目，是 Courtwork docx 修订预览/编辑环节最值得优先评估的底座（需核实 AGPL 对闭源商用的传染性影响，大概率需买商业授权） |
| **OnlyOffice / Collabora** | 重量级路线，需要独立 Docs Server 进程，通过 API/WOPI 嵌入；OnlyOffice AGPL 3.0 开源 + 商业版（本地部署商用需一次性授权费）；内置 AI assistant 可接 OpenAI/Anthropic/DeepSeek 等多家 provider | AGPL/商业双轨 | Word 兼容度最高、真·所见即所得，但需要独立服务进程，是重资产部署；适合"要完整 Office 编辑体验"的场景，对"法律 agent 里嵌一个修订预览面板"这种轻量需求偏重 |

## 5. 裁定与建议

1. **单向编译假设成立。** 通用 agent 产品全走"专有编辑面 + 显式单向导出"；Office 原生 AI 走"域内编辑 + 原生修订态"；法律垂类头部产品 Legora 直接给出"起草环境（单向导出）+ Word 修订环境"两阶段先例；Harvey 更是公开说明了为什么放弃"md 作为定稿中间层"。没有找到任何生产级产品做 md↔docx 持续双向同步的成功案例。

2. **起草画布技术底座建议**：md 源 + ProseMirror（或 TipTap）渲染编辑，叠加 `prosemirror-suggestion-mode` 或 `prosemirror-suggest-changes` 类插件承载"AI 建议 vs 用户编辑"的合并展示。这条路线技术成熟、可控、且与"起草阶段唯一权威源是 md"的假设完全匹配，不需要在这个阶段引入 docx。

3. **docx 修订预览/编辑推荐方案**：分层看——只读预览优先用 `docx-preview`（轻量、Apache-2.0、无授权顾虑）；若未来需要"在服务端/无头环境直接生成或编辑 tracked changes 而不依赖 Word 客户端"，`SuperDoc` 是目前最匹配的开源底座，但需要单独评估 AGPL 授权成本（大概率需买商业授权，属于需要过 review 的决策）；`OnlyOffice`/`Collabora` 整套文档服务器路线过重，只有在需要"完整 Office 级所见即所得编辑"时才考虑，不建议作为 MVP 阶段的默认选择。

4. **"手动编译定稿"的交互呈现**，两个可直接参照的先例：
   - **Draft-Publish 模式**（内容管理系统通用范式）：草稿可反复迭代，一个显式"发布"动作把草稿转成不可逆的最终态；再次修改会新建草稿而非编辑已发布对象。
   - **AIA Contract Documents 的 "Finalize" 机制**（建筑合同文档行业标准产品）：草稿在线可编辑，用户点击 "Finalize" 后生成不可再回退编辑的最终文档，并配 variance checker 提示定稿前后差异——这与 Courtwork "编译为 Word" 动作本身也可以顺带做一次"定稿前差异核对"高度契合。
   - 结合 docs/02 已收录的 Garfield AI（英国 SRA 批准的 AI 律所，每步不可逆动作都需人工批准）与深圳法院系统（每环节勾选确认）先例，"编译为 Word"这个按钮本身应该设计成一次需要用户主动确认的"冻结仪式"，而不是自动触发的后台动作，这与 CLAUDE.md 里"留人确认是产品纪律"的工程纪律直接对齐。
