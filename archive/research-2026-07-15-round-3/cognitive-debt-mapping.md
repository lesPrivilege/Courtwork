# 认知债三部曲对照（2026-07-15）

来源：产品负责人转交的行业文章（Coding Agent 后产研需偿还认知债：UL / 可执行 PRD / Headless POC）。调研原稿，不具约束力。

## 对照结论：三步已有实现，且各有更硬形态

- **UL** ≅ schema engineering（字段=业务判断的形状）+ 各包词表——我们有 drift 机器门，语义漂移触红，不靠团队自律；
- **可执行 PRD** ≅ ADR/SPEC/验收标准 + 机器门——SPEC 留痕连反例注入都是强制项；
- **Headless POC** ≅ demo-runtime CLI（demo:s3/demo:legal/real:s3）——文章描述的演化路径（原型→测试编排层→可执行业务说明书）已走完：golden 字节锁定、验收编排、真跑入口三态俱全。「**可执行业务说明书**」是 demo-runtime 职责的最佳命名，采用。
- 「生成能力压倒验证能力」的病理即本轮多次实证的「声称与行为漂移」（样式断言≠行为发生、错误归因三传成真）——验收体系的逐帧采样/基线复跑口径就是解药。

## 一个真实 gap（eval 线候选，挂起）

文章第七节：Agent 在 headless 层模拟不同类型用户自动探索业务路径（正常用户走通核心 / 恶意用户绕不过约束 / 历史用户故事不回归 / PRD 承诺的收益仍被兑现）。我们的反例目前全部由架构师手工设计；agent 驱动的探索型路径测试是 eval 线自然增量——在 demo-runtime 之上跑探索 agent，与 fast-check 式随机交互序列验证在业务层同构。待 eval 线扩容时立单。
