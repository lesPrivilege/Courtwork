# 架构观察：数据维护即场景（2026-07-10）

状态：已拍板方向，排期 MVP 后（deep research 能力就绪后启动）。

## 观察

案例集/法条库/语料的维护工作本身声明为一族**内部 curation 场景**，跑在同一 core 上：agent 承担采集、比对、初筛、格式化（"新判例发现""法条修订同步""条目复核""语料扩充"等），律师/研究团队/专家型顾问在确认门禁上做专业判断。

## 机制（零新代码）

- 留人确认是 core 通用机制（一份实现）；门禁开在哪、确认什么由场景 YAML 声明——现行架构原样复用。
- 确认协议的 actor 已是通道无关身份标识，增加 role 语义区分"办案确认"与"专家复核"（进注册表 v2 的 gate 定义考虑项）。
- C 级升级漏斗（docs/20：web 发现 → 专家复核 → 入库升级）即此工作流雏形；deep research 场景（docs/06）是其采集引擎。

## 战略含义

意见稿"线性苦活"论证不变（无自动验证、专家判断仍是瓶颈与稀缺输入、没有飞轮），变的是**单位维护成本被自己的 agent 压低**：护城河从"雇得起人做苦活"升级为"苦活单位成本数分之一 + 专家门禁保证判断质量"。同时这是产品的第一个内部 dogfooding 用户——维护团队用 Courtwork 维护 Courtwork 的数据。

## 首个试点（已挂账）

67 条演示法条复核微工单：agent 抓官方文本 → diff 比对 → 专家逐条确认入库。它既销掉既有挂账，也验证 curation 场景形态。

## 边界

- 专家判断不可被门禁默认值替代——curation 场景的门禁无"批量确认"路径（比办案场景更严：库是全体用户的信任基础）。
- 入库动作是不可逆动作谱系的一员，按 docs/09 分级确认设计。
