# Tauri 法律工作台专业级 UI/UX 设计提案 (已定稿)

本提案旨在解决当前 Tauri 桌面法律工作台存在的“AI 生成感（Slop）”视觉噪音，并针对右栏五个工作面（时间线、关系图谱、矩阵表格、docx 修订预览、起草画布）在有限空间下的管理给出系统性的交互与空间方案。**本文档已集成用户决策，作为 B 阶段验收及 polish 工单的直接设计规范依据。**

---

## 💥 第一部分：「AI 生成感」清理清单 (De-slop Audit Checklist)

“AI 生成感”通常来源于主流 AI 代码/UI 生成工具（如 v0, Lovable 等）默认生成的模板化、过度装饰、为移动端或消费级 Web 设计的视觉规范。对于处理高密度、严谨法律事务的专业用户（律师、法务、合规官），这些特征会传递出“业余”与“不可靠”的视觉信号。

以下是针对该工作台的 **De-slop 审计清单**，按严重度从高到低排序，并参考 Linear、Stripe Dashboard 及专业 eDiscovery 软件（如 Relativity）的视觉成熟度进行了修正：

| 序号 | 诊断特征 (AI Slop 表现) | 严重度 | 专业用户的负面感受 | 修正方向与设计规范 (Ref: Linear / Stripe) |
| :--- | :--- | :--- | :--- | :--- |
| **1** | **卡片左侧彩色竖线过密**<br>（每个卡片或列表项都带有一条明艳的装饰性竖线） | 🚨 **致命** | 视觉污染、缺乏重点、像低端任务看板。 | **全面确认：取消法理之线静默态。**<br>竖线**仅作为状态或属性的语义指标**（如“未读”、“有冲突冲突”或“当前选中态”），宽度控制在 2px。普通卡片仅靠 1px 的中性灰边框 (`#E2E8F0`) 或背景色差来区分。 |
| **2** | **过度使用卡片嵌套 (Card-ception)**<br>（卡片包卡片，带有多层圆角、阴影和外边距） | 🚨 **致命** | 空间利用率极低，界面臃肿，无法展示长案情。 | **扁平化信息层级。**<br>消除嵌套卡片的投影和圆角，改用 **1px 水平分割线** (`#F1F5F9`) 进行区域划分。除主面板外，内部元素采用无背景的 Tabular Layout（表格布局），减少空间浪费。 |
| **3** | **填充型彩色图标乱用**<br>（使用彩色或厚重的填充图标作为装饰，色彩混杂） | 🔴 **高** | 视觉幼稚、干扰视线，像儿童玩具或非专业 SaaS。 | **统一图标系统与色彩预算。**<br>全量采用极细的双色线框图标（如 1.25px - 1.5px 描边），默认颜色统一为中性深灰 (`#64748B`)。**图标不着色**，仅在悬停、选中或表示特定语义（如“红色警告”/“绿色已签”）时才应用单色。 |
| **4** | **消费级大间距 (Loose Padding)**<br>（采用 16px-24px 的移动端/消费端间距） | 🔴 **高** | 屏效极低，单屏承载信息少，专业人员需要频繁滚动。 | **实施紧凑型栅格系统 (Compact Density)。**<br>列表项垂直内边距 (Padding-Y) 压缩至 6px-8px，数据行压缩至 4px-6px。参考 Relativity 和 Excel，在保障文字底线对齐的前提下，尽可能提高首屏可见数据量。 |
| **5** | **大半径圆角 (Over-rounded Corners)**<br>（卡片、按钮、标签普遍采用 `12px` 或 `16px` 圆角） | 🔴 **高** | 界面圆润可爱，缺乏法律行业所需的“严谨、精准、工业感”。 | **收紧圆角半径。**<br>标签与微型按钮采用 `2px` 或 `4px` 圆角；工作区卡片采用 `6px` 圆角；外层主面板最大不超过 `8px`。折线和直角的精密感更符合金融与法律用户的心理预期。 |
| **6** | **大面积漫反射投影 (Soft Shadows)**<br>（使用模糊半径大、透明度高的灰色投影） | 🟡 **中** | 界面脏乱、缺乏物理实感，在浅色模式下尤为明显。 | **采用极细投影或无投影设计。**<br>对静态或平铺面板**完全禁用投影**，仅通过 1px 细线分隔。仅对悬浮菜单、下拉框使用 Stripe 式的极细双层投影：`box-shadow: 0 1px 2px rgba(0,0,0,0.05), 0 0 1px rgba(0,0,0,0.1)`。 |
| **7** | **未进行等宽对齐的数值与时间**<br>（案号、日期、金额使用标准无衬线字体，导致纵向错位） | 🟡 **中** | 杂乱无章，难以进行上下快速对比 and 视觉扫描。 | **启用等宽数字 (Tabular Numbers)。**<br>在 CSS 中对所有包含数字、日期、案号的区域设置 `font-variant-numeric: tabular-nums`，或直接使用 `JetBrains Mono` / `SF Mono` 等专业等宽字体。 |
| **8** | **色彩无序蔓延 (Color Pollution)**<br>（标签、状态、按钮随机分配黄蓝绿橙等高饱和度色彩） | 🟡 **中** | 视觉疲劳，丧失了“颜色传递信息”的语义功能。 | **实行严格的色彩预算（Color Budget）。**<br>界面 90% 区域保持白、灰、深灰的单色调。颜色只做语义表达：红色仅代表“冲突/时效超期”，黄色代表“待补充/待审阅”，蓝色代表“操作聚焦”，绿色代表“归档/生效”。 |
| **9** | **生硬的交互反馈与动效**<br>（鼠标滑过时元素生硬闪烁，或动画范围过大） | 🟡 **中** | 闪烁感干扰思考，界面廉价，降低工作专注度。 | **微动效与静止数据区。**<br>数据区域完全静止，禁止闪烁。仅在按钮悬停时使用 `150ms ease-out` 的微弱背景变深（如 `bg-slate-50`）或边框加深。 |
| **10** | **通用型“插画”空状态 (Empty States)**<br>（无案件或无内容时，展示一张巨大的彩色 AI 插画） | 🟡 **中** | 充满幼稚感，浪费核心工作区空间。 | **Typographic（文字型）空状态。**<br>去除插画。改用 1px 虚线边框矩形，内置一行简练的提示文字及快捷键引导（例如：“暂无案件，按 ⌘N 快速创建”），保持工作台的工具属性。 |
| **11** | **长文本硬剪裁或无 tooltip 换行**<br>（超长案件名或文档名直接折行打破容器，或强行切断） | 🟢 **低** | 视觉精致度坍塌，关键信息缺失。 | **单行截断 + Hover 悬浮显示。**<br>使用 `text-overflow: ellipsis; white-space: nowrap; overflow: hidden;`，并在悬停时展示原生或定制的高性能轻量 Tooltip。 |
| **12** | **粗糙的系统默认滚动条**<br>（Windows/Mac 默认的粗滚动条在三栏布局中割裂视线） | 🟢 **低** | 破坏了 Tauri 桌面应用的“客户端原生感（Native Look）”。 | **自定义极细滚动条。**<br>通过 CSS 修改滚动条样式，使其在静止时隐形，仅在滚动或悬停时呈现宽度为 4px-6px 的淡灰色圆角条，避免截断右侧内容。 |

## 🗺️ 第二部分：右栏多工作面的空间管理方案对比

针对右栏的五个工作面（**时间线、关系图谱、矩阵表格、docx 修订预览、起草画布**），在有限的物理宽度下，我们需要权衡它们各自的呈现效率与 35-55 岁非科技背景用户的认知心智。

### 空间组织方案对比表

| 空间方案 | 适用条件 | 认知成本 (35-55岁非科技背景) | 优势 (Pros) | 劣势 (Cons) | 行业先例与口碑 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **1. Tab 切换** | 5个工作面完全独立，用户单次仅专注于单项任务。 | 🟢 **极低**<br>(符合浏览器和 Excel Sheet 的直觉，无需学习) | 界面最整洁；单面板独占完整宽度，展示密度最高。 | 无法同时对照查看（如：无法边看关系图谱边在画布起草）。 | **Relativity (eDiscovery)**: 默认采用 Tab 切换不同文档属性，口碑极稳。<br>**Bloomberg Terminal**: 基础窗口间切换。 |
| **2. 手风琴折叠** | 各工作面间有主次之分，用户需快速展开/收起某一块查看摘要。 | 🟡 **中**<br>(折叠后内容被截断，容易找不到内容) | 可以在一屏内纵向排列多个面，提供宏观概览。 | 纵向空间被极度压缩，关系图谱或矩阵表格在此模式下几乎无法阅读。 | **Salesforce / Jira Sidebar**: 用于展示案件元数据，口碑一般，用户经常抱怨“不停地展开和折叠”。 |
| **3. 可拖拽分割条** | 用户需要频繁进行“跨面板协同”（如边看时间线/修订，边在画布撰写）。 | 🟡 **中**<br>(操作简单，但手动管理多窗口会带来“视窗整理焦虑”) | 自由度极高，支持并排或上下对照，最符合专业办公的需要。 | 拆分过多会导致界面极度拥挤，需要提供一键复位机制。 | **VS Code (Split Panels)**: 程序员标配。<br>**LiquidText (Desktop)**: 文档与脑图的左右分栏对照，口碑极高。 |
| **4. 无极缩放画布** | 需要将时间线、关系图谱、画布融合成一张大图，进行空间化思考。 | 🔴 **极高**<br>(35-55岁用户常因鼠标滚轮和空间平移而“迷路”) | 极佳的视觉延展性，适合复杂线索的整理。 | 矩阵表格和 docx 预览这种文本密集型工作面不适合被缩放，排版容易模糊或错乱。 | **Miro / Heptabase**: 科技工作者极度喜爱。<br>**传统法律用户**: 口碑极度分化，非科技背景律师经常发生“找不到画布上的内容”的问题。 |
| **5. Overview + Detail** | 专用于超大图表（如包含数百个节点的关系图谱或超长案件时间线）。 | 🟡 **中**<br>(需要理解小地图与大图的映射关系) | 解决大图场景下的空间迷失问题。 | 占用额外空间，不适合作为 5 个不同工作面的全局管理方案。 | **Palantir Foundry (Graph)**: 关系图谱分析的标准配置，专业分析员反馈极佳。 |

## 🛠️ 第三部分：推荐与拍板方案落地细则

针对 35-55 岁律所主流用户群（使用 1080p 笔记本与普通单屏，非科技行业极客），将空间管理方案落实为以下具体交互物理规范：

### 1. 核心物理架构：双能动上下分栏 (Split-Tab Grid)

*   **默认分栏方向：默认上下对切**
    *   **立论依据**：法律文书与 docx 修订的行宽是硬需求（至少保持 `600px+` 才能保证排版及修订线可读）。由于右栏为三栏中最窄，若左右对切必然使两边沦为不可读的“细条”。因此，必须采用**上下对切分栏**（保留宽度、牺牲高度——高度可以通过纵向滚动解决，宽度受限则根本无法阅读）。
*   **自适应横向分栏**：
    *   仅当窗口物理宽度 $\ge$ `1600px`（大显示器或宽屏）时，**左右分栏**选项自动解锁可用。
*   **左/中栏主动让宽机制（协同工作流的核心）**：
    *   当右栏启动“双能动对照工作流”（即拆分为上下两个面板）时，系统自动或允许手动触发：
        1. **左侧案件列表折叠**：缩为极窄的“图标条”（Icon-only Bar），仅显示图标与未读气泡。
        2. **中栏对话框收窄**：将对话流与场景卡片收缩至紧凑形态，从而为右栏腾出至少 `200px-300px` 的额外横向宽度。

### 2. 起草画布 (Drafting Canvas) 的绝对定位：结构化富文本

*   **底座规范**：遵循 `docs/23` 的决策，基于 `Markdown 源 + ProseMirror 类编辑器` 渲染。
*   **物理属性**：它是一个**线性文本编辑器**，坚决不采用任何 Canvas 画布技术。
*   **视觉与交互**：
    *   保持**绝对静止**、高密度排版、紧凑的文本栅格。
    *   **零视口缩放**。禁止任何通过 Ctrl/Cmd + 鼠标滚轮对该面板进行放大缩小的交互，所有字号大小由系统级标准 CSS 字体阶梯控制（静止态）。

### 3. 隔离缩放：无极缩放 (Zoomable Canvas) 专属沙盒

*   **唯一性隔离**：**无极缩放交互只允许存在于 `[关系图谱]` 面板内部**。
*   由于图谱属于空间拓扑数据，缩放及平移是必需的。而时间线、矩阵表格、docx 预览及起草画布作为线性或表格式文本，必须维持静态排版。
*   **交互物理学隔离**：两种面板的交互物理学（Interaction Physics）严格隔离，禁止将图谱的拖拽、滚轮缩放逻辑渗透至其他 4 个面板中，以杜绝 35-55 岁非科技用户“意外滑滚轮导致文字消失/迷路”的负面体验。
