# 架构决定：信源分级与检索策略（2026-07-09）

状态：已拍板，分期落地（等级语义随 W5/W6，retrievalPolicy 字段待 MVP 后场景注册表 v2）

## 决定

1. **Web search 是 `packages/tools` 契约下的普通工具**，不是 core 内置特权能力。同一套输入/输出 schema、超时、缓存、降级纪律。
2. **信源分级（evidence grade）**：
   - **A 级**：权威确定性接口（官方库、企查查/天眼查、类案同判库）——可作为文书批注依据、正式产出的事实来源。
   - **B 级**：自建/演示库——可用，携带覆盖边界语义（`out_of_coverage`），演示库结果自我标识。
   - **C 级**：web search——**永远是参考线索，不是已核验事实**。结构上属 `verified:false` 家族（`reason:"web_reference"` 携带结果），类型层面进不了"已核验"通道。Web search 降低漏检，不豁免核验。
3. **平衡点写在场景声明里**：注册表场景定义未来增加 `retrievalPolicy`——本场景允许的工具集 + 各信源等级可流入的产出槽位（如 S3 主体信息槽只收 A 级；研究类场景允许 C 级进正文但强制标注）。**调整检索/库的平衡 = 改 YAML 声明**，core 只做通用的等级传播与门禁执行，不含任何场景特判。
4. **C 级升级通道 = 库的采编漏斗**：律师确认 web 检索结果 → RevisionEvent 记录 → 专业团队复核 → 纳入自建库升级为 B/A。覆盖缺口由真实使用暴露，按需补库。
5. **平衡的裁判是 eval**：W7 按场景跑开/关 web search 与不同策略配置的对比，用跑分决定各场景默认声明。

## 分期

- W5（当期）：契约的判别联合与 reason 枚举已具备承载能力，不需为此改动；web-search 工具本体不在 W5 范围。
- W6（core）：等级传播与确认门禁的通用执行逻辑预留（不做 web-search 工具本身）。
- 注册表 v2（MVP 后）：`retrievalPolicy` 字段正式进场景定义 schema。
- UI（W9）：信源等级角标（已核验 / 库内 / 网络参考）。

## 不许做

- 不许让 web 结果以任何路径获得 `verified:true`。
- 不许在 core 写场景特判的检索逻辑（平衡点属于声明层）。
- 不许让 C 级事实未经确认流入 docx 批注依据。
