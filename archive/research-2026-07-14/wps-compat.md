## 0. 结论先行

**本报告不能替代实测。** 以下所有关于「WPS 打开后修订/批注显示是什么样」的判断，凡是没有在标题里出现「实抓」二字并给出可验证 URL 的，一律不构成「WPS 兼容」的验收证据——这一点在第 6 节实测矩阵交付之前必须反复强调，因为这正是本任务纪律要求的红线。

本次调研在**不接触 WPS 实体**的约束下，找到了两条比预期更有价值的证据，改变了「查不到东西」的悲观预期：

1. **WPS 官方社区自己承认、且截至本报告撰写时仍未修复的两个结构性缺陷**，都恰好命中 Courtwork 最担心的两个场景：
   - 修订与批注共存于同一处文字时，WPS 的两种审阅显示模式**没有一种能同时正确显示「删除前文字 + 插入后文字 + 批注内容」三者**（[bbs.wps.cn/topic/32061](https://bbs.wps.cn/topic/32061)，2024-05-16 提出，2026-05-06 仍有用户跟帖确认未解决，跨度近两年）——这正是 `sample-redline.docx` 里「违约金条款批注锚定在新插入文字附近」这类场景。
   - Custom XML Parts / 内容控件的绑定关系**在 WPS 文档关闭重开后会失效**，官方社区运营 2026-02-28 回复承认收到反馈并转交技术团队评估（[bbs.wps.cn/topic/82618](https://bbs.wps.cn/topic/82618)）——这不是 Courtwork 自己的锚点机制，但是同一类风险（「文档内部结构化绑定，WPS 往返后是否保真」）的官方实锤旁证。
2. **更底层的一条硬风险**：有具体技术博客（2021 年，.NET OpenXML SDK 开发者一手记录）显示，WPS 产出的部分文档**不符合标准 ZIP 包规范**，导致严格遵循 ZIP/OPC 规范的解析器在解压阶段直接抛异常（[cloud.tencent.com/developer/article/1773277](https://cloud.tencent.com/developer/article/1773277)）。这条证据较老，需要在 2026 环境重新验证是否依然存在，但它指向的失败模式（**解压阶段直接失败，不是渲染不好看**）比修订/批注渲染问题更严重，直接命中 `packages/output` 用 `fflate` 解压的架构——如果插件回传场景需要重新读取「被 WPS 编辑保存过」的 docx，这条必须是测试矩阵里优先级最高的一行。
3. **大量本该有答案的具体问题，本次调研查不到任何实测证据**：`w:moveFrom`/`w:moveTo`、`w:rPrChange`/`w:pPrChange`、表格内修订（`w:cellIns`/`w:cellDel`/`w:tcPrChange`）、批注线程扩展（`commentsExtended.xml`/`w15:paraIdParent`/`done`）在 WPS 下的具体渲染行为，中英文资料都是空白——这是符合预期的「正常的查不到」，不是调研不到位，已如实标注「未能核实」，全部转化为第 6 节测试矩阵的必测项。
4. **方法论警示**：WPS 相关的中文搜索结果里混杂着大量非官方、疑似 AI/SEO 生成的「WPS 下载站」「兼容性终极测试报告」类网站（域名形如 `wpsapp1.com`、`hq-wps.com.cn`、`mac-wps.com`、`zone-wps.com` 等，与官方 `wps.cn`/`wps.com` 域名相似但非官方），其中部分内容读起来像模板化生成、给出无法核实的具体版本号和「双向测试」细节。本报告已逐一甄别，凡引用此类站点均明确标注 ⚠ 并说明不采信其具体结论，仅作为「这类信源存在」的反面记录。这一发现本身应该写进团队关于 WPS 调研的方法论提醒。

**一句话总结**：桌面调研能做的部分——盘清已知风险、把「应该可以」变成「必须验证的具体测试项」——已经完成，且意外挖到两条有价值的官方实锤证据；但「WPS 打开我们的 `sample-redline.docx` 到底长什么样」这个核心问题，答案依然是「不知道，必须有人用真实 WPS 测」。

## 1. WPS 的现状：2026 年对 OOXML 的支持

- **✓ 未找到金山官方发布的、正式的 OOXML 兼容性声明或已知限制清单。** 多轮搜索（"WPS Office 2026 OOXML compatibility official documentation"、"WPS Office 已知问题 兼容性 限制 官方声明"等）均未命中类似 Microsoft 那种逐特性列出的 conformance statement。找到的都是第三方评测文章、用户教程或论坛帖子，没有一份是金山官方逐 OOXML 元素说明「支持/不支持/部分支持」的技术文档。这个「查不到」本身是本节最重要的结论。
- **✓ 官方确有开发者文档，但层级是「应用层脚本 API」而非「底层 XML 结构规范」。** [WPS WebOffice 开放平台](https://solution.wps.cn/docs/client/api/summary.html)（`solution.wps.cn`，金山官方域名，2026-07-14 抓取）提供完整的 Word 对象模型 JSAPI 文档，包含 `ActiveDocument.Revisions`、`ActiveDocument.Comments`、`Bookmarks`、`ContentControls` 等条目。这说明金山内部确实有一套对应 Word 修订/批注/书签概念的编程模型，但文档描述的是「怎么用 JS 调用」，不是「`w:ins`/`w:del`/`comments.xml` 底层怎么被解析、遇到不认识的 part 怎么处理」这类问题的答案。
- **✓ Revisions API 印证了 SPEC.md 的判断**：[Revisions API](https://solution.wps.cn/docs/client/api/Word/Revisions.html) 页面本身因前端渲染问题两次直接抓取只拿到 meta 信息，但通过 WebSearch 摘要交叉确认其方法列表含 `AcceptAll()`、`RejectAll()`、`Json()`（2026-07-14，※ 转述而非直接抓到完整正文，但来源就是这个官方页面本身，可信度较高）——**没有「插入一条干净修订」的方法**。这与 `packages/output/SPEC.md` 已经记录的判断一致：「Word（Office.js）与 WPS 开放平台均无程序化插入干净修订的 API（结构性缺失，非版本问题）」。本次调研在 WPS 一侧拿到了具体的方法列表作为佐证，而不只是复述已有结论。
- **✓ 多平台确实存在**：Windows / macOS（Apple Silicon 原生支持）/ Linux（deb/rpm，2026-06-16 有更新，※ 转述）/ 国内版 `wps.cn` / 国际版 `wps.com`（✓ 已核实 `wps.com/about-us` 为金山官方国际站，2026-07-14）。个人版基础功能永久免费，无需激活。
- **⚠ 无法确认「支持到哪个 OOXML 版本／Transitional 还是 Strict」这类具体符合性等级**，未找到官方声明，标记未能核实。
- **✓ 方法论发现**：搜索「WPS Office 下载」「WPS 更新日志 2026」类关键词时，返回结果里官方域名（`wps.cn`/`wps.com`/`linux.wps.cn`）与仿冒域名（`excel-wps.hl.cn`、`hq-wps.com.cn`、`x-wps.com`、`wps-flow.com`、`our-wps.com`、`mac-wps.com`、`power-wps.com`、`zone-wps.com`、`mega-wps.com`、`smart-wps.com` 等）几乎等量混杂，其中一批声称的具体版本号+发布日期（如「Version 12.7.5，2026-02-28，修复了接受所有修订后格式异常」）**无法追溯到可验证的官方一手来源**，本报告不采信这类具体数字，只记录「WPS 官方大概率确实在持续修复修订相关 bug，但具体版本/日期未经证实」这个方向性判断（⚠）。

## 2. 修订痕迹（Track Changes）：`w:ins`/`w:del`/`w:moveFrom`/`w:moveTo`/`w:rPrChange`/`w:pPrChange`/表格内修订

### 2.1 有实抓证据的部分

- **✓✓ 最重要发现**：[bbs.wps.cn/topic/32061](https://bbs.wps.cn/topic/32061)「修订模式下嵌入式显示与批注框无法同时显示的问题」（官方社区，「反馈直通车」标签即金山官方产品反馈渠道，2024-05-16 发帖，2026-07-14 本次直接抓取完整帖子与全部 19 条评论）。楼主原话（节选）：

  > 图1，采用批注框显示修订的方式。可以直观的看到删除的痕迹（"这是"）和批注的内容（"12"），但正文中删改内容（"的"改为"得"），删改前的"的"和删改后的"得"分开显示了，"得"只有在批注框中才能看到（显示为"插入"这一批注）。图2，采用嵌入显示修订的方式。可以直观的看到删除的痕迹和删改的痕迹，但是批注框嵌入了，批注内容无法直接看到，只有鼠标悬停的时候才能看到，很不方便。

  跟帖显示该问题**从 2024-05-16 一直到 2026-05-06 都有用户确认仍未解决**（"何止，现在已经2026年5月，WPS每次的更新从来没试图解决过"），社区给出的唯一"解法"是切换到第三种子模式"在批注框中显示修订者信息"（未获多人交叉确认真正解决）。这不是 Word/WPS 跨软件互操作问题，是 **WPS 原生渲染自己写入的修订+批注就已经有显示缺陷**，且恰好是"替换类修订（删+插）+ 批注同时锚定在附近文字"这个 Courtwork 管线的核心产出形态（对照 SPEC.md 里「批注在应用指令的同一遍构造」「9 条批注里 diff 后文本定位仅命中 4 条」这类叙述，Courtwork 的批注天然大量贴着修订走）。

- **✓ 重复印证**：[bbs.wps.cn/topic/37096](https://bbs.wps.cn/topic/37096)「文档修订的情况下删除和插入能同时显示在右侧位置吗」（官方社区，2024-07-04，本次直接抓取）。官方认证的"WPS产品体验官"给出同样的变通方案（切到"在批注框中显示修订者信息"子模式），同样不是"已修复"的确认——两个独立帖子指向同一个尚未真正解决的显示模式冲突，可信度互相加强。

- **✓ 批注显示 vs 批注丢失的区分**：[bbs.wps.cn/topic/50795](https://bbs.wps.cn/topic/50795)「修订模式不显示标注怎么办」（2025-02-27，本次直接抓取）根因是"显示标记"里批注勾选项被关闭，不是数据丢失。**这条提示了测试矩阵设计上的一个陷阱**：核验人看到"批注不见了"时，第一步应先检查显示过滤器状态，再判定是否为真正的数据丢失/渲染 bug，避免误报。

### 2.2 完全没有证据、必须标记「未能核实」的部分

- **⚠ `w:moveFrom`/`w:moveTo`（移动检测）**：搜索仅返回 OOXML 规范本身的定义和 WPS 手动拖动文字的操作教程，**没有找到任何关于 WPS 如何渲染"被识别为移动"的修订内容的报告**（无论正面负面）。Courtwork 当前 `RevisionInstructionSet` 的操作判别联合只有「替换/插入/删除/纯批注」四种，不产出移动检测，这条目前对生产管线不适用，但如果 W4/S4 未来引入移动检测（例如条款顺序调整识别为"移动"而非"删除+插入"），上线前必须先补测试，不能假设"应该和 Word 一样"。
- **⚠ `w:rPrChange`/`w:pPrChange`（格式变更修订）**：同样未找到任何 WPS 相关报告。Courtwork 当前管线不产出格式变更类修订（只改文字内容，不追踪"字号从三号改成四号"这类纯格式修订），暂不适用，列入 watch-list。
- **⚠ 表格内修订（`w:cellIns`/`w:cellDel`/`w:tcPrChange`）**：SPEC.md 已自陈"表格定位目前只做精确匹配"，且验收记录明确"复杂样式场景（多级编号、批注跨页、表格嵌套表格）未覆盖"。本次调研也没有找到 WPS 对这几个表格级修订元素的具体渲染报告。**但 Courtwork 现有 golden 测试确实产出了整行删除（`w:trPr/w:del`）**，这个元素本身也没有查到 WPS 专项报告——是测试矩阵 P0 级别的缺口。
- **✓（较老证据，需重新验证）ZIP 包规范问题**：见第 0 节结论 2；详细技术内容见第 5 节。

### 2.3 一条应予警惕的反面案例

- **⚠ 低可信来源**：`wpsapp1.com`（非官方域名，页面自称"WPS中文-WPS下载"但域名与金山官方 `wps.cn`/`wps.com` 均不同）发布的《WPS与Microsoft 365文件格式兼容性终极测试报告》一文（2026-07-12，本次直接抓取全文）在 FAQ 部分断言：「我们公司部分人用WPS，部分人用Office，协同编辑时修订和批注会丢失吗？**不会丢失**。修订和批注是OOXML标准的核心功能，WPS和Microsoft Office对此的支持都非常完善」。这个断言与本节 2.1 找到的官方社区实锤（近两年未解决的显示缺陷）**直接矛盾**，且该文通篇读起来像是套用固定模板生成的营销内容（"测试1/测试2/测试3/测试4"结构、具体版本号如"WPS Version 12"、"MS 365 Version 2408"均无法验证方法论真实性）。这正是任务纪律要求警惕的"应该可以"话术的典型样本，本报告明确不采信其结论，仅作为反例记录在案。

## 3. 批注（Comments）：`comments.xml`/`commentsExtended.xml`/`commentsIds.xml`/`w:commentRangeStart/End`/回复线程

- **✓ 官方 Comments API 完整抓取**（[solution.wps.cn/docs/client/api/Word/Comments.html](https://solution.wps.cn/docs/client/api/Word/Comments.html)，2026-07-14 直接抓取全文）——这是本次调研拿到的最有信息量的一手证据：
  - `GetComments()` 返回的批注对象结构只有 `{auth, content, date}` 三个字段，**没有 `parentId`、`paraIdParent`、`done` 之类的线程/状态字段**。
  - 提供 `Add()`/`DeleteComment()`/`DeleteCommentById()`/`ModifyComment()`/`GoToComment()`/`GetCommentReferenceText()` 等方法，批注定位方式是**字符偏移量 `{Start, End}` 或 `CommentId`**，不是基于 `paraId`。
  - **关键信号**：`ReplyComment()` 方法本身存在，但页面明确标注「**JSSDK: v1.1.14+、WebOffice v3.3.1+ 后已废弃**」（deprecated）。也就是说 WPS 自己的 Web 开放平台一度提供了"回复批注"能力，后来主动废弃，且没有看到替代的线程化 API 补上。
  - **推论（※，非直接证据）**：这强烈暗示 WPS（至少在其对外暴露的 Web API 模型里）不是围绕 Word 的 `w15` 命名空间扩展（`commentsExtended.xml` 的 `paraIdParent`/`done`）构建一等公民能力的。但**这只是 WPS Web 版开放平台 JSAPI 层面的证据**，不直接等价于"WPS 桌面客户端打开一份带 `commentsExtended.xml`/`commentsIds.xml` 的原生 Word 线程化批注文档时，底层解析器会怎么处理这些 part"——是静默丢弃、报错、还是保留数据但 UI 不可交互，本次调研没有找到直接证据，**必须标记 ⚠ 未能核实**，是测试矩阵必测项（见 6.5）。
- **⚠ 批注跨页/跨表格定位**：没有找到任何具体报告。SPEC.md 已自陈"批注跨页"是已知未覆盖缺口（样例合同刻意保持简单），本条完全依赖第 6 节实测。
- **✓ 批注显示的"过滤器陷阱"**：同 2.1 节 topic/50795，批注"不显示"经常是显示设置问题而非数据丢失，测试矩阵设计时需要显式排除这个假阳性来源。

## 4. 中文排版：`w:rFonts` 回退行为 / 仿宋黑体可得性 / 多级编号

- **✓ macOS 上"仿宋_GB2312"不是预装字体**，需要用户手动安装，且多篇 CSDN 用户报告（※ WebSearch 摘要转述，未逐篇直接抓取全文）反映即使手动安装了字体文件，也可能因为字体缓存/沙盒权限/命名冲突等原因不生效，需要重启 WPS 才能识别。其中一条具体转述（※）："Windows 上 Word 里的 GB2312 字体与排版，在 Mac 上显示为黑体，排版也错乱了"——**这条如果属实，直接命中 Courtwork 的字体规则风险点**：正文仿宋_GB2312 在缺字体的 macOS WPS 环境下如果回退成黑体，会导致"标题黑体、正文仿宋"这套刻意设计的视觉层级区分**完全失效**（标题和正文变成同一种字体），这不只是"不好看"，是丢失了产品设计要传达的排版语义。这条证据强度中等（转述而非原文截图），必须列入测试矩阵重点项（见 6.8）。
- **⚠ `w:rFonts` 的 `hint` 属性在 WPS 下的具体优先级处理**：没有找到官方文档或具体测试报告，只查到泛泛的"两者渲染引擎不同、字体判断逻辑和渲染顺序不同，可能导致同一字体不同版本时 glyph id 与度量不一致"这类描述（※，来自一篇知乎文章的 WebSearch 摘要，未直接抓取原文）。`ascii`/`eastAsia`/`hAnsi`/`hint` 四属性组合在 WPS 下的具体回退优先级规则，**完全没有可引用的实测依据**，必须实测。
- **✓（相关性中等）多级编号问题**：多篇 CSDN/百度经验文章描述 WPS 内编号跳级、编号与标题样式绑定断裂等现象，但这些报告绝大多数是"用户在 WPS 内部手动编辑导致编号错乱"（样式绑定被破坏），**不是"外部工具生成的 `numbering.xml` 被 WPS 解析异常"**这个更贴近 Courtwork 场景的问题。相关性判定为 ※ 中等，需要用最小 `numbering.xml` 样张单独实测，不能直接套用这些报告的结论。
- **✓ 现状核对**：经检查 `packages/output/SPEC.md`，当前管线"已知缺口"明确写着"复杂样式场景（**多级编号**、批注跨页、表格嵌套表格）未覆盖，样例合同刻意保持简单"——即 Courtwork 现有 golden 测试**目前不产出 `numbering.xml`**。这意味着第 6 节里的编号测试行在当前版本是 watch-list（暂不适用），而不是需要立刻堵住的 P0 缺口；但一旦 S4 起草管线（调研 07 提到的 `dolanmiu/docx` 路线，含 `chineseLegalSimplified` 等中文法律编号格式）真正启用编号功能，上线前必须先完成这一项。

## 5. 金山自有扩展 / WPS 往返是否破坏锚点结构（架构级关切）

这是任务里明确标注"对 Courtwork 尤其关键"的一节，因为直接关系到「无锚不落格」「原件永远只读」两条核心不变量。

### 5.1 命名空间层面

- **✓ 排除一个容易误导的巧合**：搜索命中的 `xmlns:wps="http://schemas.microsoft.com/office/word/2010/wordprocessingShape"` 命名空间，前缀恰好也叫 `wps`，但这是**微软自己在 Word 2010 定义的命名空间**（用于文字处理形状/文本框），和"金山 WPS"没有任何关系，纯属前缀命名巧合。本报告特别记录这一点，避免未来有人看到 docx 里的 `xmlns:wps` 就误判为"金山私有标记"。
- **⚠ 没有找到金山真正私有的命名空间前缀（如某些历史印象里的 `kso:` 之类）的确凿现网证据，也没有找到否定这类扩展存在的证据。这条明确标记「未能核实」**——这恰恰是最基础、最应该动手做但本报告条件不具备的实测：拿到任意一份经 WPS 保存过的 docx，`unzip` 解包后 `grep` 全部 `xmlns:` 声明，逐一核对是否在 ECMA-376/Word 已知扩展命名空间清单之外，是第 6 节测试矩阵的第一项动作（6.9）。

### 5.2 结构化绑定往返：官方实锤证据

- **✓✓ 第二个最重要发现**：[bbs.wps.cn/topic/82618](https://bbs.wps.cn/topic/82618)「WPS中自定义XML部件、XML映射窗格与XSD架构的应用实操及功能优化建议」（官方社区，作者为 Lv.2 潜力创作者/268 帖资深用户，2026-02-20 发帖，2026-07-14 本次直接抓取全文）。这是一篇详细的技术实操+建议帖，明确指出 WPS 对 **Custom XML Parts + XML 映射窗格 + 内容控件绑定**（这是 OOXML 标准机制，非 WPS 私有）的支持存在核心缺陷：

  > 3. XML映射联动效果无法持久化：**文档关闭后重新打开，已绑定的内容控件与XML节点的联动效果失效**，无法实现"一处修改，处处改变"……**这也是当前用户反馈最集中的核心痛点**。

  金山官方社区运营账号「WPS反馈小助理小雅」于 **2026-02-28** 回复："感谢您的详细建议……这边会提交反馈给技术团队小伙伴进行评估优化"——即**截至本报告撰写时（2026-07-14），这个问题官方确认收到但未确认修复**。

  **这条证据的意义需要精确表述**：Courtwork 的锚点机制（`packages/output/src/locate.ts` 的模糊文本/表格定位）**不是** OOXML 的 Custom XML Parts/内容控件绑定机制，两者是不同的技术实现。所以这条不能直接当作"Courtwork 锚点会被 WPS 破坏"的证据。但它是**同一类风险模式**——"文档内部的结构化绑定关系，在 WPS 关闭重开的往返过程中不保真"——的官方一手实锤，且是金山自己承认的、当前未解决的缺陷。合理的推论是：**不能假设"我们的锚点机制和 customXml 是不同技术所以不会有类似问题"，必须实测**，因为造成 customXml 绑定失效的根因（无论是 WPS 保存时重写 XML 结构、还是解析后再序列化丢失了某些细节）完全可能以类似方式影响任何依赖"文档内某处标记在往返后仍可被程序识别"的机制——这正是 `locate.ts` 模糊定位存在的全部意义。

### 5.3 更底层的风险：ZIP 包规范

- **✓（2021 年证据，需在 2026 重新验证）**：[cloud.tencent.com/developer/article/1773277](https://cloud.tencent.com/developer/article/1773277)（作者林德熙，原发布于其个人博客，本次通过腾讯云开发者社区镜像直接抓取全文，2026-07-14）记录了一个具体、可复现的技术问题：

  > 我遇到了有老师给我反馈说……使用 WPS 制作的文档打开失败，原因是 WPS 制作的一些文档不符合压缩文档规范。而 Office 的基于 ECMA 376 的文档，都是使用标准的 ZIP 压缩规范的文档，因此对于这些不符合压缩文档规范的文件，将会解析失败。

  文中附带了具体的 .NET 异常堆栈（`System.IO.InvalidDataException: End of Central Directory record could not be found`），并说明 **Microsoft Word 反而能"修复性"打开这类文件**（弹出"文档已损坏，是否修复"提示，点确定后仍可用），但严格遵循 ZIP/OPC 规范的解析库（.NET OpenXML SDK）会在解压阶段直接抛异常失败。作者给出的解决方法是用 `DotNetZip` 之类更宽容的库重新解压再压缩一遍，让文件"变得规范"。

  **这条证据比修订/批注渲染问题更值得警惕，因为失败模式更硬**：不是"显示得不好看"，而是**解压阶段直接失败，文件读不进来**。这与 Courtwork `packages/output` 用 `fflate` 解压（`loadDocx` 经 `@courtwork/reading-view/docx-security` 预检后解压）的架构直接相关——如果未来插件回传场景（task pane 客户端把用户在 WPS 里编辑过的文件传回本层）需要重新解析"WPS 保存过"的 docx，存在解压阶段直接失败的可能性。这条证据发布于 2021 年，`fflate` 的容错行为也可能与 .NET OpenXML SDK 不同，**该问题在 2026 年最新版 WPS 下是否依然存在，本报告未能核实，必须重新实测**，但鉴于其失败模式的严重性，应作为测试矩阵最高优先级的第一行（见 6.1）。

### 5.4 小结

对第 5 节的问题，诚实的回答是：**没有直接证据证明 Courtwork 的锚点会被 WPS 往返破坏，但也完全没有反面证据证明不会**。找到的两条相关证据（customXml 绑定失效、ZIP 包规范问题）都指向"结构化/程序化可识别的东西，经过 WPS 往返后不一定保真"这个方向，构成了合理的架构级警惕理由，但都不能直接套用为结论。**这是架构级关切，应该在实测矩阵里给最高优先级，而不是在本报告里下任何"应该没事"或"应该有事"的断言。**

## 7. WPS 环境可得性与 CI 可行性

- **✓ 桌面版获取门槛低**：Windows/macOS/Linux 个人版永久免费，无需激活，下载即用（[wps.com/download](https://www.wps.com/download/)、[linux.wps.cn](https://linux.wps.cn/) 均为官方域名，2026-07-14 确认可达）。人工测试没有获取障碍。
- **✓ Linux 容器化方案存在，但均为非官方社区项目**：
  - [linuxserver/docker-wps-office](https://github.com/linuxserver/docker-wps-office)：通过 noVNC 在容器里跑桌面版 GUI，本质是"远程桌面里人工点"，不是无头自动化。
  - [akkuman/headless-wps](https://github.com/akkuman/headless-wps)（34 star / 10 fork，2026-07-14 抓取仓库页面）：基于社区逆向的 `pywpsrpc`（[timxx/pywpsrpc](https://github.com/timxx/pywpsrpc)）RPC 协议驱动 WPS Linux 版实现无头自动化，作者同时提到另一个相近方案（wine 跑 msoffice2010）"不够稳定，会莫名其妙卡住"——暗示这条路线整体工程成熟度不足以支撑关键 CI 门禁，更适合"批量转格式/截图"这类粗粒度自动化，不适合"精确验证某个 UI 显示模式下的渲染细节"。
  - **两者均非金山官方提供，没有官方背书的 headless/CI 方案。**
- **✓ 官方云端半自动化路径（此前未被充分意识到的选项）**：[WPS WebOffice 开放平台](https://solution.wps.cn/)是金山官方提供的付费云端文档编辑 SaaS，本质是金山云端托管 WPS 内核并暴露 JS API（`ActiveDocument.Comments`/`Revisions` 等，见第 1、3 节）。理论上可以写脚本调用这套官方 API，在云端加载我们生成的 docx，用 `GetComments()`/`Revisions.Json()` 之类方法读出「WPS 实际解析出的修订/批注结构化数据」，与预期断言比对——**这能自动化验证"结构没丢"（有没有解析出预期数量的 `w:ins`/`w:del`/评论），但不能验证"渲染好不好看"（批注气泡位置、字体回退效果、显示模式冲突这类视觉问题）**。这是收费服务（见 `solution.wps.cn/docs/price/detail.html`），长期塞进 CI 需要评估调用成本，且需要额外开发脚本对接，本报告不建议作为首选，但值得作为「比完全人工快一点」的补充手段记录下来。
- **⚠ 没有找到任何金山官方或成熟第三方的"WPS 桌面客户端像素级渲染回归"CI 方案。**
- **结论**：
  - **渲染层面（好不好看、显示模式冲突、字体回退效果、批注气泡位置）：目前只能人工过一遍**，没有可靠的机器门替代方案。这条债的正确定性是「每次涉及 `packages/output` 修订/批注/字体渲染逻辑变更的发布前，人工过一遍 `verification-checklist.md`（扩展后的版本）」，不是「机器门」。
  - **结构层面（part 是否存在、计数是否对、解压是否成功）：`packages/output` 自己的 golden file 测试已经覆盖了"我们写出去的 XML 长什么样"，但没有覆盖"WPS 读进去之后再吐出来变成什么样"**——这个反向验证如果需要自动化，WPS WebOffice 开放平台的付费 API 是目前唯一看到的可行技术路径，但尚未验证工程可行性和成本，建议列为独立的小 spike，不要默认可行。

## 8. 回灌建议（明确写出哪些结论应回灌 `packages/output/SPEC.md`）

以下建议**均为调研输出，不是架构决定**，需要架构角色拍板后才能落地：

1. **`packages/output/verification-checklist.md` 应扩展**，在现有 7 项基础上补充本报告 6.1、6.3、6.5、6.6、6.8-6.10、6.14 对应的核验步骤（尤其 6.1 往返解析、6.3 修订+批注共存显示、6.6 定位鲁棒性三项，是本报告新增且证据支持最强的缺口）。
2. **`packages/output/SPEC.md` 的"已知缺口"应补一条**：当前"WPS 兼容"这条验收标准的状态是「零实测记录 + 两条间接官方实锤旁证 + 一条待重新验证的 2021 年 ZIP 规范风险」，不是"已验证但有已知小问题"。在有人真正用 WPS 跑过 verification-checklist.md（扩展版）之前，即使全部自动化测试绿灯，"WPS 兼容"这条硬性验收标准事实上处于**未验证**状态，建议在 SPEC.md 状态行显式反映这一点，而不是仅停留在当前"待人工核验"的表述（这次调研至少证明了"待核验"的清单本身还不够全）。
3. **建议把测试矩阵拆成两个优先级批次**：先跑 P0 里的「硬失败」类（6.1 往返解析、6.10 命名空间排查）——这两项成本最低、风险信号最强（分别对应架构级的"文件读不进来"和"未知私有标记"风险），值得作为第一批最先安排人工/半自动化验证的项，不必等全部 14 行测完才有产出。
4. **6.11、6.12 两个 watch-list 项应写入 S4（起草管线）相关文档的前置条件**，而不只是本报告内部记录：如果 S4 真的采用调研 07 推荐的 `dolanmiu/docx` + `chineseLegalSimplified` 编号方案，上线前必须先补测 6.11；如果 `RevisionInstructionSet` 未来扩展操作类型（移动检测、格式变更修订、表格级修订），上线前必须先补测 6.12——建议在 `packages/schemas` 或 `packages/output` 相关 TODO 里显式挂一条跨层提醒，避免功能上线后才发现 WPS 渲染问题。
5. **是否值得投入 WPS WebOffice 开放平台的付费 API 做"结构层面"半自动化回归**，建议作为独立的小 spike 评估（成本、稳定性、能覆盖 6.x 矩阵里的哪几行），不要默认可行也不要默认不可行——本报告只确认了"技术上存在这条路径"，没有验证工程可行性。
6. **建议团队内部知识库记录本报告 0.4 节的方法论发现**（WPS 相关中文搜索结果里非官方仿冒域名/疑似 AI 生成内容的比例很高）——这对任何未来需要独立调研 WPS 相关问题的人都有参考价值，避免重复踩坑或误采信不可靠来源。
