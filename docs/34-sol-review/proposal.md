# 探索性提案：从“组件证明稿”推导成审阅工作台

## 一句话方向

把右栏从“多个 artifact 的纵向展览”改为**证据驱动的主从审阅台**：列表负责扫读，详情负责判断，原文负责确定；对话只保留任务上下文，不与结构化产出争主视觉。

示例见 [exemplar-s3.html](./exemplar-s3.html)。它不是产品代码，只是 B 阶段前的布局论证。

## 1. 三栏职责更硬

- 左栏：案件与阶段，稳定导航，不展示产出细节。
- 中栏：任务日志与 AI 解释，只回答“为什么现在做这件事、系统做到了哪一步”。已完成过程折叠。
- 右栏：唯一工作面。顶部在“风险审阅 / 修订核对”间切换；默认不纵向拼接两种任务。

这使“生成”和“确定”不仅靠字族/底纹区隔，也靠空间职责区隔：AI 在中栏说话，证据和门禁在右栏落地。

## 2. 右栏改为主从双区

右栏内部水平分成：

- 左侧 360–420px 风险清单，30px 行高，负责扫读与队列；
- 右侧弹性详情，显示选中风险的解释、全部依据、修订影响与门禁。

用户切换风险时，详情原位更新；不再把展开详情插入列表导致行位移，也不需要在长列表与文书预览之间来回滚动。

## 3. 分层确认从规则变成可见进度

高危条目详情头固定显示：

```text
逐条确认 · 2/2 依据已展开 · 已审阅 48 秒
```

未展开全部依据前，“确认”禁用并解释原因；达到条件后 0ms 解锁。批量栏只在确有合法条目时出现，明确列出范围和排除项。确认质量埋点由此获得用户可理解的产品表面，而不是纯后台监控。

## 4. 法理之线只在一个地方完整发光

dense row 保留 2px 实色线与 100ms 灰底 hover，不做 40px 折光。选中详情卡才实现完整折光 + 右缘 citation。这既解决 30px 行容不下 40px 光段的问题，也让唯一亮点集中在“正在作出法律判断”的时刻。

## 5. 证据底纹改为“证据台”而非小灰块

详情中的每条依据统一为三行：

1. A/B/C 角标 + 规范 citation；
2. 文书名 + 定位动作；
3. 原文引语（冷灰底、中文正文 CJK 字体；日期/编号 mono）。

短引文不和长引文争同一行，35–55 岁用户可以顺着“法条—文书—原文”完成判断。

## 6. 冷静不等于发灰

保持浅色、藏青黑、4–6px 圆角、零阴影，但把边界分成两档：

- 面板骨架用可见结构线；
- 列表内部用 hairline。

信任来自稳定对齐和状态证据，不来自把所有分隔线都压到接近不可见。

## 7. 五年感来自“历史”

成熟度应补在状态历史，而不是再加装饰：

- 谁在何时确认；
- 哪次卷宗增量使确认失效；
- 为什么驳回；
- 当前版本与上次定稿差异；
- 退出后回来仍能定位到上次审阅项。

这些都应该由事件流提供，UI 只渲染，不把业务逻辑藏进壳。

## B 阶段实现顺序建议

1. 先做事件回放器和 8 个关键 UI 状态，不先铺页面。
2. 做右栏 risk master-detail 纵切；用真 fixture 验证长中文、7 条依据、无锚点错误态。
3. 做门禁状态机视觉映射与键盘路径。
4. 做 S1 时间线；图谱推迟到证据抽屉与命中区规范完成后。
5. 最后接法理线折光、长任务反馈和图标等 polish。
