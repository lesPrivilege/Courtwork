# 调研报告：组件与图标库选型（办公场景简约视觉组件提前建库）

调研日期：2026-07-10。范围：图标库选型、法律域图标缺口、模型编写 SVG 的工程规范、组件层（headless + 皮肤 vs 纯手写）策略。前提沿用既有架构裁决：**产品内素材一律 SVG-as-code（模型编写、吃 tokens 变量），image-gen 不入产品 UI**，本报告不重开该问题。

依据读过的仓库文档：`docs/32-设计语言包/tokens.json`（color/typography/radius/motion 全表）、`docs/32-设计语言包/principles.md`（12 条组件设计原则）、`docs/32-设计语言包/icon.md`（应用图标 16px 缩小可辨自查法先例）、`docs/35-设计素材-GeminiUX提案.md`（de-slop 12 条清单，第 3 条即图标着色纪律）、`apps/desktop/src/workbench/Icon.tsx`（现状：12 个手写线框图标，viewBox 24、stroke="currentColor"、strokeWidth 1.35，已经是 SVG-as-code 的正确落地范式，只是量少）。

## 一、图标库选型（重点）

### 1.1 四家横向对比

| 维度 | Lucide | Tabler Icons | Phosphor | Remix Icon |
|---|---|---|---|---|
| 图标规模 | 1,747（v1.0 后精简，移除品牌 logo） | 6,146（Outline 5,093 + Filled 1,053） | 9,000+ | 数千（line/fill 双风格） |
| 网格与描边可调性 | 24×24 网格，`stroke-width` 连续可调（默认 2，可设 1.35），React 端 `LucideProvider` 或 CSS `.lucide{stroke-width}` 全局统一 | 24×24 网格，`stroke`/`strokeWidth` prop 连续可调（官方文档建议编辑型 UI 用 1.5），`IconContext` 可全局设默认值 | 256×256 网格，**离散 6 档 weight**（thin/light/regular/bold/fill/duotone），无法连续设到 1.35px 这种精确值，只能挑最接近的 weight 近似 | 图标多为**填充路径拼出的线形**而非真实描边路径，`stroke-width` 在 CSS 层面不生效，无法统一到目标描边值 |
| License | ISC（`lucide.dev/license` 实测确认），无需产品内可见署名，仅需保留源码内版权声明 | MIT（`tabler-icons` GitHub README 确认） | MIT | Remix Icon License v1.0（2026 年 1 月更新过），商用免费、鼓励但不强制署名 |
| Tree-shaking / 包体积 | ESM 原生 tree-shakable，单图标约 1KB；v1.0 移除 UMD 构建后 `lucide-react` 包体积从 11.4MB 降到约 1MB（gzip），30M+ 周下载量 | 支持，`@tabler/icons-react` 单独按需导入 | 支持，但主入口直接 import 会因 9,000+ 模块拖慢开发期编译，需按路径导入子模块或开 `optimizePackageImports` | 支持但生态偏 webfont/雪碧图，纯 SVG 组件化路径不如前两者成熟 |
| 办公/法律隐喻覆盖度 | 覆盖通用 UI 语义（文档、搜索、归档、锁等）齐全，但法律专属隐喻（盖章、卷宗、法理之线类）为空——这是意料之中，任何通用库都不会有 | 图标规模最大，通用文档/办公类覆盖度略优于 Lucide，法律专属隐喻同样为空 | 覆盖度与 Lucide 相当，法律专属隐喻为空 | 覆盖度尚可，但风格不统一（部分图标视觉密度偏高），法律专属隐喻为空 |
| React 集成质量 | 官方 `lucide-react`，TypeScript 全量类型，v1.0 新增跨框架 context provider、`aria-hidden` 默认 true（装饰性图标可及性更省心） | 官方 `@tabler/icons-react`，类型完整，多框架支持广（Vue/Angular/Svelte/Compose） | 官方 `@phosphor-icons/react`，Context API 完整，但 SSR/RSC 场景需要切 `/ssr` 子路径，心智负担略高 | 社区维护的 React 封装质量参差，官方主推 webfont/雪碧图路线 |
| 2026 维护状态 | 活跃：v1.0 于近期发布（移除品牌图标、精简构建、新增多框架 context provider），GitHub 23.4k star，截至 2026-07-09 仍有提交；shadcn/ui 在 2026 年默认图标库仍是 Lucide | 活跃，赞助制维护，release 节奏稳定 | 活跃，但因发布 `@phosphor-icons/react` 替代旧版 `phosphor-react`，生态有一次迁移断层 | 相对不活跃，风格更新慢于前三家 |

### 1.2 裁定：首选 Lucide，备选 Tabler Icons

**首选 Lucide**：
- 唯一能把 `stroke-width` 精确锁到 `1.25–1.5px` 且提供全局 context provider（`LucideProvider` / CSS `.lucide` 类）一次性统一整站描边的库——这正是 principles.md de-slop 第 3 条的硬性要求。Phosphor 的离散 weight 与 Remix 的填充路径都做不到这一点。
- ISC 协议比 MIT 更宽松（连署名要求都限定在源码层面，不涉及产品 UI），法务风险最低。
- v1.0 精简后包体积对 Tauri WebView 友好（gzip 后约 1MB 且 tree-shakable，实际引入的图标数远小于全量）。
- shadcn/ui 生态默认选择，意味着后续如果引入 Base UI/Radix 生态组件，视觉语言天然对齐，不需要二次协调描边风格。

**备选 Tabler Icons**：图标规模是 Lucide 的 3.5 倍，办公类隐喻（文件柜、审批流、签核）覆盖更广，同样支持连续描边调节和 MIT 协议。当 Lucide 的 1,747 个图标出现明显缺口、且缺口不属于"法律专属隐喻"（即通用办公语义但 Lucide 恰好没有）时，允许单个图标降级取用 Tabler，但不建议整站混用两套视觉基因——**混用前必须验证两家在 1.35px 描边下的视觉密度是否接近**，否则会出现"同一屏两种粗细的线"这种新的 de-slop 违规。

**不采用 Phosphor 与 Remix Icon** 作为主力：Phosphor 的离散 weight 机制与本项目"描边统一到某个精确 px 值"的硬性要求结构性冲突；Remix Icon 的填充路径风格无法用 CSS 描边变量统一控制，且当前维护活跃度最低。

## 二、领域图标缺口清单（模型编写 SVG 自绘清单）

通用库（Lucide/Tabler）大概率没有、需要按 `docs/32/icon.md` 的方法自行设计的法律办公专属隐喻，共 17 个，按使用场景分组：

| 分组 | 图标 | 说明 |
|---|---|---|
| 卷宗与文书 | `dossier`（卷宗/案卷袋，区别于通用 folder） | 通用库的 folder 图标承载不了"卷宗"这个特定行政物件的心智，需要侧边带装订线的变体 |
| | `case-file-tab`（分类文书标签页） | 五工作面 Tab 切换需要一组统一母题的小图标（时间线/图谱/矩阵/修订/画布），不能各自取用通用库不同来源的图标拼凑 |
| 核验与留痕 | `seal-check`（盖章核验，几何抽象，禁用天平/法槌等陈词滥调） | 呼应 `tier` 语义色（A/B/C 信源分级）旁的核验状态图形 |
| | `provenance-link`（溯源跳转/引用锚点） | principles.md §6 "溯源必达"要求任何数据点都有可点击溯源入口，需要统一的锚点小图标 |
| | `source-tier-mark`（信源等级角标图形容器，非文字部分） | 与 `tokens.component.tierBadge` 配合，图标只提供角标的几何外框 |
| 门禁三态 | `gate-pending` / `gate-confirmed` / `gate-rejected`（确认门禁三态） | 与 `color.semantic.gate.*` 一一对应，需与徽章文字解耦的独立图形语汇（不能直接偷用 Lucide 的 check/x，那是通用语义，门禁三态需要一眼可辨且和"处置"心智绑定的专属形态） |
| 定稿与冻结 | `compile-freeze`（编译为 Word 冻结仪式） | principles.md §8 "冻结仪式"的主按钮图标，需要传达"单向、不可逆" |
| | `archive-lock`（定稿后只读归档态） | 冻结后画布顶部归档徽章配图 |
| 修订与矛盾 | `revision-track`（docx 修订痕迹预览入口） | 区别于通用"编辑"图标，需要暗示"红删蓝增"心智但图标本体仍单色 |
| | `conflict-mark`（时间线矛盾点标记） | 时间线工作面的矛盾行专属标记，通用库没有"法律事实冲突"这个概念 |
| 图谱与摄取 | `entity-graph`（关系图谱工作面 Tab） | 通用库的 network/graph 图标可用但需绑定到"实体关系"这一具体语境，建议自绘以统一线宽 |
| | `ingest-pipeline`（摄取/OCR 进度） | 长任务微反馈组件（骨架呼吸/边框微光）旁的状态图标，需要表达"文书正在被结构化" |
| 生成与确定区隔 | `ai-generated-mark`（AI 解释性文字通道标记） | principles.md §11 两条通道视觉区隔的图形侧，配合 sans + 冷灰底 |
| | `verified-mark`（工具核验结果通道标记） | 配合 mono + 冷灰蓝底，与 `ai-generated-mark` 成对出现，避免用户靠记颜色/字体区分 |
| 分层确认 | `batch-eligible-mark`（可批量确认范围标记） | principles.md §10 分层确认——"不入批量·逐条确认"徽章需要一个专属小图标 |
| 阶段与续行 | `stage-continue`（开启新阶段续行） | 状态条红色态"开启新阶段继续"按钮的专属图标 |

## 三、模型编写 SVG 的工程规范

业界最成熟的可参照实践就是 **Lucide 自己的贡献者规范**（`lucide.dev/contribute/icon-design-guide`）——它本质上是一套给"设计师+自动化工具"共同遵守的机器可校验规则集，与"让 LLM 产出长期可维护 SVG"的诉求高度同构，可以直接移植：

### 3.1 网格与几何约定（移植自 Lucide 14 条设计规则，收窄到本项目描边值）

- **画布**：24×24 viewBox，与现有 `Icon.tsx` 一致，不新起网格。
- **留白**：画布内至少 1px padding。
- **描边**：`stroke-width` 固定 **1.35**（落在 de-slop 第 3 条 1.25–1.5px 区间中值，与 `Icon.tsx` 现状对齐，不必每个图标各选一个值制造不一致）。
- **端点/连接**：`stroke-linecap="round"`、`stroke-linejoin="round"`，描边居中（centered stroke，非 inside/outside）。
- **圆角**：形状（矩形等）≥8px 用 2px 圆角，<8px 用 1px 圆角。
- **元素间距**：不同图形元素之间至少 2px 间隙，避免糊成一团。
- **光学体量**：与同网格的 circle/square 基准图形做模糊对比，视觉重量不应明显偏轻或偏重（这条是人审目测项，写进验收记录而非自动化脚本）。
- **视觉居中**：按重心而非几何中心对齐。

### 3.2 代码层硬约束（可写成 CI 脚本机械校验的部分）

- `viewBox="0 0 24 24"`，`fill="none"`，`stroke="currentColor"`，`stroke-width="1.35"`（或不写死，靠外层 `<Icon>` 组件统一注入，图标源文件本身不带 `stroke-width` 属性以避免和全局设置打架——参照 Lucide 允许元素列表的思路，图标文件只描述几何，样式属性交给消费端容器）。
- **禁止内联色值**：图标源文件里不允许出现任何 `#RRGGBB`/`rgb()`/命名色，颜色一律由外层 `currentColor` 继承 tokens（这与 principles.md §5 零技术概念暴露、§12 命名纪律的精神一致：图标不携带自己的配色决策）。
- **禁止 fill**（除非明确设计为面性 tint 徽章场景，本项目线框图标默认 `fill="none"`）。
- **元素白名单**（直接抄 Lucide 规则）：只允许 `<path d>`、`<line x1 x2 y1 y2>`、`<polygon points>`、`<polyline points>`、`<circle cx cy r>`、`<ellipse cx cy rx ry>`、`<rect x y width height rx>`；禁止 `<image>`、`<use>`（ID 唯一性在多图标同页面嵌入时会冲突）、`<filter>`、`<linearGradient>`、`transform` 属性、内联 `<style>`。
- **path 优化**：用 svgo 跑一遍再入库，配置建议基于 `preset-default` 收窄：
  ```js
  // svgo.config.mjs
  export default {
    multipass: true,
    plugins: [
      { name: 'preset-default', params: { overrides: {
        removeViewBox: false,      // 保留 viewBox，响应式缩放依赖它
        cleanupIds: false,         // 组件级复用场景避免 ID 冲突时误删
        collapseGroups: false,     // 部分图标用 <g> 分层，避免破坏可读性
      }}},
      'removeDimensions',          // 去掉硬编码 width/height，靠外层 CSS 控制
    ],
    floatPrecision: 2,             // 图标级足够，避免路径数据膨胀
  };
  ```
- **命名规范**：直接移植 Lucide 的第 3 条命名原则——"图标应按其描绘的形状命名，而非按用途命名"（如 `floppy-disk` 而非 `save`）。这与 `principles.md` §12 命名纪律（组件/token 命名领域无关，法律语义只存在于文案与数据）天然吻合：即使是"盖章核验"这种法律专属隐喻，文件名/组件名也应描述**图形本身**（如 `seal-check` 而非 `case-verification-icon`），法律含义留给使用它的按钮文案和数据。kebab-case，不含数字（除非数字本身是图形内容）。

### 3.3 16px 缩小可辨自查法（复用 `docs/32/icon.md` 先例，从应用图标扩展到全部 UI 图标）

- 方法：每个新图标 SVG 直接渲染至 16×16 与 24×24 两档位图人工截屏观察。
- 通过标准：16px 档主体轮廓仍可辨识、与工作台内相邻图标（尤其五工作面 Tab 那组）不混淆；24px 档细节可数。
- 若 16px 端点发虚，参照 `icon.md` 的降级预案思路——简化图标内部细节（合并线条、减少元素数），不改变母题，无需回架构重议。
- 验收记录格式建议沿用 `apps/desktop/SPEC.md` 已有的销账/截图索引惯例：每个新图标一行，附 16px/24px 截图对照。

### 3.4 CI 可查项清单（供后续写 `tools/verify-icons.mjs` 之类的自研小脚本）

1. viewBox 必须是 `0 0 24 24`。
2. 不含 `fill="#..."` / `stroke="#..."` 等硬编码色值（正则扫描十六进制色值 + 常见命名色）。
3. 不含白名单外的元素标签（`image|use|filter|linearGradient|radialGradient|style`）。
4. 不含 `transform` 属性。
5. 文件名满足 kebab-case 且已登记进图标清单（可仿 Lucide 的 JSON metadata descriptor，给每个自绘图标配一个 `{name, tags, addedInSpec}` 元数据条目，便于后续检索复用，避免重复造近似图标）。

说明：通用 SVG lint 生态目前没有活跃维护的现成 eslint 插件可直接套用（`eslint-plugin-react-svg` 已 5 年未更新），建议自写轻量 Node 校验脚本而非引入不活跃依赖——这与 CLAUDE.md "难看但自有的代码不允许为省事绕过" 的工程纪律精神一致（虽然图标校验不在文件列出的五个必须自研加固节点之内，但同一原则适用于这类小型确定性校验）。

## 四、组件层策略

### 4.1 Headless 库 2026 现状（截至 2026-07）

| 维度 | Radix UI | React Aria (Adobe) | Base UI (MUI) |
|---|---|---|---|
| 组件覆盖 | 30+ 组件，asChild 组合模式 | 40+ 交互模式，Hooks 而非组件（`useButton`/`useDialog`），需自行拼组件，代码量更大但渲染输出控制力最强 | 35 个组件，v1.0 于 2025 年底稳定，2026 年内持续扩展（OTPField、Drawer 等陆续转正式） |
| 可及性 | 完整键盘导航/ARIA/焦点管理，RTL 需手动处理 | 业界公认最严格，WAI-ARIA 合规性最高，适合政府/强合规场景 | 完整键盘导航/ARIA，v1.0 起原生 RTL 支持（本项目中文场景非刚需） |
| 维护状态 | 被 WorkOS 收购后部分组件更新放缓 | Adobe 长期投入，稳定但重 | MUI 全职工程团队维护，2026 年是当前发布节奏最快的一家 |
| API 心智 | `asChild` 组合 | 组合 Hooks，样式与逻辑分离彻底但接线工作量大 | render prop 模式，显式声明渲染元素，被认为是吸收 Radix 教训后的迭代 |
| 生态位置 | shadcn/ui 2026 年 7 月起默认迁移到 Base UI，但未废弃 Radix | 独立生态，常见于对 WCAG 有合同级要求的产品 | 2026-07 起是 shadcn/ui 新项目默认 headless 层 |

### 4.2 判断

Courtwork 当前组件面需求集中在**对话框（编译为 Word 二次确认弹层）、下拉菜单（场景选择）、Tooltip（KBD 提示/溯源引语悬浮）**，表格已由 AG Grid 拍板解决，不重复造轮子。可及性要求是"focus/键盘已是验收项"（工程纪律层面），尚不是政府/强合规场景下的合同级 WAI-ARIA 义务，因此不需要 React Aria 这种最重的方案；Radix 因被收购后更新放缓，且 2026-07 时点生态权重已明显转向 Base UI（shadcn/ui 官方默认迁移）。

**判据结论**：引入 **Base UI** 作为 headless 基础层，只取当前缺的几个 primitive（Dialog / Popover / Menu / Tooltip），皮肤完全自绘、吃 `tokens.json` 变量。注意 Base UI 组件默认可能带自己的进入/退出过渡类，需要按 `principles.md` §3/§3a/§3b（数据区绝对静止、动效八禁、0ms 硬切等硬性条款）逐一覆盖或禁用其默认 transition，不能假设 headless 库的默认动效符合本项目纪律。

存量的简单展示型组件（如 `Icon.tsx`、`SplitView.tsx`、`Panels.tsx`）不需要推倒重写；新增的、涉及焦点陷阱/键盘循环/ARIA modal 等边角案例多的交互组件（对话框类）优先走 Base UI primitive，手写这类组件的边角案例成本高、易漏（如 Esc 关闭、Tab 循环、`aria-modal`、焦点归还），是明显的可及性性价比洼地。

### 4.3 是否抽 `packages/ui`

**判据**：抽共享包的前提是至少有第二个消费方去验证接口边界是否正确，否则是过早抽象。当前 `apps/desktop` 是唯一的 UI 壳（`docs/10` 里 `apps/desktop` 明确"B 阶段 UI 任命"，且尚无第二个壳的工单或时间表）。

**结论**：现阶段**不抽 `packages/ui`**。图标与 headless+皮肤组合先在 `apps/desktop/src` 内部沉淀出清晰边界（例如集中到一个 `design-system/` 或 `workbench/icons/` 子目录），等真正出现第二个消费方（如未来的 Web 版本或其他任命的 UI 壳）时，再把已经被两个使用方验证过的接口抽到 `packages/ui`，避免为假设的未来买单，也避免在 `packages/` 下过早新增一层未经消费验证的依赖。

---

## 五、裁定建议汇总

1. **图标库**：首选 **Lucide**（ISC 协议、`stroke-width` 可连续锁定到 1.35px、v1.0 后包体积对 Tauri 友好、tree-shakable、shadcn 生态默认），备选 **Tabler Icons**（图标规模更大，用于填补 Lucide 缺口，但需人工核对 1.35px 描边下与 Lucide 视觉密度一致后才可混用，不整站替换）。不采用 Phosphor（离散 weight 无法连续调描边）与 Remix Icon（填充路径无法用 CSS 统一描边）。
2. **领域图标缺口**：17 个（卷宗/核验/门禁三态/定稿冻结/修订矛盾/图谱摄取/生成核验区隔/分层确认/阶段续行/场景入口九组），均走模型编写 SVG + 人审，不外购、不用 image-gen。
3. **SVG 工程规范要点**：24×24 网格、1.35px 描边、圆角/间距/端点/光学体量对齐移植版 Lucide 14 条规则；代码层禁内联色值、禁 fill、元素白名单、svgo（`preset-default` + `removeViewBox:false` + `floatPrecision:2` + multipass）；命名按"描绘形状而非法律用途"与本项目命名纪律天然吻合；16px/24px 缩小可辨自查法从应用图标扩展到全部 UI 图标，验收记录仿 SPEC.md 截图索引惯例；CI 校验走自写轻量脚本（通用 eslint-svg 插件已停止维护，不引入）。
4. **组件策略一句话**：headless 基础层引入 **Base UI**（MUI 维护、2026-07 shadcn 默认、RTL 非本项目刚需但不减分）只取 Dialog/Popover/Menu/Tooltip，皮肤自绘吃 tokens，其默认动效需按本项目动效八禁逐一覆盖；存量简单组件不推倒重写；`packages/ui` 暂不抽，等第二个 UI 壳出现再议。
5. **分期**：
   - **分期 0（当前 polish 工单内）**：引入 Lucide 替换/补充 `Icon.tsx` 通用图标，统一 context provider 设 `strokeWidth=1.35`；落地 svgo 配置与 icon 校验脚本。
   - **分期 1（下一轮设计/polish 迭代）**：按缺口清单产出 17 个法律域自绘图标（模型编写 + 人审），进图标目录，纳入同一套 lint 校验，16px 自查记录写入验收记录。
   - **分期 2（触发式，非固定时间）**：当出现新的强焦点管理交互需求（对话框/下拉/tooltip）时引入 Base UI 对应 primitive，按需取用，不做一次性全量替换。
   - **分期 3（第二个 UI 壳出现时）**：评估把 design-system（图标 + headless 皮肤组件）抽取为 `packages/ui`。
