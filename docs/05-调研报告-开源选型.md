## 1. OCR 与文档解析（重点节点）

### 1.1 百度新 OCR 确认：Unlimited-OCR（用户所指"unlimited ocr"）

已通过 GitHub 官方仓库确认，**不是 PaddleOCR-VL，而是一个独立的新项目**：

- 仓库：`github.com/baidu/Unlimited-OCR`（注意组织是 `baidu`，不是 `PaddlePaddle`）
- 全称："Unlimited OCR Works: Welcome the Era of One-shot Long-horizon Parsing"
- 发布时间：2026/06/22（代码）、2026/06/23（arXiv 论文 2606.23050、ModelScope）、2026/06/24（HuggingFace Spaces demo）—— 是一个**上个月刚发布的全新模型**
- **License：MIT**（GitHub 仓库首页明确标注 "MIT license"，商用无风险）
- Star：约 6.7k，Fork 550，Watch 30（查证时点，无正式 Release 版本号，仍以 main 分支迭代）
- 核心能力：**3B 参数，BF16 精度（约 6GB 显存/磁盘占用）**，上下文窗口 32,768 tokens，主打"一次性解析长文档/多页 PDF"（不用逐页调用），支持 Transformers 推理和 SGLang 高并发服务化部署，输出结构化文本+表格+版面
- 技术脉络：官方 README 明确致谢 **DeepSeek-OCR、DeepSeek-OCR-2、PaddleOCR**，说明是站在这几个项目基础上做"长文档一次性解析"的改进，而非 PaddleOCR 官方新模型
- 与 PaddleOCR-VL 的关系：两者是**平行的、都来自百度体系但不同团队/仓库**的产品。PaddleOCR-VL 侧重版面+文字+表格+公式的高精度结构化解析（OmniDocBench 榜首），Unlimited-OCR 侧重长文档单次推理的效率与上下文压缩
- **评价**：非常新（发布仅约2周），生产级验证不足，尚无 GitHub Release、无中文法律场景（印章/手写）专项评测数据。建议**先跟踪观察 1-2 个月，不作为 MVP 首发依赖**，但因 MIT 协议+3B 轻量，值得作为第二引擎做技术预研和交叉验证。

### 1.3 自研 vs 用开源判断

- **可直接用**：印刷体中文文字识别、常规版面检测、表格结构识别 —— PaddleOCR-VL/PP-StructureV3 已足够工业级。
- **必须自研加固**：
  1. **印章场景**：即使 PP-OCRv4_seal 是目前唯一原生方案，法律卷宗中印章与文字重叠、多枚印章叠加、红色印泥褪色扫描件等场景仍需要针对性训练集微调 + DeepSeek-OCR 交叉验证 + 规则兜底（例如印章区域先分割再单独识别）。
  2. **手写批注**：无论 PaddleOCR、DeepSeek-OCR 还是 MinerU，官方均未给出手写体在法律文书场景（法官批示、代理词手写修改、签字捺印）下的量化指标，DeepSeek-OCR 官方甚至明确当前不支持完整手写文本识别。这是**全行业开源空白**，必须自建标注数据 + 微调或采购专项手写 OCR 能力（如结合科大讯飞/腾讯优图等商业 API 做兜底对比）。
  3. **卷宗级别的多页拼接、装订顺序错乱、扫描倾斜校正**：需要在开源版面分析基础上叠加自研的"卷宗结构还原"规则引擎。

**自研 vs 开源判断（本节点结论）**：
- NER/句法分析的**技术底座**（分词、实体抽取骨架、依存句法）可以直接用 HanLP/LTP，不需要重新造轮子。
- 但"法律域实体消歧"——例如同一自然人在起诉状写"张某"、在判决书写"张三（身份证号 XXX）"、在委托书写"张三丰"曾用名、公司"某某科技有限公司"与其简称、股东与法定代表人的关系推断——**这是行业公认难题，目前没有可直接商用的开源方案**，必须视为核心自研壁垒：
  1. 用 HanLP/LTP 做实体抽取的原材料层；
  2. 参考 LightRAG/GraphRAG/iText2KG 的增量图谱构建与实体消歧设计模式；
  3. 自建法律实体消歧规则引擎+小模型（结合身份证号、统一社会信用代码等强特征做确定性匹配，模糊匹配用向量相似度兜底）；
  4. 这是本产品最应该投入自研资源、构筑技术壁垒的节点之一。

## 6. docx 修订/批注输出

法律场景对"生成带 Word 原生修订痕迹的 docx"要求刚性（律师/法务的审阅工作流强依赖 track changes），这是**开源工具能力参差不齐、必须重点验证的节点**。

| 项目 | GitHub | License | 活跃度 | 能力与局限 | 匹配度 |
|---|---|---|---|---|---|
| **python-docx** | python-openxml/python-docx | MIT | 中，长期基础库 | **原生不支持修订痕迹（track changes）和高版本批注（解决/回复）**，只能操作文档 body，写入修订需要手写底层 OOXML，工程成本高 | 仅作为基础文档读写库，不能满足"生成带修订痕迹输出"的核心需求 |
| **Python-Redlines** | JSv4/Python-Redlines | 待核实（多次搜索未拿到明确 LICENSE 文本，作者其他项目历史上用 MIT，**使用前需直接查看仓库 LICENSE 文件确认**） | 中，star 约 100+，仍有 issue/PR 活跃，2026 年迁移到新默认引擎 **DocxodusEngine**（.NET 8 对 Open-XML-PowerTools 的现代化 fork） | **核心能力对口**：对比两个 Word 文档，生成第三个包含所有插入/删除（可选移动检测）的**原生 Word 修订痕迹文档**，含 LCS 表格行匹配、格式变更检测 | **首选**：目前开源生态里唯一"开箱即用生成原生 track changes docx"的活跃项目，直接命中本产品"输出带修订痕迹文书"的核心需求 |
| **docx4j** | plutext/docx4j | **Apache-2.0** | 中，Java 生态，长期维护（最新已用 Jakarta XML Binding 3.0） | JAXB 级别对 OOXML 完全可编程控制，可精细操作修订/批注节点，但需要自己实现"diff 生成修订"的业务逻辑（不像 Python-Redlines 开箱即用） | **备选**：当 Python-Redlines 能力不够、需要更细粒度控制（如批注定位到特定字符范围、自定义修订作者/颜色）时的底层工具 |
| **redlines（houfu）** | houfu/redlines | 需核实（PyPI 有独立发布，多个输出格式：JSON/Markdown/HTML/终端富文本） | 中 | 仅做**文本级 diff**（增删标记），**不生成 docx 原生修订格式**，适合"AI 编辑建议展示层"而非最终 Word 交付物 | 备选：可用于 UI 层展示 AI 修改建议（类似 diff 视图），最终落盘 Word 时仍需 Python-Redlines/docx4j |

**License 与技术风险提示**：
- **python-docx 局限明确**：不能满足修订痕迹需求，只能作为基础读写库。
- **WPS 兼容性**：所有上述工具均基于 OOXML 标准生成文件，理论上 WPS 应可读取，但**中国用户主力办公软件是 WPS 而非 Word**，微软 OOXML 标准之外的 WPS 私有扩展（如部分版式、字体回退、修订标记的显示细节）需要**自行用 WPS 实测验证**，这是任何官方文档都不会覆盖的风险点，必须自研测试用例矩阵。

**判断**：**必须自研加固**——Python-Redlines/docx4j 提供了"生成原生修订痕迹"的技术底座，但（1）WPS 兼容性验证、（2）法律文书复杂样式（多级编号、表格内修订、批注跨页定位）、（3）批注的"解决/回复"协作语义，都需要在开源库上大量补丁和测试，不能假设开箱即用。

```
│  主引擎：PaddleOCR-VL + PP-StructureV3（含印章识别）        │
│  交叉验证：DeepSeek-OCR（印章遮挡场景）                     │
│  长文档效率通道：跟踪 Unlimited-OCR（暂不做主依赖）          │
│  自研加固层：手写批注专项微调 + 卷宗结构还原规则引擎         │
├─────────────────────────────────────────────────────────┤
│ 实体对齐/知识图谱：                                        │
│  底座：HanLP（NER/句法）                                   │
│  图谱：LightRAG（增量构建，双层架构）                       │
│  自研核心壁垒：法律域实体消歧规则+小模型（强特征优先，向量兜底）│
├─────────────────────────────────────────────────────────┤
│ RAG 底座：Qdrant + BGE-M3 + bge-reranker-v2-m3            │
├─────────────────────────────────────────────────────────┤
│ docx 修订输出：Python-Redlines（主）+ docx4j（细粒度兜底）  │
│  自研加固层：WPS 兼容性测试矩阵 + 法律文书复杂样式补丁      │
├─────────────────────────────────────────────────────────┤

└─────────────────────────────────────────────────────────┘
```

### 必须自研加固的节点（技术壁垒清单）

1. **OCR 的印章/手写场景**：全行业开源空白，需自建标注数据+微调+多引擎交叉验证兜底。
2. **跨文档法律实体对齐**：无现成开源方案，是本产品最应投入资源构筑壁垒的节点。
3. **docx 修订的 WPS 兼容性与复杂法律文书样式**：开源库解决了"生成原生修订痕迹"的技术底座问题，但兼容性和复杂样式必须自行测试加固。
4. **卷宗结构还原**（多页拼接、装订顺序、目录层级）：版面分析开源工具不覆盖这一法律卷宗特有需求。
5. **法律垂类 Eval 数据集**：任何通用评测框架都需要配上自建的领域标注数据才有意义。

### 开源基本够用、直接拿来用的节点

1. Agent Core 框架（pi-mono/smolagents）
2. RAG 检索底座（Qdrant + BGE 系列）
3. 前端可视化组件（时间线/图谱/表格/PDF 高亮）
4. 通用版面分析与结构化（PP-StructureV3/Docling）
5. Eval 工具链本身（promptfoo/DeepEval/Ragas）
