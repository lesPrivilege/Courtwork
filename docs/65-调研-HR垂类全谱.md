# 65 调研 · HR 垂类全谱（2026-07-11）

> 定位：比 docs/63 三垂类评估更重的一次全谱调研——HR 域 AI 产品与传统 workflow SaaS 一网打尽，场景按 HR 真实工作流分域。评估目的：HR 轻量包与重度包的分界与可行性。
> 机制回指（不复述）：容器+阶段、三区（只读证据区/工作稿/产出）、schema 字段契约+溯源锚、渲染词表 18 原语（docs/57）、确认门禁、计算器族（docs/56）、时效族。依赖分级沿用 docs/60 图例：**声**=纯声明｜**构**=需新产出 schema｜**具**=需新工具接口｜**资**=需外部数据资产。

## 〇、给 HR 读者的阅读图例

本文希望 HR 从业者不借工程师也能读懂。四个反复出现的词，用 HR 的话翻译一遍：

- **容器**：一个"事"的文件夹。招聘里是"一个职位的招聘项目"，员工关系里是"一名员工的人事档案袋"，就像你桌上按事分的文件夹。
- **三区**：文件夹里的三格——**证据区**放原件（简历、合同、考勤导出表，AI 只能看不能改）；**工作稿**放过程稿（筛选表草稿、测算草稿）；**产出**放最终件（待你确认的 offer 草案、处分决定草案）。
- **渲染原语**：产出长什么样——表格（table）、清单（checklist）、时间线（timeline）、日程提醒板（schedule）、组织图（tree）、对照表（compare）、文书（document）、填报表单（form）、图表（chart）等，一共 18 种固定版式，不会给你花哨的自由画布。
- **门禁**：所有"发出去就收不回"的动作——发 offer、发处分决定、出具离职证明、报送申报——AI 只准备到"待发"状态，**点发送的永远是人**。这是产品纪律，不是可选项。

### 1.4 竞品谱系的三个结论

1. **竞品全部是"记录系统"（system of record）**：数据库 + 审批流是本体，AI 是长在系统上的助手，价值兑现前提是"企业已上系统、数据已在库"。我方是"工作文件夹"：吃的是**还没上系统、或系统覆盖不到的那层工作**——中国绝大多数中小企业的人事现状是"Excel 花名册 + Word 合同 + 钉钉审批"，这层恰好是文件夹形态。两者不正面竞争。
2. **AI 火力集中三点**：AI 视频面试（红海+合规争议缠身）、简历打分（黑盒评分是负资产，见第三章）、员工问答 chatbot（流量件）。**文档与证据链侧无人做带锚结构化**——处分证据链、离职结算带依据的计算书、制度合规审查、面试记录到评估表的可溯源装配，全是空白。这是 docs/60"表格审阅空白"在 HR 域的精确复现。
3. **Lattice 与 Workday 已验证"证据装配 + 按工作切 agent"两个形态**，可作为对客叙事的海外先例引用。

## 二、场景全谱（按 HR 真实工作流分域）

分域按 HR 的"入转调离 + 选用育留"心智，不按工程模块。每场景五件套：输入→产出｜原语｜门禁点｜依赖分级｜轻量/重度归属。编号 H1–H24。

### 2.1 招聘域（选）

容器 = **一个职位的招聘项目**（开缺 → 寻访 → 面试 → offer → 到岗关闭）；证据区 = 简历原件、面试录音转写、候选人提交材料。

| # | 场景 | 输入 → 产出 | 原语 | 门禁点 | 分级 | 归属 |
|---|---|---|---|---|---|---|
| H1 | JD/岗位说明书起草 | 岗位需求要点 + 组织既有 JD → JD 草案（任职资格结构化） | document + list | **发布到门禁为止**（发出=对外要约表述） | 声 | 轻 |
| H2 | 简历筛选矩阵 | 简历包（文件夹拖入）× JD 要求 → 要求×候选人矩阵，每格带简历原文锚；**只出排序建议与理由，不出"淘汰"结论** | table(matrix) + checklist | 淘汰名单逐人确认（偏见防线，见 3.3） | 声 | 轻 |
| H3 | 面试记录 → 评估表 | 面试转写/笔记 → 结构化评估表预填（维度×评语，每条评语锚到原话）——Lattice 证据装配同构 | form + table | 面试官逐项确认后署名 | 声 | 轻 |
| H4 | 面试题库定制 | JD + 候选人简历 → 结构化面试题单（追问点标注） | list | 使用前确认 | 声 | 轻 |
| H5 | 背景核对单 | 候选人材料 → 一致性核对（学历/任职经历/证书，材料内互证） | checklist | 疑点仅登记，处置在人 | 声 + 资（学信/背调接口=资·可后置为纯文档核对，party-verify 思路复用） | 轻（接口档=重） |
| H6 | 定薪与 offer 装配 | 定薪参数（band/司龄/试用期折算）→ 定薪测算表 + offer 草案 + 审批件 | form + table + document + gate | **发 offer 到门禁为止**（offer 是要约，撤回有缔约过失风险——法条锚：民法典 §473–476） | 声 + 构（OfferSheet） | 轻 |
| H7 | 招聘漏斗看板 | 候选人台账 → 漏斗/周期报表 | chart + statcard | 知悉 | 声 | 轻（实时数据=重，需招聘平台接口） |
| H8 | 拒信/沟通函 | 筛选结论 → 拒信草案（不含任何评价性理由外泄） | document | **发送到门禁为止** | 声 | 轻 |

**战略弃项：AI 视频面试不做。** 红海（大易/海纳/近屿已卷成量级战），且人脸/声纹/微表情属敏感个人信息 + 算法歧视高风险区（见第三章）。我方只做"面试之后的记录装配"，不做"面试本身的自动化"。

### 2.2 员工关系域（入·离·纪律）——与法律包 S17 的交界区

容器 = **一名员工的人事档案袋**（入职 → 在职 → 离职，生命周期即阶段）；证据区 = 劳动合同原件、签收记录、考勤记录、违纪材料、往来通知。

| # | 场景 | 输入 → 产出 | 原语 | 门禁点 | 分级 | 归属 |
|---|---|---|---|---|---|---|
| H9 | 劳动合同/协议起草与审查 | 要素表 → 合同草案；存量合同 → 风险清单 | document + list | 用印外发门禁 | 声（**=法律包 S3/S4 本行，直接复用**） | 轻 |
| H10 | 入职 checklist | 录用信息 → 入职材料核对单（身份证/学历/离职证明/体检/竞业核查）+ 缺项跟踪 | checklist | 办结确认 | 声 | 轻 |
| H11 | 离职 checklist + 离职结算 | 离职申请/解除决定 → 交接核对单 + 结算清单（未休年假折算、经济补偿 N/N+1——**计算器族 A4 直接复用**，docs/56） | checklist + form + table | **出具离职证明/解除通知到门禁为止**（离职证明是法定义务文件，劳动合同法 §50） | 声（A4 复用） | 轻 |
| H12 | 处分与解除证据链 | 违纪材料 → 事件时间线 + 制度依据锚（员工手册条款，须民主程序生效）+ 法条锚（劳动合同法 §39/40/41）+ 证据缺口提示 + 处分/解除决定草案 | timeline + list + document | **送达到门禁为止**；**无制度依据与证据锚不出决定草案**（硬约束，见 3.2） | 声 | 轻 |
| H13 | 仲裁应对包 | 仲裁申请书 + 档案袋 → 争议焦点 + 证据清单 + 答辩草案 | timeline + list + document | 提交确认 | 声 | **划归法律包 S17**（=S1 阅卷+S15 焦点+S4 文书的劳动换皮），HR 包内只留入口 |
| H14 | 员工手册/制度合规审查 | 制度文本 → 合规风险清单（民主程序缺失/公示缺失/违法条款，逐条带法条锚） | list + checklist | 逐条确认后出修订建议 | 声 + 资（劳动法规要点参数，与法律包共建） | 轻 |
| H15 | 竞业限制/服务期管理 | 协议 + 培训记录 → 义务台账 + 补偿金/违约金计算（劳动合同法 §22/23/24） | table + form + schedule | 启动/豁免通知门禁 | 声 + 构（协议义务条目 schema） | 轻 |

**S17 交界标注**：H12/H13/H15 与法律包"劳动争议应对包"（docs/60 S17）是**同一资产的两个入口**——争议发生前的预防与合规动作归 HR 包，进入仲裁/诉讼程序后归法律包；两侧共用 A4/A10 计算器、劳动法规参数表、证据链 schema。企业客户从 HR 包升级买法律包劳动专项，是最自然的复利路径。

### 2.4 考勤假期域——时效族同构

法律包时效族（开庭/上诉期零 LLM 提醒）在 HR 域的镜像：**假期与期限也全是确定性规则**。

| # | 场景 | 输入 → 产出 | 原语 | 门禁点 | 分级 | 归属 |
|---|---|---|---|---|---|---|
| H21 | 年休假折算与假期余额 | 入职日期/累计工龄 → 应休天数折算（职工带薪年休假条例，确定性规则零 LLM）+ 余额台账 | form + schedule + table | 知悉；清零前提醒留痕 | 声 + 构（假期规则参数一张，全国性、极稳定） | 轻 |
| H22 | 考勤异常核对 | 考勤导出表（xlsx）→ 异常清单（缺卡/连续加班超 36 小时预警——劳动法 §41 锚） | table + list | 异常处理确认 | 声 | 轻（考勤机直连=重） |
| H23 | 加班费测算 | 加班记录 → 加班费计算书（1.5/2/3 倍规则，**A4 直接复用**） | form + table | 采用确认 | 声 | 轻 |
| H24 | 医疗期/三期/合同到期提醒板 | 花名册 + 员工事件 → 期限提醒板（医疗期 3–24 个月规则表、孕产哺乳三期保护、合同到期续签窗口——全部确定性） | schedule | 知悉留痕；**到期不自动发任何通知** | 声 + 资（医疗期参数表一张，数十年未大改） | 轻 |

**排班不做**：盖雅的智能排班是运筹优化问题（蓝领场景、算法资产重），不是文档工作，词表也不为它开原语。

### 2.7 场景全谱小结

29 个场景：**24 个归轻量包**（其中 H13 划归法律包、H20 后置、AI 视频面试/排班/测评三项战略不做）。原语消费全部落在既有 18 原语词表内——tree 原语获得首个真实租户（组织图），**零新增原语诉求**，词表封闭性在第五个垂类继续成立。

**HR 域门禁点全谱**（不可逆动作清单，全部"自动到门禁为止"）：发布 JD、发 offer、发拒信、签订/变更/解除劳动合同、送达处分决定、出具离职证明、发薪指令、社保/公积金/个税**报送**、启动竞业限制通知、发布组织调整公告。比法律域更密——因为 HR 的每个外发动作都直接落在具体自然人身上。

## 三、HR 域特殊纪律

### 3.1 员工隐私数据承诺——比法律卷宗更敏感

HR 档案袋里是**成建制的敏感个人信息**：身份证号、银行卡、健康体检、婚育状况、薪酬、处分记录。按个人信息保护法，其中健康、金融账户属敏感个人信息，处理须单独同意；[AI 招聘场景下人脸/声纹更是严格限制区](http://legalinfo.moj.gov.cn/zhfxfzzx/fzzxyw/202412/t20241218_511303.html)。产品承诺（沿 docs/28 数据承诺分层，HR 包措辞加重）：

- **本地存储为默认拓扑**：员工档案袋不出企业边界；云端只走推理不落盘（与法律包同一承诺，对 HR 客户要写进第一页而非附录）。
- **永不训练**：员工个人数据不用于任何模型训练——这句话在 HR 采购决策里的分量高于一切功能点（HiBob 的"责任 AI"叙事已验证此卖点）。
- **最小必要与脱敏预览**：场景只读取声明所需字段（H2 简历筛选读不到婚育字段，见 3.3）；演示与测试夹具一律虚构花名册。

### 3.2 劳动法合规锚——"无锚不落格"在 HR 域的形态

法律包纪律"无引注不出观点"在 HR 域翻译为：**无制度依据 + 无证据锚 + 无法条锚，不出任何处分/解除文书草案**。具体硬约束：

- H12 处分场景：决定草案的每一项指控必须锚到证据区材料；引用的员工手册条款必须标注民主程序与公示状态（缺失即红色缺口提示，不是静默放行）——这是败诉率最高的两个坑，恰好是确定性核对。
- H11 离职结算：经济补偿 N/N+1/2N 的选择必须显示适用法条（§46/47/87）与封顶规则（社平三倍），计算过程可核——A4 计算器的既有纪律。
- 与法律包的复利联动：劳动法规参数表（各地社平工资、医疗期表、假期规则）**一次维护、两包共用**；HR 包的 H12/H13 产出物 schema 与法律包 S17 证据链 schema 同形，企业从"预防"升级到"应诉"时档案袋直接续行。

### 3.3 偏见防线——模型只生成排序建议，不裁决

HR 是全平台**算法歧视法律风险最高的域**：[就业算法属高风险算法，使用者负周期性风险评估义务；PIPL 赋予个人拒绝纯自动化决策的权利](https://www.stcn.com/article/detail/1264934.html)；HireVue 弃用面部分析是国际前车之鉴。我方三条线：

1. **只排序不裁决**：H2 简历筛选输出"匹配度排序 + 逐项理由带锚"，永远不输出"淘汰"；H18 绩效输出"建议区间"，评级由人落笔。淘汰与评级是门禁后的人类动作——这与招投标"AI 不得代替评委"（docs/63）同构：竞品的黑盒打分是负资产，我们的"预填+带锚+人裁决"是卖点。
2. **禁用维度白名单**：筛选矩阵的维度只能来自 JD 声明的任职资格；性别、年龄、婚育、户籍、地域不进入矩阵字段（就业促进法 §3/26 锚），schema 层面不给这些字段留位。
3. **裁量区间纪律**：凡涉主观评价（绩效评语、面试评估），产出一律呈现为"证据 + 建议区间"，区间边界带依据——与法律包 A6 违约金"裁量区间呈现范式"（docs/56）同一范式。

## 四、轻量包/重度包分界

### 4.1 纯本地可跑占比

HR 工作层的输入天然是文件：简历（PDF/Word）、合同（Word/扫描件）、花名册/考勤/工资表（**xlsx**）、制度文本。24 个轻量场景中约 **22 个"拖文件夹即可跑"**（H5 背调、H7 实时漏斗降级为文档档后也能跑），占比 ≈ 22/24。

**唯一真实的"具"级新需求：xlsx 结构化摄取。** 现有 ingest 面向 OCR 文书、reading-view 覆盖 docx/md/txt/PDF，而 HR 域近半场景（H16/H17/H21/H22/H27/H28）的输入是表格文件。这不是 HR 专属需求（PM 台账、供应商报价单、工程 BoQ 同样受益），建议立为**通用包层级的表格摄取器**工单，不算 HR 包私有成本。OCR 依赖反而比法律/建工轻一档（扫描件只在合同与违纪材料出现）。

### 4.2 重度包接口清单（垂类 provider 驻场对接项）

| 接口 | 服务场景 | 性质 |
|---|---|---|
| 考勤机/门禁硬件数据 | H22 实时化 | 具·驻场 |
| 薪酬核算系统/银行代发 | H17 从"复核"升级为"对账"（发薪指令仍永远在门禁外） | 具·驻场 |
| 社保/公积金/个税申报通道（电子税务局等） | 申报表预填到门禁为止 | 具+资·驻场 |
| 招聘平台（BOSS 直聘/猎聘/前程无忧）| H2/H7 简历自动入夹、漏斗实时 | 具·驻场（各平台 API 开放度不一） |
| 学历核验（学信）/背调服务商 | H5 升级 | 资 |
| 电子签 | H9 签署环节（签署本身外包给法大大/e签宝，不自建） | 具 |
| 企业 IM（钉钉/企微/飞书） | 通知与审批流转 | 具 |
| 薪酬市场数据 | H20 解锁 | 资·重（唯一建议永久后置项） |

### 4.3 "资"级依赖可后置性

全部可后置，且大半**已由法律包建成**：社平工资/五险一金参数（A4/A11 已维护）、医疗期表与假期规则（各一张、极稳定）、劳动法规要点（与 S17 共建）。HR 轻量包的"资"级新增维护成本≈**两张静态表**，是五个已评垂类中最低的。

## 五、结论

### 5.1 零 core 成立否——**成立，且输入形态最顺**

24 场景中 22 个纯本地文件即可跑；渲染零新增原语（tree 首租户）；"具"级唯一新需求（xlsx 摄取）是通用包资产；"资"级几乎全部复用法律包。风险不在技术在市场：HR 是竞品最密集的域，差异化必须钉死在**竞品不做的证据链/带锚文档层**，绝不去碰 AI 视频面试、智能排班、测评、薪酬数据四个重资产红海。

### 5.2 最小轻量包建议（7 场景）

H2 简历筛选矩阵、H3 面试评估表装配、H10+H11 入离职 checklist 与离职结算（A4 复用展品）、H12 处分与解除证据链（差异化旗舰）、H21+H24 假期折算与期限提醒板（时效族镜像）、H27 组织架构图（tree 首秀）、H6 offer 装配。约 5–6 张新 schema（OfferSheet、AdjustmentSheet、假期/期限参数、评估表、义务台账），其余全部复用。

### 5.3 与法律包 S17 的资产共享点

A4/A10 计算器、劳动法规参数表、证据链 schema、劳动文书模板四项双包共用；分工界面 = **仲裁受理**（之前 HR 包、之后法律包）；商业上 HR 包是 S17 的获客漏斗，S17 是 HR 包的纠纷升级出口——docs/60"S17 以组合包形态试点"的最佳试点宿主可能就是 HR 包而非法律包自身。

### 5.4 泛化序位置建议

排在招投标之后、供应商之前：招投标仍是第三垂类首选（7/7 纯本地 + 政策窗口），但 HR 因 S17 复利、市场最宽（每个企业都有 HR）、输入形态最顺（xlsx+Word 即现状）应升为**第四位**，且目标客群明确为"用 Excel 和 Word 管人事的中小企业"——恰好是竞品记录系统够不着、文件夹协作范式即其现状的那一层。工程施工维持缓议。
