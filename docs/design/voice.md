# 文案与用语规范（Voice & Content）

状态：现行规范。与 `principles.md §9`（零技术概念暴露）合并成册，§9 为其中一节，本文是同一约束的完整展开与用例。可机器断言的条款由静态门 `apps/desktop` 的 `lint:voice` 强制（见文末）。

用语面向中国律所与企业法务：文案说人话、说办案话，不说工程话、不说营销话（营销腔黑名单另由站点 `deslop-scan` 守）。标识符仍用英文，本文只约束**呈现给用户的文案**。

## 1. 动作命名 = 动词 + 名词

动作控件（按钮、菜单项、场景触发）的文案必须是「动词 + 名词」，让用户脱离上下文也能读懂按下会发生什么。

- 禁止裸的无内容确认词：`确认`、`确定`、`OK`。确认动作要带宾语：说清确认的是什么。
- 好：`存入卷宗`、`确认此项`、`编译为 Word`、`打开时间线`、`归档案件`；成对处置动词 `驳回`、`修正` 语义自足，可不带宾语。
- 坏：`确认`、`确定`、`OK`、`好的`、`提交`（宾语缺失，读者要回上文才懂）。
- 例外：纯告知性浮层（无副作用，如导览气泡）的关闭动作可用一句知会（如 `知道了`），但仍优先前向动词（如 `开始导览`）。此例外**不适用**于任何触发副作用或落盘的确认。

## 2. 错误文案 = 发生了什么 + 下一步

失败与阻断文案先陈述用户能懂的事实（发生了什么），再给一个可执行的下一步；用 ` · ` 分隔或写成一句。

- 好：`这份文件没有可读取的文字内容 · 请改用含文字的 Word/PDF 或直接粘贴内容`；`文件过大 · 请拆分后重新选择，或换用摘要文本`。
- 坏：`处理出错`（只报错不给出路）；`Error 413`、裸错误码或堆栈（属诊断层，见 §6）。
- 无「下一步」可给时（纯客观阻断），也要把「为什么不能继续」说清，不留用户困在原地。

## 3. 完成提示不喊「成功」

结果提示（toast、状态行）用「已 + 动词」的结果陈述，让用户看到系统处在什么状态，而不是自我表扬。

- 好：`已存入卷宗`、`已归档`、`已定稿`、`已删除`、`已连接`。
- 坏：`成功删除`、`删除成功`、`保存成功`、`操作成功`、`连接成功`。「成功」是冗余的自评，结果本身已说明成功。
- 失败对应态用 §2 的「发生了什么 + 下一步」，不用 `未成功` 这类以「成功」取反的绕口说法。

## 4. 进行态用「……中」或「正在……」

长任务与异步态用持续体：短标签用 `……中`（`识别中`、`载入中`、`验证中`），成句用 `正在……`（`正在识别文书类型…`、`正在对齐当事人关系…`），并配品牌化思考反馈或事件流进度（`principles.md §5`），不让 spinner 裸奔、不用 `请稍候` 之类无信息占位。

- 好：`关系图谱载入中…`、`正在核对合同条款与当事人信息…`。
- 坏：`加载`、`处理`（静止词冒充进行态）、`请稍候`（无内容）。

## 5. 空态指向第一动作

空状态除说明「暂无 X」外，必须指向用户在此处的第一个动作，用 ` · ` 接一句引导。

- 好：`刚建立，尚无卷宗内容 · 从对话或场景开始整理`、`尚无工作稿 · 点击「新建工作稿」开始笔记`。
- 被动数据位——用户在此处本无可发起的动作（历史列表、数据格、引语位）——可只述状态：`暂无历史会话`、`暂无可引用原文`。
- 坏：一个可被用户填充、却只写 `暂无内容` 而不给入口的主区空态。

## 6. 零技术概念暴露（`principles.md §9`）

普通用户界面不出现 `schema`、`JSON`、`token`、`prompt`、`command`、`trace`、`payload` 等工程词；wire id、错误码、内部结构名只进入诊断层。用户文案用办案词表：案件、材料、卷宗、依据、来源、核对、确认、定稿、归档、**文字识别**（不用 `OCR`）。

- 有意的技术呈现——如「模型运行了某工具」的工具调用行（英文 affordance，对标成熟 agent 产品）、诊断/日志面板——是经设计裁定的透明化展示，不是文案泄漏，走设计评审，不受本节机器门约束。
- 判据：中文办案文案里夹进工程词才是泄漏；纯英文的工具/诊断 affordance 由设计评审把关。

**通用 chrome 的语言口径（产品裁定 2026-07-20，出处：`ARCH-SCOPE-CLOSE-1` 票面第 9 项，产品负责人于该票内直接下达；R-17 只授权「盲区登记」，本段是与之同批下达的独立产品裁定，非 R-17 的推论）**：MVP 阶段通用 chrome 走**英文原文**，控件命名对齐 frontier agent 惯例（利于后续功能下放时的概念对齐），**不硬翻译**。本节的中文办案词表约束的是**案件内容面**——schema 表、风险与依据、确认与处置、错误与空态；chrome 与内容面的语言分层由 `RP-2.7` 定义并由 `lint:rp27` 锁定，本裁定不改动 RP-2.7。防呆设计与文案统一（含 chrome 与内容面衔接处的措辞一致性）斟酌为**后续便利单**，不在当期范围。

**i18n 显式取舍（`HARNESS-CORE-1` Stage C R-10 准，2026-07-20）**：当期**不引入 i18n 框架**，chrome 英文与内容面中文均为硬编码字面量。这是取舍不是欠账——目标用户单一（中国律所与企业法务），chrome 英文由 `lint:rp27` 锁为闭集词表、内容面中文由本文与 `deslop-scan` 守，两侧都已有机器约束；此时引入 i18n 只会把「一处改字面量」换成「改 key + 改词表 + 对齐两份文件」，复杂度上升而可读性下降（复杂度节制条：能用平铺解决的不上抽象）。**重启判据**：出现第二个真实目标语种，或 chrome 词表需要按租户替换（`RP-2.7` 的原始动机之一）——满足其一再议，不预研。

## 可机器断言的条款（静态门 `lint:voice`）

`apps/desktop/scripts/assert-voice-copy.mjs` 扫描 `apps/desktop/src` 的用户文案（注释与 `${…}` 插值已剔除，只取字符串字面量与 JSX 文本），对下列可判定条款触红。规则库与反例见 `assert-voice-copy.test.mjs`。

| 条款 | 机器判据 | 触红例 |
|---|---|---|
| §1 裸确认词 | 文案**整体**等于 `确认`／`确定`／`OK`（去空白后精确匹配） | `<button>确认</button>` |
| §3 成功自评 | 出现「动作动词 + 成功」「成功 + 动作动词」「操作成功」，或整体为 `成功`/`成功了` | `成功删除`、`保存成功`、`操作成功` |
| §6 工程词泄漏 | 同一条中文文案里出现工程词黑名单（英文按词界匹配，须与中文同串） | `解析 schema 失败`、`token 用完` |

**门的边界（如实登记，非绝对声明）**：

- 扫描面为 `apps/desktop/src`（产品 UI 宿主，用户文案的著作面）。垂类包 `packages/*/src/schemas/` 是携技术描述与校验消息的诊断层，`presentation/` 词表随 desktop 渲染，二者不在本门扫描面内，其用语由评审与 `deslop-scan` 兜底。
- §2 错误体例、§4 进行态、§5 空态**不转硬门**：其「是否给了下一步/是否指向第一动作」需要语义与相邻 DOM 判断，机器易误伤，留人工评审；本门只守 §1/§3/§6 三条可判定项。
- 已知盲区：含 `{表达式}` 的混排 JSX 文本不入扫描（toast 多为字符串字面量，已覆盖）；`知道了` 一类告知性关闭词按 §1 例外不入黑名单。
- **已知盲区 · 英文 chrome 零覆盖（2026-07-20 登记，R-17）**：本门三条判据均以中文为形——§1 要求整体等于中文裸确认词、§3 匹配中文动作动词、§6 明定「工程词黑名单**与中文同串**」。故通用 chrome 的英文文案（`src/chrome/copy.ts` 的导航、欢迎态、composer、账户等词条）**整体不进本门**。这是产品最大的一块用户文案面，而 voice 门看不见它。如实登记，不据此宣称本门覆盖全部用户文案；chrome 词表的约束真源是 `lint:rp27`（锁其纯英文）与设计评审，不是本门。
- 本门当前对全树 §3/§6 为零违例（回归护栏）；§1 由本单一处整改后归零。逐处违例登记见 `apps/desktop/SPEC.md` 的 `VOICE-SPEC-1` 节。

- 容器双词表（案件用「卷宗」、通用工作区用「资料」）见 `apps/desktop/src/case/container-copy.ts`，是 §1 动词+名词的领域落地。
