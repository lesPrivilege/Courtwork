## 三、发工单的手艺（模板语义）

- **实现工单**：必读清单（CLAUDE.md + docs/10 + 本层 SPEC + 相关 ADR）→ 复述范围确认 → 交付清单 → 验收标准 → 纪律提醒（TDD、显式路径分层提交、共享索引核对暂存、跨层写对方 SPEC TODO 不越界）。
- **验收工单**：先读清单 → 逐项验收清单（含假绿防护：核对真实用例计数；干净环境自跑不采信自述）→ 处置规则 → 结论必须明确回答放行问题。报告写本层 ACCEPTANCE.md，补验纯追加。产物校验方法论：DMG/安装包级哈希不可复现（hdiutil 嵌时间戳/UUID），用 codesign + hdiutil verify + 前端内容哈希做确定性校验（B 阶段验收判例）。
- **拍板落笔纪律**：任何裁决必须落进权威文档正文（ADR/SPEC/规格文件），提交信息与待执行工单不算记录（B 阶段验收抓过一次"提交信息声称、diff 未包含"的失误——引以为戒）。
- **调研分发**：subagent（Sonnet）写入 docs/编号，**不 git 操作**（多会话并行），收尾 ≤8 行摘要；回报后本会话整合成 ADR/拍板再 commit。**调研分发定则（2026-07-11 用户拍板）：Sonnet 通用调研、Grok X 域顺藤（开源分享类可摸源码）与调研+落版连体、Fable 不派调研（但可亲笔高价值设计时 output，判例 docs/62）**；旧注：Grok 适派"调研 + 落版"连体任务（算力足/速度快/Twitter 域数据可顺藤摸瓜，调研结果在其 context 内直接落实现，省一次交接）——实测校准已过（合批验收 + P-3 待 F-1 回报后二证）。
- **commit 惯例**：显式路径 add；提交前**机器门而非目视**——本会话只该提交 docs/、AGENTS.md、CLAUDE.md、*/SPEC.md，用 `git -c core.quotePath=false diff --cached --name-only | grep -vE '^(docs/|AGENTS\.md|CLAUDE\.md|README\.md|.*SPEC\.md)'`（非空即 BLOCKED 中止；**必须 quotePath=false**——git 默认对中文路径加引号转义会使正则失配，首跑即误拦自证）拦截越界路径（2026-07-10 二次触发判例 b9d5c14→96efdb3：&& 链里的检查"打印不等于把关"）；**共享索引高峰期（并行会话正在暂存）的根治协议**：不碰共享索引，用临时索引 + 原子比对更新——`GIT_INDEX_FILE=.git/arch-index-tmp` → `read-tree HEAD` → add 自己文件 → `write-tree` → `commit-tree -p HEAD` → `update-ref refs/heads/main $COMMIT $OLDHEAD`（CAS，他人先提交则以新 HEAD 重试；判例 dcdd731 首用成功，零扰动）；**收尾必做**：`git restore --staged <自己的文件>` 把共享索引条目同步到新 HEAD——否则共享索引残留旧版，他人下次提交会把你的改动回滚（判例 4d9e202 后发现补堵）；若此前误动他人暂存，先 `git add` 归还再走本协议；**排单纪律（2026-07-10，F-2 三会话同编 App.tsx 教训）**：涉及同一热点文件（App.tsx/styles.css 类）的工单不并行派发、串行排队——手术暂存救得了一致性，救不回协调成本；`git -c user.name="Courtwork Architecture (Cowork)" -c user.email="lesprivilege@gmail.com" commit`；完后清理 .git 锁残留。不碰 `usecase/`、`.obsidian/`、他人未提交文件。

## 五、核心哲学速查（拍板时的一致性基准）

1. 模型只生成不裁决；确定事实归接口（信源 A/B/C，docs/20）。
2. 一切不可逆动作是用户的决定（留人确认/定稿编译/续行按钮/记忆写入）。
3. 静默降级零容忍（out_of_coverage、adapter_error、MiniMax 出局的理由）。
4. 契约先行，schemas 是根；改契约走提案 + 架构 review。
5. 厚薄算法：上游会替你变强的做薄（loop/skill/模型），上游永远不做的做厚（数据/判断/场景 schema/专家门禁）。
6. 声明高于代码：场景/路由/租户/缺口全是声明，改声明不改 core。
7. 通用层禁渗领域语义（docs/22）；demo 数据走装配点。
8. 零技术概念暴露给普通用户；可预期性即产品力（docs/24）。
9. 品味是流程不是天赋：多源实测、数据压倒散文、跨模型对抗 review。**时序律：设计语言与基调先行一次做重（tokens/规格/禁令），后期功能只是地基上的便宜增量**；slop 的根源是泛滥 + 依赖（无调研无 polish 地外包给模型默认先验）——有好发现即派工单，keep polish。
10. 缺口三态诚实声明（演示/未核验/禁用，docs/41），不隐藏不假装。

## 七、本手册的维护纪律

每逢：工单发出/验收关账/重大拍板 → 顺手更新第六节板面（一两行）。compaction 前若有预感，主动全文复核一遍第六节。这份手册本身就是 docs/25"结构化再水化"哲学的自我应用——产品怎么给律师续行 session，本会话就怎么给自己续行。

**Shell 判例（2026-07-10 第二次出现即入册）**：链式命令中尾随 echo/管道会掩盖前序命令退出码（F-4 build、D-1 e2e 两案）——验证类命令单独执行或显式检查退出状态，"打印了结果"不等于"命令成功"。

**共享检出分支切换判例（2026-07-11）**：并发会话可能把共享检出的 HEAD 切到自己的分支（sol 首例 codex/sol-courtwork-rp2ui-b）——此后本会话所有 git 读写必须**显式针对 refs/heads/main**（rev-parse/read-tree/log 一律带全名，禁用裸 HEAD），CAS 的 OLDHEAD 取 refs/heads/main；提交成败以 `git log refs/heads/main` 为准，HEAD 显示的是别人的世界。
