# ADR-019：卷宗容器与本地缓存分区

- 状态：**Accepted（2026-07-20）**
- 日期：2026-07-20
- 关系：ADR-013 的逻辑终点（会话窗口与 memory 已定，容器归属未定）；继承 ADR-005 §1 的容器隔离；不改 ADR-011 决定一的双账本
- 提出单：`HARNESS-CORE-1` Stage A；裁决：R-9 准（2026-07-20）

## 背景

ADR-013 已经拍板了 Chat 的**时间维**划界：1 小时连续性窗口自动分 session、历史 transcript 只读、memory 为可撤销缓存层、用户零管理面。但它没有拍板**空间维**的归属：这些 session、transcript、memory 与派生缓存**属于谁**。

当前实现里这个归属是含混的：

- `case-list.v1` 持久案件列表，`work-session-store` 持久 Work 会话，chat memory 另有单键——三处版本化单键各自为政，容器语义靠调用方自觉携带 caseId。
- ADR-005 §1 已定「容器是隔离边界；数据访问必须携 container/case id，不提供无作用域全局读取」，但这是**约束**，不是**归属模型**。
- 归档容器同构论（`chat-as-dossier-thesis.md`）把结论推到底：chat 目录整体即一个卷宗/垂类项目，内部不做 session 管理，而是按工作语义改变每次请求的注入。

实现就绪图 Round 5 方向④把它登记为 ADR 议题（「本地缓存容器化：不按 session 分，按 chat 卷宗」）；原型盘点又把版式两项（卷宗容器为组织单元、产出先入卷再确认）移交本议题作素材（就绪图皮层迁移批次账 B4 行）。本 ADR 收口这一条线。

## 决定一：容器是唯一持久组织单元，session 不是

- **卷宗/项目（container）是持久分区的唯一单位**。所有本地缓存——transcript、蒸馏 memory、阅读视图派生、渐进披露 index、工作稿——按 container 分区，不按 session 分区。
- session 降级为**容器内的时间切片**：它决定注入什么，不决定存在哪里。ADR-013 §1 的 1 小时窗口语义完全保留，只是不再充当存储边界。
- 无容器归属的 chat（「先聊后建」态）落**未归档区**，是容器化仪式的待迁入源，不是第四种持久分区。
- 用户面不变：ADR-013 §1「不提供显式 session 管理入口」继续有效；容器化是用户已有的显式动作（建案/建项目），不新增管理面。

## 决定二：注入按工作语义分级，不按 session 边界切

同一容器内，每次请求的注入由系统按工作语义确定性编译：

| 相态 | 注入内容 | 现状 |
|---|---|---|
| 连续请求（窗口内） | 完整上下文，服务前缀缓存命中 | 已落地（ADR-013 §1） |
| 跨窗续行 | 蒸馏 memory + 案语境段，不回灌历史全文 | 已落地（ADR-013 §2、`workContextSegment`） |
| 窗口内超长 | 旧 turn 摘要化并保留锚点，原 transcript 不变 | ADR-013 §2 已列，未实现 |
| 原始层召回 | 渐进披露 index，按需检索片段 | 未实现，本 ADR 定契约 |

- 渐进披露 index 是**确定性检索索引**，不是模型摘要：它可以由规则/substring/结构化字段建立（沿 CHAT-MEMORY-1 的 substring 检索 hook 先例），不上 embedding、不上模型。
- 压缩与索引产物一律是**派生层**：可整体重建、可整体丢弃，原 transcript 与账本不因其存在而改变（不变量 6）。

## 决定三：产出先入卷，再确认

- 场景与对话的产出**先落入所属容器的工作稿分区**，然后才进入确认流程。落卷是持久动作，确认是裁决动作，两者不同时、不合并。
- 这不是新纪律，是 ADR-010 durable-before-effect 在产出面的同一条：**先可撤销地持久，再做不可逆的事**。落卷可撤销（工作稿可删可改），定稿不可撤销（ADR-004：定稿后 docx 是权威表示）。
- 落卷本身不构成用户已确认，不得据此跳过任何 gate；UI 不得把「已入卷」呈现为「已确认」（voice 门适用）。

## 决定四：roleplay 与其他轻量垂类是同态，core 零改

容器同构论的可证伪形态：换 substrate 不改机制。一个角色 = 一个容器，四层现成机制对位——角色档案 = schema 化结构、摘要记忆 = 蒸馏缓存层、原始对话 = 渐进披露原文、可视化 = preview schema（ADR-014 tab = 一张 schema 表）。

- 判据：若某垂类需要改 core 才能表达，则容器同构论对该垂类**不成立**，须显式登记，不得靠 core 加特例掩盖。
- 本 ADR 不立 roleplay 包，只登记该判据作为容器模型的验证条件。

## 决定五：隔离约束原样继承，不因容器化而松动

- 案件与 Work 内容不写入全局 chat memory（ADR-013 §3）——容器化只改分区，不改跨容器可见性。
- 跨容器读取一律无授权面；跨案晋升仍是用户显式不可逆动作（不变量 3）。
- 密钥与凭证永不进入任何分区的缓存或蒸馏输入。
- demo 容器与真实容器双向隔离（不变量 7），派生缓存同样分区，不共享 index。

## 门禁

- 出现不携 container id 的持久读写必须触红。
- 出现第四种持久分区单位（按 session / 按 turn / 按 provider 分区）必须触红。
- 派生层（压缩产物、index）被当作真源消费必须触红；删除派生层后功能须可重建（注入删除的反例）。
- 「已入卷」被投影为「已确认」必须触红。
- 跨容器检索命中必须触红（index 分区反例）。

## 明确拒绝

- 按 session 分区持久层，或给用户 session 管理面（ADR-013 §5 不变）；
- 用模型摘要替代确定性索引，或让压缩产物改写原 transcript；
- 云同步、跨设备容器、跨容器聚合 memory（本地优先不变）；
- 借本 ADR 把三处版本化单键合并成一个通用 KV 抽象——分区语义统一不等于存储实现统一，合并须另证其必要。

## 未决项的处置（2026-07-20 裁决后）

1. ~~就地补 container 维 vs 迁统一分区布局~~ → **已裁：就地补维。** 现有版本化单键（实测**五处**逐字同构，非三处，见 Stage A A2.2 S1）各自补 container 维，不做统一分区布局迁移——零迁移风险优先，既有先例已多次复用且稳定。`PERSIST-BACKEND-1` 的 backend 归并按此前提执行：**归并的是同构 backend 实现，不是分区布局**。
2. ~~渐进披露 index 的建立时机~~ → **已裁：惰性建立 + 可整体重建起步，失效即重建。** 不做写入时增量、不做定期重建；确有性能证据再议升级。
3. ~~未归档区的保留期与容量上限~~ → **已裁：设上限；超限诚实提示归档/清理，不拒绝新建、不静默丢弃。** 具体数值归实现票。
