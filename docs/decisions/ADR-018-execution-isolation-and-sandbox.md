# ADR-018：执行隔离与沙箱边界

- 状态：Proposed
- 日期：2026-07-20
- 关系：泛化 ADR-005 §3 已有的隔离先例；约束 ADR-017 的受控命令执行；不改 ADR-009 决定一的 Rust 宿主定位
- 提出单：`HARNESS-CORE-1` Stage A

## 背景

本仓已经有一条隔离纪律，但它只写在单一场景上：ADR-005 §3 第 1 条「OCR、宏和其他高风险解析留在 ingest 隔离进程；**进程内纯库不得宣称自带沙箱**」。这句话包含两个可泛化的判断——隔离靠进程边界，以及**同进程内的库调用不构成隔离**——但从未被提升为通用判据。

随着 ADR-017 把受控命令执行纳入能力面，「什么算隔离」必须有闭集答案，否则会出现「用了某个库所以是安全的」这类不可验证声明。

实现就绪图 Round 5 方向②对 bash 明写「**沙盒后期**」。本 ADR 因此**只定边界与判据，不排期、不要求实现**——它的作用是让 ADR-017 在无沙箱条件下能诚实地收窄能力面，而不是等沙箱就位才动手。

## 决定一：隔离等级是闭集，且不得自我声明

隔离等级穷举为四级，每级的判据是**机制性的**，不接受「使用了某库/某框架」式声明：

| 等级 | 判据 | 本仓现状 |
|---|---|---|
| `none` | 同进程调用。库自称的能力边界不算隔离 | 全部 `packages/tools` 工具 |
| `process` | 独立 OS 进程，崩溃与内存不互穿；仍共享文件系统与网络 | `services/ingest` 的设计定位（尚无实现） |
| `os_confined` | 进程 + 内核级约束（macOS Seatbelt、Linux seccomp/namespaces），文件系统与网络按策略收窄 | 无 |
| `container` | 独立文件系统命名空间与资源限额 | 无 |

- **依赖隔离不是执行隔离**：语言级虚拟环境（Python venv、Node 的模块隔离）只隔离依赖解析，不隔离文件系统、网络与进程，一律计 `none`。此条来自归档 `research-2026-07-19-agent-pedagogy/survey.md` 的同一论断，与 ADR-005 §3「进程内纯库不得宣称自带沙箱」同源。
- 任何声称提供隔离的实现必须能被**反例证伪**：给定越界操作（读策略外路径、连策略外地址），未被拒绝即该等级不成立。无反例即无等级。

## 决定二：沙箱是外采面，不自研

按 `HARNESS-CORE-1` 方针二（外采面＝基础设施层），执行隔离属**优先选用社区成熟方案**的一类。本仓不自研沙箱。

- 采纳任何隔离方案须出选型对比：≥2 候选、许可证、维护活性、体积与传递依赖、逃逸风险与供应链面、Node 22 + Tauri v2 兼容实测点。
- **不接受把「我们做了检查」当作隔离**：命令白名单、路径校验、参数过滤都是**准入控制**，不是隔离。两者互补，不互相替代——这是本 ADR 与 ADR-017 的分工线。

## 决定三：本地优先约束下的取舍

桌面 Tauri 分发对隔离方案构成硬约束，与 `packages/output/SPEC.md:23` 拒绝 JVM 子进程的论据同构：**不能为一项能力让安装包扛一个运行时**。

- `container` 级要求用户机器上有容器运行时。对 ToB 桌面产品，这是长期运维税，**当期拒绝**。
- `os_confined` 级是本地优先约束下唯一现实的目标：macOS 的 Seatbelt 属系统自带、零分发成本。**但其可用性须实测后才可写入能力声明**——`sandbox-exec` 的弃用状态与在当前 macOS 版本、当前 Tauri 打包形态下的实际行为，属就绪图「需要实测，不再泛化调研」一类，本 ADR 不预判（裁决请求见 Stage A 清单）。
- `process` 级零额外分发成本，是 `services/ingest` 已声明的形态，也是任何执行面的**最低起点**。

## 决定四：隔离属受控宿主能力，不属 core

- 沙箱进程的创建、策略装载与生命周期归 Rust 宿主层（ADR-009 决定一：Rust 只做受控宿主能力）。core 不认识隔离等级的实现，只认识**声明的等级与其反例证据**。
- renderer 永不直接触达隔离原语；TS 侧只经既有 port 拿到结构化结果与结构化失败。
- 隔离失败（策略装载失败、进程起不来）一律 fail closed 并显式：不得静默降级到更低等级执行。此条是不变量 4 在隔离面的落点，也是归档记录的 TRAE 反面参照的正解——沙箱失败时应**显式询问**而非静默提权。

## 决定五：无沙箱不阻塞能力，但必须收窄能力面

这是本 ADR 的实用核心。**隔离等级与可授予的能力是绑定的**：

| 隔离等级 | 可授予的最高能力 |
|---|---|
| `none` | 只读、无副作用；一切写入与外发仍走既有 gate/confirmation |
| `process` | 加：ADR-004 定义的无损级动作 |
| `os_confined` | 加：策略内的受限写入 |
| `container` | 本 ADR 不预授——需求到来时另裁 |

- 当期隔离等级为 `none`，故 ADR-017 的受控命令执行**只能落在最窄档**：白名单 + 逐次确认 + 授权先于执行。这不是保守，是等级绑定的必然结果。
- **不得以「将来会有沙箱」为由预先放宽当期能力面。** 能力随隔离等级晋升而晋升，且每次晋升须有该等级的证伪反例作证据。

## 门禁

- 任何声称隔离的实现，缺越界反例必须触红。
- 隔离失败后仍继续执行（而非 fail closed）必须触红。
- 能力面超出当前隔离等级所允许的最高档必须触红（等级—能力绑定表可机器校验）。
- renderer 直达隔离原语必须触红。
- 把依赖隔离（venv 类）计为执行隔离必须触红。

## 明确拒绝

- 自研沙箱；
- 当期引入容器运行时依赖；
- 用准入控制（白名单/路径校验）冒充隔离；
- 沙箱失败后静默升权重试；
- 以未来沙箱为由预先放宽当期能力面。

## 未决（须架构拍板）

1. 是否现在就投入 `os_confined` 的可行性实测（macOS Seatbelt 在当前系统版本 + Tauri 打包形态下的实际可用性与弃用风险）。不实测则当期永久停在 `none`——这是可接受的，但须显式选择。
2. `services/ingest` 的 `process` 级隔离是否随本 ADR 一并升格为已声明等级（该服务当前只有 `SPEC.md`，零实现）。
3. 等级—能力绑定表是否需要机器可读形态（供门禁直接校验），还是文档级约束即可。
