# 调研报告：专业桌面工具的交互反馈动效（2026-07-10）

状态：调研完成，供 B 阶段 UI 最后一轮 UX debug 出规格参考。不改动 `docs/32-设计语言包/` 现有拍板，仅在既有分界线内补细粒度数值与技术裁定；涉及需架构拍板的点单列于文末。

## 〇、问题定义（复述，不重新论证）

现状诊断：点击"木讷"——**缺交互确认反馈**（按压态、切换过渡、落定感），**不是缺装饰动画**。既有硬性拍板不可推翻：

- 数据区（表格、时间线、图谱、文书、数字）渲染完成后绝对静止，`principles.md §3`。
- 长任务（2–5s）必须有边框微光流转或骨架呼吸，`motion.longTaskGlow` 1800ms linear infinite / `motion.longTaskBreath` 2000ms ease-in-out alternate；>5s 升级为事件流进度组件；<800ms 瞬发不做动效。
- 状态变更（门禁落章、风险定级、修订接受）永远 0ms 硬切，不淡入淡出；悬停反馈允许 100ms（`motion.hover`）。
- `docs/35` 已明确否决 Rauno 式物理弹簧拖拽（"弹性物理学 Spring，操作时晃动解压"）——理由原文：*"物理拖拽晃动适合个人艺术站点和泛娱乐消费产品。法律工作台需要绝对的'庄重与确定性'，如果法条卡片拖拽时伴随弹簧抖动，会让律师产生一种产品不够严肃、数据不稳定的轻浮感。"*

本报告要解决的缺口：在"数据静止 + 状态硬切"这条红线**之外**，界面仍然需要对**用户动作本身**（不是数据本体）给出即时应答——按下去有没有反应、切走了有没有过渡、确认了有没有落定感。这一层反馈目前完全空白，读成了"死机感/木讷感"。

## 一、Apple HIG 动效原则

来源：[Apple HIG · Motion](https://developer.apple.com/design/human-interface-guidelines/motion)、[To Spring or Not To Spring](https://www.userinterface.wiki/to-spring-or-not-to-spring)、[SF Symbols 动效（createwithswift）](https://www.createwithswift.com/animating-sf-symbols-with-the-symbol-effect-modifier/)、[NN/g Button States](https://www.nngroup.com/articles/button-states-communicate-interaction/)。

**Purposeful motion**：动效服务体验，不为动效而动效；系统已为标准控件提供精细动效，**高频交互不应额外加动效**——每次点击都要看一遍动画，用户会觉得被拖慢。这对 Courtwork 是直接约束：J/K 逐条浏览、⌘K 命令面板这类高频动作，反馈必须落在"瞬时"档（<150ms），不能做"入场动画"式的强调。

**Feedback 是刚需，不是可选项**：每个动作需要"立即、诚实"的响应；一次点击如果没有任何视觉变化，"感觉像坏了"（HIG 原文精神）。这句话直接命中"木讷"问题的病灶——当前 Courtwork 的按钮/Tab/确认操作大概率就是"点了但没有视觉回执"。

**Spring vs Ease 的选择依据，不是审美偏好，是输入类型**：
- Spring 用于动效与用户输入**直接绑定、可被打断**的场景——拖拽、滑动、二维手势；spring 能从任意初速度启动，跟手感来自"承接手势的能量"。
- Ease（预定义曲线，无初速度）用于**由程序触发、非手势驱动**的过渡——面板出现/消失、状态切换、Tab 切换。用 ease 硬揻一个手势动作会显得生硬（"jerks to a halt"）；反过来，给一个非手势触发的状态切换用 spring 会显得轻浮失控——这正是 Rauno 式弹簧被否决的技术侧理由，不只是审美判断。

**结论**：Courtwork 右栏五个工作面、Tab、按钮、门禁——全部是程序触发而非手势拖拽（`docs/35` 已裁定关系图谱之外禁止拖拽物理），**因此全站不存在 spring 的适用场景**，ease 曲线覆盖 100%。这不是"克制选择"，是"输入类型决定了没有 spring 的用武之地"，可以作为对未来"要不要加点弹性"提案的一句话挡箭牌。

**时长参考阶梯**（HIG 实践值，非官方强制表格，来自多方 HIG 解读整合）：按压/悬停落在 120–180ms；小状态变化 180–260ms；较大过渡 200–400ms，超过 400ms 会显迟滞。

**SF Symbols 动效谱系可借�4类语义，不借用视觉**：Bounce（动作已触发/完成的确认）、Pulse（进行中）、Variable Draw（精细进度）。这与 Courtwork 现有的"长任务微光流转=进行中"、"骨架呼吸=结果未到达"已经是同构映射，无需新增；唯一空缺的是 **Bounce 对应的"确认已完成"这一档**，当前 principles.md 只讲了状态本体 0ms 硬切，没讲状态之上是否允许一层"确认回执"的独立动效——这是本报告第五节要补的缺口，且论证它与硬切原则不冲突（见下）。

## 二、专业工具的反馈谱系

### Linear（技术实测，来源：[performance.dev 技术拆解](https://performance.dev/how-is-linear-so-fast-a-technical-breakdown)、[925studios 设计拆解](https://www.925studios.co/blog/linear-design-breakdown-saas-ui-2026)）

这篇实测直接扒了 Linear 生产环境的 CSS 变量，是本次调研含金量最高的一手材料：

```css
--speed-highlightFadeIn: 0s;      /* 高亮进入：瞬时，不淡入 */
--speed-highlightFadeOut: .15s;   /* 高亮退出：150ms 渐隐 */
--speed-quickTransition: .1s;     /* 快速过渡：100ms，hover/按钮为主 */
--speed-regularTransition: .25s;  /* 常规过渡：250ms */
--speed-slowTransition: .35s;     /* 慢过渡：350ms，用得极少 */
```

三条硬纪律，与 Courtwork 现有分界完全兼容甚至互相印证：

1. **只动画三类属性**：`transform`、`opacity`（compositor 专属，GPU 独立于主线程）；偶尔 `background-color`/`border-color`（触发重绘不触发重排）。**永不动画会触发 layout 重排的属性**（`width/height/top/left/margin/padding`）——文中原话："I mean never"，举的反例正是 `margin-left` hover 位移，会导致长列表逐帧重排、每帧都卡顿。这对 Courtwork 高密度表格/紧凑列表场景是刚性红线，比"庄重感"更硬——是性能问题。
2. **非对称进出时长**：状态"进入"可以是 0ms（如高亮标记出现即刻可见），但"退出"给 150ms 渐隐——用户需要时间注意到"发生过一件事"，但不需要时间等待"这件事开始"。这个非对称结构正好是本报告要给 Courtwork 补的"确认落定"反馈模型：状态本体瞬间到位（0ms，不违反硬切），但可以有一层伴随的确认标记以 150ms 衰减退场。
3. **动效"有出处"（spatial continuity）**：状态弹层从触发它的状态药丸位置"缩放展开"，Agent 面板从切换按钮方向滑入——动效在做空间指示工作（"新东西从哪来的"），不是纯装饰性淡入。列表行本身没有进出场过渡，保持絕对干脆。

Linear 列表行**没有**进出场动效正是"数据区静止"原则在业界最接近的对标实现——这条已经被顶级效率工具验证过，不是 Courtwork 独有的保守选择。

### Things 3 / Craft（macOS 原生感标杆，来源：[MacStories Things 3 评测](https://www.macstories.net/reviews/things-3-beauty-and-delight-in-a-task-manager/)、[MacStories Craft 评测](https://www.macstories.net/reviews/craft-review-a-powerful-native-notes-and-collaboration-app/)）

两者共同点是"自研动画工具包 + 小而讨喜"：勾选任务的 checkmark 动效短促且带一点弹性回弹感，但作用范围极小（仅图标本身，不牵动整行位移），完成后整行随即从列表淡出/移出。这个模式对 Courtwork 的启示是：**弹性感可以局部保留在"图标级微动效"上**（如确认徽章打勾的一瞬间），只要不牵动卡片/行的整体位移或缩放，就不违反"数据不许晃动"的红线——晃动的是 12–16px 的小图标，不是承载数据的容器本体。Craft 的拖拽动效属于其笔记块重排场景，与 Courtwork 数据区无关，不移植。

### Stripe Dashboard（来源：[Design Details: Stripe Dashboard](https://brianlovin.com/design-details/stripe-dashboard-ios)）

按钮按压反馈普遍是 `transform: scale(0.97)` 的 active 态，100–160ms，且业界惯例是**缩放幅度不低于 0.95**（再小会显得像误触/抖动而非按压）。Stripe 用动效给数据异步加载"买时间"——卡片从触发处滑出展示，趁着这个 200–300ms 的窗口后台把数据取回来，用户感知不到等待。加载态用 Stripe/Linear/Notion 共通的骨架屏 shimmer，这与 Courtwork 已拍板的骨架呼吸同构。

**scale 按压态是否适用于 Courtwork：不适用于数据卡**，因为"法理之线"卡片是数据容器，缩放=容器形变=违反数据静止/容器有界原则；**适用于非数据的操作型按钮**（工具栏按钮、门禁的"确认/驳回"按钮本体）——这类元素不承载数据，缩放是安全的确认手势。区分标准：**按钮缩放 vs 卡片变暗**，见第五节表格。

### Figma（面板/Tab 切换编排，来源：[Figma Blog: Smart Animate](https://www.figma.com/blog/announcing-smart-animate-and-advanced-transitions/)、[Figma 幻灯片过渡文档](https://help.figma.com/hc/en-us/articles/24244588378007-Use-slide-transitions)）

Figma 的 Smart Animate 靠图层命名匹配自动补间两帧之间同名图层的位置/尺寸差异；Push/Slide 转场模拟物理位移，用来表达"内容从哪个方向来"。这套手法本质是"指示器/chrome 层可以有位移动效，内容层瞬切"的活案例：Figma 页面/画板切换时画布内容通常是硬切，动效开销花在"过渡本身讲故事"的场景（原型演示），不代表其生产力工具主界面（图层面板、属性面板切换）本身有大幅位移动效——面板切换在 Figma 生产 UI 里同样是瞬切或极短交叉。对 Courtwork 的映射：**Tab 选中指示器**（下划线/背景块）可以有 80–120ms 的位移动效表达"焦点移动方向"，但**Tab 切出的内容区**必须瞬切，不做 push/crossfade——这是"数据静止"原则向 Tab 场景的自然延伸。

## 三、动效技术选型（Tauri WebView 语境）

来源：[MDN CSS/JS 动画性能对比](https://developer.mozilla.org/en-US/docs/Web/Performance/Guides/CSS_JavaScript_animation_performance)、[Motion.dev 动画性能分级](https://motion.dev/magazine/web-animation-performance-tier-list)、[Motion 缩减包体积文档](https://motion.dev/docs/react-reduce-bundle-size)、[framer-motion vs CSS](https://www.matthewswong.com/en/blog/framer-motion-vs-css-animations/)、[View Transitions API 浏览器支持](https://caniuse.com/view-transitions)、[Chrome Dev View Transitions](https://developer.chrome.com/docs/web-platform/view-transitions)、[tauri-plugin-macos-fps（WKWebView 60fps 封顶实测）](https://github.com/userFRM/tauri-plugin-macos-fps)。

**CSS-only（transition + 少量 @keyframes，必要时配合 Web Animations API 做程序化触发/取消）**：0 bundle 体积，跑在 GPU compositor 线程，独立于主线程（不与 React 渲染/状态更新抢时间片），是 Linear、Stripe 等专业级 Web 应用的主力方案（Linear 生产代码全站只动 `transform/opacity/background-color/border-color` 四种属性）。业界 2026 年估算 CSS 覆盖约 70–80% 的 UI 动效需求。

**View Transitions API**：Chrome/Edge 111+、Safari 18+（同文档场景）支持良好，但跨文档转场（MPA 场景）在 Safari 上仍处于"预期 2026 年内支持"状态，尚未稳定。Tauri 在 macOS 上通过 WKWebView 承载（内核即 Safari WebKit），版本绑定用户系统 Safari/WebKit 版本，不受应用自身控制——**不能作为唯一实现路径**，只能作为渐进增强（同文档 SPA 内的 Tab/面板切换可以尝试，失败自然回退到即时切换，不影响功能）。

**framer-motion / motion**：核心库约 34KB gzip，`LazyMotion` + `domAnimation` 可压至约 6KB，覆盖 90% 常见需求；其存在价值在于 spring 物理、手势驱动、可打断动效、复杂时间轴编排。**第一节已论证 Courtwork 全站没有 spring 的适用场景**（关系图谱之外禁止拖拽物理，所有过渡都是程序触发的 ease），因此引入这个库的边际收益趋近于零，代价是包体积 + 与 React 状态同步的额外心智成本（effect 清理、AnimatePresence 生命周期）。

**WKWebView 语境的额外考量**：macOS 13–15 的 WKWebView 曾把 `requestAnimationFrame` 强制封顶在 60fps，与部分律所办公环境的显示器实际刷新率不符；macOS 26（Tahoe）起 Apple 已解除此限制，但不能假设全部用户已升级。这进一步佐证**克制、短时长、compositor-only 属性**的路线比追求"丝滑 spring 手感"更稳妥——旧系统上 spring 物理动效本就更容易掉帧，而 CSS transform/opacity 过渡即使在受限帧率下也能保持视觉平滑。

**技术裁定（一句话）**：**全站动效只用 CSS transition/animation（辅以 Web Animations API 做程序化控制，如长任务微光的启停），不引入 framer-motion/motion 等第三方动画库**——需求形态（进入瞬切、退出渐隐、长任务循环微光）用变量 + 关键帧完全覆盖，且天然把动效锁死在 compositor 属性范围内，是护栏而非限制。

## 四、空态与空路由判据

仓库已拍板文字型空态 + 快捷键引导（`docs/35` 第 10 条：去插画，改用 1px 虚线框 + 简练文案 + 快捷键提示）。本节只补"何时该隐藏入口 vs 何时该空态"的判据，来源：[NN/g 复杂应用空态设计](https://www.nngroup.com/articles/empty-state-interface-design/)、[Ant Design 空状态规范](https://ant.design/docs/spec/research-empty-cn/)、[UX 资料库·空状态](https://uxlib.net/content/guidelines/empty-state.html)。

英文资料共识：空态应该**替换掉本该出现的内容本体**（如表格连表头一起换成空态，不要只在数据区域中间放一行小字"暂无数据"、表头和工具栏还在傻等），但**保留主导航**，让用户能自由离开当前空视图。中文资料补充了隐藏 vs 展示的具体判据：**入口出现/消失的频率**是关键——如果一个入口"一旦有数据就会长期存在"（低频状态翻转），可以隐藏，用户不会因为"入口时有时无"而迷失；如果入口频繁在有/无数据间切换，隐藏会让用户"找不到路"，此时必须保留入口 + 空态兜底。

**对 Courtwork 的判据（两条，可直接执行）**：

1. **结构位（三栏帧、右栏五工作面 Tab、案件列表）永远不隐藏**——这是 `principles.md §1` "完整工作台帧"的硬性延伸：四横带、三栏是永久帧，孤立漂浮或缺失结构位=演示稿层级的错误。空态文案落在这一层。
2. **场景级/子功能级入口**（如某场景卡片、某项操作按钮，依赖前置阶段是否完成）**用禁用态（置灰 + tooltip 说明门槛），不用物理隐藏**——理由正是上面的"频率判据":这类入口会随案件推进频繁地"从不可用变可用"，物理隐藏会让用户每次都要重新找位置；禁用态保留了空间记忆的连续性，且与门禁组件"下游动作禁用态注明剩余数量"（`principles.md §4`）是同一套语言，不新增模式。

简言之：**帧级永不隐藏用空态，功能级频繁翻转用禁用态**，两者都不允许"完全消失又重新出现"这种第三种形态。

## 五、规格建议：反馈时长阶梯表

前提声明：`motion.hover` 在 `principles.md §3` 写作 100ms，在 `docs/35` 第 9 条写作 150ms ease-out，两份既有文档表述不一致。本报告不擅自改动已拍板 token，仅指出这一处需要 Design 收敛，并给出参考值（120ms，落在 Linear `--speed-quickTransition` 100ms 与 Stripe 100–160ms 按压区间的交叠地带）——**这是唯一一处建议列入 `[需架构拍板]`，其余行都是在既有分界内新增，不触碰已拍板项**。

| 场景 | 时长 | 缓动 | 允许变化的属性 | 备注 |
|---|---|---|---|---|
| **按压态**（mousedown→释放） | 瞬时触发，保持到释放 | 无过渡或 60–80ms ease-out | 操作型按钮：`background-color` 加深一档；**数据卡片本体不做按压态**（无 scale/位移） | Stripe 的 `scale(0.97)` 仅移植到工具栏/门禁按钮等非数据元素；法理之线卡片按压只允许底色变化，不缩放，呼应"容器不晃动" |
| **hover** | 100–150ms（建议收敛 120ms，见上）| ease-out | `background-color` / `border-color` | 已拍板 `motion.hover`，本表仅补收敛建议 |
| **Tab 切换**（工作面/场景 Tab） | 内容区 0ms 瞬切；选中指示器 80–120ms | 指示器：ease-out；内容：无 | 仅指示器 `transform: translateX/scaleX`；**内容区不做 crossfade** | 对标 Linear"列表行无进出场动效"+ Figma 指示器位移手法的组合 |
| **面板对切**（右栏五工作面互斥展示） | 0ms 硬切 | 无 | 无 | 数据静止原则向面板级延伸；不做交叉淡入淡出，理由同 principles §3 "状态变更 0ms 硬切" |
| **确认落定**（门禁确认/接受修订） | 状态本体 0ms 硬切 + 外加确认光效 150ms 衰减 | 光效衰减用 ease-out | 仅新增一层 `border-color`（沿用法理之线 hover 折光的同一套视觉语言，用绿色态），**不改变文字/色块本体的到达时序** | 呼应 Linear `highlightFadeIn:0s / highlightFadeOut:.15s` 的非对称模型：状态瞬间到位，确认回执渐隐退场，二者不冲突 |
| **驳回落定**（逐条驳回，不批量） | 同上，150ms | ease-out | 同上，光效改用 danger 红线色；卡片随即从待确认队列移出或转灰 | 驳回无批量按钮已是纪律（`principles.md §10`），落定动效需明确"移出"而非"停留" |
| **续行跳转**（状态条红色态→开启新阶段） | 200–260ms | ease-out | `opacity` + `transform: translateY(≤4px)` 入场 | 属于路由级大过渡，仍需 <300ms 阈值内，不可显迟滞 |

已有、本报告不改动、仅并列展示以保证阶梯完整：长任务微光流转 1800ms linear infinite；骨架呼吸 2000ms ease-in-out alternate；<800ms 瞬发无动效。

**关于"确认落定"与"0ms 硬切"是否冲突的说明**：硬切原则约束的是**状态的视觉表征本体**（徽章文字、颜色、门禁三态）必须瞬间到达最终值，不允许中间态、不允许颜色渐变穿越色相。本表新增的"确认光效"是**叠加在硬切之上的独立图层**，只负责告诉用户"刚刚发生了一次确认动作"，其退场不影响、不延迟状态本体的可读性——用户在光效出现的第一帧就已经看到了最终态。这与 Linear 的非对称模型（进入 0s，退出 150ms）是同一个结构，可以安全采纳而不违反既有硬性拍板。

## 六、绝不许做清单

1. **整卡/整行位移或缩放**（抬升、回弹、放大缩小）——违反容器有界与数据静止原则；按压态最多允许底色加深，数据卡片零位移零缩放。
2. **任何弹簧/物理回弹曲线**（overshoot、阻尼振荡、rubber-band 回弹）——`docs/35` 已明确否决 Rauno 式物理弹簧拖拽；本报告第一节补充了技术侧理由：spring 只该绑定手势输入，Courtwork 除关系图谱外没有拖拽手势，spring 无适用场景。
3. **加载转圈 spinner 裸奔替代进度事实**——已拍板长任务须用边框微光/骨架呼吸/事件流进度组件，重申不得开倒车。
4. **状态本体的淡入淡出或颜色穿越**（如风险等级红→绿渐变、门禁三态之间过渡色）——0ms 硬切是纪律，"确认光效"是独立叠加层，不是给状态本体加过渡的借口。
5. **Tab/面板内容区的交叉淡入淡出（crossfade）**——数据静止原则向 Tab/面板延伸；只有 Tab 指示器本身可以有位移动效。
6. **hover 态用阴影升起模拟"卡片浮起"**（box-shadow elevation on hover）——阴影全站归零是硬性纪律（`principles.md §1`），hover 反馈一律用 border/background 差异表达，不得用阴影模拟层级。
7. **动画会触发 layout 重排的属性**（`width/height/top/left/margin/padding`）——不是审美问题是性能问题，Linear 实测证明这类属性动画在长列表场景会逐帧重排导致卡顿，比看起来"轻浮"更糟；一律锁定在 `transform/opacity/background-color/border-color` 四种 compositor/paint 层属性内。
8. **入口的"物理消失又重新出现"**——结构位（三栏帧、五工作面 Tab）永不隐藏，只用空态；功能级入口频繁翻转时用禁用态，不用隐藏，避免用户反复"找入口"。
