# 架构决定：数据承诺分层（2026-07-10）

状态：已拍板。依据 docs/15 调研。回答"分层还是确认制"：**分层同意为骨架，关键授权处用确认制**——留人确认哲学在数据授权上的应用。

## 承诺矩阵（数据类型 × 部署形态）

| 数据类型 | SaaS | 私有化 |
|---|---|---|
| 案件内容本体 | **永不训练**（红线） | **永不训练**（红线） |
| 脱敏行为数据（RevisionEvent 字段级修正） | opt-in（条款明示、可随时关、不溯及既往） | 默认不回传；"仅指标回传"作为客户主动选项 |
| 公开判例上的专业判断标注 | opt-in；使用权归产品方，"个人贡献记录"可见性归律师本人 | 同左，回传经客户机构同意 |
| 使用遥测 | 默认开启 | 默认仅存客户本地 |

**"永不训练案件内容"的法理依据**：律所/律师无权代当事人同意将案件实质内容用于第三方训练——这不是我们的让步，是委托代理关系的必然。对外叙事把它做成信任卖点，写进主协议正文而非隐私政策附件。

## 复利数据的合规定位

可复利的只有**不可逆脱敏后的字段级修正行为**（改了哪个字段、否决了哪个风险点），在"具体化合法利益/统计分析"框架下用于产品改进。三个硬条件：脱敏不可逆（真匿名化，非可复原假名化——docs/25 脱敏节点的验收标准就此升级）、目的具体化、用户可见可退出。web fetch 记录属使用遥测档，不单独成类。

## 试点前法务硬件（四件，缺一不上线）

① 隐私政策（对律师个人）② 服务主协议（对律所机构，含永不训练条款）③ DPA 数据处理补充协议 ④ **模型 Provider 侧"不训练"书面确认**（先于上线拿到；provider 无关架构在此的合规含义：每个接入的 provider 都要有这份确认）。私有化补充条款与脱敏技术说明文档（供客户尽调/监管检查）随需补。

## 不许做

- 任何"默认勾选案件内容相关授权"的路径。
- 用可复原的假名化冒充脱敏。
- 把数据授权埋进长文本——关键授权（行为数据 opt-in、标注贡献）逐项确认，与门禁同一交互语言。
