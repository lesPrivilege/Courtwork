# MVP 规划：法律垂类 Agent（内部对齐口径，2026-07-09）

承接《规划稿-产品技术商业化》与《调研报告-场景与功能全景》。MVP 只回答一个命题：**普通法律从业者第一眼觉得方便，并愿意把整套卷宗扔进来。** 摄取发生了，沉淀才开始；沉淀开始了，北极星（团队标配）才有起点。其余一切从这个命题倒推。

## 一、滩头场景（四个，一条摄取管线养活）

选择标准：调研确认的市场空白 × 高频刚需 × 右侧交互可演示 × 现有数据/接口可支撑。四个场景共享同一条卷宗摄取管线，工程上是一份投入、四份产出。

**S1 卷宗摄取与结构化阅卷（旗舰演示场景）**
用户把一套扫描件卷宗（倾斜、印章遮挡、手写批注）拖进一个案件文件夹 → OCR → 卷内文书分类 → 跨文档实体对齐 → 右侧自动产出：**事件时间线（每条超链接回原文并高亮）、当事人关系图谱、供述/证据矛盾清单**。对标 Wexler + Clearbrief 模式；调研确认国内仅有雏形（智合AI），卷宗级 OCR 摄取则是完全真空——律所现状是自拼 PaddleOCR 私有方案。这是"第一眼方便"的主承载场景。

**S2 表格审阅（矩阵视图）**
行 = 卷内文件或批量合同，列 = 要素问题，单元格带溯源。Legora Tabular Review / Harvey Vault / Hebbia 已验证这是批量文档的最佳交互，**国内无任何对标产品**。复用 S1 管线，边际成本低，对企业法务客户（批量合同）与律所（批量证据筛查）双向可演示。

**S3 合同审查（立场化 + 预置行业 playbook）**
红海场景（MeCheck/法宝来签/iTerms 已卷），但它是采购清单必查项、日常最高频入口，不做等于把日活让给别人。差异化三点：甲/乙/中立立场一键切换 + **行业级预置 playbook**（目标用户没有历史 DMS，要求自建清单等于劝退——机会在预置）+ 合同主体自动核验走企查查/天眼查确定性接口。**产出为带修订痕迹与批注（附法条/依据）的 .docx**——调研确认国内产品普遍只给建议文本，修订痕迹交付成熟度低，这是低成本的体验代差。

**S4 诉讼文书起草（要素式，基于已摄取卷宗）**
起诉状/答辩状从**案件文件夹的结构化上下文**生成（当事人、事实时间线直接回填要素），而非竞品的"对话里描述案情再生成"。与 S1 形成闭环：阅卷产物直接变成文书原料，让用户体感"这个东西认识我的案子"。

**明确不做（MVP 边界）**：不自建类案检索门户（法宝/得理/MetaLaw 已卷且数据壁垒高，MVP 阶段以接入现有库做引用溯源为限）；不做监控预警、多人协作门户、自动谈判、心跳式自主任务；不做 2C。

## 二、UI 形态

三栏结构：**左侧 = 案件/session 列表**（一个案件 = 一个文件夹 = 持续上下文）；**中间 = 对话流 + 场景卡片**（进入案件后由场景识别分类器推荐场景按钮：整理卷宗 / 审这批合同 / 起草答辩状——按钮是主入口，聊天框是兜底，不是反过来）；**右侧 = 结构化产出与专门交互**（时间线、关系图谱、矩阵审阅、修订预览，全部可点击溯源回原文）。

两条交互纪律：**每个关键产出节点留人确认**（Garfield AI 每步批准、深圳中院每环节勾选的范式，既是低容错场景的护栏，也是低认知用户的安全感来源）；**所有修正是 schema 级操作**（改时间线上的一个事件、否决一个风险点、修一条图谱边），从第一天起就是反馈标注，不是自由文本。

## 三、架构最小集

- **Agent core**：开源极简框架（pi 类）验证交互，不自研 loop。
- **摄取管线 v1**（★自有仓库）：OCR（含印章/手写鲁棒性）、文书类型分类、跨文档实体对齐。这是 MVP 最大的工程投入项，也是唯一不许省的。
- **场景注册表 v1**（★）：四个场景可以硬编码实现，但 schema（触发条件 + 输入 schema + 工具集 + 产出 schema + UI 模板）必须按注册表规范定义——Phase 1 的解耦迭代靠它。
- **确定性接口 v1**（★）：企查查/天眼查主体核验 + 一个法规/判例引用校验源。
- **产出管线**：markdown 底层 → .docx 修订/批注输出（开源工具链）。
- **模型**：甜点区中档模型起步，provider 无关封装；eval 集 v0 与试点同步启动（试点案卷即评测素材）。

## 四、验证指标

- **摄取意愿**（北极星前置条件）：试点团队真实卷宗的主动摄取率；一套卷宗从拖入到出时间线的端到端时长（目标：一次午休之内）。
- **高频**：席位周均场景触发次数；场景按钮 vs 自由聊天的使用占比（按钮占比低说明场景识别或场景设计失败）。
- **质量**：文书/修订采纳率（用户保留的修订比例）；时间线/图谱的人工修正密度（既是质量信号也是标注产量）。
- **商业**：试点续费/付费转化意向；摄取 credits 包的实际消耗。

## 五、试点与定价试验

试点对象：3–5 家中型律所诉讼团队（S1/S4 主战场）+ 1–2 家企业法务（S2/S3 主战场）。定价按既定框架试验：席位费保底覆盖 S3/S4 高频场景，credits 只计量 S1 摄取与 S2 批量任务；**历史卷宗批量摄取作为签约动作打包**（一次性 credits 包，收入与切换成本同时落袋）。试点期配驻场/半驻场产品角色——目标用户表达不出需求，场景挖掘机制本身是 MVP 交付物之一，不是附属品。

## 总结

MVP = 一条摄取管线 + 四个场景 + 三栏 UI + 留人确认。S1 负责"第一眼方便"，S2 负责国内无人有的交互，S3 负责日活和采购清单，S4 负责闭环体感。验证的不是功能完整度，而是那个单一命题：用户愿不愿意把整套卷宗交给它。
