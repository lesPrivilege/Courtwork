# [需架构拍板] 清单

## 1. 信源 A/B/C 与“已核验”资格从哪里来

**问题**：`risk-list.json`、`timeline.json`、`party-graph.json` 的 `sourceAnchors` 只有 `fileId/quote/textRange`，没有 tier、verified 或 authority 字段；北极星屏却展示 A/B/C，并据此决定批量确认资格。

**理由**：这是跨层契约，不是 UI 可自行推断的样式。用文件名、citation 文案或来源域猜 tier 会把业务逻辑塞进壳，并可能错误放行高风险批量确认。

**替代方案**：由 schemas/core 明确提供引用级 provenance，例如在事件流伴随的展示模型中给 `tier` 与 `verificationStatus`；或由工具结果经 core 映射进既有 artifact。具体落点由架构决定，UI 只消费结果。

**影响面**：所有 A/B/C 角标、authority 绿线、未核验强制逐条确认、批量资格、证据底纹说明、溯源详情。

## 2. 确认质量埋点属于 UI 事件还是 core 门禁协议

**问题**：硬约束要求记录确认停留时长并识别“秒批”，但当前设计包只有视觉规则，没有事件名、计时边界、脱敏/存储策略或何时阻断确认的契约。

**理由**：若 UI 自行计时并决定阻断，会把门禁业务逻辑放进壳；若只上报原始鼠标/键盘行为，又涉及敏感行为数据与可解释性。

**替代方案**：core 定义最小审阅遥测事件（如打开条目、展开依据、提交处置）及时间戳，规则计算在 core/分析侧；UI 只显示“已展开 N/N 依据”和服务端返回的门禁资格。是否真正阻断或仅告警需拍板。

**影响面**：门禁组件、事件流协议、审计日志、隐私说明、回归测试、批量确认。

## 3. 语义色可访问性修正与法理线同色约束

**问题**：现有语义 fg 在 tint 上多项未达到 12px 正常文本 AA；验收记录的对比度数字不正确。最明显的琥珀仅 2.96:1。

**理由**：硬约束既要求语义色预算，又要求专业可读性。继续使用当前 fg 只能把彩色降为装饰并另加深色文字，或修改语义 fg；两者都会影响“线色、徽章字色、状态文字同色”的既有一致性。

**替代方案**：优先保留 hue 预算但下压 fg 明度，琥珀候选 `#B45309`（在 `#FCF6E8` 上 4.66:1）；红/绿/板岩统一做同样复算。法理之线可继续使用原高纯度色，徽章文本使用 accessible fg，或全域同步换色。两种策略需架构选择。

**影响面**：tokens、法理之线、A/B/C、风险等级、门禁、修订标记、视觉回归基线、图标（若复用语义色）。

## 非拍板项但必须修正

- 图谱 10.5px 违反 12px 下限；直接修设计稿即可。
- HTML 中未登记 hover/glow 色值应改为 token/透明度组合。
- 补 `:focus-visible`、可点击图谱边命中区、状态圆盘键盘入口。
- dense row 不实现 40px 折光；完整签名动作只放展开详情。这是规格退化细化，不改变母题。
