# 架构观察：agent 与 workflow 的演进阶梯（2026-07-10）

状态：战略归档。招投标评审/供应商管理/合同审批/电子签章等办公场景确认为"接口 + 注册表"之事，其与企业 workflow 的关系分三段演进。

## 三段阶梯

1. **伴随（现在～Stage 1）**：企业 OA/签章平台为主干，agent 做节点增值（审批前审查报告、签前核验、供应商资质核查）。形态 = 适配器，自动到门禁为止（docs/48）。**电子签章永远停在此档**：牌照生意，对接 e签宝/法大大，不自建。
2. **内置（Stage 2–3）**：workflow 引擎的智能节点由我们供货（审查节点出 RiskList、评审节点出矩阵），人审保留。形态 = 我们做 MCP server/API 被调用方（docs/29 第三触发的商业含义）——从外挂变节点供应商。
3. **取代（长期，被拉动不强推）**：workflow 引擎 = 预定义流程 + 人工节点 + 流转日志，**与场景执行器同构**——场景声明=流程定义、确认门禁=审批节点、事件流=实例日志、通道无关 actor=会签地基。差三个声明式扩展（多角色/会签门禁、串并行分支、超时升级——注册表 v3 材料）。声明即部署 + 节点原生智能 + 天生留痕，是下一代 workflow 引擎的形态。

## 时序纪律

1→2 赚信任，3 由客户拉动——**永不主动打"替代 OA"旗号**（触发采购防御），只做"你的 OA 里最聪明的节点"，直到客户自己要求整条流程进来。

## 场景确认

招投标评审（S9 族扩展：多方评审即多角色门禁的首个用例）、供应商管理（party-verify + 私域库 + 监控层的组合场景）、合同审批（阶梯 3 的旗舰试验场）——全部是声明 + 适配器，零新机器。
