# 北极星屏逐屏评审

## 总体判断

二轮修正有效解决了“漂浮卡片拼贴”的第一层 AI 生成感：完整横带、三栏、紧凑列表和贯穿网格让页面开始像软件。然而两屏仍有一个共同问题：**所有设计决定都被同时摆上台面证明**。标题栏、KBD、法理之线、AI callout、进度光、批量栏、状态条、徽章说明、图谱和修订预览都在争取被看见，形成“设计语言验收墙”，而不是一个用户已经使用五年的自然工作状态。

成熟软件通常不会解释自己的语法。比如 S3 面板头的“底纹 = 核验内容”适合首次引导，不适合常驻；S1 的“琥珀边 = 涉矛盾点关联”应进入图例抽屉或帮助提示。默认屏应让规则通过重复使用自然显现。

## S3 · 合同审查

### 已成立

1. 风险清单由卡片流改为 30px 列表，是最关键的成熟度提升；编号、等级、法条、状态、操作列符合律师扫读路径。
2. R3 展开行把解释、依据和门禁放在同一局部，正确体现“看完证据再处置”，比独立弹窗更有连续性。
3. 红删蓝增忠实 Word 心智；文书纸面使用 15px、两端对齐，和 UI 表面形成合适的阅读模式切换。
4. “白底 2px 线”与“蓝 tint 4px AI callout”在结构上已严格分工，这是整包最有价值的视觉语法。

### 工艺缺陷

1. **签名动作没有被完整证明。** 列表 hover 只把整条线改成三段渐变，没有实现规格所写的“鼠标 Y 附近约 40px 局部折光”；卡片没有折光；citation 右缘浮现完全缺席。30px 行本身也不足以承载 40px 线段，规格需要区分 dense row 与展开卡。
2. **颜色预算发生漂移。** `#1D4ED8`、`#3E6FA8`、`#E3E9EF`、`#F08A8A`、`#EBB35C`、`#9DB0C4` 直接写在 HTML，未登记为 token；这与“彩色只能来自 semantic/line token”冲突。
3. **批量资格的证据链断裂。** 批量栏称 R2/R6/R5“依据已核验”，但 `risk-list.json` 只有 `citation/sourceAnchors`，没有 tier 或 verified 字段。屏幕能画出来，真实客户端无法从当前 artifact 稳定推导。
4. **静态状态与真数据混用未标识。** fixture 中 6 项风险均为 `pending`，屏上把 R1 改为已确认、R4 改为已驳回并附时间。这可以作为状态演示，但应在 handoff 中明确“交互演绎状态”，否则实现者容易误以为来自 artifact。
5. **可访问性交互近乎空白。** 无统一 `:focus-visible`，状态圆盘只靠 `title`，可点击溯源多为 `href="#"`，展开行不是可聚焦控件；键盘提示存在，但键盘路径没有语义载体。
6. **对比度声明错误。** 琥珀 `#D97706` / `#FCF6E8` 实测 `2.96:1`；红 tint `4.41:1`、绿 tint `3.15:1`、板岩 tint `4.34:1`。12px 正常字重徽章不能声称全部通过 WCAG AA。

### “AI 生成感”残留

1. 右栏把“风险表 + 展开详情 + 修订预览 + 4 张批注卡”纵向一次铺满，像为了证明组件覆盖而拼接的长页面。真实工作时律师一次更可能在“风险处置”或“修订核对”之间切换，而不是同时滚动两个任务域。
2. 中栏出现“启动卡—结果卡—建议 callout—正在起草”四种样式连续陈列，像组件样板册。长期使用界面应压缩已完成步骤，把对话历史降噪为一行摘要。
3. “信源 A/B/C + 底纹 = 核验内容”的常驻图例把培训信息放进生产界面，显得原型感重。

### 与五年成熟软件的差距

- 缺少稳定的主从导航：选择哪条风险、对应哪段修订、当前证据定位到哪里，三者没有持续的选中关系。
- 缺少恢复与历史感：已确认条目只有时间，没有确认人、变更缘由、撤销窗口或审阅轨迹入口。
- 缺少窗口尺度策略：`body min-width:1280px` + 左栏 240px + 中栏最小 440px + 右栏 47% 在 1280px 已产生潜在横向溢出，而 `overflow:hidden` 会直接裁切。
- 24px 高的小按钮承载连续法律处置，35–55 岁目标用户在高 DPI 与 Windows 缩放下可能过紧；至少需要密度档与命中区分离。

## S1 · 阅卷整理

### 已成立

1. 时间线列宽顺序正确：日期—编号—事件—金额—来源；金额右对齐、日期/金额等宽，扫读效率明显高于卡片时间线。
2. >5 秒摄取使用具体步骤、数字进度和办案文案，没有裸 spinner，符合“系统没有死机”的信任要求。
3. 展开 E31 同屏展示生成叙述与原文引语，已经接近“解释可疑、证据可核”的产品承诺。

### 工艺缺陷

1. **图谱文字违规。** `.gelabel` / `.gtag` 为 `10.5px`，低于设计包 12px 全站下限；长关系文案也因此难以在常见办公屏幕上稳定清晰。
2. **图谱不是可用交互。** 规格要求图谱边可点击溯源，但 SVG 的 `<line>` / `<text>` 没有链接、按钮语义、命中区或焦点态。1–1.5px 线条也不可能成为可靠鼠标目标。
3. **确认门禁过粗。** 工具栏“确认时间线”看起来可立即点击，但右栏又写“4 处逐条核对后方可整体确认”。没有 4/4 核对进度、禁用态或未读项状态，防呆原则尚未闭环。
4. **摄取状态与 fixture 相反。** `case-file.json` 的 20 个文件均为 `done`，屏上演示 16/20 摄取中。作为状态样例合理，但必须从事件流回放 fixture 提供，而不是由 UI 临时改写。
5. **仍有未登记色值与签名动作缺项。** 与 S3 相同，折光用了硬编码浅色，citation 浮现未实现。
6. **金额列存在语义空值问题。** 用长破折号表示无金额可读，但实现层需要区分“无金额”“未知”“不适用”，否则排序、筛选和读屏语义会混淆。

### “AI 生成感”残留

1. 右栏同时纵向放时间线与关系图谱，像展示两件交付物；成熟软件会让时间线保持主任务，图谱作为可切换视图或关联检查抽屉。
2. 静态手排图谱把 14 个主体都塞进一张图，并在细线上写完整关系句，接近“演示时看得懂、工作时点不准”的典型原型。
3. “基于已完成 16 件”与“剩余完成后自动补全”给出自动变更承诺，却没有增量变化提示、版本号或重新确认门禁，容易破坏已确认结果的确定感。

### 与五年成熟软件的差距

- 缺少新增卷宗后的增量审阅模型：哪些事件是新增、哪些关系被改写、原确认是否失效，应有稳定差分状态。
- 缺少筛选/排序/定位：47 个事件只有“展开全部”，没有按主体、文书、矛盾点、金额过滤的成熟工作流。
- 图谱没有拥挤退化、长名称省略、缩放、框选、键盘导航和边证据抽屉规格；手排坐标不能进入量产。

## 两屏共同建议

1. 默认视图只保留当前任务：S3 用“风险 / 修订”视图切换，S1 用“时间线 / 关系”切换；不要在同一滚动面纵向堆叠两种主任务。
2. 中栏历史完成卡折叠成一条“审查完成 · 6 项风险 · 4 项待确认”，把空间留给当前处置上下文。
3. 法理之线只在展开详情完成 40px 折光 + citation；dense row 仅做 0ms 颜色与 100ms hover 底，不勉强复制完整动作。
4. 把确认质量反馈做成用户可理解的进度：“2/2 依据已展开 · 已审阅 48 秒”，而不是暗中只采集埋点。
