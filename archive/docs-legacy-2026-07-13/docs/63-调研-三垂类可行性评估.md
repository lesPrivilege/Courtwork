# 63 调研 · 三垂类最小实现可行性评估（2026-07-11）

> 定位：验证"垂类即依赖包"的泛化能力，候选三垂类——供应商管理、招投标管理（企业采购侧）、工程施工管理。**只做最小评估，不展开设计**；不是实施立案，docs/62 的规格深度不在本文范围。
> 机制回指（不复述）：垂类包 = schema 字段契约 + 场景声明 YAML + 渲染词表消费（docs/57 七族 18 原语）+ 续行投影（docs/25）。依赖分级沿用 docs/60 图例：**声**=纯声明｜**构**=需新产出 schema｜**具**=需新工具接口｜**资**=需外部数据资产。核心纪律预设：模型只生成不裁决、无锚不落格、留人确认、原件只读。

---

## 一、供应商管理（企业采购侧 SRM-lite）

### 1.1 容器与三区

- **容器 = 供应商档案**（一个供应商一容器）。与 docs/61"容器对齐有始有终的工作"存在张力：供应商是长周期实体，但有明确终态（退出/拉黑），生命周期本身即阶段序列，可接受。**阶段 = 准入 → 在营考评 → 整改 → 退出**。备选切法（一次准入评审/一次年审 = 一容器）粒度过碎，续行投影反而断裂，不取。
- **三区**：只读证据区 = 供应商递交的资质原件（营业执照/体系认证证书/审计报告/银行资信扫描件）与往来函件；工作稿 = 考评表、整改跟踪台账；产出 = 准入评审报告、年度考评报告、黑名单提案（docx）。

### 1.2 核心场景（6 个）

| 场景 | 输入 → 产出 | 原语 | 门禁 | 分级 |
|---|---|---|---|---|
| 资质核验与准入核对 | 资质文件夹 → 准入核对单（证照 × 效期 × 齐备 × 主体一致性） | checklist + table | 准入结论确认 | 声 + 具（party-verify **复用**；工商数据=资·可后置为纯文档核对） |
| 供应商尽调摘要 | 工商/涉诉/资质材料 → 尽调报告（S14 同构） | list + graph | 写入档案确认 | 声 + 资（涉诉数据后置，先吃用户投放材料） |
| 多家报价对比 | 报价单若干 → 对比矩阵 + 差异摘要 | table(matrix) + compare | 采纳建议确认 | 声 |
| 绩效考评打分 | 交付/质量参数 → 评分卡（S10 计算器族第三租户，权重表团队自定义零年更） | form + table + statcard | 考评结果确认后写档案 | 声 + 构 |
| 证书效期板 | 档案内证照 → 到期日程板（效期计算零 LLM） | schedule | 知悉确认留痕 | 声 |
| 整改闭环跟踪 | 整改通知/回复函 → 整改项清单（≈PM-4 carryOvers 跨文档接续） | checklist + timeline | 销项确认 | 声 |

### 1.3 字段推理改造点

- **新字段族**：SupplierFile 容器档案（≈CaseFile 同形）；资质字段族（证照类型/证号/发证机关/效期——checklist 条目的 evidence 扩展）；SupplierScorecard（CalcArtifact 同族）；RemediationSet（≈ActionItemSet 换皮）。约 4 张 schema。
- **直接复用**：主体核验 party-verify（法律包既有工具的第一个跨垂类租户）、矩阵审阅、schedule、checklist、CalcArtifact、S3 风险清单机制（供应商合同审查即 S3 本行，甚至不算新场景）。

### 1.4 接口与数据依赖

- 外部依赖：企业 ERP/SRM 台账（Stage 1 适配器层，后置）；工商/涉诉核验数据（资，**可后置**——先做"文档内一致性核对"，外部核验升级为增强档）。扫描件走既有 ingest OCR（具·现成）。
- 纯本地起步：6 场景中约 5 个"拖供应商递交的资质包进文件夹即可跑"（尽调的外部数据档降级后也能跑），占比约 **5/6**。

### 1.5 可行性结论

零 core 上架**成立**。最小包 ≈ 5–6 场景 / 4 schema。壁垒判断：**数据为主**（工商/涉诉核验资产与法律包尽调线共用，复利）、判断为辅——纯文档核对层竞品易追，护城河在外部核验数据接上之后。

### 1.6 竞品一瞥

SRM SaaS 市场成熟（2024 年国内 147 亿规模，[IDC 数据](https://www.ithome.com/0/960/957.htm)；致远互联等平台化玩家，[趋势综述](https://www.zhengyuansz.com/blog/p-docs-2199/)），AI 应用集中在 OCR 证照自动回填与流程自动化，形态是"供应商注册门户 + 审批流"。我方差异：无需供应商上门户走注册流，**采购方拖文件夹即得带锚核对单**——轻部署侧切入，不与全流程 SRM 正面竞争。

## 二、招投标管理（企业采购侧）

### 2.1 容器与三区

- **容器 = 一个采购项目/标段**——天然"有始有终"（发标 → 开标 → 评标 → 定标 → 签约），与"一个案件"同粒度，是三垂类中容器语义最干净的。**阶段 = 需求确认/招标文件编制/发标答疑/开评标/定标/签约**（docs/49 第十章"招投标包可声明标段"的预言即此）。
- **三区**：只读证据区 = **各家投标文件原件**（PDF/扫描件）、招标文件定稿、澄清答疑函——评审证据语义强，与法律"卷宗只读"几乎同义（质疑投诉时原件即证据）；工作稿 = 评分表草稿、澄清问题清单、招标文件草案；产出 = 评标辅助报告、废标提案、澄清函、中标通知草案。

### 2.2 核心场景（7 个）

| 场景 | 输入 → 产出 | 原语 | 门禁 | 分级 |
|---|---|---|---|---|
| 招标文件审查 | 招标文件草案 → 缺陷清单（歧义条款/评分标准冲突/资格条件违规风险） | list + table(matrix)（S3 换皮） | 逐条确认后出修订稿 | 声 |
| 投标响应性核对 | 投标文件 × 招标要求 → 逐项响应核对单（**S9 原型场景**，要求 × 各投标人矩阵 + 偏离标注） | checklist + table(matrix) | 废标提案确认 | 声（扫描件走既有 OCR） |
| 投标人资质核验 | 资质证照 → 核验单 + 主体一致性；围串标线索（同一联系人/地址/关联股权）**仅提示不裁决** | checklist + graph | 线索仅登记，处置在人 | 声 + 具（party-verify 复用）+ 资（股权关联数据可后置为文档内比对） |
| 报价对比与算术复核 | 各家报价清单 → 对比矩阵 + 算术错误核对（确定性计算零 LLM） | table(matrix) + compare | 修正认定确认 | 声 + 构（报价行 schema） |
| 评分表预填 | 评分办法 + 响应材料 → 预填评分表（每分项带锚，**评分永远是提案**——评标须法定评委独立打分，留人确认在此是合规刚需不是产品洁癖） | form + table | 评委逐项确认 | 声 |
| 澄清函起草 | 核对缺口 → 澄清问题函草案 | document | 外发到门禁为止（docs/47） | 声 |
| 程序时间线与法定期限板 | 项目材料 → 程序时间线 + 期限板（招标投标法期限：文件发售/答疑/开标间隔，确定性规则零 LLM） | timeline + schedule + flow | 知悉确认 | 声 + 资（法定期限参数表，一张、极稳定） |

### 2.3 字段推理改造点

- **新字段族**：TenderProject 容器档案；BidResponseMatrix（要求条目 × 投标人 × 响应/偏离——矩阵审阅 cells 的字段换皮）；BidPriceSheet（报价行：项/量/单价/合价）；ScoringSheet（CalcArtifact 同族）。约 4 张 schema。
- **直接复用**：S9 招投标核对本就是法律包场景族候选（docs/49 明示"复用 party-verify，一批声明而已"）；矩阵审阅、checklist、CalcArtifact、timeline/schedule、S3 风险清单、S4 起草全部现成。**这是三垂类中"既有资产命中率"最高的一个**。

### 2.4 接口与数据依赖

- 外部依赖：公共资源交易平台/招采平台（资）——**完全可后置**：招投标材料天然以文件包形态流转，用户从平台下载标书拖进文件夹即是标准工作流，平台接口只省一步下载。工商关联数据（资）后置为增强档。
- 纯本地起步占比：**7/7**——三垂类中唯一全场景纯本地可跑的。

### 2.5 可行性结论

零 core 上架**成立，且是三垂类中最顺的**：材料形态（一个标段=一个文件包）、证据语义（原件只读）、留人裁决（评标法定人评）与法律包三条纪律逐条同构。最小包 ≈ 5–6 场景 / 4 schema。壁垒判断：**判断为主**（响应性偏离判定、评分办法解析的专业判断沉淀），数据为辅。

### 2.6 竞品一瞥

政策东风明确：发改委 2026 年 2 月发布[招投标领域 AI 推广应用实施意见](https://www.ndrc.gov.cn/xxgk/zcfb/tz/202602/t20260210_1403680.html)；工具侧有钛投标/剑鱼/广联达清标（[评标软件实测](https://gitcode.csdn.net/6a28248110ee7a33f279d12a.html)）与采购垂直大模型["灵析"](https://cnsocic.com/news-detail/2/24)，但重心压在**投标人写标书**与**交易平台侧智能评标**两端；企业采购方自用的"带锚响应性核对矩阵"是空白——docs/60"表格审阅空白"在招采域的复现。我方差异：合规红线（AI 不得代替评委）恰好与留人门禁宪法同向，竞品的黑盒评分是负资产、我们的"预填+带锚+人裁决"是卖点。

## 三、工程施工管理

### 3.1 容器与三区

- **容器 = 一个工程项目（合同段/标段）**——有始有终（开工 → 竣工结算 → 保修），但周期以年计，远长于案件/迭代/标段；续行投影在此承压最重（也可能是最强展品）。**阶段 = 开工准备/施工/竣工验收/结算/保修**。
- **三区**：只读证据区 = **签证单、监理指令、往来函件、会议纪要、施工日志、影像的扫描件**——工期与费用索赔的证据链宿主，只读红线的证据法意义与法律卷宗等强；工作稿 = 索赔计算书、结算书草案、台账；产出 = 签证索赔报告、结算资料、项目月报。

### 3.2 核心场景（6 个）

| 场景 | 输入 → 产出 | 原语 | 门禁 | 分级 |
|---|---|---|---|---|
| 签证索赔台账 | 签证单/函件扫描 → 台账（事项 × 金额 × 依据锚 × 时效状态——28 天索赔期是确定性规则） | table + schedule | 登账确认 | 声 + 具（**手写签字/印章 OCR**——法律包五自研节点之一的第二租户，方向现成、难度真实） |
| 索赔事件时间线 | 函件/纪要/日志 → 事件因果时间线（S1 阅卷同构，工期索赔的证据链装配） | timeline | 确认 | 声 |
| 施工合同风险审查 | 施工合同 → 风险清单 | list + table(matrix)（S3 直接复用） | 逐条确认 | 声 |
| 工程量清单对比 | 投标清单 vs 结算清单 → 量差矩阵 + 算术复核（确定性） | compare + table | 量差认定确认 | 声 + 构（BoQ 行 schema；**万行级数据量**，AG Grid 触发条件 docs/46 裁决 4） |
| 计划 vs 实际进度对照 | 进度计划 + 周报 → 偏差对照 | compare + timeline（**gantt 为天然需求方，挂账原语的首个解挂候选**） | 确认 | 声 + 构 |
| 索赔计算器 | 工期/费用参数 → 索赔额计算书（S10 租户，分项 + 依据 + 溯源） | form + table | 采用确认 | 声 + 构 |

### 3.3 字段推理改造点

- **新字段族**：ProjectSite 容器档案；签证索赔字段族（事项/金额/时效起算/证据链锚）；BoQ 工程量清单行（编码/项目特征/量/价——行数量级是新问题）；进度条目族。约 5 张 schema，其中 BoQ 是三垂类中唯一"字段不难、量级难"的。
- **直接复用**：S3 合同审查（施工合同即法律包本行）、S1 时间线、CalcArtifact、checklist（竣工资料核对）、compare。

### 3.4 接口与数据依赖

- 外部依赖三层：现场数据（IoT/影像/劳务实名）——**明确不做**，智慧工地红海；造价数据（定额库/信息价）——**资·重**，省级且月更，维护承诺比法律 17 张参数表更重，造价审核类场景必须后置；手写签证单 OCR——具·难，但与法律包"OCR 印章/手写"自研节点同一条壁垒线，投入可复用。
- 纯本地起步占比：约 **5/6**（造价审核不做的前提下，签证/索赔/合同/时间线/清单对比均可"拖竣工资料文件夹进来就跑"；但扫描件质量对 OCR 的依赖比前两垂类重一档）。

### 3.5 可行性结论

零 core 上架**基本成立，带两处真实压力**：①gantt 原语挂账需解挂（唯一触碰 18 原语词表边界的垂类，docs/57 挂账理由"多线并行需求出现再评估"在此被触发）；②BoQ 万行量级考验 table 原语性能路径。最小包 ≈ 5–6 场景 / 5 schema。壁垒判断：**数据 + 判断双重**（造价数据资产重、索赔判断专业深）。战略注记：工程索赔本是建工律师业务——本垂类更宜定位为**法律包的场景族延伸**（建工专项，类比 docs/60 劳动专项组合包）而非独立第三垂类，先以 S3+S1+S10 的既有能力吃合同审查与索赔证据链，独立成包等 gantt/BoQ 两块补齐。

### 3.6 竞品一瞥

[广联达](http://zhgd.glodon.com/)以"造价 + BIM + 智慧工地"垄断性覆盖（合同/签证/索赔已在其项目管理套件内），品茗等跟随（[方案盘点](https://blog.csdn.net/qq_35004943/article/details/145243338)）；AI 投入集中在现场监控与算量，**文档侧的"签证索赔证据链装配"无人做带锚结构化**。我方差异：不碰现场与算量，只做"扫描件文件夹 → 带锚索赔证据链 → 待确认计算书"——与广联达是互补切口而非正面竞争。

## 四、泛化可行性矩阵

| 垂类 | 纯本地起步占比 | 声构具资分布 | 零 core 结论 | 建议优先级 |
|---|---|---|---|---|
| 供应商管理 | ≈5/6 | 声 5｜构 1｜具 1（party-verify 复用）｜资 2（均可后置） | **成立** | 中（等外部核验数据线与法律尽调共建后升档） |
| 招投标管理（采购侧） | **7/7** | 声 6｜构 1｜具 1（复用）｜资 2（均可后置） | **成立·最顺** | **高——第三垂类首选**（S9 预言兑现，材料/证据/门禁三语义全同构，政策窗口期） |
| 工程施工管理 | ≈5/6（OCR 依赖重一档） | 声 3｜构 3｜具 1（难·自研节点复用）｜资 1（重·必须后置） | 基本成立（gantt 解挂 + BoQ 量级两处压力） | 低——建议降维为法律包建工专项延伸，独立成包缓议 |

三垂类共同印证 docs/61 的判断：**渲染词表 18 原语零新增诉求**（唯一例外是工程侧触发 gantt 解挂，且 gantt 本在挂账清单而非词表外）；四级分级中"资"全部可后置、"具"全部是既有工具/自研节点的复用——"换垂类=换声明"在三个采购/工程域再次成立，成立强度：招投标 > 供应商 > 工程施工。
