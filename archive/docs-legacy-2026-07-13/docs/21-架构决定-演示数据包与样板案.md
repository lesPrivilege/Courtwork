# 架构决定：演示数据包与样板案（2026-07-09）

状态：已拍板。承接 docs/20（信源分级）与 W5 会话中"demo-fixture 适配器"的拍板，并调整其数据落位。

## 决定

1. **所有演示数据独立成包：`packages/demo-data`（`@courtwork/demo-data`）**，与 src 完全解耦。包内只有数据 + 薄的类型化访问器（typed accessors），不含业务逻辑。此前"fixtures 放 packages/tools 内"的说法作废，tools 只保留 demo 适配器本体，其数据源以注入方式来自 demo-data。
2. **src 只认接口，不认数据**：任何 src 包（tools/core/output/ingest 消费侧）**不得直接 import `@courtwork/demo-data`**，唯一例外是显式的装配点（composition root / 配置装配）。[2026-07-10 澄清，源自 W6 验收观察] 本禁令对象是**生产链路运行时代码**；测试文件（*.test.ts）与 eval 的数据集构建脚本是合法消费方（评测夹具本应取材语料），不受此限。日后接入真实数据与接口 = 换装配点的一行 wiring + 适配器实现，其余板块零改动、各自持续迭代。
3. **演示数据的组织核心是"样板案"**：一套内部自洽的虚构案件贯穿 S1–S4——虚构当事人与企业谱系（供 party-verify 库与 PartyGraph）、一份有纠纷的合同（供 S3 审查与 S4 起草）、配套诉讼文书与证据材料（供 S1 摄取出时间线/图谱）、可复核的引用（真实公开法条，供 cite-check）。演示必须讲**一个连贯的故事**，而不是一堆互不相关的假数据——目标用户不具备拆解能力评估，但对"这东西是不是真的懂我的案子"极其敏感，连贯性就是"看起来真的有用"的来源。
4. **虚构纪律沿用 W5 拍板**：主体全虚构（虚构名称 + 非真实格式信用代码），法条用真实公开文本并注明版本；所有 demo 结果自我标识（`source:"demo-fixture"`），覆盖缺口走 `out_of_coverage`，不许自信否定。
5. **数据形态按消费方分层**：tools 的 fixture 库（JSON）、ingest 的样板卷宗（合成 PDF/图像，初期干净版式，印章/手写样张待 W3 有真实脱敏样本后补）、output 的样例合同 docx、以及一组**预生成的合规 artifact 样例**（Timeline/PartyGraph/RiskList/ReviewMatrix JSON）——后者供 UI 开发与 W7 评测在上游未就绪时先行。
6. **真实数据到位后 demo-data 不退役**：转为新客户试用沙盒、培训素材、UI 开发夹具与 eval 基线，长期随 schemas 演进维护。

## 分期

- 当期（W5 在途）：demo-data 包骨架 + party-verify/cite-check 两个 fixture 库随 W5 落地（W5 范围顺延，数据放新包）。
- W4/W7：样例合同 docx、预生成 artifact 样例按需补入。
- 样板案全量（卷宗级）：与 W3/W8 协同，作为独立小工单排期。

## 不许做

- 不许在任何 src 包内 hardcode 演示数据或直接 import demo-data（装配点除外）。
- 不许用真实企业/人名/案号构造演示数据。
- 不许让演示路径绕过 schemas 校验与信源分级——demo 数据走的管线必须与真实数据完全同一条。

## 终局增补（2026-07-11）：demo 即 onboarding

演示数据的终局不是被清理而是转正为产品导览：正式版默认不注入，空态/上传面显式邀请注入 isDemo 样板案（角标 + 容器隔离 + 装配点边界即既有机制），用户在虚构案件中学会全部操作后归档。样板案的叙事连贯性（docs/21 第 3 条）由此获得第二个消费方：演示之外，教学。
