## 一、发现①:keychain 反复弹窗多次才过——归因成立,H1×H2 叠加

**结论:ad-hoc CDHash 跨 build 漂移(H1,主因)× 双条目结构性双弹(H2,放大器)。两者都有实锤。**

### 证据

1. **H1 漂移直接对照**:FIX-KC-1 验收(0.1.0 时代)记录 CDHash `04d33791…`(ACCEPTANCE.md:597);本轮实测 `/Applications/Courtwork.app`(0.1.1)CDHash `a721141417ac0f11cc084c794f2433052504c12f`,`flags=0x10002(adhoc,runtime)`、`TeamIdentifier=not set`。钥匙串条目 ACL 记的是写入时的应用身份,ad-hoc 下每次出包身份必变 → 0.1.0 写入的条目,0.1.1 每次读取都触发系统授权弹窗。
2. **H2 双条目结构性双弹(代码级)**:macOS 钥匙串仅在**读取 secret data(get)**时执行 ACL 授权;delete/set 不读 data 不弹。而凭证链的两条主路径各含**跨两个条目的两次 get**:
   - `credential_status`(设置页/探针):get `active-source`(lib.rs:533)→ get `provider-secret`(lib.rs:544);
   - `active_secret`(「验证连接」与将来每次真请求):get `active-source`(lib.rs:593)→ get `provider-secret`(lib.rs:595)。
3. **弹窗算术**(旧条目 + 新 build 场景):打开设置页 2 弹 + 点「验证连接」2 弹 = **首轮最少 4 弹**;弹窗点"允许"(单次)不更新 ACL,每次重试再 +2;点"始终允许"本 build 内止弹,但下一个 ad-hoc build 再来一轮。与用户体感"反复弹窗多次才过"完全吻合。
4. save 路径(F2:delete→set 整组重写)不弹,且重写后条目 ACL 归当前 build——save 出口的 reprobe 读新建条目,零弹。

### 用户即时止血(现有能力,无需改码)

**在设置页把 key 重新粘贴保存一次** → F2 强制以当前应用身份重建两条目 → 本 build 内所有后续访问零弹窗。此路径 FIX-KC-1 已交付并验收。

### trace 验证指引(F3 预批门所需)

真机日志目录 `~/Library/Logs/cn.courtwork.desktop/` 不存在——0.1.1 未开 trace,首轮无 log 可读。请照 ACCEPTANCE.md「FIX-KC-1 验收」节真机采集剧本:`COURTWORK_CRED_TRACE=1` 启动 → 走一轮设置页+验证连接 → 回传 `credential-probe.log`。log 中 `op=get` 行的 `account` 字段直接实证两条目各弹,`fail_kind` 记录拒绝/取消(user_canceled=-128 / acl_denied=-25315/-25308)。

### F3(单条目合并)升格候选的处置

预批门为"**trace 证实**两条目各弹"(docs/55 追加①);trace 缺位,但静态分析已证明双弹是**代码结构必然**(上述证据 2)。本会话守门不擅动存储语义,方案先行:

- **形制**:单条目 `service=cn.courtwork.desktop.provider / account=credential`,value 为 JSON `{"source":"pasted","secret":"…"}` 或 `{"source":"environment:NAME"}`;status/active_secret 均一次 get 完成 → 每轮弹窗 2→1。
- **迁移二选一**:a) 首读 NoEntry 时回读旧双条目(一次性再弹一轮)→ 重写单条目 → 删旧;b) **不迁移**——检测到旧条目时走 F5 恢复文案引导用户重存(推荐:零迁移读取、零半迁移状态,语义变更最小)。
- **损益如实**:止血已由"重存 key"覆盖;F3 的边际收益仅在"新 build 首次访问"场景(弹窗 2→1),Developer ID 到位后该场景整体消失。**是否值得在 0.1.x 花一张安全敏感件验收,请架构基于静态证据裁决**:认静态证据即批可开单;否则等 trace 回传再定。

## 二、发现②:API 真请求未成功——`deepseek-v4-flash` 预设疑点**排除**,根因收窄至凭证链

**结论:quirk 预设表没有写错。`deepseek-v4-flash`/`deepseek-v4-pro` 是 DeepSeek 现行官方模型名;修表动作取消。**

### 排除证据(三方一致)

1. **官方文档原文**(api-docs.deepseek.com/quick_start/pricing,本轮抓取):model 参数可用值为 `deepseek-v4-flash`、`deepseek-v4-pro`、`deepseek-chat`(将弃用)、`deepseek-reasoner`(将弃用);旧名弃用时刻 "2026/07/24 15:59 UTC",弃用前兼容映射到 v4-flash 的非思考/思考模式。
2. **docs/18:21 调研注记**(2026-07-10):"旧名将于 2026-07-24 停用,新名 deepseek-v4-flash/deepseek-v4-pro……MVP 接入需直接用新模型名"——预设表照调研落表,链路自洽。
3. **无凭证端点探测**(本轮实测):`POST https://api.deepseek.com/v1/chat/completions` → 401(端点存在,鉴权先挡);`/v1/models` → 401。**`/v1` 前缀兼容仍在,baseUrl 无错**;若路径错会是 404。

注:架构预判"真实 API 模型是 deepseek-chat/deepseek-reasoner"基于旧名时代经验——旧名此刻仍活(还有 12 天),但表里的新名同样有效且更长寿。**运营警报:2026-07-24 后旧名失效**;预设已是新名,无需动作,但用户若在下拉手选了发现列表里的旧名,7/24 后会转 model 型失败。

### 失败根因收窄

冒烟链路(lib.rs:647-688)实现无错:请求体 `model/messages/max_tokens=1/stream=false`(+DeepSeek 为 model_switch,extraBody 为空)对 V4 API 合法;超时 20s;分型完备。排除表错、端点错后,失败最可能环节按概率排序:

1. **凭证链失败**(与发现①同根):弹窗序列中有一次"拒绝"/取消 → get 失败 → 走钥匙串分型,根本没发出网络请求;
2. **key 未通过服务商验证**(401/403):弹窗全过、key 读出,但 key 本身无效/过期/欠费;
3. 网络/超时。

### 定位对照表(请用户对号回传 UI 显示的原句)

| 失败环节 | UI 文案原句 | 分型 |
|---|---|---|
| 弹窗被拒绝/取消 | 需要允许访问安全凭证库才能连接 | user_canceled |
| 钥匙串锁定/密码 | 无法解锁电脑的安全凭证库,请确认钥匙串密码后重试 | auth_failed(钥匙串) |
| ACL 漂移拒访 | 凭证库访问未授权,请重新完成连接;若刚更新过应用,请删除旧凭证后重试 | acl_denied |
| key 未过服务商验证 | 访问凭证未通过服务商验证,请检查后重试 | auth_failed(HTTP 401/403) |
| Base URL 路径 | 服务地址无法完成请求,请检查 Base URL | endpoint(404) |
| 限流 | 服务商暂时限制了请求,请稍后重试 | rate_limited(429) |
| 模型不可用 | 当前模型不可用,请从模型列表选择或手动填写 | model(400/422) |
| 超时/网络 | 服务商响应超时…/暂时无法连接服务商… | timeout/network |

**推荐用户动作序列:先重存 key(一并解决①)→ 点「验证连接」→ 若仍失败,回传上表中命中的原句;命中前三行则连同 trace log 一起回传。**

### [需架构拍板] 观察项(登记不擅动,docs/52 #40/#41)

1. **model_switch 覆盖所选模型**:`resolveReasoningRoute` 对 model_switch 档忽略用户所选 modelId,直接发 quirk 表值(quirk-profile.ts:45)。用户在下拉选了发现列表中的其他模型(如旧名或未来新模型),实发仍是表值——UI 所选与实发不一致,PRV-1 拍板的映射语义 vs 用户选择权,契约级。
2. **V4 推理路由形态再评估**:官方 V4 示例携带 `thinking: {"type":"enabled"}` 与 `reasoning_effort` 请求字段,提示 DeepSeek 推理开关已有 request_field 形态;现行 model_switch(flash/pro 档位切换)仍有效但语义或非官方本意(flash/pro 更像能力档位)。是否改 quirk 声明为 request_field 型,契约级(docs/18 增补 + PRV-1 映射表变更)。
