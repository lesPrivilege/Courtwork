# 调研报告：Agent 记忆系统与用户偏好个性化（2025 下半年–2026）

状态：调研完成，供 Stage 1 用户层偏好设计与 packages/core / packages/schemas 后续 TODO 参考。内部对齐口径，2026-07-09。

## 零、结论先行

Courtwork 已经拥有一套记忆底座：**案件层记忆**——CaseFile/Timeline/PartyGraph/RiskList 等 schema 化 artifact，配合 RevisionEvent 修正流水，构成"卷宗即记忆"。跑遍 2025 下半年到 2026 的开源记忆件（mem0/Letta/Zep-Graphiti/LangMem/A-MEM）与消费级产品记忆（ChatGPT/Claude/Gemini/豆包/Kimi/元宝）之后，判断是：**这套底座在架构上已经优于直接嵌入通用记忆中间件**，因为通用记忆件的核心卖点（LLM 自主判断记什么、何时更新、何时遗忘）恰恰是 Courtwork 的防呆纪律要规避的不确定性来源。真正该补的缺口是**用户层偏好**——从 RevisionEvent 里的重复修正模式中提炼可复用规则，这一层现在完全空白，且是 Stage 1"从工具变成长在工作流里"命题的直接抓手。机构层留给 Stage 2，现在只需要在 schema 上不堵死"利冲隔离维度"这一件事。

三层记忆提案、单窗折中、sleep-time 早做与否的分级结论见第六节，可直接跳读。

## 一、开源记忆架构盘点

### 1.1 mem0——事实抽取 + LLM 自主判定的三态操作

mem0 的核心是一条抽取管线：每一轮对话消息被蒸馏成原子化的自然语言"事实"（memory），而非存储原始对话块，随后进入 Add/Update/Delete 三态判定——检索出相似历史记忆后，**由 LLM 通过 tool call 自主决定新事实是新增、覆盖旧值、还是删除冲突项，mem0 本身不加任何产品级编排**。存储层是向量库 + 知识图谱（Pro 版）的双栈架构，支持 Qdrant/Chroma/Milvus/pgvector/Redis。遗忘机制分时间衰减、频率衰减、重要性驱动三类。2026 年 1 月发布的 1.0.3 版新增了按项目配置的 inclusion/exclusion prompt 与 memory depth，把行为配置从全局下沉到项目级。License 为 Apache 2.0，可自托管商用；~4.8 万 GitHub star，2025 年 10 月完成 2400 万美元 A 轮。

关键判断：mem0 的"LLM 自主拍板 ADD/UPDATE/DELETE、无产品级编排"这一设计恰好是 Courtwork 不想要的——法律场景需要每条记忆变更都可解释、可追溯到具体触发事件，而不是模型自由裁量后的黑箱结果。

### 1.2 Letta（原 MemGPT）——OS 隐喻 + sleep-time compute

Letta 把记忆当操作系统资源管理：主上下文是 RAM，归档记忆是磁盘，agent 自主管理内存分页。2025 年 4 月的论文提出 **sleep-time compute** 范式：与其把记忆整理塞进对话轮次里拖慢响应，不如让 agent 在"睡眠"（无用户输入的空闲期）里后台重写记忆状态。落地形态是 MemGPT 2.0 里的双 agent 结构——主 agent 负责对话，不持有改写核心记忆的工具；**sleep-time agent 异步跑，专门负责整理、压缩、去噪主 agent 的记忆块**，可配置比主 agent更强、更慢的模型（因为不受延迟约束）。效果是把记忆维护的计算量从高延迟的用户交互期挪到本来就空闲的时段，论文证明这是帕累托改进（性能不降、成本更优）。License Apache 2.0。

关键判断：sleep-time compute 的价值前提是"记忆条目持续增长到需要定期整理的规模、且高频交互"，Courtwork 现阶段案件量和对话频率远达不到；更重要的是，"LLM 在无人监督时自主重写记忆状态"与防呆三原则里的"留人确认"存在直接张力——除非给 sleep-time agent 的输出也套确认门禁，但那样它"异步不打扰用户"的核心卖点就被削弱了。

### 1.3 Zep / Graphiti——双时态知识图谱，"关闭旧区间"而非删除

Graphiti 是 Zep 的核心引擎，开源、2 万+ star、Apache 2.0，是当前最主流的时序知识图谱记忆方案。核心特性是**双时态模型（bi-temporal）**：每条图边同时记录"事件发生时间"和"信息摄入时间"，并携带显式的有效区间。当新信息与已有事实冲突时，**Graphiti 关闭旧事实的有效区间窗口而不是删除它**，agent 推理时用当前状态，但历史全程可查询、可回放到任意时间点。企业级 Zep Cloud 在此之上跑 Context Graph Engine，号称百万级图谱、p95 检索小于 200ms。在 LongMemEval（当前记忆系统的事实标准评测）上，Zep/Graphiti 用 GPT-4o 拿到 63.8%，显著高于 mem0 的 49.0%，尤其在"这件事在某个时间点是否成立"类查询上优势明显。

关键判断：这个"关闭区间、不删除"的语义**正是 RevisionEvent 该有的形状**——RevisionEvent 目前已经记录 previousValue/newValue，本质上是这个模型的手工简化版。不需要引入 Graphiti 本体（引入一整套时序知识图谱库对 MVP 规模是过度工程），但设计"同一字段被反复修正"的历史链查询接口时，应该直接照抄这个思路：返回"当前有效值 + 按时间范围查历史值"，而不是任何形式的删除语义。

### 1.4 LangMem——三分记忆类型，功能原语而非托管服务

LangMem 是 LangChain 生态的长期记忆 SDK，提出语义记忆（事实）、情景记忆（具体经历）、程序性记忆（学到的行为规则/prompt 调优）三分类，可脱离 LangGraph 独立使用，也有原生存储层集成。截至 2026 年 6 月仍活跃维护，但最新 PyPI 版本停在 0.0.30（2025 年 10 月），未到 1.0。它更像一套"记忆类型词汇表"而非成熟产品，对 Courtwork 的参考价值主要是**分类思路**：程序性记忆（学到的行为规则）这一类目与"用户层偏好"高度对应。

### 1.5 A-MEM——Zettelkasten 式动态记忆网络

NeurIPS 2025 论文，借鉴卡片盒笔记法，让记忆条目动态互相链接、索引；新记忆进入时可以触发已有记忆的上下文表征更新，形成持续演化的知识网络，在六个基础模型上的长对话任务中优于当时的基线。这是一种更强调"记忆间关联演化"的方案，复杂度对 Courtwork 当前规模偏重，暂列为观察对象。

### 1.6 其他值得记录的新作

Supermemory（MIT 协议，单二进制本地跑、声称在 LongMemEval/LoCoMo/ConvoMem 三个评测集第一）和 OpenMemory（Apache 2.0 的本地 MCP 记忆服务，CaviraOSS 版本仍活跃；mem0 官方那支 OpenMemory MCP 已在收缩，官方转推荐直接用 mem0 自托管）代表了"本地优先、MCP 协议化"的轻量记忆件趋势，license 都宽松、可嵌入，但功能定位仍是通用记忆层，判断同 1.1。

### 1.7 全景对照表

## 二、产品级记忆 UX：用户如何感知"它记住了什么"

### 2.1 ChatGPT——从可见列表到不透明后台合成，正是要避免的反面案例

ChatGPT 记忆分两层：**"saved memories"**（显式、用户可编辑删除的事实列表）和 **"reference chat history"**（隐式，从历史对话模式中召回，2025 年 4 月上线）。2026 年 6 月 4 日起，OpenAI 推出 **Dreaming V3**：用一个读取多年历史对话的后台合成过程取代人工维护的显式列表，**在用户不知情、不确认的情况下自动更新对用户的画像**。2026 年 CHI 会议一篇论文把这个现象总结为"个性化-便利悖论"：用户最看重的功能恰恰是用户最无法审计和约束的功能。与此同时，已披露的问题包括：2024 年测评中记忆的事实召回准确率仅 41.5%，错误信息一旦进入记忆会以与事实同等的置信度被复用，"记忆摘要页"官方承认不显示全部留存内容，且对话删除后记忆层的数据仍可能保留。

关键判断：Dreaming V3 这条路线——**后台不透明合成、用户看不到全貌**——正是法律场景必须规避的反面案例，直接违背"记忆可见可审计可删除"的律师职业要求。

### 2.2 Claude——四类目显式记忆 + 分区隔离 + 即将上线的结构化 Memory Files

Claude 记忆在 2026 年 3 月起对所有用户（含免费层）默认开启，自动总结跨会话上下文；官方界定四类留存内容：陈述过的格式偏好、工作/学习背景（编程语言、项目目标）、以及其他个性化细节。**Projects 与常规对话的记忆是隔离的**——只有 standalone 对话汇入通用记忆，Project 内对话有独立记忆空间，这是一种天然的"案件级隔离"雏形。2026 年 5–7 月披露的下一步是双模式 **Memory Files**：允许用户在当前单一记忆模式和更细粒度的、按主题/项目/上下文拆分成多个结构化文档的模式之间选择——这比单一 blob 更接近"可审计、按边界组织"的方向。7 月新增的 Settings > Reflect 月度回顾，把"最近用了 Claude 做什么、活跃时段、使用模式观察"直接呈现给用户，是一种建立信任的透明化设计。

关键判断：Claude 的"Project 隔离记忆空间"与"Memory Files 按主题/项目拆分"两个设计方向，与 Courtwork"案件是记忆边界"的判断是同构的，可以作为佐证。

### 2.3 Gemini——选择性接入而非全量摄取，附带记忆可迁移

Gemini 的 Personal Intelligence（2026 年 1 月上线）让模型可以跨用户的 Google 应用（邮件、照片）检索，但**默认关闭，用户逐个授权连接的应用**；官方强调是"按需选择性调取相关信息回答，不是把全部数据摄入模型权重或训练"。2026 年 3 月上线的 Import Memory 功能允许用户把 ChatGPT/Claude 等其他平台的记忆一次性搬进 Gemini——这是对"记忆锁定单一供应商"批评的直接回应，释放了记忆的可迁移性诉求。

### 2.4 国内单窗产品：显式指令 + 隐式提炼 + 可见删除列表，是行业收敛的低认知范式

豆包、Kimi、元宝三家的记忆设计高度收敛为同一套三件套：
- **隐式自动提炼**：对话中判断出对未来回复有帮助的信息，自动存为记忆并随对话推进刷新，用户无感知、无需操作；
- **显式自然语言指令**：用户直接说"记住我喜欢……"或"忘记……"即可写入/删除，零 UI 学习成本，语音指令即操作；
- **可见可管理的记忆列表**：统一在"记忆空间/个性化设置"页面展示全部条目，支持单条删除、一键清空，且明确声明"不会用于模型训练，随时可关闭"（Kimi 的措辞）。

元宝额外提供**临时会话模式**：敏感对话不进入历史/不沉淀记忆，是"案件级隔离"在 C 端产品里的朴素对应物。豆包把 256K 长上下文与记忆功能分开陈述，二者不是同一回事——长上下文解决"这次对话别断"，记忆解决"下次还记得"。

关键判断：这套"隐式提炼 + 显式指令 + 可见删除列表"三件套，是国内产品在"不让用户写 prompt"约束下收敛出的通用答案，**可以直接借用于 Courtwork 轻对话模式的记忆 UI**，不需要重新发明。但"隐式提炼"这一半必须被法律域的可审计要求改写——见第六节。

## 三、不让用户写 prompt 的个性化：从行为学习偏好

### 3.1 PRELUDE——从编辑动作学习潜在偏好

PRELUDE（PREference Learning from User's Direct Edits）框架的思路是：观察用户对 AI 输出的直接修改（跨多个写作任务），学习其潜在偏好，目标是让未来的输出减少编辑量。这与 Courtwork RevisionEvent 的设计假设几乎一模一样——fieldPath + previousValue + newValue + reason 本身就是"直接编辑"的结构化记录，天然是这类偏好学习的原始输入，不需要额外埋点。配套的 NeurIPS 2024 论文（Ge Gao 等，"Aligning LLM Agents by Learning Latent Preference from User Edits"）证实了这条路径的可行性。

### 3.2 GhostWriter——隐式风格学习 + 显式标注旋钮并存

GhostWriter 让 AI 隐式学习作者写作风格的同时，保留显式手动标注的控制权，兼顾个性化与用户能动性（agency）。这提示一个设计原则：**不要只做隐式学习，要留一个显式微调的旋钮**，即便这个旋钮的操作成本极低（比如从几个候选风格里选一个，而非写自由文本）。

### 3.3 一个必须诚实面对的限制：LLM 模仿隐式个人文风的能力仍然有限

2025 年的研究（"LLMs Still Struggle to Imitate the Implicit Writing Styles of Everyday Authors"）明确指出，个人风格往往细微而隐性，仅从少量样本里模仿一个人的写作风格，当前 LLM 表现并不可靠。这为 Courtwork 的"文风画像"类野心设了一个现实边界：**不要期待"隐式风格迁移"自动work**，更稳妥的落地是把学到的偏好表达成可读、可审计的显式规则条目（"在起诉状事实部分偏好短句陈述、避免从句嵌套"），而不是训练一个隐式的"文风向量"去悄悄影响生成——后者既不可审计，效果也未必可靠。

### 3.4 低认知的偏好采集：选项优于自由文本

研究证实，让用户从预设选项里选择比要求其准确描述偏好的认知负荷低得多——这也是"选项卡/示例选择"能替代 custom instructions 自由填空的依据。对 Courtwork 而言意味着：用户层偏好规则候选生成后，确认界面应该是"是/否 + 可选调整项"的轻量交互，而不是让用户重新用自然语言描述一遍偏好。

### 3.5 一个必要的警示：记忆更新的"应用缺口"

2026 年的 Supersede 论文（"Diagnosing and Training the Memory-Update Gap in LLM Agents"）指出一个常被忽视的问题：即便正确捕捉到了用户的修正信息，模型在后续生成时也不一定会可靠地应用这条更新——记忆写进去了，不代表下次真的会被正确用上。这提醒 Courtwork：把 RevisionEvent 聚类成偏好规则只是第一步，规则如何在场景执行器的生成节点里被显式注入（而非寄望于放进 context 就自动生效）需要单独的工程设计，不能假设"记住了=会用上"。

## 四、专业工具的记忆合规：律所 ethical wall 对记忆系统意味着什么

### 4.1 Harvey：Matter 是记忆的原子边界，fail-closed，审计要经得起法庭质证

Harvey 2026 年 3 月的博客《Long Horizon Agents and Ethical Walls》是目前对"AI 记忆 × 律所信息隔离墙"讲得最系统的公开材料，核心论点直接可用：

- **传统访问控制在人速操作下成立，在 agent 规模下失效**：律所现有的伦理墙靠 Intapp（利冲检索）+ iManage/NetDocuments（文档级访问控制）+ 物理/行政隔离实现，边界单位是"人访问一份资源"。但 agent 自主决定读什么、且能跨会话维持上下文，意味着"agent 越权读取一份墙后文档"这个违规动作，即便律师本人从未看到结果，违规也已经发生。
- **长时程 agent 的核心新增风险是跨时间的上下文污染**：在 Matter A 上跑过的 agent，如果同一上下文被后续用于 Matter B（且二者之间存在伦理墙），历史上下文本身就构成信息泄漏，即便两次任务的调用方是"同一个 agent 实例"而非同一个人。
- **解法是 fail-closed，不是 fail-open**：agent 无法确认某文档是否在当前 matter 边界内时，必须跳过并标记不确定性，而不是像传统 DMS 常见的做法那样"默认放行、事后记录待查"。
- **Matter 必须是产品的原子隔离单位**，而不是一个元数据标签——每个 agent、每份文档、每次对话、每份产出都要 scope 到具体 matter，agent 运行时只能访问该 matter 边界内的资源。
- **审计轨迹要经得起法庭质证**：不仅要证明律师本人被隔离，还要证明代表该律师行动的 AI 也被同等隔离；每次文档检索、每个上下文窗口、每个 agent 会话都要留痕到能在事后证明"没有访问墙后信息"的粒度。
- **不要重复造轮子**：正确做法是深度对接律所已有的利冲/DMS 基础设施（如 Intapp、iManage），继承其墙配置，而不是自建一套平行的隔离系统。

### 4.2 对 Courtwork 的直接映射

Courtwork 当前"案件文件夹 = 一个 CaseFile"的架构，天然就是 Harvey 说的"matter 是原子隔离单位"——这一点上 MVP 的架构判断是对的，不需要改。**真正的风险点在用户层偏好和未来的机构层知识**：如果直接把 RevisionEvent 的 previousValue/newValue 原样搬进"跨案件生效"的用户层记忆，这些字段可能携带具体案件的实质内容（金额、当事人、条款原文），那么用户层记忆本身就变成了信息穿透案件边界的渠道——即便技术上只在"同一律师账号"下生效，也违背"案件是记忆边界"的产品纪律，且当同一律师未来在有利冲关系的案件间切换时，这层"个人记忆"会成为事实上的隐性伦理墙漏洞。这不是危言耸听式的合规问题，而是当前 RevisionEvent schema 已经具备的字段（含 fieldPath/previousValue/newValue 这类可能载有实质内容的通用字段）decides 了这一层设计必须显式处理，见第六节的脱敏边界设计。

### 4.3 国内律所现状与 HIPAA 的旁证

国内调研显示 60% 律所缺乏明确的 AI 应用指引，利冲检索依赖客户数据库匹配（得理等厂商的产品方向），本地化部署被反复强调为数据不出所的优先选项；从业者的一线共识是"不要相信 AI 的记忆，要相信原始卷宗"——这恰好印证了 Courtwork"卷宗即记忆、结构化再水化而非聊天摘要"的架构判断本身就是对的，是这次调研里少有的"现有设计已经比通用做法更审慎"的正向确认。医疗领域 HIPAA 对 PHI 的要求（加密存储、访问控制、操作性访问与合规性访问要分别留存、不同保留期）提供了另一个可直接借鉴的模式：RevisionEvent 的审计日志设计可以照此区分"日常调试访问"与"合规审计访问"两类日志留存策略。

## 五、Courtwork 三层记忆对照评估（报告核心）

### 5.1 案件层（已有，无需引入通用记忆件）

现状：CaseFile/Timeline/PartyGraph/RiskList artifact + RevisionEvent 构成的结构化记忆，写入时机是**同步、随留人确认发生**——这与所有通用记忆件的"后台异步抽取"范式相反，但恰好吻合防呆三原则①分层确认：写入动作本身就是确认动作，不存在"抽取是否准确"的二次风险。

与 Graphiti 的对照：法律执业留痕要求全量可溯（含被否决的历史值），不能"遗忘"只能"修正"——这比 mem0 的 ADD/UPDATE/DELETE 三态更接近 Graphiti 的"关闭旧区间、不删除"语义。建议：RevisionEvent 目前已经具备 previousValue/newValue 字段，未来如果要支持"同一字段被多次修正"的历史链查询，直接照抄 Graphiti 思路（返回当前有效值 + 按时间范围查历史值），不需要引入图谱库本体。

结论：**不引入 mem0/Letta/Zep 任何一个通用记忆中间件到案件层**。这些框架的核心操作靠 LLM 自主判定，而 Courtwork 案件层的受众窄、字段已知、写入时机已经锚定在确认动作上，通用记忆件反而会引入不必要的不确定性，与"生成与确定视觉区隔"纪律冲突。

### 5.2 用户层偏好（新，Stage 1 做）

**来源**：不需要额外埋点，RevisionEvent 本身就是 PRELUDE 式偏好学习的原始输入。

**写入时机**：不做同步抽取（每次修正都问"要不要记住"打断心智，且与"不写 prompt"的产品调性冲突）。借鉴 sleep-time compute 的"后台批处理"思路，但做成轻量版——不需要常驻的"随时苏醒"agent，挂在既有产品节点上做规则聚类即可（案件归档时、或固定周期批处理任务），比如识别"该律师过去 N 次审查里反复把违约金上限从 20% 改成 10%"这类重复模式。

**写入形态**：不是自由文本 memory blob，落地为 schema 化的"偏好规则候选"——字段类型 + 触发条件 + 默认修正 + 关联的 RevisionEvent 溯源集合。这样天然满足可审计（每条规则都能回溯到产生它的具体修正事件），也规避第三节确认的"隐式风格迁移不可靠"问题——规则是显式、可读的，不是一个隐式的"风格向量"。

**确认门禁**：候选规则不自动生效，必须过一次性确认（"发现你总是这样处理，以后默认这样，确认吗？"），且规则候选作为 LLM 聚类的生成物，需要按防呆三原则②的视觉区隔规则明确标注为"待确认"，不与 schema 化确定产出混淆。

**可见可删界面**：直接借用豆包/Kimi 的"记忆空间"范式（列表 + 单条删除 + 一键清空），但比消费级产品多两个法律场景特有字段：①这条偏好从哪些案件的哪些修正学来的（溯源，复用 SourceAnchors 的形状）；②这条偏好的生效范围是仅本案件还是跨案件（这是利冲边界问题，见下）。

**利冲边界（对照 Harvey）**：RevisionEvent.previousValue/newValue 可能携带具体案件的实质内容（金额、当事人、条款原文），不能把这些内容原样搬进跨案件生效的用户层记忆——那样用户层记忆本身就成了信息穿透案件边界的渠道。设计上必须在写入用户层前插入一个**显式的脱敏/泛化生成节点**：只沉淀"流程/风格类偏好"（如"审查合同偏好保守表述""该律师习惯把利息条款单独成段"），过滤掉实质性当事人信息与具体数值/条款原文；这个脱敏节点的输出本身也应该留人确认——不能因为是"内部技术步骤"就跳过防呆纪律。

### 5.3 机构层（Stage 2，现在只需预留接口）

对照 Harvey 的教训：机构层知识（本所历史案卷、文书先例索引）在数据模型上必须显式维护"利冲隔离组"这一维度，不能只有 case/user/org 三层扁平结构——需要在 org 层预留 conflict-group/wall 标记字段，未来对接律所已有的利冲检索系统（如得理）而非自建平行系统。Courtwork 当前 schemas 未涉及此维度，属于 Stage 2 前必须补的 TODO，现在只记录判断、不动 schemas。团队级图谱库资产化（docs/04 Stage 2）落地时，还要重新评估"用户层偏好是否团队共享"——如果共享，脱敏边界要求比个人版更严格，需要把"个人偏好"和"团队 knowledge"分层，不能合并成一层。

### 5.4 轻对话模式（Stage 1）的记忆与上下文策略

**记忆 UI 借用行业收敛范式**：豆包/Kimi/元宝的"隐式自动提炼 + 显式自然语言指令 + 可见删除列表"三件套可以直接套用，用户侧交互成本极低。但"隐式提炼"这一半必须被法律域的可审计要求改写：**不做 ChatGPT Dreaming V3 式的完全不透明后台合成**——CHI 2026 论文点名的"个性化-便利悖论"正是要规避的反面案例。折中方案：轻对话模式的记忆条目统一来自与用户层偏好同一套显式聚类流程产出的、用户可见的规则列表，不做无法审计的"隐式画像常驻后台改写"。

**上下文管理与记忆是两个不同层次的问题，UI 上要分开**：Claude Code/Copilot Chat 那类"老对话摘要化、近期逐字保留"的 compaction 机制解决的是"这一次单窗对话别爆掉"，与"下次对话还记得住什么"的用户层偏好记忆是两件事。很多国内产品把两者混为一谈，导致用户分不清"删记忆"和"清空对话"的区别。建议 Courtwork 在轻对话模式的 UI 上明确拆成两个入口："清空本次对话"（对应 compaction/重开一轮）与"清除记住的偏好"（对应用户层记忆的可见可删界面）。案件工作台内的 session 续行（packages/core/SPEC.md 已规划）走的是"结构化再水化"而非聊天压缩，不受这条影响，只有轻对话模式因为没有 artifact 可挂，才需要这套机制。

### 5.5 sleep-time 式后台整理：不早做

结论：**不值得早做**，理由三点：①sleep-time compute 的价值前提（Letta 论文）是记忆条目持续增长到需要定期整理的规模、且交互频率高——Courtwork MVP 阶段的案件数与对话频率远达不到这个量级，过早引入后台 agent 本身是运维负担（额外调度、额外模型调用预算、额外失败模式），边际收益很低；②"LLM 在无人监督时自主重写记忆状态"与防呆三原则的"留人确认"直接张力——除非给 sleep-time agent 的输出也套确认门禁，但那样其"异步不打扰用户"的核心卖点就被削弱，等于自己否定了引入它的理由；③一个更朴素的替代方案已经够用：批处理规则聚类完全可以挂在明确的产品事件上（案件归档时、每 N 次修正后的定时任务），不需要做成通用的"随时后台苏醒"agent。Stage 2 视用户层偏好的规则聚类规模再评估是否需要更重的自动化，且即便做，也应该保留"聚类候选 + 强制确认"的形态，不做真正自主的"做梦时静默改写"。

### 5.6 最值得借的开源件（设计形状，非代码依赖）

不建议直接嵌入 mem0/Zep/Letta 任何一个通用记忆中间件本体——理由如 CLAUDE.md 既定的依赖方向判断：这些框架的核心操作（LLM 自主判定 ADD/UPDATE/DELETE、无产品级编排）恰恰是 Courtwork 想规避的不确定性来源。但两个设计形状值得借（不引入代码依赖）：
1. **Graphiti 的双时态"关闭旧区间、不删除"模型**——RevisionEvent 已经是这个模型的手工简化版，未来支持历史链查询时直接照抄这个思路即可。
2. **mem0 的 Add/Update/Delete 语义分类法**——可以借来作为用户层偏好规则候选生成节点内部表达"这条新规则和已有规则冲突，是否覆盖"的分类词汇，但决策权收回给用户确认，不像 mem0 那样让 LLM 自主拍板。

### 5.7 分级结论

- **MVP（Stage 0）不做**：案件层记忆（artifact + RevisionEvent）已是足够的记忆底座，唯一动作是确保 RevisionEvent 字段完整、写入时机锚定在确认发生的同步节点——packages/core/SPEC.md 已覆盖。
- **Stage 1 做**：用户层偏好——① RevisionEvent 批量聚类生成候选规则（挂在案件归档/固定周期任务上，不做常驻 sleep-time agent）；② 脱敏/泛化生成节点（案件实质内容与流程偏好强制分离，输出需人工确认）；③ 记忆空间可见可删 UI（借豆包/Kimi 范式：列表 + 溯源 + 单条删除 + 清空）；④ 轻对话模式复用同一套用户层偏好记忆，不做 ChatGPT Dreaming V3 式不透明合成；⑤ 单窗对话内的 compaction 作为独立机制单独实现，UI 明确区分"清空对话"与"清除偏好"两个入口。
- **Stage 2 做**：机构层知识——预留 conflict-group/利冲隔离维度的 schema 字段（对照 Harvey matter-centric isolation 与 Intapp/iManage 集成教训：接入律所已有利冲系统而非自建平行系统）；团队级图谱库资产化时重新评估用户层偏好的团队共享边界。
- **sleep-time 式后台"做梦"整理**：不早做，Stage 2 视聚类规模决定是否需要更重的自动化，且即便做也保留强制确认，不做真正自主的静默改写。
