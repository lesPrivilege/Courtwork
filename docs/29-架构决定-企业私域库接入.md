# 架构决定：企业私域库接入（2026-07-10）

状态：已拍板，排期 MVP 后（通用适配器契约随 Stage 1 客户需求启动）。依据 docs/16 调研。

## 四块分工（每块干什么）

- **库 = 适配器（数据在哪）**：tools 契约下的 B 级检索适配器（sourceId 自标识、out_of_coverage、不许自信否定，全部继承）。**双接入形态**：实时检索端点 或 批量快照上传（国内律所多无 API，快照是现实主流）。
- **schema = 条目形态（数据长什么样）**：最小条目 schema `{id, content, sourceRef, finality, updatedAt}`——**`finality`（草稿/终稿/已作废）为必填**，防同库脏数据无差别检索（GIGO 教训）。artifact 侧的结构化引用（internalRuleRef 类）届时走 schemas 提案。
- **路由 = retrievalPolicy（什么场景用哪个库到什么槽）**：租户级策略覆盖层，注册表 v2 承载；改声明不改代码。
- **skill/场景 = 工作流（拿数据干什么）**：企业合规审查 = 场景变体（企业 playbook + 企业库进声明），docs/24 纪律不豁免（补产出 schema 与门禁）。

## 注册与披露

- 租户注册 = 声明文件（YAML/JSON）+ per-tenant 装配点读取（docs/21 模式推广）；管理界面 W9 后。
- 分层披露：库元数据（名称/范围/覆盖声明/索引摘要）常驻 → 检索按需 → 原文按 SourceAnchor 取。
- 利冲/租户边界**透传企业侧既有标记**（Intapp/iManage 类），不自建判断逻辑。

## MCP 取舍

MVP 不做 MCP 化（单一消费方，收支平衡点约 3 个消费方未触及）；注册表 v2 后加一种 MCP 适配器实现（对标 Harvey 接 iManage 的模式）——届时它只是适配器三分（mock/demo-fixture/真实）之外的第四种实现，契约不动。

## 分期

MVP 后首期：通用适配器契约 + 三种实现（demo-fixture / http-endpoint / snapshot-upload）。延后：MCP 适配器、管理界面、更新通知机制。
