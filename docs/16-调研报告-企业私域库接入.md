# 调研报告：企业私域知识库接入 agent 产品的范式（2025–2026）

状态：调研稿，供租户级库注册机制设计参考。日期：2026-07-10。

## 〇、背景与调研问题

docs/20（信源分级与检索策略）已拍板：B 级 = 自建/演示库，携带 `out_of_coverage` 覆盖边界语义；场景注册表 v2 将引入 `retrievalPolicy` 字段，声明本场景允许的工具集与各信源等级可流入的产出槽位。docs/21 确立装配点模式：src 只认接口不认数据，真实数据接入 = 换装配点的一行 wiring。docs/24 明确场景是声明式固定编排，skill 兼容不豁免"产出 schema + 确认门禁"两个强制字段。docs/25 预留机构层 `conflict-group` 维度字段，利冲隔离对接律所既有系统（Intapp/iManage）而非自建。

本报告要回答：当 Courtwork 从"演示库"走向"企业接入自己的私域知识库"（律所的内部审查规则、合同台账、历史 playbook）时，注册机制应该长什么样——企业库在架构上算 tools 契约下的哪一类适配器、条目 schema 定在哪层、路由靠什么字段、MCP 要不要支持、租户注册的物理配置形态是什么。**本报告不改架构决定，只验证/挑战架构预置并为落地提供依据。**

调研方式：WebSearch 中英文合计 24 次，覆盖 MCP 行业现状、法律 AI firm knowledge 接入先例、多租户隔离、检索质量工程、契约设计五个方向；对 Harvey 官方 Connector Library 博客、Draftwise "Garbage In Garbage Out" 精选文章做了 web_fetch 全文核对。

## 一、MCP 作为行业标准的现状

### 1.1 采用度：2026 年已是跨厂商默认协议，但仍在快速变形

MCP 由 Anthropic 于 2024 年 11 月推出，2025 年 12 月捐赠给 Linux Foundation 旗下的 Agentic AI Foundation，到 2026 年已成为 OpenAI、Google、Microsoft、Salesforce、Snowflake 等跨厂商默认支持的协议，SDK 月下载量约 9700 万，公开 server 数超 5800 个。Fortune 500 企业 18 个月内落地率约 28%，78% 的企业 AI 团队已有 MCP 支撑的 agent 在生产环境跑。企业级授权（Enterprise-Managed Authorization）扩展已转正为 stable，被 Anthropic、Microsoft、Okta 采纳，允许企业通过自己的身份提供商集中管控 MCP server 访问权限——这是"MCP 能不能过企业 IT 安全评审"这道门槛在 2026 年基本迈过去的信号。

但协议本身仍在演进：2026-07-28 的规范候选版本引入了无状态协议核心、Extensions 框架、Tasks、MCP Apps、鉴权加固与正式的弃用策略，说明这不是一个已经定型可以照单全收的标准，接入时要预留跟版本走的成本。

### 1.2 三原语与渐进式披露：resources 与 tools 的分工，以及上下文经济学

MCP 的核心区分是**交互模型**而非能力边界：tools 是"模型控制"的——模型基于对话上下文自主发现和调用；resources 是"应用控制"的——宿主应用决定何时以及如何把哪些数据塞进上下文，本质是只读的结构化数据供给（文件、schema、参考资料），不执行动作。经验法则：**模型需要自主决策时用 tool，需要限定范围的参考数据、避免上下文膨胀时用 resource**。

2026 年生产环境 MCP 设计的默认做法是渐进式披露（progressive disclosure）：server 先只暴露极简的工具描述，模型需要时再按标准化的 resource 模式取完整文档/schema，实测能把初始上下文占用降低 80%–90%；Anthropic 的"代码执行 + MCP"模式把同一任务的输入 token 从约 15 万降到约 2000（降幅 98.7%）。这与 docs/20 拍板的"分层披露"设想（库元数据常驻 → 检索按需 → 原文按锚取）在设计动机上完全一致——MCP 生态已经把这个模式沉淀成协议层惯例，不是 Courtwork 独创，可以直接对齐其命名与形状，降低未来对接 MCP 生态的翻译成本。

### 1.3 企业自建 MCP server 暴露内部库：已是成熟套路

多个参考实现（企业知识库列表/检索、Docling 解析 DOCX/PDF/PPTX 后分块建混合索引再包成 Remote MCP Server 等）显示企业自建"内部知识库 MCP server"已经模式化：server 位于 AI 客户端与内部系统之间，负责工具发现（告知模型有哪些操作可用）与参数校验；关键工程难点不是协议本身，而是"设计出让模型能用、又不会越权泄漏数据的正确抽象"——认证、多租户、权限范围裁剪、健康检查与监控才是真正的工作量所在。

### 1.4 MCP 适配器 vs 自有契约包一层：各自代价

行业判断的分界点很朴素：**如果同一套工具要被多个客户端复用，或要作为可复用的内部平台/对外开放给别人的自有 assistant 调用，MCP 化收益明显**（一次更新、多方受益，鉴权/错误响应/发现逻辑标准化）；**如果只是单一产品内部对接一两个已知端点，直接写确定性接口反而更简单、更好调试**，行业估算的收支平衡点大约是"3 个消费方"或"2 个消费方用满 6 个月"。业界总体判断仍是"对约 90% 的团队，直接写进代码的自定义 API 集成才是对的选择，MCP 的标准化收益只在多消费方场景显著"。

**对 Courtwork 的含义**：`packages/tools` 已经是"确定性接口 + 三种适配器（mock/demo-fixture/真实骨架）"的自有契约设计，这与"MCP 适配器"并不冲突——**MCP 可以作为企业库的第四种适配器实现方式（`sourceId` 例如 `mcp:<server-id>`），而不必让自有契约整体迁移成 MCP**。理由对齐上面的收支平衡点：Courtwork 当前只有一个消费方（自己的 core 编排器），MCP 化整个工具层收益不明显；但**企业客户一侧很可能已经自建或将要自建 MCP server 暴露 DMS/知识库**（Harvey 的案例证明这是律所生态的现实选择，见下节），此时"支持把一个 MCP server 接成一种 `party-verify`/企业库检索适配器"是低成本高杠杆的动作——复用企业已有的鉴权与发现基础设施，Courtwork 不用重新发明连接器。这是对架构预置"企业库 = tools 契约下的检索适配器"的**验证而非挑战**：调研没有找到任何证据要求把 tools 契约本身改造成 MCP-native，MCP 更适合定位为一种可选的传输/发现层，包在现有适配器抽象之下。

## 二、法律 AI 的 firm knowledge 接入先例

### 2.1 Harvey Connector Library（2026-06 发布，Early Access）：原生 API + MCP 双轨

Harvey 的做法是官方对本报告问题最直接的先例，全文核对如下：

- **连接器分两类**：原生 API 集成（Gmail、Google Drive、Outlook、SharePoint）"行为更结构化、能力更深入，支持更精准的工作流"；**MCP 集成**（iManage、NetDocuments、PitchBook、SS&C Intralinks DealCentre AI、Datasite、Box）——Harvey 官方原话是"MCP 连接器带来速度与规模：供应商能更容易地构建和增强集成，管理员能设置工具治理策略，用户能在工作流中跨多系统获得洞察"。**这印证了上一节的判断：MCP 被 Harvey 用在"多个第三方系统、各自独立演进"的场景，而不是自家核心工具契约。**
- **索引 vs 实时检索**：iManage/NetDocuments 的集成方式是"检索并引用工作区内容，文档留在原地，全程保持权限感知"（search-and-reference with full permission-awareness），不是把客户全部文档搬进 Harvey 自己的索引——这是律所客户对数据主权高度敏感的必然选择，与 Courtwork docs/21 装配点模式"数据源可插拔、消费方零导入"的解耦哲学同构。
- **客户侧部署形态是共享责任模型**：Harvey 负责鉴权流程、凭证处理与访问控制的框架，**律所管理员保留"启用什么、对谁启用"的决定权**，每个连接器上线前都过 Harvey Security & Trust 团队的安全评审（评估 server、工具能力与权限范围）。这个"平台管鉴权框架、客户管开关"的分权模式，直接对应本报告要建议的"租户注册=声明文件由客户侧维护，装配逻辑由 Courtwork 平台侧统一执行"。

### 2.2 Legora / DraftWise：DMS 索引 + firm playbook 分层，强调数据质量而非接入广度

- **Legora**：DMS 集成（iManage、SharePoint）、文档摄取、内容源、第三方法律服务与 MCP 连接器共同喂给 agent 的上下文；firm playbook、判例库、条款库、历次谈判立场与完整 matter 历史被分层注入每一次 agent 操作——这与 docs/20"平衡点写在场景声明里"的思路一致：不是所有库内容都无差别流入所有场景，而是按场景/按操作分层调用。
- **DraftWise**：DMS 集成覆盖 iManage、NetDocuments、Opentext、SharePoint（不做 LPMS 对接）；检索技术栈是 Cohere Embed（多模态嵌入，可处理图表/表格）+ Cohere Rerank，官方披露接入精调模型后检索质量提升约 30%。**其官方博客《Garbage In, Garbage Out: How to Build a Precedent Database Lawyers Can Trust》直接对应"企业内部文档脏乱"这个调研关切**，核心结论：
  - "私有的、律所自有的数据始终优于通用网络数据"，但前提是数据质量有保障；
  - 最直接的质量手段是**过滤**——只认律所自己律师撰写的文档（排除对方草稿）、优先终稿而非中间稿、按法律概念而非关键词做概念检索；
  - 更进阶的手段是**"黄金标准"精选集**——由知识管理律师/资深合伙人挑出各领域范例文档，供模型优先参考，同时兼作新人培训素材；
  - **透明度是信任的前提**：每条建议都要有精确引用，指明来自哪份文档、哪个条款，方便律师核实来源、判断是否契合当前 matter；
  - Thomson Reuters 的对照案例佐证了"证据不足时承认不知道比瞎编更重要"——这与 docs/20 的 `out_of_coverage` 语义、"C 级降低漏检不豁免核验"的纪律在方法论上完全对齐，是外部证据对 Courtwork 既有设计的独立支持，而非新增需求。

### 2.3 国内律所/企业法务内部库现状：信息孤岛是常态，AI 接入前提是先补数字化的课

国内检索显示中国律所内部管理仍处于相对传统阶段：案件信息散落在各律师本地电脑形成"信息孤岛"；合同审批、报销依赖纸质流转；卷宗/判例/法律文书检索困难。超过 70% 的中小型律所仍依赖纸质文档和电子表格管理，版本控制混乱（同一份合同多轮修改产生多个版本、误用旧版本引发法律后果）是普遍反馈的痛点。2025 年市场上出现的多款律所文档管理系统正在往"以案件为中心、集成项目协同/知识库构建/审批流程"的方向补课，但普及率仍有限。

**对 Courtwork 的含义**：国内客户侧"企业私域库"大概率不是像 iManage/NetDocuments 那样结构化、有权限模型、有增量同步 API 的成熟 DMS，而更可能是共享盘/OA 附件/微信文档/散落 Excel 台账。这意味着**"通用库适配器"的最小兼容集不能假设客户能提供实时检索端点**——demo-fixture 式的"批量导入 + 定期离线重建索引"很可能是国内客户的现实主路径，而不是 Harvey/DraftWise 那种"文档留在原地、实时检索"的高阶形态。这是对架构预置的一个**需要分层的修正**：注册契约应该同时容纳"实时检索端点"与"批量快照上传"两种接入形态，不能只设计前者。

## 三、多租户知识隔离

### 3.1 SaaS 通用实践：Silo / Pool / Bridge 三模式，命名空间是隔离的主要技术手段

行业把多租户 RAG 隔离归纳为三种模式：**Silo**（每租户独立索引，隔离最强，企业级部署首选）、**Pool**（共享索引 + 元数据过滤，成本效率最优，适合 SMB）、**Bridge**（混合，适合客户构成混杂的场景）。命名空间被反复强调为推荐做法：查询成本通常按命名空间规模计费（一租户一命名空间比在大命名空间里做元数据过滤便宜得多），且命名空间让租户下线简化为一次删除操作。安全实践的共识是：**租户隔离必须在向量数据库层用命名空间/强制过滤 + JWT 做确定性保证，"依赖 LLM 自己做访问控制"被明确列为架构反模式**——这条对 Courtwork 有直接约束：企业库检索适配器的租户边界不能靠 prompt 层约束或模型自觉遵守，必须是装配点注入时就物理隔离的命名空间/连接凭证。

### 3.2 与利冲隔离的交叠：律所现有基础设施优先，不建平行系统

docs/13（已完成的调研）与 docs/25（已拍板）已经指出关键教训：**传统访问控制在人速操作下成立，在 agent 规模下失效**——律所现有伦理墙靠 Intapp（利冲检索）+ iManage/NetDocuments（文档级访问控制）实现，边界单位是"人访问一份资源"；但 agent 自主决定读什么、且能跨会话维持上下文，意味着"agent 越权读取一份墙后文档"这个违规动作即便律师本人从未看到结果也已经发生。本轮调研进一步确认：SaaS 层的租户隔离（Silo/Pool/Bridge、命名空间）解决的是"客户 A 的库不能被客户 B 看到"这个粗粒度问题，**利冲隔离要解决的是同一客户内部"这份文档不能给参与冲突 matter 的这位律师看"这个细粒度问题，两者维度不同、不能互相替代**。docs/25 已经拍板"利冲隔离对接律所既有系统（Intapp/iManage），不自建"，本报告调研结果支持维持这一决定不变——企业私域库接入时，`conflict-group` 字段应该做的是**透传/继承企业侧既有的伦理墙标记**，而不是 Courtwork 自己重新实现一套冲突检测逻辑。

### 3.3 对 Courtwork 租户注册的含义

租户级 `retrievalPolicy` 需要在两个维度上都留字段位：**租户边界**（命名空间/连接凭证，粗粒度，SaaS 架构层）与**利冲边界**（conflict-group，细粒度，继承自客户系统）。二者都不应该成为 core 里写死的特判逻辑，而是场景声明层可读取、装配点写入的配置数据——这与 docs/20"平衡点写在场景声明里，core 只做通用传播与门禁"的纪律完全一致，调研没有发现需要突破这条纪律的理由。

## 四、检索质量工程

### 4.1 Contextual Retrieval + 混合检索：千~万级私域小库仍然适用，且是 2026 年的默认起点

Anthropic 2024 年提出的 contextual retrieval（分块前为每个 chunk 前置一段 50–100 token 的上下文说明，同时用于向量嵌入和 BM25 索引）在 2026 年的多个基准测试中仍然是大多数企业语料的最优起点，能把检索失败率降低约 49%，叠加重排序后降低约 67%。成本上依赖 prompt caching：索引阶段父文档被缓存，每个 chunk 只需付出增量 token 成本。对千~万级条目的私域小库而言，这个方案的边际成本很低，且不需要 GraphRAG 那种面向多跳全局推理的复杂基础设施——**只有当企业库涉及大量图表/版式复杂文档（扫描件、盖章合同）时，才需要额外引入视觉检索（如 ColPali）**，这与 Courtwork 自身"OCR 印章/手写"是五个必须自研加固节点之一的判断吻合，不是新增负担，而是同一个难点在检索侧的另一面。

### 4.2 库质量差时的兜底：置信标注 + 拒答，与 out_of_coverage 语义直接对接

行业对"企业内部文档脏乱"问题的共识兜底架构是三层：**检索置信度评分 → 引用校验 → 基于 LLM 的拒答（abstention）**，核心原则是"可靠的系统不仅要答得好，更要知道什么时候不该答"。已知的失败模式根源多在上游数据质量：过期文档产生误导性相似度分数、上下文贫乏的资产产生弱嵌入、未核验来源产生噪声结果——这些失败模式恰恰是"库覆盖不全"与"库内容本身不可信"两类问题的交叠，而不是纯检索算法能解决的。

DraftWise 的"Garbage In, Garbage Out"方法论提供了具体的工程手段（authorship 过滤、终稿优先、黄金标准精选集、元数据驱动的自动分类），这些手段本质是**在检索之前先做条目质量分级**，与 docs/20 的 A/B/C 信源分级异曲同工，但粒度更细——A/B/C 分的是"信源整体的权威性"，DraftWise 分的是"同一信源内部条目的可信度"（终稿 vs 草稿、本方 vs 对方文书）。**这提示条目 schema 需要一个"文档状态/可信度"字段（如 `finality: draft|executed|superseded`），不能只有信源等级，否则同一个 B 级库内部的脏数据仍会被无差别检索**——这是对架构预置"条目 schema 在适配器输出层"的一个具体补充建议，而非推翻。

### 4.3 与 out_of_coverage 语义的对接

架构已经拍板"库里没有 ≠ 不存在"（`out_of_coverage` reason），本节调研进一步支持这一设计：企业库覆盖不均是常态而非例外（"知识库可能在某些领域覆盖详细，某些领域存在空白，使得难以预判检索何时可靠"），因此 `out_of_coverage` 不应该是一次性判断，而应该允许**按查询主题/按条目类别分别声明覆盖范围**——这需要企业方在注册时提供一份"覆盖声明"（哪些业务领域/文档类型被索引了，哪些明确没有），而不是笼统的"库已接入"布尔值。

## 五、建议：最小兼容集契约草案

### 5.1 通用库适配器的最小兼容集

企业方接入需要提供（按必要性排序）：

1. **检索端点或快照上传二选一**：实时检索端点（HTTP，输入查询/过滤条件，输出候选条目 + 相关性分数）**或**批量快照包（定期上传的结构化文档集，Courtwork 侧离线建索引）——2.3 节已论证国内客户现实情况更可能需要后者，契约必须两者都覆盖，不能假设企业已有实时 API。
2. **条目 schema**：最小字段集 = `id`（企业侧稳定标识）、`content`（正文或摘要）、`sourceRef`（企业侧原文定位方式，如 DMS 内部路径，不是 Courtwork 的 `SourceAnchor`——后者是卷宗内页码坐标，语义不同，参照 `packages/tools/SPEC.md` 已有的教训不要混用）、`finality`（草稿/终稿/已作废，对应 4.2 节发现）、`updatedAt`。
3. **覆盖声明**：企业方在注册时声明库覆盖的业务领域/文档类型清单，作为 `out_of_coverage` 判断依据的基线（而非每次查询临时猜测）。
4. **更新通知（可选，分期）**：webhook 或增量拉取端点，MVP 阶段可以没有，退化为"定期全量重建"，与国内客户现实（大概率没有增量 API）匹配。
5. **租户/利冲边界标识（透传，非必须新建）**：如企业侧已有 conflict-group/部门等隔离维度，注册时随条目一并声明，Courtwork 侧只透传继承，不做二次判断（呼应 3.2 节"不建平行系统"）。

不要求的：企业方不需要理解或适配 Courtwown 的 `SourceAnchor`/`RiskBasis` 等内部 schema——3.2 节和 `packages/tools/SPEC.md` 已确认的架构纪律是"嵌入形状定义在 schemas，映射发生在 core"，企业库适配器只需要交出上面 5 项，映射投影责任在 Courtwork 侧，不转嫁给企业方。

### 5.2 MCP 支持放哪一期

**不建议在 MVP 阶段把企业库适配器整体建成 MCP-native**，理由见 1.4 节：Courtwork 当前只有单一消费方（core 编排器），MCP 化的收支平衡点（约 3 个消费方）尚未触及，自建的确定性契约（`ToolDefinition`/`sourceId`/六种 `reason`）已经覆盖企业库适配器的核心需求。**建议把"MCP 作为企业库适配器的一种可选传输层"列为注册表 v2 之后的独立小工单**：当有企业客户明确已经自建或计划自建 MCP server（对标 Harvey 案例中的 iManage/NetDocuments 模式）时，再实现一个 `createMcpXAdapter(mcpServerConfig)`，对上暴露和 mock/demo-fixture/真实骨架完全同构的适配器接口，对下把 MCP 的 resources/tools 调用包一层——这条路径不需要现在预先设计，装配点模式本来就支持"未来加一种适配器实现"零改动其余板块。

### 5.3 租户注册的配置形态

建议**声明文件（YAML/JSON）+ 装配点读取，不做管理界面**（W9 之前）：与场景注册表本身的形态保持一致（docs/24："场景声明本质是数据"），一个租户库注册声明包含：库 id、`sourceId`、适配器类型（demo-fixture/http-endpoint/snapshot-upload，MCP 留作未来第四种）、条目 schema 版本、覆盖声明、conflict-group 透传字段、连接凭证走配置注入（不硬编码，沿用 `packages/tools` 已确立的凭证纪律）。管理界面（律所 IT 自助注册库、查看覆盖范围、开关连接器权限，对标 Harvey"管理员保留启用范围决定权"的分权模式）留给 apps/desktop 或独立的租户管理后台，属于 W9 之后的产品面工单，不在当前架构调研范围内替它拍板。

### 5.4 各块验收标准（各一句）

- **库（适配器）**：能在缺少真实凭证/端点不可达时正确降级为已有六种 `reason` 之一（尤其 `out_of_coverage`/`not_configured`），不返回伪造的 `verified:true`。
- **schema（条目）**：能装下 Harvey/Legora/DraftWise 三家先例中出现的字段集合（正文、来源定位、终稿状态、更新时间）而不需要为任何一家单独扩展。
- **路由（retrievalPolicy）**：同一场景在"开企业库"与"关企业库，仅用 demo-fixture/mock"两种配置下产出的产出 schema 形状完全一致，只有内容与信源标注不同——可预期性不因接入真实企业库而破坏。
- **skill（场景变体）**：企业 playbook + 企业库组成的场景变体在注册表校验时，与内置场景一样必须补齐产出 schema 与确认门禁两个强制字段，不存在因为"这是企业定制场景"而豁免的裸 skill 路径。
