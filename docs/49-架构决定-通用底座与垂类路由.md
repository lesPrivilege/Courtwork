# 架构决定：通用底座与垂类路由（2026-07-10）

状态：已拍板。核心律：**用户选名词（建什么容器），不选形容词（什么模式）**——"通用/垂类模式切换"永不出现在 UI。

## 三层机制

1. **通用底座 = 无容器默认态**：轻对话 + composer + 通用动词（上传/工作稿/fetch/文件整理/系统动词）。FileOpsPlan/工作稿轨/阅读视图本为领域无关（docs/22 红利），底座是摘掉法律皮肤后的剩余全部，零垂类 Button，无需单独实现。
2. **垂类 Button = 场景卡片，双信号点亮**：容器类型（新建案件 vs 新建工作区——new task 的选择点，选容器不选模式；案件 → S1–S4 常驻，工作区 → 通用场景族）+ 内容识别（分类器推荐置顶，推荐不触发，docs/24）。
3. **路由 = 声明过滤非代码分支**：场景声明增加 `context: case | workspace | any`（注册表 v2 清单 +1），货架按容器类型 × trigger 匹配过滤展示。边界：普通工作区点法律场景卡 → popover 轻确认"升级为案件"（单向仪式，与存入卷宗同族）。

## 视图归属速查

- 通用底座视图：对话流、工作稿画布、整理计划表、附件 chip、产出卡。
- 垂类视图（随案件容器出现）：五工作面（时间线/图谱/矩阵/修订预览/起草画布的法律模板）、法理之线、信源角标、门禁卡。
- 两者共用：状态条、⌘K、专注模式、设置——全局层永远容器无关。

## 落地清单

- 注册表 v2：`context` 字段（与 retrievalPolicy 同批）。
- UI：新建工作区动词 + 容器升级 popover（F-4 之后切单，热点文件排队纪律）。
- MVP demo 不受影响（演示即案件容器态）。

## 第二章：容器内的事项类型路由（2026-07-10 增补）

法律事项多形（诉讼/合同专项/招投标/尽调），裁决：**容器不分叉，类型是推断标签非前置选择**（选名词不选形容词的延伸）。

1. 新建案件 = 通用法律事项容器，零类型选择；三区语义（卷宗/工作稿/产出）与红线对所有事项类型同构。
2. 事项类型 = 分类器推断的可改元数据（改动即 RevisionEvent 标注）；标签影响：场景卡排序、产出目录模板、时效规则集挂载（诉讼挂开庭/上诉期，招投标挂投标截止——docs/48 时效族按类型取规则集）。
3. 路由零新机制：类型标签即 classifierTags 之一种，注册表 trigger 三维已覆盖。招投标 = S9 场景族候选（招标文件审查/投标响应核对/资质核验——复用 party-verify），一批声明而已。
4. 文案注记：律所称"案件"泛称接受度高；企业法务偏"项目"——容器名称随 persona 配置属租户级词表（Stage 1 定），语义层统一。

## 第三章：右栏模块栈（2026-07-10 增补）

裁决：**右栏 = 声明渲染的折叠模块栈**，通用三件与垂类工作面同栈共存、互不取代。

1. **通用三件折叠常驻**（frontier 习惯锚点，不引入门槛）：progress = todo 快照+摄取进度（docs/12 事件驱动）；working folders = 原件/工作稿/产出三区（F-3 已建）；context = 用量圆盘明细+附件来源。案件场景下降位为面板头（名称+计数+状态点），不消失。
2. **展开事件驱动**：全模块默认面板头可见（空间记忆+计数预告）；场景推理节点完成 → artifact_produced 自动展开对应 schema 模块并铺好数据；用户手动折叠/展开优先于自动行为。
3. **新场景 = 栈内加模块零重排**：按场景声明 uiTemplateId 渲染，排序由声明字段定——信息架构不随场景增长重设计。
4. **与既有 Tab/对照演进不推翻**：Tab = 快速跳转、Split = 双开、折叠头 = 未激活模块的可见态，同一栈三种视图密度，实现为增量。

排期：RP-1 工单（右栏模块栈改造），UX-1/SET-1 之后（热点串行）；docs/46 登记面板头组件族。

## 第四章：画布-浮面三层模型（2026-07-10 增补——"像产品而非 Demo"的空间钥匙）

- **L0 画布**：对话流直接坐在页面底色上（冷灰系），不装卡——对话是其余一切浮于其上的地面。
- **L1 浮面**：左栏容器卡 / 右栏模块栈 / composer 输入卡——上浮语法 = 纯白面 + 边缘留白（inset 不贴边）+ 圆角 12 + 细描边，**零投影**（de-slop #6 不破，靠色调差与留白悬浮）。标题栏透明化融入红绿灯 chrome 层，只留 wordmark + 全局动作。
- **L2 瞬时层**：popover/⌘K/菜单（surface-popover 既有）。
- **收缩态 = 通用底座的视觉形态**：左栏折叠 + 右栏全折 → 画布 + composer 浮卡 + 折叠按钮，全部收敛到底色（docs/49 底座论的视觉表达）。
- **模型服务声明位置**：常驻归状态条（模型名可点）；非 connected 态才在标题栏浮现琥珀警示——正常安静、异常上浮。
- **复杂工作面与通用对齐**：靠浮面卡外壳语法全栈统一（头/留白/圆角），卡内密度自治。
- 排期：并入 RP-1（右栏模块栈与层级改造同一次重排）；tokens 增 elevation 组（底色级联 + inset 间距）由 RP-1 提案、架构过目。

## 第五章：frontier 实测参照与 Build 门（2026-07-11，据用户 Cowork 真机截图）

### 左栏骨架参照（第三批次 #11 的对照基准）

Cowork 实测结构：顶部动词区（New）→ 一级导航（Projects / Artifacts / Scheduled / Dispatch·Beta）→ Pinned 区 → Recents 区（带排序控件）。Courtwork 映射：新建（既有）→ 案件/工作区列表（既有，即 Projects 位）→ 产出（Artifacts 位，跳工作区级产出目录）→ 定时（Scheduled 位，docs/47 sources 声明的 Stage 3 触发器，禁用态骨架）→ 派发（Dispatch 位，Stage 1 团队协作，禁用态骨架）。Customize 不做（宪法排除）。Pinned/Recents = 案件置顶 + 最近打开，轻量本地态。

### 右栏模块栈参照（第三章的对照基准）

Cowork 实测结构：Progress（`17 of 17` 计数 + chevron，面板头即信息）→ Working folders（容器名可折叠 + 文件树，逐文件图标+名称）→ Context（Uploads 文件列表 + Connectors chips）。印证第三章裁决：通用三件折叠常驻、面板头带计数预告。Courtwork 对应：progress = 阶段进度 `0/6` 从底部状态条**升入右栏面板头**；working folders = 三区树（原件只读标记随行）；context = 用量圆盘 + 附件来源 + 已连接服务 chip（模型服务 connected 态的常驻声明位，与第四章"正常安静"一致）。

### Build 门（用户拍板 2026-07-11）

**左右栏分层（RP-1）跑通 = base 定形，出第一个正式 Build**（打包分发版）。序列：UX-1/SET-1 验收放行 → RP-1（右栏模块栈 + 画布-浮面层级 + 左栏骨架 + 本章参照）→ 验收 → Build。工作面 polish（批次三）与 Build 解耦，后续版本独立迭代。

## 第四章补遗（2026-07-11）：elevation 组过目放行

RP-1 提案的 elevation 全量（11 token，见 apps/desktop/SPEC.md）架构过目放行。三项拍板：floatRadius=12 = 宪法补 token 非破例（仅 L1 外壳、仅经变量消费）；floatInset/shellGap 锁 8（4 基阶，小屏优先；升 12 须真机视觉证据）；warn 三轨复用 gate.pending 琥珀（预算不破），**语义色使用白名单同步 +1 行：系统凭证异常（标题栏，仅 failed 态）**——此为琥珀在审阅面之外的唯一合法使用位，再扩须架构拍板。判例：热点强耦合下 A/B/C 合一枚 feat 接受，分层纪律的单位是职责不是字母编号。

## 第四章修正（2026-07-11 真机 Debug 1）：模型声明位唯一化

原"非 connected 态标题栏浮现琥珀警示"被真机证伪——实际出现标题栏 + 工具栏两处重复警示，噪音大于警示。修正：**模型服务声明位全 app 唯一 = 状态条 chip**，探针三态驱动（connected 显模型名·推理强度／failed 就地转琥珀"连接失败"／pending 显"待连接"），点开 = model-config popover（含状态详情与重试）。标题栏琥珀 chip 与工具栏重复项撤销。假态零容忍：未 connected 不得显示模型名。

## 第四章再修（2026-07-11 深夜）：声明位归 composer

同日第一次修正（声明位=状态条）经用户真机再裁：**唯一声明位 = composer 下缘、发送键旁**（frontier composer 惯例，贴近"发送前确认模型"心智），探针三态与假态零容忍不变；状态条不再承载模型名。布局定调：三列纵向贯通，上下不分横带；双侧可折叠，展开钮驻原位。

## 第三章修正（2026-07-11，RP-2.5 修宪）：模块栈 → 双宿主

原"通用三件与垂类工作面同栈共存"被 RP-2.5 取代：**右栏 = UtilityRail（通用三卡，领域无关，L0 上三个独立 L1 浮卡）+ SchemaPreviewHost（Preview 宿主）双宿主**。Preview 打开时为主承重浮面、通用三卡收敛为顶部 utility dock（一行摘要可临时展开）；关闭恢复三卡布局。保留不变：artifact_produced 自动打开对应 Preview、用户手动关闭优先、新场景 = 声明加渲染器零重排。修宪理由：双宿主比同栈更忠实于通用层禁渗领域语义（docs/22）与垂类即依赖包（docs/53）——垂类渲染器可独立演进不碰底座。

**宿主语义（架构批注）**：SchemaPreviewHost 本身领域无关，渲染器按场景声明 uiTemplateId 挂载，法律五工作面只是首个渲染器集；宿主代码内出现法律语义即违宪。UtilityRail 与垂类渲染器互不 import，此边界由 grep/lint 守卫测试机器化（与 floor 同级门禁）。

## 第六章：空态首屏与模块位（2026-07-11，声明预留不落版）

1. **首屏 = 通用底座空态**：chat + composer + 左栏。样板案演示非首屏——空态内显式路由（"从样板案开始体验"），用户自选进入（docs/21 转正裁决的首屏面）。
2. **首屏模块位 = 声明式接口**（与 PreviewHost 同族）：中央与右侧各留模块位，热力图（案件活动/用量）、日程看板（时效族接口取数）、引言位等皆为可选模块，按声明挂载、用户可选装。宿主通用，模块即声明 + renderer。
3. **默认态克制**：中央引言/欢迎 + 右侧至多单模块；不做仪表盘杂货铺。
4. 排期：本期仅评估 + 接口声明（注册表/声明 schema 预留字段），零实现；落版另单，据首批用户反馈选默认模块。

## 第七章：渲染形态封闭集（2026-07-11，立词表）

1. **schema 渲染形态 = 有限封闭词表**：类 PPT 的版面 + cheap 可交互 HTML 形态，有限但足够；场景声明从词表选用 renderer 原语，**首期不开放自定义**（封闭性即防呆与反 slop 地基）。
2. **SVG 一切原生绘制**：控件、图形、标记沿 P-4 流水线（manifest/SVGO/审计），禁临时内联、禁位图拼贴。
3. 词表全谱、各原语的 token 边界与排版架构由专项调研立案（docs/57）；设计语言/色系/图标等通用元素随词表自然生长，仍过架构过目。
