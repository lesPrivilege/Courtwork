# 观察：协作范式的泛化条件（2026-07-10）

状态：经验归档（实证基数：本仓库 20+ 会话、3 种模型、两次不放行、一次治理拒令、零人工逐行 review）。

## 三个修正

1. **稀缺的不是 review 带宽，是决策的时间一致性。** 架构角色的产出不是审查代码，是维护决定空间（ADR 宪法 + 工单纪律），让第 N 个会话的判断与第 1 个一致。
2. **"发工单难命中"由结构化检查点消解，不由完美规格消解。** 复述确认（动手前）+ [需架构拍板]（执行中）+ 独立验收（完工后）三道网；coding cheaper 使迭代误差成本低于事前完美规格的成本。
3. **自测 ≠ 可放行。** 实现者不得宣告自己完工是可泛化不变量（W6 判例：自测全绿仍被验收抓出两个契约漏洞）；异模型互查抓的问题类型宽于同构互查（实测结论）。

## 泛化五前提（缺一退化为昂贵仪式）

契约先行的分层（有 schemas 类的根）｜成文宪法（决定在 repo 不在会议）｜验收可机器判定（"实测不采信自述"可执行）｜工单即数据（docs/11 模式）｜有人愿把判断成文化（真瓶颈——多数组织的第一课是把决定写下来，不是上 agent）。

## 角色收敛

架构师可以是 agent（本仓库即是）。人类保留三样：要什么（方向/优先级）、价值观拍板（品味否决/时序/对外承诺）、终审。成本诚实：三角制 ≈ 2–3 倍单会话 token，错误代价高的领域值回票价，低风险内部工具未必。
