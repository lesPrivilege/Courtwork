# 场景设计：S5 合同对比（2026-07-09）

状态：已设计，排期 MVP 后首个新场景（场景注册表机制的示范扩展）。

## 用户问题

对方回传修改后的 Word 但未开修订痕迹（现实常态，可能一次回传多份版本）；用户需要知道改了哪里、意味着什么。

## 架构（三层，复用优先）

1. **确定性 diff 层（新件，唯一需要新建的引擎）**：双 docx 解析 → 段落对齐（含条款移动检测，重排不显示为删+增）→ 段内字符级 diff → 格式性改动单独分类（不进主视图）→ 表格独立 diff。**产出 = RevisionInstructionSet**——把"未知的修改"编译为"已知的指令"。
2. **渲染层（全复用）**：修订指令集经 W4 著录器生成带修订痕迹的 docx（等于替对方补上没开的修订标注）；右栏对比视图（Vincent diff 视图为参照，docs/02 §3.6）。
3. **判断层（LLM + 规则，垂类价值所在）**：规则先行——金额/期限/比例/管辖等高危增量用规则精确捕捉，不许漏、不靠 LLM；LLM 做语义标注——噪音分组（编号连锁）、实质改动的立场化影响（对我方不利/有利/中性，复用 S3 立场机制）、与 playbook 比对。LLM 标注一律呈现为待确认判断，不混入确定性 diff 结果。

## 多版本链

多份 word 先后不明时：两两相似度矩阵 → 推导版本谱系 → **呈现给用户确认**（留人节点）→ 按链逐段对比。

## 场景声明草案

trigger：用户拖入 ≥2 份同名/高相似 docx，或场景按钮"对比合同版本"。inputArtifacts：[]（原始文件）。toolIds：[]（diff 是本地引擎非外部接口；playbook 比对复用 S3 工具位）。outputArtifacts：[RevisionInstructionSet]（+ 对比报告，产物类型届时定）。确认门禁：版本谱系确认、立场标注确认。

## 测试素材

样板案主合同 + 10 份变体即天然对比测试对；另需补"同一合同的多轮修改版本链"夹具（demo-data 微工单 +1）。

## 不许做

- LLM 不许参与"哪里改了"的判定（diff 是确定性层）。
- 高危字段（金额/期限/管辖）的变动检测不许只靠 LLM，必须有规则兜底。
