# 架构决定：sandbox 与 web fetch 分期（2026-07-10）

状态：已拍板。依据 docs/14 调研。

## 前提认定

既有架构（docs/24 固定编排 + 执行器调工具、LLM 不自主选工具 + 留人确认）已命中业界公认的最强注入防御组合（Rule of Two），"模型被注入后自主调用危险工具"这条最大攻击面不存在。剩余风险 = 生成节点文字被 fetch 内容诱导 → 用内容级消毒补齐，**不做 CaMeL/FIDES 级架构投入**（过度设计）。

## MVP 最小集（六条，全部在既有契约边界后，解耦成立）

1. ingest 解析进程隔离 + 解压比例上限（防 zip 炸弹）；Python 服务资源限制。
2. docx/OOXML 解析禁宏、禁 XXE（output 与 ingest 双侧）。
3. web fetch 工具内置 SSRF 拦截（私有网段/云元数据端点黑名单）；**证书校验失败一律 `adapter_error` 降级，绝不静默放行**（复用 tools 六 reason，无新契约）。
4. fetch 内容进生成节点前做 spotlighting 标记（结构化内容隔离：标记 + datamarking），并携带信源等级（docs/20 的 C 级在 prompt 层显式传播）——"fetch 内容永远是数据不是指令"的工程落点。
5. Tauri capability/permission 最小化声明（W9 打包时执行）。
6. 确认 UI 展示真实依据原文而非模型自然语言摘要（与防呆②/溯源必达 §6 同一条，此处从安全角度再锁一遍）。

## 分期与不做

- Stage 1–2：egress 白名单（堵注入后的外泄出口）、第三方 skill 的执行边界（进注册表 v2 一并设计）、服务端 fetch 代理选项（私有化客户合规审计需要时）。
- 不做：模型层防注入微调（已被证明不可靠，不许拿它顶替消毒层）；MVP 阶段的全容器化沙箱。

## Provider 凭证处理（2026-07-10 增补）

三形态（客户端最终零 key）：① MVP/本机——用户自填，存 OS 钥匙串（Tauri keychain/stronghold），绝不落明文文件，UI 掩码，RuntimeGuard.maxUsd 默认保守值兜住用量失控；② 试点 SaaS——后端代理持 key，客户端零 key（同时是 credits 计量落点、模型路由中枢、docs/28"Provider 不训练确认"的集中签署处）；③ 私有化——企业后端/网关提供，key 在客户租户内。

两条红线：**不做静默扫描本地第三方配置**（读别家工具的凭证文件与产品的保密姿态相悖；首启引导允许的上限是"粘贴 key 或指定环境变量名"的显式动作）；**key 永不进事件流/日志/遥测**（事件可落盘可回放，混入即持久化泄露——进验收项）。

## 红线

- 生产 web fetch 在任何情况下不许关闭证书校验、不许绕 WAF——人工调研阶段的手法（Playwright 忽略证书）不得进入产品代码路径。
- 消毒层是 fetch 工具实现的一部分，收在工具契约后面：换消毒方案不动 core、不动场景。
