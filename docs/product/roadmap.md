# 产品路线图

状态：现行定本（2026-07-15）

本路线图按**证据条件**而不是日期推进。阶段编号表示依赖顺序，不表示可以并行对外承诺的功能清单。每一阶段的当前状态只认[当前基线](../status/current.md)；本文只定义进入条件、最小交付、退出证据与禁止抢跑项。

当前所处阶段只在[当前基线](../status/current.md)登记。无论当前阶段为何，包级能力、demo fixture、已发布界面和未来接口都不能替代对应阶段要求的真实证据。

## Stage 0 — 真实 MVP

**进入条件**

- 通用包、Legal 垂类包、桌面宿主和发布链已有可执行基线；
- 产品继续把原件、系统锚点、结构化结论与人工决定分成不同事实层。

**最小交付**

- 真实材料经宿主授权进入案件，并形成可复验的材料引用与阅读视图；
- 真实 Work 经 production command/store/binding 运行，不读取 recording、fixture 或 demo 原文；
- 模型引语由系统解析为锚点，风险与修订逐项呈现并由人确认、驳回或修正；
- 已确认结果安全生成或修订 docx，未应用指令不会静默落盘；
- 试点使用能够留下任务完成、人工修正与交付结果的证据。

**退出证据**

- 至少一条“真实材料 → 真实 Work → 证据与确认 → docx”端到端链路，在重启、切案与异常注入下仍可复验；
- Word/WPS P0 往返与结构门有精确环境记录；
- 真实试点持续使用并形成可审计的采纳、驳回、修正和交付基线。

**禁止抢跑**

- 不以 demo-runtime 全链、catalog preview、静态 Pages 或单次截图宣称 MVP 已退出；
- 不用 recording fallback、占位材料或模拟 identity 填补 production 缺口；
- 不在真实链路未成立时扩建团队协作、自动监控或生态平台。

## Stage 1 — 长入既有工作流

**进入条件**

- Stage 0 的真实链路与试点证据成立；
- 场景上新、来源接入与入口扩展都有可观测需求，而不是内部想象。

**通用底座补齐线（2026-07-18 产品定调，Stage 0→1 之间的独立方向）**：宣称「不加载 schema 包仍有通用底座」必须可证伪——**卸下全部垂类包，产品仍应是一个合格的通用 work agent**（读写工作文件夹、通用文档任务、受控工具、诚实失败）。路径三条纪律：①开源社区轮子尽可能用（借形不接管真源，pi 对照先例）；②相比 coding agent 只做减法（封闭工具集、声明式编排、留人确认——不暴露任何 /command 式上下文操作，见减法调研）；③ schema 契约是预留接口——垂类包缺席时插槽显式空、不降级不伪装。定位澄清（2026-07-18）：Chat=轻聊天机器人（无文件夹权限、context 手动上传、memory 走本地缓存）；Work=agent 本体（文件夹授权、harness、场景、确认账本）——垂类包此前的装载已验证协议与编排，底座补齐即让 Work 侧成为自足 agent。能力基线以竞品实测清单为准（不按想象铺），逐项经架构拍板入队。

**已裁不做（2026-07-20，`HARNESS-CORE-1` Stage C 逐项裁决）**——记在这里是因为「刻意不做」与「还没做」是两回事：欠账会被补上，裁定不会。凡竞品有而我方无的功能，若属下列两条，**不再重新提案**；要翻案须携新论据对着本节提修订，不得从零辩论。

- **重新生成 / 多回答分支（`C/R-5` 准，判定二成立）**：不做。多回答择优是**以文采代替依据**；重新生成对一个成功轮产生两份等权产出，而系统**没有任何裁决依据可据以选择**——两者都与原则 1（模型只生成不裁决）、原则 2（无锚不落格）相悖。正确形态是逐条确认 + 引语回跳 + 驳回/修正：让人对着依据裁，而不是对着两份文采挑。**边界（不在被裁之列）**：失败轮的 Retry、用户改写输入后的 fork editing 均照常——被裁的只是「**同一输入再摇一次**」。
- **`/command` 式上下文操作（`C/R-3` 裁，限定词有效）**：被**裁死**的是 `/command` 式上下文操作（`/to-goal`、`/handoff`、先缩再跑一类把上下文管理暴露给用户的命令）——这正是减法调研「减 UI 暴露面不减能力」的正面主张：该做的下沉为结构，不做成用户要学的命令。**非上下文类斜杠触发不属被裁项，但当期同样不做**：快速动作统一归 ⌘K 命令面板，守入口单一性（`RP-2.7` 已把重复入口收敛为单一路径）。未来若要斜杠须**新裁**，不得援引本条自动放行。

两条均适用「刻意不做须有用户可见说明」通则（`C/R-4`ⓑ）中的工程内部项一档：文档留痕即可，不占用户面席位——用户不会主动去找「重新生成为什么没有」，但会主动去找会话导出与 memory 编辑，故后两者才升到 Settings 可见面。

**最小交付**

- 在 Word/WPS、邮件或客户已有工作流中提供薄入口，执行与著录仍回到同一权威底座；
- 场景继续通过版本化 descriptor、schema、门禁与 eval 接入，不复制 core；
- 真实任务沉淀为脱敏、授权明确的 eval；
- 法规、案例或企业信息只接入可核验的官方或正式授权来源，并保留来源级别。

**退出证据**

- 至少一个工作流入口被真实用户持续使用，且与桌面端共享同一案件、身份、证据和确认语义；
- 新场景在不改通用底座领域语义的前提下通过包准入、回归与人工验收；
- eval 与官方来源实质参与模型、场景或引用质量决策。

**禁止抢跑**

- 插件、邮件和通道不得另造修订器、状态机或授权语义；
- 不恢复任意 provider/base URL、任意可执行 package 或模型自主选择不可逆流程；
- 不以场景数量、入口数量或未经授权的材料规模替代真实采用证据。

## Stage 2 — 团队化与受治理资产

**进入条件**

- Stage 1 已证明个人工作链与入口稳定；
- 真实团队需求足以要求多人共享，而不只是把单机数据同步到云端。

**最小交付**

- authenticated principal、组织与角色成为真实依赖，不再使用本地字面 actor；
- 多写者状态有明确并发语义、审计、恢复和冲突处理；
- 案件、材料、artifact、来源与动作具备 ACL、伦理墙和租户隔离；
- 跨案图谱使用独立治理容器，只有经显式审查和晋升的事实可进入；单案账本保持封闭。

**退出证据**

- 多人对同一案件的读取、修正、确认与冲突处理在真实组织权限下通过；
- 跨租户、跨伦理墙、跨案件引用的反例全部 fail closed；
- 跨案资产的来源、授权、撤回和晋升记录可以独立审计。

**禁止抢跑**

- 不把 v1 单机单写者的 CAS 语义宣称为协作能力；
- 不以共享文件夹、同步目录或 actor 字符串代替 identity、ACL 与伦理墙；
- 不打开 case 账本做全局检索，也不自动把单案推理提升为机构事实。

## Stage 3 — 受控监控与主动工作

**进入条件**

- Stage 2 的团队图谱与治理容器已经真实运行；
- 官方数据接口和至少一个受治理通道已经落地；
- scheduled invocation、trigger context 和 effect authorization 已由新 ADR 拍板。

**最小交付**

- 对合同义务、主体风险或法规变化进行来源可核验的定时检查；
- 调度、重试、暂停与恢复沿同一 session 累积 steps、toolCalls、usd 与 executionSeconds 预算；
- 每一次 `external_send`、`file_write` 或改变权限的 effect 都在执行前取得对应授权；
- 通道只推送受权限约束的事实与待确认动作，不替人作专业判断。

**退出证据**

- 图谱、官方数据与通道三条依赖均有真实生产证据；
- 调度重复、宕机恢复、预算耗尽与授权拒绝的反例不会重复执行或越权；
- 客户把告警和确认纳入真实合规或业务流程，并可追溯到来源和责任人。

**禁止抢跑**

- executor 对 trigger 不敏感不等于支持 scheduled invocation；
- 不按每个 resume 重置预算，不用事后 gate 追认已经发生的 effect；
- 不在图谱治理、官方数据或通道任一缺席时宣传持续监控。

## Stage 4 — API 与生态

**进入条件**

- Stage 2–3 的身份、治理、监控和实际采用已经成立；
- 外部集成需求可以由稳定、版本化、可撤销的契约承载。

**最小交付**

- 对外提供版本化 API、薄适配器和垂类包准入工具；
- 第三方只接入序列化 command/event wire、schema 与受限 effect，不取得进程内 callback、secret 或任意执行能力；
- playbook、数据源和垂类扩展具备发布、兼容、撤回与审计机制。

**退出证据**

- 至少一个外部系统在兼容门、权限门与故障恢复下稳定集成；
- package/API 升级、降级和撤回不破坏既有案件账本；
- 生态贡献可以独立验收，且不改变 Courtwork 的人工裁决边界。

**禁止抢跑**

- 不在内部 ABI 尚未被真实场景稳定使用时开放市场或任意插件；
- 不把企业 SDK 编排放入通用 core；
- 不为生态想象提前引入第二 runtime、远程后端或跨进程协议。
