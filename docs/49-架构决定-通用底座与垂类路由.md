# 架构决定：通用底座与垂类路由（2026-07-10）

状态：已拍板。核心律：**用户选名词（建什么容器），不选形容词（什么模式）**——"通用/垂类模式切换"永不出现在 UI。

## 三层机制

1. **通用底座 = 无容器默认态**：轻对话 + composer + 通用动词（上传/工作稿/fetch/文件整理/系统动词）。FileOpsPlan/工作稿轨/阅读视图本为领域无关（docs/22 红利），底座是摘掉法律皮肤后的剩余全部，零垂类 Button，无需单独实现。
2. **垂类 Button = 场景卡片，双信号点亮**：容器类型（新建案件 vs 新建工作区——new task 的选择点，选容器不选模式；案件 → S1–S4 常驻，工作区 → 通用场景族）+ 内容识别（分类器推荐置顶，推荐不触发，docs/24）。
3. **路由 = 声明过滤非代码分支**：场景声明增加 `context: case | workspace | any`（注册表 v2 清单 +1），货架按容器类型 × trigger 匹配过滤展示。边界：普通工作区点法律场景卡 → popover 轻确认"升级为案件"（单向仪式，与存入卷宗同族）。

## 视图归属速查

- 通用底座视图：对话流、工作稿画布、整理计划表、附件 chip、产出卡。
- 垂类视图（随案件容器出现）：五工作面（时间线/图谱/矩阵/修订预览/起草画布的法律模板）、法理之线、信源角标、门禁卡。
- 两者共用：状态条、⌘K、专注模式、设置——全局层永远容器无关。

## 落地清单

- 注册表 v2：`context` 字段（与 retrievalPolicy 同批）。
- UI：新建工作区动词 + 容器升级 popover（F-4 之后切单，热点文件排队纪律）。
- MVP demo 不受影响（演示即案件容器态）。
