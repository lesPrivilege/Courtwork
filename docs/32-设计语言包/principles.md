# 组件设计原则（设计语言包 4/6，2026-07-09）

约束后续所有组件量产（Opus 四件套）与 UI 代码（GPT-5.6）。与 docs/30 硬性条款同效力。

## 1. 容器有界

一切数据容器用 `1px border.hairline` 划界，**阴影全站归零（含弹层，二轮修正⑥）**——弹层的层级靠 1px 描边 + 底色差表达。律师需要信源容器边界——无边框的"法条雪崩"是规避对象（docs/31 采信 Gemini 反方理由）。圆角上限 6px（卡/行 4px，面板与 callout 6px，表格与文书纸面 0）。

**完整工作台帧（二轮修正①）**：任何屏必须是完整帧，四横带自上而下：标题栏 40px（品牌 + 案件名 + 案号 mono + ⌘K）、工具栏 40px（阶段面包屑 + 文档级动作）、三栏工作区（每栏带 40px 面板头：标题 + mono 计数）、全宽状态条 32px（含上下文用量圆盘）。孤立漂浮的内容区 = 演示稿，打回。

**贯穿式网格线 + 8px 间距审计（二轮修正④）**：四横带分隔线与三栏竖分隔线贯穿整窗；面板头等高对齐，横线在栏间连续不断；列表行分隔线通面板全宽。一切 padding/gap/margin 取 8 的倍数（微尺寸内部对齐允许 4），奇数间距即违规。

## 2. 颜色预算

- 彩色只能来自 `color.semantic.*` 与 `color.line.*` token；组件不得引入新色值。
- 语义色严格双轨：`*.graphic` 只用于法理之线、图谱边、圆盘等图形；`*.fg` 只用于 12px 及以上文字。组件不得用图形色渲染文字，也不得为追求“同色”绕过 AA。
- 全站色相恒为 4（红/琥珀/蓝/绿）+ 板岩灰。同色相跨域复用是纪律：红=危险/删除，琥珀=待人处理/未核验，蓝=新增/可点击，绿=已核验/已确认，灰=中性/退出工作集。
- 主操作按钮用 ink（`action.primaryBg`），不占语义预算。
- 低危、已驳回等"不需要注意"的状态一律灰——**不给颜色也是给信息**。

## 2a. 划界：法理之线 vs AI callout（二轮修正③，硬性）

两种左缘竖线元素，职责互斥，禁止混用：

- **法理之线（数据容器）**：纯白卡/行（`bg.raised`）上贴边通高 **2px** 竖线。承载一切结构化数据（风险、事件、修订、文书），配徽章/门禁/溯源。**语义色 tint 禁止做卡片或列行背景**——状态只由线色 + 徽章/chip 表达，卡面永远纯白。
- **AI callout（发言容器）**：`border-left 4px` + 语义 tint 底 + `radius 6`。只承载 AI 的建议/提醒/解释性插话，**绝不承载徽章、门禁、溯源数据**。tint 作底仅在此处（及徽章/chip/修订字符底纹）合法。

一眼规则：**白底细线 = 数据，彩底粗线 = AI 在说话**。用户永不需要猜一块内容能不能当作事实引用。

## 2b. 紧凑列表优先（二轮修正②）

同构数据集合（风险清单、事件时间线、卷宗列表、矩阵）默认用 **28–32px 行高的紧凑列表 + 列头**（`component.listRow`），不用卡片流：单行省略号截断，完整内容靠展开行（行内展开，非弹窗）；展开行内才出现描述段落、依据底纹区、门禁动作。卡片保留给异构内容（对话流产出卡、摘要卡）。列宽纪律：编号/日期/法条/金额列用 mono + tabular，金额右对齐。

## 3. 动效分界（含长任务微反馈规格）

两半各居其位（docs/31 成文裁定）：

**数据区绝对静止**（Midday 原则）：表格、时间线、图谱、文书、数字——渲染完成后没有任何自发动效。数据晃动 = 数据不可信。

**长任务微反馈为必做组件**：任何预期 2–5 秒的运算（审查、核验、生成、摄取单步）**必有**以下之一，否则会被读成死机：

- **边框微光流转**：承载卡片的 1px 描边上，一段亮色（ink 提亮）沿边框匀速流转，周期 1800ms linear infinite（`motion.longTaskGlow`）。用于"正在对这张卡运算"。
- **骨架呼吸**：占位骨架块 opacity 0.55⇄1.0，2000ms ease-in-out alternate（`motion.longTaskBreath`）。用于"结果尚未到达"。

分档：< 800ms 的操作瞬发完成，不做任何动效；2–5s 用上述微反馈；> 5s（如整卷摄取）升级为**事件流驱动的进度组件**（进度数字 + 当前步骤文案 + 骨架呼吸），进度文案用办案语言（"正在识别文书类型…"），禁止转圈 spinner 裸奔。

**状态变更永远 0ms 硬切**：门禁落章、风险定级、修订接受——瞬间完成，不淡入淡出。悬停反馈全站统一 120ms ease-out（`motion.hover`）。

### 3a. 交互确认反馈阶梯（P-2 架构审定）

数据本体保持绝对静止，但用户动作必须即时得到独立回执。全站只允许 CSS transition/animation 与 Web Animations API，不引入 motion 库；可动画属性严格限于 `transform`、`opacity`、`background-color`、`border-color`。

| 场景 | 时长与缓动 | 落地纪律 |
|---|---|---|
| 按压态 | 70ms ease-out | 操作型按钮只加深底色；数据卡零位移、零缩放 |
| hover | 120ms ease-out | 全站统一，只动背景色/边色 |
| Tab 切换 | 指示器 100ms ease-out；内容 0ms | 只动画指示器 transform，内容区瞬切 |
| 面板对切 | 0ms | 无过渡、无 crossfade |
| 确认/驳回落定 | 本体 0ms + 光效层 150ms ease-out 衰减 | 光效层只动 border-color，不延迟最终态 |
| 续行跳转 | 240ms ease-out | 回执只动 opacity + `translateY(≤4px)`，入口保留原位 |

### 3b. 动效与路由八禁（P-2 逐条自查基线）

1. 禁止整卡/整行位移或缩放；数据容器不得抬升、回弹、放大缩小。
2. 禁止任何弹簧、overshoot、阻尼振荡或 rubber-band 物理回弹。
3. 禁止 spinner 裸奔替代边框微光、骨架呼吸或事件流进度事实。
4. 禁止状态本体淡入淡出或跨色相过渡；确认光效必须是独立叠加层。
5. 禁止 Tab/面板内容区 crossfade；只有 Tab 指示器可动。
6. 禁止 hover 阴影升起；反馈一律由 background-color / border-color 表达。
7. 禁止动画 `width/height/top/left/margin/padding` 等 layout 属性。
8. 禁止结构位或功能入口物理消失后重现；三栏帧与五工作面永不隐藏，空数据用文字型空态，依赖前置条件的功能入口用禁用态 + tooltip 保留空间记忆。

## 4. 留人确认门禁

每个关键产出节点是一个门禁组件：三态徽章（`gate.pending/confirmed/rejected`）+ 三动作（**确认 / 驳回 / 修正**）。修正是 schema 级操作（选中元素改字段），不是自由文本框。不可逆动作（编译为 Word、开启新阶段）永远需要显式点击，永不自动触发。门禁未清空时，下游动作（如"编译为 Word"）呈禁用态并注明剩余数量（"3 项待确认"）。

## 5. 零技术概念暴露（硬性）

UI 任何位置不出现：终端/命令/路径/token/markdown/JSON/schema/上下文 等词。词汇表以办案语言为准：**案件、卷宗、阶段、文书、依据、溯源、定稿、归档、开启新阶段**。系统动作按钮化："打开 Word 文档""在文件夹中显示"。状态条红色态文案："本阶段内容较多，开启新阶段继续"。

## 6. 溯源必达

任何结构化数据点（事件、风险依据、矩阵单元格、图谱边）必须携带可点击的溯源入口：文件名 + 原文引语，点击跳原文并高亮。引语用 mono meta 档展示。信源角标 A/B/C 常伴引用出现（规格见 signature-line.md §五）。无锚点的数据不得进入右栏——这是 schema 契约（sourceAnchors）在视觉层的镜像。

## 7. KBD 常驻

高频动作旁常驻快捷键提示（规格见 typography-density.md §五）。首批保留位：`⌘K` 场景/命令、`J/K` 列表逐条、`⏎` 确认、`⇧⏎` 驳回、`Esc` 收起面板。

## 8. 「编译为 Word」冻结仪式

显式、单向、一次性（docs/23）。按钮用 ink 主按钮 + 明确动词（"编译为 Word 文档"）；点击后二次确认弹层说明后果（"定稿后本画布转为只读存档"）；完成后画布 0ms 硬切只读态：底转 `bg.surface`、编辑控件消失、顶部出现归档徽章（mono，含定稿时间）+ "打开 Word 文档"按钮。永不提供"转回编辑"。

## 9. 状态条

中栏底部常驻，上缘 1px 网格线。左：上下文用量圆盘（12px 环形，三态 `usage.*`；悬停浮层展开卷宗占用/对话占用/可压缩项明细）+ 摄取余量（credits 数字，tabular）。右：当前阶段名。红色态时圆盘旁直接展开"开启新阶段继续"按钮。

## 10. 分层确认（docs/30 §二.6 防呆原则①）

确认门禁**不设全局"全部确认"按钮**。批量确认的合法范围有硬边界：仅限 severity 为中/低、且全部依据不含 tier.c（未核验源）的待确认项；批量栏必须枚举其范围与数量（"中/低危且依据已核验 · N 项"）。**severity.high 与含未核验依据的条目永不入批量**——必须逐条展开卡片、可见其全部依据后才能确认，卡片上以"不入批量 · 逐条确认"徽章明示。驳回不提供批量：驳回是逐条决定。

## 11. 生成与确定的视觉区隔（docs/30 §二.6 防呆原则②）

两类内容在任何容器里都保持稳定区隔，规则只有一条、全站通用：

- **AI 生成的解释性文字**（风险描述、事件叙述、修订说明）：sans 字族、常规底、`text.primary/secondary`。
- **工具核验结果与引用**（原文引语、法条引用、核验结论、编号/日期/金额）：mono 字族 + `provenance.verifiedBg` 底纹（#F5F7F9，radius.sm）。

区隔靠**字族 + 底纹**两个通道冗余表达，不靠颜色（彩色预算属于语义状态）。用户应在三天内习得"有底纹的 = 机器核验过的原文，没底纹的 = AI 的话"这一条规则，且永不遇到反例。

## 12. 命名纪律

组件与 token 命名领域无关（docs/22）：tier/severity/revision/gate/usage/line。法律语义只存在于文案与数据。评审时在 token 名或组件名里看到法律词汇即打回。
