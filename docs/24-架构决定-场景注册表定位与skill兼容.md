# 架构决定：场景注册表定位与 skill 兼容（2026-07-09）

状态：定位阐明，已拍板。

## 定位

场景 = **预编排的固定 pipeline**（声明式：触发条件 → 输入 schema → 工具集 → 生成节点 → 产出 schema → 确认门禁 → UI 模板），与 core 解耦（core 只是通用执行器）。**不靠 LLM 在推理中触发或选择调用**——选择权属于用户与声明，不属于模型：

- 入口 = 场景按钮 + 触发条件匹配；执行 = 声明写死的编排骨架；LLM 只在骨架内的生成节点工作。
- 场景识别分类器只做**推荐**（按钮排序/置顶），不做自主触发；推荐错误零代价，用户点击才算数。
- 聊天兜底中 LLM 可建议跑某场景，执行仍走注册表管道，无旁路。

与通用 agent skill 机制的本质区别是选择权归属：可预期性（同输入 → 同流程 → 同形状产出）是机构采购与低认知用户的产品力，优先于模型自主的灵活性。

## Skill 生态兼容

场景声明本质是数据，与 skill 生态双向转换无技术障碍（skill 包装为场景 / 场景导出为 skill）。**硬条件：外部 skill 进注册表必须补齐产出 schema 与确认门禁两个强制字段**——不存在绕过 schema 校验与留人确认的"裸 skill"路径。兼容的是入口，不是纪律豁免。
