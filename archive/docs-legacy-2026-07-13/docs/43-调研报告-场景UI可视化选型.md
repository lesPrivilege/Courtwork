# 调研报告：专业工具内嵌可视化——关系图谱/时间线/图表的库选型与品味来源（2026-07-10）

状态：调研完成，供架构会话拍板量产版可视化选型；不改动 docs/05 已核 license 的既定选型（矩阵 AG Grid Community、PDF 高亮 react-pdf-highlighter），只处理本报告范围内三项：关系图谱、时间线、图表通用层（含 Craft 品味拆解与布局算法专项评估）。

## 〇、背景复述（不重新论证）

- docs/05 §7 已拍板前端可视化组件基线：时间线 vis-timeline、图谱 AntV G6、表格 AG Grid Community、PDF 高亮 react-pdf-highlighter，license 均已核（MIT/Apache-2.0/双授权）。
- docs/32 tokens.json + principles.md 硬性约束：浅色唯一、语义色预算 4 色相（红/琥珀/蓝/绿）+ 板岩灰、`color.semantic.*` 双轨（graphic 图形 / fg 12px+ 文字）、数据区绝对静止（`motion.dataRegion: none`）、`tabular-nums` 硬性、圆角上限 6px、阴影全站归零、法理之线贴边 2px 与 AI callout 4px 划界。
- docs/19 硬性：全站动效只用 CSS transition/animation，不引入 framer-motion 等动画库；数据区静止 + 长任务微光/骨架呼吸 + 状态 0ms 硬切三段式。

## 一、Craft 拆解：AI 生成图表为什么仍然好看

### 1.1 关键发现：Craft 把"品味"开源了

Craft（craft.do）团队为其 Craft Agents 产品自研并开源了图表渲染引擎 **beautiful-mermaid**（`github.com/lukilabs/beautiful-mermaid`，MIT，9.9k star，339 fork，2026-02 发布 v1.1.2，持续维护）。这是本次调研含金量最高的一手材料——**Craft 官方在 README 里直接写明了"为什么默认 Mermaid 渲染器不够好看"以及他们的修复方法**，等于把品味来源公开成了工程规格。

### 1.2 机制拆解：约束语法 + 收口渲染，AI 从不触碰"怎么画"

1. **输入层收窄到声明式语法**：AI/Agent 不直接输出 SVG 或像素指令，只允许输出 Mermaid 文本（flowchart/state/sequence/class/er/`xychart-beta` 六种图）。这是品味的第一层来源——生成自由度从"任意像素"收窄到"图的拓扑/数据对不对"，AI 训练语料里 Mermaid 语法样本量巨大，出语法错误的概率远低于"自由画图"。
2. **渲染与主题完全独立于 AI 输出**：`parseMermaid` 解析文本 → 布局引擎（flowchart 用 **ELK.js**，同步跑，通过 FakeWorker bypass 避免异步）→ 渲染成 SVG，颜色/字体/圆角/网格风格全部由一套**两色派生系统**统一套皮：只需提供 `bg`（背景）与 `fg`（前景）两个颜色，其余 10 个视觉角色（正文、次级文字、边线、箭头、节点填充、描边等）通过 `color-mix()` 按固定比例派生（如"次级文字 = fg 60% 混入 bg"）；15 个内置主题 + 任意 VS Code Shiki 主题可直接复用。**AI 永远碰不到颜色/字体/网格这些"品味"参数，只能决定拓扑结构**，主题切换是 CSS 自定义属性实时生效，不触发重渲染。
3. **XY 图表（对应 Courtwork 未来监控/统计挂账）明确写了设计哲学**，原文："The chart renderer follows a clean, minimal design philosophy **inspired by Apple and Craft**"：点阵网格（非实线网格线）、圆角柱、三次样条平滑曲线（非折线/阶梯线）、浮动无轴线标签（无可见坐标轴与刻度线）、线条下方投影、单色系配色（仅从主题 accent 色派生深浅，不引入多色相）。这套具体清单与 Courtwork tokens.json 的"语义色预算 4 色相 + 数据区静止 + tabular-nums"高度同构，可作为图表通用层默认审美的直接参照基准。

**一句话结论**：Craft 的 AI 生成图表好看，不是因为模型审美好，而是**语法层收窄（Mermaid）+ 渲染层收口（自研固定主题系统）**——AI 唯一的自由度是"画什么"，不是"怎么画"。这直接印证 Courtwork 的既定架构：场景执行输出 typed artifact（schemas），渲染皮肤在 output/reading-view/apps/desktop 侧自控，不给 LLM 任何样式自由度。这不是巧合的相似，是同一个工程判断在两个不同产品里各自独立得出的结论。

### 1.3 同类参照

- **Notion charts**：2026 年 3 月新增"Breakdown by"堆叠维度与仪表盘聚合能力，但样式选项克制、非 AI 自由生成，是 BI 式声明式配置——同样不给用户/AI 挑颜色的自由，只挑图表类型与筛选维度。
- **Linear Insights/Dashboards**：仪表盘卡片"模块化 + 一致克制视觉"，颜色/字体不可由用户改，只能选 chart 类型与 filter，交互聚焦在"hover 看数字、点选筛选"而非视觉定制。
- **Observable Plot**：与 Craft 相对的另一极端——不是产品是库，默认继承宿主 `system-ui` 字体与 `currentColor`，本身不下审美判断，把"品味"外包给宿主页面的 CSS。这是"极简底座，等宿主来精修"的路线，与 Craft"内置固定精修主题"的路线互补，不是同一件事：**Plot 适合已经有一套设计语言、想低成本继承的场景（正是 Courtwork 的处境）；Craft 式固定主题适合独立产品自己就是终端消费者**。

## 二、关系图谱（最弱项，重点）

### 2.1 现状与问题

apps/desktop 当前是手绘 SVG 手排坐标（B 阶段验证态），10 节点验证通过，但案件图谱量级是 14–100+（Stage 2 团队图谱库会继续增长）。手排坐标在节点数超过约 20–30 个后不可维护：新增节点需要人工重新排布坐标，这与"数据区静止但内容会随案件推进持续追加"的产品形态直接冲突——**不是要不要换库的问题，是手绘路线本身在当前 MVP 量级末端已经到了工程上不可持续的临界点**。

### 2.2 库选型：AntV G6 5.x vs Cytoscape.js vs xyflow vs sigma.js

| 维度 | AntV G6 5.x | Cytoscape.js | react-flow/xyflow | sigma.js |
|---|---|---|---|---|
| 生态规模 | npm 周下载约 14.7 万，GitHub star 约 12.1k | 周下载约 205.7 万（10 倍+），star 约 10.7k | 面向"可编辑节点 UI"生态，非同类对比基准 | star 约 12k，WebGL 专项 |
| 维护状态 | 高度活跃，5.x 持续迭代（5.1.1 于 2026-05 发布），5.0 起内置浅/深主题一键切换 | 高度活跃，图论算法生态成熟（中心性、社群发现、最短路径） | 高度活跃，定位"可编辑节点式 UI"而非图分析 | 中，专注大规模图渲染 |
| 中文文档 | **一手中文官方站**（g6.antv.antgroup.com），案例库含"企业关系""决策场景"现成模板 | 纯英文生态 | 英文 | 英文 |
| API 范式 | 声明式配置，开发效率高，部分实现细节与文档存在"中文为主"局限 | 命令式，需手写更多交互响应代码，学术/生物信息学场景验证深 | 每个节点是 React 组件，主打"可拖拽表单/可编辑流程图" | 图论能力弱，专注渲染 |
| 性能上限 | 中等规模（千级）DOM/Canvas 混合渲染 | 中等规模，图论计算开销随算法而定 | 面向编辑场景，非大规模渲染优化 | **WebGL，10 万–50 万节点级** |
| 定位匹配 | **只读溯源展示 + 中等规模关系网络** | 需要图论计算（中心性/社群）时更优，当前无该需求 | "可编辑表单式节点"与当事人关系图（只读溯源）错配，不建议引入 | 性能远超当前及可预见量级需求，引入 WebGL 调试/皮肤定制门槛不划算 |

**裁定：换库到 AntV G6 5.x**。理由：(1) 5.x 并非停滞项目，持续发版且内置主题系统；(2) 中文文档是团队协作效率的实际杠杆，非虚指标；(3) Courtwork 当前需求是"渲染 + 缩放 + 溯源点击"，不需要 Cytoscape 的图论计算深度；(4) xyflow 的"可编辑节点表单"定位与当事人关系图（只读展示）错配；(5) sigma.js 的 WebGL 性能天花板远超当前 14–100+、Stage 2 可预见的量级，引入 WebGL 渲染管线的调试与皮肤定制成本（着色器定制门槛远高于 DOM/SVG）不划算，列为 Stage 2 量级突破后的备选而非当前裁定。这与 docs/05 既定选型一致，本报告是对该选型的加固验证而非推翻。

### 2.3 布局算法单独评估

**预设判据"布局算法不自研、渲染皮肤必自控"——本次调研成立**，证据链：

1. **Craft 自己的实践**：beautiful-mermaid 的 flowchart 布局用的是 **ELK.js**（同步跑），Craft 团队自研的是主题/渲染层，没有自研布局算法。这是"专业工具如何处理布局"的一手证据，不是类比推断。
2. **dagre vs ELK**：dagre 代码停留在 2015 年基本不再更新（仍偶尔接受 PR），是"简单快、够用"但功能单薄的选择，对祖先→后代类连接（如容器到其子节点）处理不了；ELK 由学术团队 part-time 持续维护，更新更活跃，配置项远多于 dagre，支持分组容器等复杂约束。**若 Courtwork 需要"案件→当事人→关联企业"这类多层嵌套分组关系，ELK 更合适；若只是简单有向层次图，G6 5.x 内置的 dagre-layout 仍是最省心的起点**——两者都在 G6 生态内，切换是配置项而非重新选型。
3. **d3-force（力导向）vs dagre/ELK（层次布局）**：行业共识是力导向更适合"探索性、无预设结构"的展示（"demo 好看但代表真实数据不如 dagre" —— Hacker News 长期讨论共识），层次布局更适合"有向、有清晰层级"的真实数据场景。
4. **法律当事人图谱的先例**：中文法律实务圈公认"诉讼图表三原图"之一是"法律关系图"（另两图为时间关系图、程序结构图），股权/控制关系图行业惯例是**层次布局**（股东在上层、被控公司在下层）——企查查/天眼查等商业股权穿透图清一色用树状/层次布局，部分辅以力学布局做"关系网"总览态。Palantir Gotham 的 Graph 模块不对外公开具体布局算法，但其"Canvas 自由排布 + 分析师手动整理节点"的设计说明专业调查分析工具**默认给分析师排布自由度，不强制纯自动布局**——这对 Courtwork 是一个信号：默认态用层次布局呈现清晰的从属/控制/委托关系方向，但应保留"用户可手动微调后固定"的逃生舱（这属于图谱整理动作，与已否决的"物理弹簧拖拽美学"不冲突——拖拽用于排布不是数据本体自发动效）。

**裁定**：法律当事人/股权关系图默认用**层次布局**（G6 内置 `dagre-layout` 起步，超过约 50 节点或需要分组容器时切 `ELK.js`，同一套 G6 主题皮肤复用，只换 layout 配置）；不提供力导向作为默认视图（法律关系是确定性从属/控制/委托关系，不是探索性网络，力导向的物理弹跳观感也与 docs/35 已否决的弹簧美学方向冲突）。d3-force 仅可作为"节点数极多、用户主动切换为探索总览态"的可选二级视图，非默认，MVP 阶段不做。

### 2.4 渲染库 + 自控皮肤的现实成本

**一次性成本**：(1) 参照 beautiful-mermaid"两色派生 N 角色"的模式，写一个 `courtworkG6Theme(tokens)` 主题映射函数，把 `tokens.json` 的 `color.semantic.*`、`border.hairline`、`typography.*` 套进 G6 的 node/edge/canvas 默认 style（G6 5.x 主题是声明式 Graph Options 子集，只需维护少量语义色变量，不必逐组件写死颜色），量级约单人 1–2 周；(2) 溯源点击/hover 态交互复刻 principles.md 的 120ms hover、0ms 状态硬切等动效纪律，G6 5.x 的 behavior 系统可挂自定义事件，工程量不大。

**持续成本**：每次 `tokens.json` 改版需要同步一份 G6 主题配置，与 output/reading-view 包"契约先行、渲染层各自实现"的既有工程纪律同构，非首次接触，团队已有对应经验模式。

## 三、时间线

### 3.1 vis-timeline 现状

2026-07 检索显示最近更新到 2026-07-02，不是废弃项目，但项目量级中等（star 约 2.5k），issue 列表有约 19 个"help wanted"标记，不算高研发投入项目。样式定制依赖 CSS 覆盖官方 class，要做到 Courtwork "数据区绝对静止 + 法理之线 + tabular-nums"这套细颗粒度自定义视觉语言，需要大量 CSS override，且每次上游库演进都要重新做回归测试。

### 3.2 表格式 vs 轨道式：法律案件叙事哪个更合适

apps/desktop 当前的时间线是手绘表格式（非 vis-timeline），已在 B 阶段 UX polish 中通过验收（P-1 语义收敛微补丁、Playwright 回归、法理之线直接消费 `TimelineEvent.markers`），证明该路线技术上可行且已过产品验收，不是"权宜之计"。

行业参照（头部 legal AI 产品的选择，具有很强的说服力）：

- **Wexler.ai**（"Fact Intelligence"定位，2025 年获 $5.3M 融资）：核心产出是从证据集抽取事件/实体/时间线/矛盾点，每个事实绑定到确切来源句子，产出物是可以直接进入摘要/动议起草的"完整时间事实地图"，形态是结构化事实表而非可交互轨道。
- **Clearbrief**：核心功能"Hyperlinked Timelines"——一键在 Word 里生成带超链接（跳转回源文档）的年表（chronology），本质是**带来源超链接的结构化表格/文档**，直接生成到 Word，而非交互式轨道时间轴。
- **中文法律实务圈**：诉讼可视化三原图之一"时间关系图"的主流实践是"以时间为线索、大纲方式罗列事件"，商业工具（如聚法案例）同样是"时间轴管理"而非可视化轨道部件。

**结论**：**表格式时间线更贴合法律案件叙事**——适合 47–数百事件量级的顺序阅读、逐条溯源，用户诉求是"能塞进诉状/代理词的年表"，不是"能拖拽缩放的时间轴部件"。轨道式时间线（vis-timeline 的强项——可视化区间重叠、并行事件、缩放导航）在"多主体并行事件、需要看时间密度/重叠"的场景（如多方谈判同步进程）有独特优势，但不是当前样板案的主要叙事需求。

**裁定：沿用手绘（表格式），不引入 vis-timeline**。理由三条：(1) 表格式已过 B 阶段验收，重写成本无法被 vis-timeline 的"轨道能力"证成——该能力当前无场景消费；(2) Wexler/Clearbrief 头部同行印证表格/超链接形态是法律叙事场景的行业选择，不是"没资源做交互轨道"的将就；(3) vis-timeline 要做到与既定设计语言（法理之线贴边 2px、0ms 硬切、tabular-nums、hairline 网格）同等精细度的自定义，需要覆盖式改写其 CSS 且承担上游库演进的回归风险，成本高于收益。

Stage 2 若出现"多方谈判/多案并行需要看时间重叠密度"的新叙事需求，再重新评估 vis-timeline 或自绘轨道式时间线的必要性，不在当前 MVP 工单范围。

## 四、图表通用层（未来监控/统计，挂账）

| 库 | License | 包体积 | 默认品味 | 定制到 Courtwork 美学的成本 |
|---|---|---|---|---|
| **Observable Plot** | BSD-3（最宽松，商用无忧） | 小，声明式 marks 按需树摇 | 默认 `system-ui` 字体、颜色继承 `currentColor`，不提供固定成品皮肤——"语法糖库"而非"精修产品" | 低：一旦画好基于 tokens.json 的默认样式（hairline 灰网格线、tabular-nums 数字），后续任意图表复用同一套 grammar-of-graphics 声明式 API，长期维护成本低 |
| **ECharts** | Apache-2.0 | 约 100–300KB gzip（视引入模块） | 功能最全，默认动效丰富、渐变色多，热力图/仪表盘/大数据实时更新是专项强项 | 中高：需要逐项关闭默认动效与渐变以贴合"数据区静止 + 4 色相预算"的克制方向，选项字典式 API 比声明式繁琐，但一旦配好可复用 `echarts.registerTheme` |
| **Chart.js** | MIT | 最小（约 60KB） | API 最简单，适合快速出图 | 定制粒度不足，复杂仪表盘（多轴/热力图）支持较弱，不适合未来监控面板 |
| **Recharts** | MIT | 中等 | React 组件化友好，默认观感"够用的中间选项" | 定制天花板不够高，相对 ECharts 无热力图/大数据优势 |

**裁定：MVP 阶段图表通用层暂不引入**（当前无消费场景，矩阵/时间线/图谱已覆盖主要结构化数据展示需求）；一旦 Stage 2 监控面板/热力图/S5 对比统计视图立项，按需求量级分诊：

- **轻量统计卡片/趋势小图** → Observable Plot。与"极简底座、自己画皮肤"的路线一致，可直接复用团队已经为 G6/时间线积累的语义色映射经验，长期维护成本最低。
- **重仪表盘/热力图/大数据实时更新** → ECharts。功能最全，克制其默认动效与配色即可，不需要为热力图这类 ECharts 强项重新造轮子。

不建议 Chart.js（能力天花板不够未来监控面板需求）或 Recharts（定制天花板不足以达到本产品的精修皮肤要求，且相对 ECharts 无热力图优势）。

## 五、裁定速览表与分期

### 5.1 五工作面/关键节点一行结论

| 工作面/节点 | 现状 | 裁定 | 一句话理由 |
|---|---|---|---|
| **关系图谱** | 手绘 SVG 手排坐标 | **换库**：AntV G6 5.x + 自控浅色皮肤 | 节点量级持续增长（14→100+→Stage 2 更多），手排坐标不可扩展；G6 声明式 API + 中文文档 + license 已核 |
| **布局算法（图谱内）** | 手排坐标 | **不自研**：默认层次布局（G6 内置 dagre-layout 起步，规模/分组需求增长后切 ELK.js），力导向仅作非默认二级探索视图 | 法律当事人/股权关系是确定性从属关系非探索性网络，层次布局是行业惯例（股权穿透图/诉讼三原图），Craft 自身也用 ELK.js 而非自研布局 |
| **时间线** | 手绘表格式 | **沿用手绘**，不引入 vis-timeline | 已过 B 阶段验收；Wexler/Clearbrief 头部同行印证表格/超链接形态才是法律叙事场景选择；vis-timeline 定制成本高于其未被消费的轨道能力收益 |
| **图表通用层**（监控/热力图，挂账） | 无 | **挂账分诊**：轻量统计 → Observable Plot；重仪表盘/热力图 → ECharts | MVP 无消费场景，Stage 2 按需求量级选型，避免现在预先决定 |
| 矩阵 / PDF 高亮 | AG Grid Community / react-pdf-highlighter | 维持既定（docs/05），不在本次调研范围内变更 | — |

### 5.2 "布局算法不自研、渲染皮肤必自控"判据的成立性

**整体成立，但需要一处颗粒度修正**：不是"渲染皮肤"整体自研，而是"渲染皮肤的主题/配色/字体/动效层"自控（通过 G6/Plot 的主题系统吃 tokens.json），底层 SVG/Canvas 绘制逻辑仍完全复用库代码，自研的只是"主题映射配置"这一层——与 beautiful-mermaid"两色派生 N 角色主题系统"同构。这比"自研渲染引擎"轻得多，是 Courtwork 应该继续坚持的正确颗粒度：**自研的边界画在"主题映射函数"，不画在"渲染管线"或"布局算法"**。

### 5.3 tokens 对接方式

- **G6 5.x**：主题是声明式 Graph Options 子集，可写一个 `courtworkG6Theme(tokens)` 函数把 `color.semantic.*`/`border.hairline`/`typography.*` 映射进 node/edge/canvas 默认 style，一次性适配层，与 output/reading-view 包"契约先行"的既有工程纪律同构。
- **Observable Plot**：默认继承宿主 CSS（`currentColor`/`system-ui`），对接反而更省事——声明式 marks 直接传 tokens 里的色值即可，无需额外主题注册机制。
- **ECharts**：走 `echarts.registerTheme` 机制，同样支持完整主题化，但选项字典式配置比 G6/Plot 繁琐，一次性投入更高。

### 5.4 分期

**MVP 量产版（当前工单范围）**：

1. 关系图谱换库到 G6 5.x + 编写 `courtworkG6Theme` 主题映射 + 默认 dagre 层次布局（G6 内置），溯源点击/hover 态按 principles.md 动效纪律实现。工作量评估：中等，建议单独立项，由架构会话另立工单排入 docs/10 实施切分，不与本报告一并动手。
2. 时间线维持现状手绘表格式，不动。
3. 图表通用层不引入，保留挂账。

**Stage 2（团队图谱库继续增长后）**：

1. 图谱节点量级若持续增长超过约 50–100 节点，或需要"案件→当事人→关联企业"多层嵌套分组，评估从 `dagre-layout` 切 `ELK.js`（G6 5.x 同样内置支持，同一套主题皮肤复用，只换 layout 配置，非重新选型）。
2. 若节点量级向数千甚至万级演进（团队图谱库场景），重新评估 sigma.js 的 WebGL 渲染必要性。
3. 若 S5 对比视图/监控面板/热力图需求实际立项，按 §4 分诊表引入 Observable Plot 或 ECharts，并补一份该次选型的独立主题适配。
4. 若出现"多方谈判/多案并行需要看时间重叠密度"的叙事需求，重新评估 vis-timeline 或自绘轨道式时间线。
