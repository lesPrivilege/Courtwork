# W9 设计 Brief 与 Handoff 规范（2026-07-09）

状态：预备文档。W6（core）验收放行后启动 Design；本文先固化审美方向、handoff 格式与代码化工序，避免临时拍脑袋。

## 一、审美方向

**反面清单（硬性规避）**：暖黄/奶油色底 + 衬线字体标题的"AI 产品默认审美"（2025–2026 已泛滥）；玻璃拟态堆叠；渐变彩虹强调色；装饰性插画占据工作区。

**正面参照范式**：mimo.mi.com 一类冷调科技极简——冷白/浅灰底、高对比近黑正文、无衬线字体（中文 UI 走系统栈：PingFang SC / 微软雅黑 / MiSans 类）、大留白、**单一克制的强调色**、以排版层级而非色块承载结构。

**法律 ToB 的三条修正**（不照抄消费级官网）：
1. **信任感优先于新奇感**：目标用户是普通法律从业者，界面要像专业工具不像玩具；动效克制（状态反馈用，不做表演）。
2. **信息密度适中偏高**：律师习惯高密度数据（卷宗清单、矩阵、时间线），留白服务扫读而非空旷；正文字号与行高按长文阅读调校。
3. **零技术概念暴露（硬性）**：UI 任何位置不出现 CLI/终端/命令/文件路径/token/markdown 等技术概念。CLI 保留为开发、验收与运维（私有化部署）的内部客户端，与用户界面无关。系统动作一律按钮化：如"打开 Word 文档"按钮 = 壳层调系统默认程序（Tauri shell open），"在文件夹中显示"同理。文案词汇表以办案语言为准（阶段/卷宗/定稿），docs/30 各处已有实例（续行按钮、编译仪式）。
4. **语义色系统**：颜色预算优先给语义——信源等级角标（A 已核验/B 库内/C 网络参考）、风险等级（高中低）、修订痕迹（增/删）、确认门禁状态。装饰性用色让位。浅色主题优先（律所办公环境），深色主题不做进 MVP。

## 二、Design 输入包(交给 Design 的材料)

1. 产品语境：docs/03-MVP规划（三栏布局与四场景）、docs/02 调研报告第 3 节（UI/交互模式盘点，Legora Tabular Review 等参照）。
2. 结构清单：左栏案件列表；中栏对话流 + 场景卡片（按钮主入口）；右栏五交互——时间线（可点击溯源）、关系图谱、矩阵审阅、docx 修订预览、起草画布（md 渲染编辑，见 docs/23）；"编译为 Word"冻结仪式交互（含编译后画布转只读态）；确认门禁交互（每个关键产出的确认/驳回/修正）；摄取进度（事件流驱动）。
3. 真实填充数据：**packages/demo-data/data/artifacts/ 的五个预生成 artifact + 样板案语料**——设计稿一律用真数据形状填充，禁止 lorem ipsum 与编造字段。
4. 协议约束：UI 是 core 事件流协议的纯客户端；所有修正是 schema 级操作（选中元素改字段，非自由文本框）。
5. 状态条组件（MVP 内）：中栏底部状态条含**上下文用量圆盘**（三态阈值语义色：正常/提示压缩/**开启新阶段**，悬停展开卷宗占用/对话占用/可压缩项明细）+ 摄取 credits 余量——用量感知与付费心智同处建立。红色态直接接**一键续行按钮**（文案用办案阶段心智，如"本阶段内容较多，开启新阶段继续"，不出现"上下文/token"字样）；点击开新 session 自动续行（artifact 引用挂回 + 待确认门禁接续 + 近期摘要，见 packages/core/SPEC.md 会话链预留），旧 session 归档可回看。概念稿见架构会话 2026-07-09 的 courtwork_usage_ring_and_heatmap_mock。
6. 防呆三原则（MVP 内，依据 docs/09 调研，2026-07-09 拍板）：**①分层确认**——确认门禁不是单一按钮：低风险可撤销项走轻确认（toast+撤销），高风险/未核验（tier C / out_of_coverage）条目强制拆出批量范围、逐条展开溯源后才可确认；批量确认按钮在首次使用或异常批次要求随机抽样细读后解锁。**②生成与确定的视觉区隔**——LLM 生成的解释性文字与 schema 化确定性产出（工具核验结果、diff、引用）必须有稳定的视觉区隔规则（字体/底色/标识三选一以上），用户一眼可辨"这是机器的话还是查证的事实"。**③确认质量埋点**——记录确认停留时长，门禁退化成"秒批"即触发重新设计告警（也是反馈数据资产的一部分）。
7. 挂账（MVP 后，运营面板）：usage/API 请求/模型路由热力图（场景 × 时段，格内标注路由命中的甜点区模型，参照 Claude Code 的用量呈现）——本质是 eval 选型结论的生产环境实时验证面，与管理端一起排期。

## 三、Design 产出要求（handoff 格式，模型无关）

Prototype 交付四件套，交给任何代码实现方（人或模型）都是同一包：

1. **Design tokens（JSON）**：色板（含语义色映射）、字阶、间距、圆角、阴影——直接可导入代码。
2. **组件清单 + 状态矩阵**：每组件列全状态（默认/悬停/加载/空/错误/未核验），尤其信源角标、门禁按钮、修订痕迹渲染的全部状态。
3. **关键流程高保真**：S1 摄取→时间线、S3 审查→确认→docx 预览两条主流程的逐屏稿。
4. **交互规格**：事件流→UI 状态的映射表（进度事件、artifact 到达、确认请求、错误）；键盘与滚动行为。

## 四、代码化工序（GPT-5.6 / Codex 实验，已批准）

用 Codex（GPT-5.6）从 handoff 四件套写 UI 全量代码、直至 Tauri v2 打包成品，作为对其前端能力的实测——**批准，护栏如下**：

1. handoff 四件套模型无关，不为任何特定模型定制；5.6 未发布或表现不济时换实现方零成本。
2. UI 壳只消费 core 事件流协议 + demo-data 装配，**一行业务逻辑不许进壳**（W6 验收放行的协议是唯一接口；违者按 CLAUDE.md 算验收失败）。
3. 验收不因模型降格：Playwright 回归进 CI、协议契约测试（mock 事件流回放）、假绿防护（用例计数核对）、Tauri 打包产物可安装可运行、视觉与 design tokens 比对。
4. 验收三角不变：实现方（无论哪个模型）之外仍需独立验收会话 + 架构拍板；computer use 探索性 QA 按 docs/10 测试策略届时立 spike 工单。
5. 实验记录：5.6 的实际表现（速度/质量/返工率）写进 W9 验收记录，作为后续"哪类工单交给哪个模型"的选型数据。
