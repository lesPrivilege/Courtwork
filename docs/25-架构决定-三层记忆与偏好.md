# 架构决定：三层记忆与用户偏好（2026-07-09）

状态：已拍板。依据 docs/13 调研（Harvey ethical walls 文与"案件=记忆边界"同构为关键旁证）。

## 三层结构与分级

1. **案件层（已有，MVP，不动）**：artifact + RevisionEvent + 跨 session 结构化再水化即记忆底座（"卷宗即记忆"）。**案件即记忆边界**——案件间信息默认不互通，这既是架构事实也是利冲合规的地基。
2. **用户层偏好（Stage 1 新建）**：管线 = RevisionEvent 批量聚类出"偏好规则候选"（后台批处理，挂在案件归档等既有节点，**不做常驻 sleep-time agent**）→ **强制脱敏/泛化节点**（剥离案件实质内容，只留流程/风格/立场偏好）→ 用户一次性确认写入 → 可见可删 UI（豆包/Kimi"记忆空间"范式：每条带溯源与生效范围）。记忆写入过留人确认——该纪律的第四次适用。
3. **机构层（Stage 2）**：firm knowledge + 利冲隔离。现在仅预留：届时经 schemas 提案增加 conflict-group 维度字段；利冲对接律所既有系统，不自建。

## 轻对话单窗模式的折中

单窗无限流（豆包式零 session 心智）+ **记忆可见可审计**：隐式提炼 + 显式指令（"记住我偏好 X"）+ 可见删除列表三件套；**拒绝不透明后台合成**（ChatGPT Dreaming 式"个性化-便利悖论"为反面案例）。UI 上"清空对话"（上下文）与"清除偏好"（记忆）是**两个独立入口**，不许混淆。

## 借用（设计不借代码）

- Graphiti 双时态模型的"关闭旧区间不删除"语义（RevisionEvent 已是其手工简化版，将来演进有对照系）。
- mem0 的 Add/Update/Delete 分类词汇——仅借分类法，**决策权收回给用户确认**。

## 不许做

- 不引入任何"LLM 自主判定记忆增改删"的记忆框架本体。
- 案件实质内容（当事人/金额/事实）不得进入用户层偏好——脱敏节点是强制关卡不是建议。
- sleep-time 式后台整理不早做；即便 Stage 2 引入，写入仍过确认。
- MVP 不做任何新记忆功能——现有案件层已够，克制即路线。

## 增补（2026-07-11）：左栏即容器列表，session 概念的残余归宿

用户拍板确认并补全"卷宗即记忆"的 UI 面：

1. **左栏主序 = 容器列表，非 session 列表**。卷宗可展开，展开的是工作结构（阶段 + 三区），不是对话历史；卷宗内不做 session 管理——对话是阶段的对话流，随容器走，不是独立资产。
2. **续行由工作语义保证，不由 transcript 保证**：调好的注册表声明（场景阶段状态）+ artifact + 证据台账 + RevisionEvent 即结构化再水化的全部输入，重开案件读这些，聊天记录只是附录。
3. **session 概念的唯一残余 = 未容器化的先聊后建对话**（composer-first chip 态）：归 Recents 区收纳，容器化仪式吸收——存入容器即融入阶段流，不再独立存在。频道（对话）与资产（容器）分离，频道可弃，资产永续。
4. 对 RP-1 左栏骨架（docs/49 第五章）的修正语义：Recents ≠ 最近容器，= 未归档对话候车厅 + 最近打开容器的混合列表，前者带"存入"动词。
