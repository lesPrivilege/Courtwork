# 关于法律垂类模型与产品形态部署的几点意见（2026-07-08）



## 一、基座与训练策略：微调是手段，护城河不在模型本身

先厘清一个前提，避免把战略重心放错位置：**竞争的胜负手不在"我们有没有自己的模型"，而在"能不能把行业数据和专业判断，持续、低成本地编译成可靠的专用系统"。** 基座模型是可替换的组件——谁都能在开源基座上包一层提示词工程和 UI，这一层准入门槛低、护城河浅。真正换不掉的，是编译出来的领域系统、数据积累和判断逻辑。

在这个前提下，用自有数据做微调是一个值得系统性、持续做的**手段**，但它服务于上面这个目标，不是目标本身：

- **做法**：选体量适中、推理能力够用的开源基座，用高质量法律数据集做精调，把垂类任务的性能顶上去即可。不追求"自研基座大模型"——那在体量和技术深度上不现实，硬做还会被上游模型厂商的能力迭代持续碾压。选择微调而非自研基座，本身就是垂类公司正确的战略选择，不是退而求其次。

- **一个可参照的先例（但要注意它的边界）**：Cursor 的编程模型 Composer 就是在开源基座（Moonshot 的 Kimi K2.5 checkpoint）上、用自有代码数据做大规模后训练得到的近前沿模型。值得注意的是官方披露的算力配比——最终模型里只有约 15% 的算力花在基座、85% 花在自有数据的后训练和强化学习上。**这恰恰说明价值不在基座（基座是可换的开源件），而在叠加其上的专有数据后训练层**，与我们的判断一致：护城河在"数据 + 判断逻辑"那一层，不在模型本身。（补注：该案例当时伴随"是否充分披露使用了开源基座"的行业争议，引用时点到即可，不必展开。）

- **诚实边界——法律 ≠ 代码，别照搬飞轮叙事**：代码领域能形成自优化飞轮，是因为它同时满足三个条件：可分解的中间表示、廉价的自动验证（编译器 / 测试套件）、错误可逆且代价低。法律数据只满足第一条——文书 / 合同 / 证据高度结构化、可分解，但验证锁在司法解释与裁量里（没有编译器式的自动判对错），且法律错误往往不可逆、代价高。**结论**：我们不该期待代码那种自动飞轮，法律垂类的精调更依赖高质量数据的人工校准和"人在环上"的验证。这不是弱点，是这个领域的结构特征——也正好解释了为什么"数据 + 判断逻辑"这层壁垒比模型本身更硬、更值得长期投。

## 二、产品形态：从 Chatbot 深化到"文件夹级工作伙伴"

核心不再是一问一答的 Chatbot，也不是每次人工上传附件，而是**用户直接指定一个项目文件夹（例如一套完整卷宗），智能体持续持有该目录的全部上下文，协助发现问题、提炼要点、起草与修改法律文书，关键判断处留人确认**——整个交互更接近一个持续拥有项目上下文的**协作者（cowork / workbuddy）**，而非被动应答工具，也不是"替人自主干活的数字员工"。

**方向校准（重要）**：法律没有自动验证对错的闭环、错误不可逆且代价高，判断权本来就不该、也不能真交出去——因此这个形态的**持久内核是"协作者"那一层（持续上下文 + 深度嵌入办案流程 + 留人确认），而不是"自主数字员工"叙事**。"记忆用户偏好 + 持续持有上下文"这层是对的、值得做深；"心跳任务 / 主动干活"这类偏自主的部分，在法律域要谨慎——它是热潮退去后最容易显得过度承诺的一层。产品能量应押在协作层的深化上。

**相对通用智能体的结构性优势**：通用 Agent 只能依赖模型自身能力，而我们已具备针对专业场景的识别与适配，并能接入确定性外部接口——例如合同审查中自动识别合同主体，直连企查查、天眼查等系统实时获取工商、风险信息。这种"把关键事实交给确定性接口、而不是让模型猜"的做法，是通用产品难以实现的。

## 三、垂类自研 / 自建带来的四层壁垒

1. **降低幻觉：把确定的事实交给确定的接口。** 大模型本质是概率性生成，同一问题多次作答可能不一致。对企业信息、法规校验这类存在权威源的事实，我们通过确定性接口 / 校验库接入结果，让关键事实不依赖模型"猜测"——这是"什么该交给模型、什么该用规则兜底"这条产品判断落到实处，也是降幻觉最实在的一条路径。

2. **检索质量：数据源本身就是壁垒。** 通用 Agent 的检索受限于公开网络资源。我们持续维护专业案例 / 法规库（亿级判例、千万级法规量级），覆盖更全、质量更高；更硬的一层是**与最高人民法院合作、数百名专家型法官参与研究制定的类案同判库（享独家电子版权）**——这不是数据规模问题，而是机构级、制度性的合作关系，别人短期内复制不了。

3. **专业场景的结构化表达与内嵌。** 不止把专业场景识别做成独立 Skill，更将其内嵌进工作流：处理诉讼卷宗时，可在侧栏自动生成人物关系谱系、事件时间线、逻辑结构图，用户在此基础上继续与 Agent 交互。这种结构化呈现，极大提升了复杂案件的处理效率与可解释性。

4. **行业深耕的复利与生态合作。**
   - **数据复利（★这里恰恰是护城河的根，不是短板）**：服务大量专业用户，日常交互数据不断反哺检索校准与模型精调。要点在于：法律域这个闭环是"线性增强"，不是代码域那种可自我优化的指数飞轮——**而"没有飞轮"正是数据在位者护城河成立的前提**。有飞轮的领域，后来者砸算力就能快速追平，护城河反而浅；法律没有自动验证的捷径，能力提升只能靠持续人工维护法条修订、新判例、司法解释的线性积累。正因为不能用算力抄近路，已经在做这件线性苦活、又握有独家数据源（最高法合作的类案同判库等）的在位者，就占了别人短期追不上的位置。这也意味着上游基座模型再强，也不能自动判定"法律对不对"，对这层护城河的侵蚀比在有飞轮的领域小得多。
   - **官方与上游合作**：与立法、司法、行政等上游机构长期合作，获取独家数据源和合规通道，这是通用智能体无法获得的。
   - **ToB 工作流适配**：具备成熟的面向律所等机构的产品方案，能把完整工作流无缝嵌入律所既有作业体系，相较通用产品有更强的流程适配性和落地能力。

## 总结

通用模型能力再强，"最后一公里"的专业问题也不会自动消失——但价值不在"我们自己有一个模型"，而在把行业数据、专业判断、确定性接口和官方合作，持续编译成可靠系统的能力。基座可换，这一层换不掉。因此：**微调作为手段值得系统性、持续地做，但战略重心与对外叙事都应落在"数据 + 判断逻辑 + 接口 + 生态"这层护城河上**；产品形态坚定沿"文件夹级工作伙伴"深化，持续发挥我们在数据、接口、场景和生态上的差异化优势。
