# 产品定位与路线

状态：现行定本（2026-07-13）

## 一句话定位

Courtwork 是面向中国律所与企业法务的本地优先工作代理：以案件或工作区为容器，把文件、确定性工具、模型推理、结构化产出、来源锚点和人工确认组织成可审计流程。

通用 agent 卖能力，能力随模型进步快速折旧；Courtwork 卖的是秩序——场景、字段、门禁、来源、修正与留痕。模型越便宜，这套秩序越值钱。

## 用户与问题

首要用户是需要处理大量卷宗、合同、研究材料和正式文书的律师与法务团队。他们面对的核心问题不是“模型会不会写”，而是：

- 文件分散，案件上下文难以连续；
- 结论与原文脱节，复核成本高；
- 自动化容易越权或静默降级；
- 产出无法直接进入 Word/WPS 工作流；
- 团队判断不能沉淀成稳定、可复用、可审计的形状。

## 产品形态

- `work`：容器化工作台。左侧是容器，中间是连续工作流，右侧是由 artifact schema 驱动的结构化工作面。
- `chat`：轻量单一对话画布，用于不值得建容器的任务；通过“存入”升格为容器。
- 场景由用户显式触发或由系统推荐，模型不得自行决定执行不可逆流程。
- 所有结构化结论都可回到原文锚点；所有修正都成为事件；所有正式产出都经过确认或定稿仪式。

## 当前滩头

法律包已经覆盖样板案的阅读、时间线、主体关系、合同风险、修订建议与 Word 产出全链路；PM 包已建立第二垂类 schema 基础，用于证明通用底座不依赖法律语义。

PM 垂类的现行字段与交互边界见 [PM 垂类契约](pm-vertical.md)。

MVP 的衡量标准不是场景数量，而是一个真实材料能否走完：

1. 安全摄取并生成阅读视图；
2. 模型按场景声明推理；
3. 系统为引语铸造坐标并拒收歧义；
4. 产出通过 schema 与确认门禁；
5. 用户修正被记录并投影到续行上下文；
6. 正式文书可安全编译或修订为 docx。

## 长期资产

- 领域 schema 与渲染词表；
- 场景声明、专业门禁和确定性接口；
- 脱敏评测集与回归基线；
- 来源解析、实体对齐、文书著录和 WPS 兼容能力；
- 经真实使用确认的 RevisionEvent 与专业判断数据。

## 明确不做

- 不把法律产品做成自主行动的通用 chatbot；
- 不以模型自证代替确定性校验；
- 不让 web 线索冒充已核验事实；
- 不自动执行不可逆动作；
- 不编辑用户上传的原件；
- 不把案件内容用于训练；
- 不在没有真实数据管线时用占位内容制造“高级感”；
- 不因平台可泛化就提前做平台市场。

## 路线原则

先打穿法律真实链路，再用第二垂类证明 ABI；先做能触红的契约和验证器，再扩场景；只有真实数据撞上性能或上下文边界时，才引入 Rust、sandbox 重构、context 修剪或更重的 agent 编排。
