# MVP 缺口盘点与路由声明（2026-07-10）

状态：已拍板。目标：MVP 本体自足（可下载、选文件夹、完成普通 work agent 工作），全部外部依赖以**声明的方式**在场（不隐藏、不假装），且缺口填充不影响推理编排。

## 核心原则

1. **缺口不改编排**：场景声明照常引用工具 id，装配点决定路由到 demo 或真实实现——工具契约同形（六 reason/sourceId/信源等级），换数据源是换适配器，**推理编排零感知**。"填充字段顺延"不影响编排的担心由此消解：契约就是隔离层。
2. **缺口三态声明规范**（每个缺口必居其一，UI 可见）：
   - **演示态**：demo-fixture 顶上，`source:"demo-fixture"` 自标识 → UI 打"演示数据"角标；
   - **未核验态**：C 级/`out_of_coverage`/`officialTextVerified:false` → 未核验标记；
   - **禁用态**：功能入口存在但声明"即将支持 + 当前替代路径"（如扫描件 → copy-paste 兜底），不隐藏。

## MVP 基础 work agent 能力面（"本体自足"的定义）

选文件夹（案件文件夹级上下文）+ office 生态文件阅读（#4 阅读视图）+ 轻对话 + 四个 MVP 场景（S1 以阅读视图版运行/S2/S3/S4，注册表声明驱动，todo 由声明派生非模型自写）+ web fetch 补具体信息 + 产出编译 docx。**官方内置 skill/编排随版本分发（docs/24 供给侧），特殊场景不依赖模型自写 task（docs/12 判据）。**

## 解耦确认

框架外皆可换件，已是架构事实：数据源=适配器（tools 契约）、场景=YAML 声明、演示数据=独立包经装配点、UI=协议纯客户端、消毒层=fetch 工具内部。后续迭代 = 改声明/换适配器，main 框架不动。
